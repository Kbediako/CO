## Summary
- Problem Statement: the installed CO stop hook disables `~/.codex/hooks/co_orchestration_autocontinue.json` whenever `CO_ORCHESTRATOR_DONE`, `CO_ORCHESTRATOR_CRITICAL_BLOCKER`, or `CO_ORCHESTRATOR_DESTRUCTIVE_DECISION_REQUIRED` appears anywhere in `last_assistant_message`. Incidental or quoted sentinel text can therefore persist `enabled=false` even when no user-needed blocker exists.
- Desired Outcome: make stop persistence require a deliberate structured stop signal while preserving genuine terminal/destructive stop behavior and adding focused regressions for the false-stop cases.

## User Request Translation
- User intent / needs: prevent the CO auto-continue gate from being disabled by arbitrary sentinel substring matches, especially read-only / approval-policy language that mentions `CO_ORCHESTRATOR_CRITICAL_BLOCKER`, while keeping real stop flows able to disable auto-continue.
- Success criteria / acceptance:
  - auto-continue disables only for a deliberate structured stop signal
  - read-only or approval-policy limitations do not persist `enabled=false` unless they are genuine user-needed blockers
  - quoted sentinel text does not disable the gate
  - true user-needed `CO_ORCHESTRATOR_CRITICAL_BLOCKER`, `CO_ORCHESTRATOR_DONE`, and `CO_ORCHESTRATOR_DESTRUCTIVE_DECISION_REQUIRED` flows still disable the gate
  - focused regressions cover false read-only stops, quoted sentinel text, and real destructive-stop flows
- Constraints / non-goals:
  - do not broaden into global Codex hook redesign
  - do not remove the existing fail-open behavior on hook errors
  - do not disable normal resume prompting while the gate is enabled
  - do not change unrelated CO provider-worker sentinel semantics
