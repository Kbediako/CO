# TECH_SPEC - CO-404 Control-Host Provider-Worker Acknowledgement Timeouts

Canonical spec: `tasks/specs/linear-0354e9d0-84ee-4340-b098-a9c685c191b5.md`.

## Summary
- Split accepted recover/relaunch/nudge acknowledgement from asynchronous provider-worker launch/resume completion.
- Return deterministic `queued` or `starting` metadata without exposing raw launch tokens.
- Keep retry idempotency action-scoped: identical recover/relaunch/nudge retries coalesce, while distinct actions for the same issue/runtime queue independently.

## Protected Surfaces
- `controlHostProviderWorkerRecoverCliShell.ts`
- `observabilityApiController.ts` action-scoped recover in-flight key
- `provider-intake-state.json`
- `provider-linear-worker-proof.json`
- `provider-linear-worker-linear-audit.jsonl`

## Fallback / Refactor Decision

| Surface | Decision | Removal / Retention Condition |
| --- | --- | --- |
| CLI hard timeout after accepted side effects | `remove fallback` | Acknowledged metadata replaces timeout-after-acceptance. |
| API `202` waits for full recover action | `remove fallback` | Accepted work can return before background completion. |
| Accepted-without-run capacity counting | `remove fallback` | Null launch/run identity is not active capacity. |
| Provider-worker idempotency | `justify retaining fallback` | Retry safety remains a durable launch-safety contract. |

## Validation Plan
- Focused API/CLI acknowledgement, action-scoped retry, and duplicate-launch regressions.
- Accepted-without-run capacity regression.
- Normal build/lint/test/docs/review gates.
