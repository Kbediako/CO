## Problem
After a provider issue reaches terminal Linear truth, local provider intake can retain a released-pending-reopen `Merging` / started claim with `provider_issue_released_pending_reopen:provider_issue_released:not_active` and dead worker PID evidence. That stale terminal claim must not keep counting as a `CO STATUS running row`, re-trigger `provider_refresh_lifecycle_stuck`, set or preserve `restart_required`, or destabilize supervisor recovery.

## Intent Checksum
Preserve these exact terms and surfaces: `provider_refresh_lifecycle_stuck`, `restart_required`, `released-pending-reopen`, `Merging`, `stale terminal claim`, `dead worker PID`, `CO STATUS running row`, `provider_issue_released_pending_reopen:provider_issue_released:not_active`, `ECONNREFUSED`, `provider-intake-state.json`, `control-host`, `supervisor`, `CO-192 projection-only pruning`, and `CO-193 Ready reclaim`.

Reject these interpretations:
- This is only a presentation pruning issue.
- This is only a Ready reclaim issue.
- The fix can hide refresh-health errors, disable refreshes, delete all released claims, or kill unrelated live workers.

## Goals
- Classify or reconcile stale terminal released-pending-reopen `Merging` claims before they drive refresh-health or restart-loop authority.
- Exclude the dead stale claim from running-slot authority once terminal issue truth and dead-PID evidence are available.
- Preserve truthful `provider_refresh_lifecycle_stuck` / `restart_required` diagnostics for real stuck refreshes.
- Preserve unrelated live provider workers during refresh reconciliation and supervisor recovery.
- Test this path separately from `CO-192` projection-only pruning and `CO-193` Ready reclaim.

## Non-Goals
- Replace `CO-192` stale-row projection work.
- Replace `CO-193` Ready/unstarted reclaim work.
- Redesign scheduler/admission/provider workflow.
- Hide health errors, disable refreshes, or treat supervisor restart as the fix.
- Destructively clean provider-intake history as the primary fix.

## Not Done If
- Stale merged/Done issue truth can still leave a retained `Merging` claim that drives `provider_refresh_lifecycle_stuck` or `restart_required`.
- `co-status` can still fail because this stale claim triggers a restart loop.
- Recovery drops or kills unrelated live provider workers such as `CO-192` or `CO-193`.
- The fix only hides the `CO STATUS` row without stabilizing refresh lifecycle health.
- Tests do not distinguish this terminal/stale `Merging` refresh-health path from `CO-192` and `CO-193`.

## Acceptance Criteria
1. Reproduce terminal/Done issue truth plus retained released-pending-reopen `Merging` / started claim, dead worker PID, and unrelated live workers.
2. Provider refresh reconciles or quarantines the stale terminal claim without entering `provider_refresh_lifecycle_stuck`.
3. Provider refresh does not set or immediately re-enter `restart_required` from the same retained stale claim.
4. `CO STATUS` stops counting the dead stale claim as running once terminal truth and dead-PID evidence are available.
5. Supervisor restart/recovery preserves unrelated live provider workers and avoids re-looping on the stale claim.
6. Tests cover this terminal/stale `Merging` path separately from `CO-192` and `CO-193`.
7. Genuine refresh-health failures still surface `provider_refresh_lifecycle_stuck` / `restart_required`.
