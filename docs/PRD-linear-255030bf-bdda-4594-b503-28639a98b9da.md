# PRD - CO: shift Linear intake to webhook-first targeted reconcile with slow full recovery sweeps
## Traceability
- Linear issue: `CO-144` / `255030bf-bdda-4594-b503-28639a98b9da`
- Linear URL: https://linear.app/asabeko/issue/CO-144/co-shift-linear-intake-to-webhook-first-targeted-reconcile-with-slow
## Problem
- CO already has webhook intake plus poll-backed recovery, but ordinary refresh still uses full tracked-issue polling as the default intake path.
- That wastes request budget and increases cooldown pressure.
## Outcome
- Make webhook-triggered by-id intake the primary fast path.
- Keep startup recovery and slower full recovery sweeps for missed events and older unchanged ready issues.
- Run ordinary refresh as targeted reconcile plus bounded fresh discovery only when capacity exists.
## Acceptance Criteria
- Startup recovery still performs a full rehydrate/reconcile pass.
- Claim, retry, release, reopen, reattach, and relaunch-after-success behavior stay correct.
- Ordinary refresh ticks no longer full-scan the active set unless recovery or bounded discovery requires it.
- Fresh discovery stops after enough eligible candidates are found for current free global and per-state slots.
- The lane records enough request-burn evidence to support a later quota decision.
## Non-Goals
- Webhook-only intake.
- Removing full recovery sweeps.
- Weakening shared-budget fail-closed behavior.
- Redesigning provider mutation or review-helper semantics.
## Protected Terms
- Preserve `webhook-first targeted reconcile`, `slow full recovery sweeps`, `missed-webhook recovery`, `release/reattach`, `retry/refetch`, and `relaunch-after-success`.
- Reject narrower interpretations such as "just slow polling down" or "only trim the query payload."
