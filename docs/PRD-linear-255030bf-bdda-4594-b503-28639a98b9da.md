# PRD - CO: shift Linear intake to webhook-first targeted reconcile with slow full recovery sweeps

## Traceability
- Linear issue: `CO-144` / `255030bf-bdda-4594-b503-28639a98b9da`
- Linear URL: https://linear.app/asabeko/issue/CO-144/co-shift-linear-intake-to-webhook-first-targeted-reconcile-with-slow

## Problem
- CO already has webhook intake, shared-budget hardening, and poll-backed recovery.
- Ordinary control-host refresh still treats full tracked-issue polling as a normal intake path through `resolveLiveLinearTrackedIssues(...)`.
- `providerIssueHandoff.ts` still preloads the active tracked set before claim reconcile, and `linearDispatchSource.ts` still fetches a heavy candidate payload for every active issue.
- That burns Linear request budget, prolongs cooldown recovery, and makes operator-visible budget truth noisier than necessary.

## Desired Outcome
- Make webhook-triggered by-id intake the primary fast path for new or changed issues.
- Keep startup recovery and a slower full recovery sweep for missed events and unchanged older ready issues.
- Run ordinary refresh ticks as targeted reconcile for owned or recovery-tracked issues plus bounded fresh discovery only when free capacity exists.
- Preserve claim, retry, release, reopen, reattach, relaunch-after-success, and operator-facing budget-truth semantics.

## Acceptance Criteria
- Startup recovery still performs a full rehydrate or reconcile pass.
- Existing claim, retry, release, reopen, reattach, and relaunch-after-success behavior stays correct.
- Regular refresh ticks no longer full-scan the active Linear set when free-capacity discovery is already satisfied or unavailable.
- Fresh discovery stops after enough eligible candidates are found for the current free global and per-state slots.
- Missed webhook events and unchanged older ready issues are still recovered by a slower full sweep.
- Operator surfaces still expose shared-budget cooldown and exhaustion truthfully.
- Steady-state Linear request burn drops materially relative to current mainline behavior.
- The implementation records enough budget-backed or header-backed evidence to inform a later quota-increase decision.

## Non-Goals
- Moving to webhook-only intake.
- Removing full recovery scans entirely.
- Weakening fail-closed budget gating, shared cooldowns, or reservation behavior.
- Redesigning provider mutation, workpad, PR helper, or review-helper semantics in this lane.
- Filing the quota-increase request in the same issue.

## Intent / Protected Surfaces
- Preserve the issue framing: `webhook-first targeted reconcile`, `slow full recovery sweeps`, `fresh-dispatch polling`, `missed-webhook recovery`, `release/reattach behavior`, `relaunch-after-success`, `retry/refetch correctness`, `operator-facing budget truth`, and `measured steady-state Linear request burn`.
- Keep the implementation centered on `linearWebhookController.ts`, `controlServerPublicLifecycle.ts`, `providerIssueHandoff.ts`, `linearDispatchSource.ts`, and `providerPollingHealth.ts`.
- Reject these narrower or wrong interpretations:
  - "turn polling off" without keeping startup or missed-event recovery
  - "just slow polling down" without splitting targeted reconcile from full-scan discovery
  - "only trim the query payload" without changing when the active set is scanned
  - "widen into provider workflow or review-helper redesign"

## Parity Matrix
- Current:
  - webhook deliveries already resolve issues by id and hand them to `providerIssueHandoff.handleAcceptedTrackedIssue(...)`
  - ordinary scheduled ticks still resolve the full tracked-issue set
  - tracked-issue discovery still fetches `description`, blocker relations, and history for every active issue
- Reference:
  - webhook deliveries should cover the common fast path for new or changed issues
  - periodic polling should stay responsible for recovery, release or reopen, retry refetch, and relaunch sequencing
  - slow sweeps should catch missed events and unchanged older ready issues without becoming the normal intake path
- Target:
  - ordinary ticks reconcile owned or recovery-tracked issues by id and run bounded fresh discovery only when dispatch capacity exists
  - fresh discovery uses a leaner payload and stops once enough eligible candidates exist for the current free slots
  - startup and slower periodic sweeps still perform full recovery scans

## Not Done If
- Ordinary refresh ticks still full-scan the active Linear set by default.
- Fresh discovery keeps paginating after enough eligible candidates are already found.
- Startup recovery or slow-sweep recovery can no longer recover older unchanged ready work.
- Existing claim, retry, release, reopen, reattach, or relaunch-after-success paths regress.
- Operator-facing cooldown or next-refresh truth becomes less explicit.
- The lane cannot produce before or after request-burn evidence from shared-budget or header-backed telemetry.

## Metrics & Guardrails
- Success metrics:
  - fewer steady-state tracked-issues requests under ordinary operation
  - fewer avoidable cooldown hits caused by redundant discovery scans
  - bounded discovery returns only the candidates needed to fill free slots
- Guardrails:
  - keep startup rehydrate and slow missed-event recovery truthful and available
  - preserve existing release, retry, and relaunch contracts
  - preserve evidence strong enough for later quota decisions
