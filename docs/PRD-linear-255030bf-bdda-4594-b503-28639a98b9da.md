# PRD - CO: shift Linear intake to webhook-first targeted reconcile with slow full recovery sweeps

## Added by Bootstrap 2026-04-10

## Traceability
- Linear issue: `CO-144` / `255030bf-bdda-4594-b503-28639a98b9da`
- Linear URL: https://linear.app/asabeko/issue/CO-144/co-shift-linear-intake-to-webhook-first-targeted-reconcile-with-slow

## Summary
- Problem Statement: CO already has webhook intake, shared-budget hardening, and poll-backed recovery, but the ordinary control-host refresh loop still treats periodic full active-set polling as a normal intake path. `linearWebhookController.ts` already fetches changed issues by id on webhook delivery, yet `controlServerPublicLifecycle.ts` still calls `resolveLiveLinearTrackedIssues(...)` on regular ticks and `providerIssueHandoff.ts` preloads that full tracked-issue set before claim reconcile. `linearDispatchSource.ts` also fetches a heavy candidate payload for every active issue even though the scheduler only needs a small set of eligible fresh candidates when free capacity exists. That burns Linear request budget, delays cooldown recovery, and makes the operator-visible budget truth noisier than it needs to be.
- Desired Outcome: make webhook-triggered by-id intake the primary fast path, keep startup recovery and a slower full recovery sweep for missed events, run ordinary refresh ticks as targeted reconcile for owned or recovery-tracked issues plus bounded fresh discovery only when free capacity exists, and preserve claim, retry, release, reopen, reattach, relaunch-after-success, and budget-truth semantics.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): stop using ordinary polling as the default intake path without regressing the recovery behaviors that polling currently covers. The lane is not "delete polling"; it is "split polling into the parts that actually need to run often versus the parts that should only run at startup or on a slower recovery cadence", while making the request-burn reduction measurable enough to decide later whether a Linear quota increase is still necessary.
- Success criteria / acceptance:
  - startup recovery still performs a full rehydrate or reconcile pass
  - existing claim, retry, release, reopen, reattach, and relaunch-after-success behavior stays correct
  - regular refresh ticks no longer full-scan the active Linear set when fresh-discovery capacity is already satisfied or unavailable
  - fresh discovery stops after enough eligible candidates are found for current free execution slots
  - missed webhook events and unchanged older ready issues are still recovered by a slower full sweep
  - operator surfaces still expose shared-budget cooldown and exhaustion truthfully
  - steady-state Linear request burn drops materially relative to the current mainline behavior
  - the implementation records header-backed or budget-backed evidence that can support or reject a later quota-increase request
- Constraints / non-goals:
  - do not move to webhook-only intake in this issue
  - do not remove full recovery scans entirely
  - do not weaken fail-closed budget gating, shared cooldowns, or reservation behavior
  - do not redesign provider mutation, workpad, PR helper, or review-helper semantics in this lane
  - do not file the quota-increase request in this same issue

## Parity / Alignment Matrix
- Current truth:
  - `linearWebhookController.ts` already validates create or update issue webhooks, fetches the issue by id, and hands the tracked issue directly into `providerIssueHandoff.handleAcceptedTrackedIssue(...)`
  - `controlServerPublicLifecycle.ts` still resolves the full active tracked-issue set on ordinary scheduled ticks through `resolveProviderPollTrackedIssues()`
  - `providerIssueHandoff.refresh()` still preloads that full tracked-issue set before claim reconcile, even though existing claims can already fall back to direct by-id refresh reads
  - `resolveLiveLinearTrackedIssues(...)` supports `limit` and `stopWhenEligibleForExecution`, but the scheduler does not use them and the early-stop path only stops after the first eligible issue rather than after enough free-slot candidates
  - the tracked-issues query still fetches `description`, blocker relations, and recent history for every scanned issue
- Reference truth:
  - webhook deliveries should cover the common fast path for new or changed issues
  - periodic polling should stay responsible for rehydrate, recovery, release or reopen, retry refetch, and relaunch sequencing, but it should not keep paying for full active-set discovery when free capacity does not require it
  - slow recovery sweeps should catch missed events and unchanged older ready issues without becoming the normal high-frequency intake path
- Target truth / intended delta:
  - ordinary ticks reconcile owned or recovery-tracked issues by id and run bounded fresh discovery only when dispatch capacity exists
  - discovery uses a leaner payload and stops once enough eligible candidates are found for the current free slots
  - startup and periodic slow sweeps still perform full recovery scans
  - operator-visible polling or budget surfaces explain whether the next loop is a targeted reconcile, a bounded discovery pass, or a slow recovery sweep
- Explicitly out-of-scope differences:
  - webhook-only intake
  - removing full sweeps entirely
  - reworking provider mutation, workpad, or PR helper semantics

## Not Done If
- Ordinary refresh ticks still full-scan the active Linear set by default even when no free dispatch capacity exists.
- Fresh discovery still keeps paginating after enough eligible candidates were already found.
- Startup recovery or slow missed-event recovery can no longer recover older unchanged ready work.
- Existing claim, retry, release, reopen, reattach, or relaunch-after-success paths regress.
- Operator-facing cooldown or next-refresh truth becomes less explicit.
- The lane cannot produce before or after request-burn evidence from shared-budget or header-backed telemetry.

## Goals
- Make webhook-triggered by-id intake the primary fast path for new or changed issues.
- Split ordinary refresh behavior into targeted tracked-state reconcile plus bounded fresh discovery.
- Keep startup full recovery and a much slower periodic recovery sweep for missed or stale events.
- Reduce request burn by trimming fresh-discovery payloads and stopping discovery once current free-slot needs are satisfied.
- Preserve truthful operator budget and polling surfaces.

## Non-Goals
- Eliminating polling entirely.
- Rewriting provider-worker review, merge, or mutation semantics.
- Changing shared-budget fail-closed posture to hide real exhaustion.
- Filing the quota-increase request before this optimization lands and is measured.

## Metrics & Guardrails
- Primary Success Metrics:
  - fewer steady-state tracked-issues requests under ordinary operation

  - bounded discovery returns only the candidates needed to fill free slots

  - preserve existing release, retry, and relaunch contracts

## User Experience
- Personas: control-host operator, provider-worker owner, reviewer diagnosing Linear request burn.
- User Journeys:
  - a new or changed eligible issue arrives via webhook and is considered immediately without waiting for the next full poll
  - an existing owned or recovery-tracked issue is reconciled correctly on ordinary ticks without paying for a full active-set discovery scan
  - a missed webhook or older unchanged ready issue is still recovered on startup or the slower sweep cadence
  - the operator can still tell when the next loop is suppressed by cooldown versus when it is running a bounded discovery or full recovery pass

## Technical Considerations
- Architectural Notes:
  - the best implementation seam is the existing split between `controlServerPublicLifecycle.ts`, `providerIssueHandoff.ts`, and `linearDispatchSource.ts`
  - existing-claim reconcile already has a by-id fallback path, so ordinary refresh does not need to prefetch the whole active set just to reconcile owned claims
  - `resolveLiveLinearTrackedIssues(...)` already exposes enough shape to support bounded discovery, but it needs a free-slot-aware stop condition and likely a leaner discovery payload mode
  - `providerPollingHealth.ts` already owns interval, reason, and budget visibility, so the new intake modes should keep that truth explicit instead of inventing a separate operator channel
- Dependencies / Integrations:
  - `linearWebhookController.ts`
  - `controlServerPublicLifecycle.ts`
  - `providerIssueHandoff.ts`
  - `linearDispatchSource.ts`
  - `providerPollingHealth.ts`
  - shared Linear budget state and headers

## Open Questions
- Whether the slow recovery sweep cadence should be fixed-time only or should also accelerate when webhook freshness goes stale can remain implementation-scoped as long as the first landed policy is explicit, test-backed, and does not widen the lane.
