## Summary
- Problem Statement: `codex-orchestrator linear create-follow-up` can create or reuse the intended Linear issue and then return a non-retryable failure because strict post-create description comparison treats Linear Markdown bullet and spacing normalization as a semantic mismatch.
- Desired Outcome: post-create verification compares descriptions through a Linear-compatible normalized Markdown representation while keeping created issue id, identifier, URL, canonical marker, labels, and relations as authoritative success evidence when normalized text is otherwise equivalent.

## User Request Translation
- User intent / needs: stop successful follow-up creation from looking failed solely because Linear persisted equivalent Markdown with normalized bullets or spacing.
- Success criteria / acceptance:
  - description comparison normalizes expected and observed Markdown before classifying mismatch.
  - created issue id, identifier, URL, canonical marker, labels, and related links remain required success evidence.
  - bullet and spacing normalization regressions are covered for ordinary and canonical-owner-key follow-ups.
  - genuinely missing sections, canonical marker, labels, or relations still fail closed.
- Constraints / non-goals:
  - do not change Linear persisted Markdown rendering.
  - do not bypass canonical owner reuse.
  - do not remove required follow-up sections or labels.
  - do not create duplicate issues when a previous helper attempt already succeeded.
