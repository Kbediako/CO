## Summary
- Objective: replace the current plain-text `CO STATUS` renderer with a Symphony-parity terminal dashboard frame while preserving CO's current operator-dashboard data seam.
- Scope:
  - docs-first registration for `CO-44`
  - Symphony-parity terminal renderer changes in `controlStatusDashboard.ts`
  - additive startup wiring only where needed in `controlHostCliShell.ts`
  - focused full-frame terminal renderer coverage and startup-gating regressions
- Constraints:
  - keep the operator-visible title exactly `CO STATUS`
  - preserve JSON and non-interactive behavior
  - document each justified shipped deviation from Symphony explicitly

## Technical Requirements
- Functional requirements:
  - `control-host` text-mode startup on a TTY must render a `CO STATUS` frame with Symphony-style terminal chrome and section structure
  - the header must move from prose metadata lines to a compact operator-scanning structure closer to Symphony, including summary lines for agent/load state, throughput, runtime, tokens, rate limits, project/dashboard context, and next refresh where CO data permits
  - running work must render as a width-aware table with headers and separators rather than pipe-delimited prose rows
  - retrying work must render as a sorted backoff queue with relative due-time display and single-line sanitized errors
  - narrow terminals must degrade intentionally to smaller layouts instead of relying only on generic field truncation
  - the current `ISSUES` prose block must not remain in the primary terminal frame unless a documented parity reason emerges during implementation
  - the renderer must continue to read `OperatorDashboardDataset` rather than deriving a second terminal-only model
  - `control-host --format json` must remain dashboard-free and machine-readable
  - non-TTY text-mode runs must remain readable and non-dashboard-oriented
- Non-functional requirements (performance, reliability, security):
  - keep rendering observational and bounded
  - keep refresh behavior consistent with the existing timer plus runtime notification posture
  - sanitize rendered strings so multiline / control-sequence content cannot break the terminal frame
- Interfaces / contracts:
  - dataset input: `OperatorDashboardDataset` from `orchestrator/src/cli/control/operatorDashboardPresenter.ts`
  - startup surface: `orchestrator/src/cli/controlHostCliShell.ts`
  - focused regressions: new dashboard frame tests plus startup-gating coverage in `orchestrator/tests/ControlHostCliShell.test.ts`

## Architecture & Data
- Architecture / design adjustments:
  - extend the control-host dashboard state to maintain rolling token samples so CO can show a Symphony-style throughput line
  - replace the current line-by-line prose renderer with section formatters for summary, running table, and backoff queue
  - keep `baseUrl` additive so the frame can show a reviewer/operator-usable dashboard link analogous to Symphony's optional dashboard URL line
  - prefer explicit justified substitutions over fake parity when CO lacks a direct Symphony data point; for this lane the shipped substitutions are the `CO STATUS` title, no PID column, and `running/tracked` counts instead of a true `current/max` agent summary
- Data model changes / migrations:
  - none planned for the authoritative runtime/read-model payloads
  - implementation may add lightweight renderer-local state only for formatting and throughput calculation
- External dependencies / integrations:
  - `orchestrator/src/cli/control/controlStatusDashboard.ts`
