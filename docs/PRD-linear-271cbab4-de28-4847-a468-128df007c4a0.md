## Summary
- Problem Statement: `CO-14` documented a truthful issue-parent architecture above multiple same-issue workers, but current CO still has only one real provider-owned issue worker plus bounded review/docs child streams. There is no runtime path where one authoritative parent issue run launches multiple subordinate same-issue lanes, supervises their outputs, and still keeps Linear state, workpad, PR, review, merge, and done truth parent-owned end to end.
- Desired Outcome: one authoritative `provider-linear-worker` parent can launch more than one subordinate same-issue lane, keep those lanes inside the issue workspace boundary, enforce declared ownership scopes plus stale-output invalidation, and expose nested lane truth through parent proof and observability without surfacing subordinate runs as peer issue owners.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): turn the `CO-14` umbrella contract into a real runtime slice in the current CO lane, not another planning packet. Keep the existing control-host plus `provider-linear-worker` as the only external issue owner, add subordinate same-issue worker support beneath it, and prove the result in the current workspace with truthful docs, regressions, and runtime artifacts.
- Success criteria / acceptance:
  - one parent issue worker can launch more than one subordinate same-issue lane with `<parent-task-id>-<stream>` identity and `parent_run_id` lineage
  - only the parent lane can mutate Linear state, the persistent workpad, PR attachments, and review or merge handoffs
  - subordinate lanes require declared purpose plus file-scope or phase-scope ownership, and overlapping scopes are rejected until the parent resolves or serializes them
  - subordinate outputs can be accepted, rejected, rerun, or invalidated when base branch, issue state, or review feedback makes them stale
  - parent proof and read-side observability show nested `child_lanes` truth without turning subordinate runs into peer issue owners
  - at least one real runtime proof from this lane shows the parent issue run using more than one subordinate lane while keeping one truthful workpad and PR lifecycle
- Constraints / non-goals:
  - keep the current `codex exec` / `codex exec resume` seam; do not widen into remote app-server authority
  - do not introduce a second external owner for the same issue
  - do not replace the existing `provider-linear-worker` parent seam
  - do not widen into generic scheduler parallelism unrelated to one parent-owned issue

## Goals
- Add a real parent-managed same-issue child-lane launcher beneath `provider-linear-worker`.
- Keep subordinate execution bounded to lane-local workspace artifacts inside the authoritative per-issue workspace boundary.
- Enforce parent-only Linear mutation ownership even when subordinate runs have the same issue binding.
- Record subordinate lineage, scope, worktree or patch evidence, integration state, and staleness decisions in parent proof.
- Surface nested child-lane truth in read-side observability while preserving one authoritative issue owner row.
- Produce real runtime proof from `CO-35` using more than one subordinate lane.

## Non-Goals
- Replacing control-host or provider-worker ownership with a new scheduler model.
- Building a generic queueing or scheduler-wide parallel execution system.
- Expanding the authority model for remote app-server commands or watchers.
- Letting subordinate lanes talk to Linear as peer owners.

## Technical Considerations
- Architectural Notes:
  - current subordinate support is limited to `linear child-stream` and an allowlisted review/docs pipeline set
  - parent-owned runtime truth currently lives in `provider-linear-worker-proof.json`, `provider-linear-worker-child-streams.json`, and control-host read models
  - `discoverProviderIssueRuns` already ignores bounded child-stream manifests when they carry `parent_run_id`; CO-35 can extend that rule for real same-issue child-lane pipelines
  - child-lane integration should prefer lane-local worktree plus patch artifact flow so parent acceptance stays explicit and auditable
  - parent-only Linear ownership likely belongs in the Linear CLI helper surface so subordinate same-issue lanes fail closed even if they try to call mutation helpers directly
