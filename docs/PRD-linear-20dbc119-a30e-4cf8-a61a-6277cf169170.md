# PRD - CO: Reuse canonical owner issues before create-follow-up for recurring baseline debt

## Summary
- Problem Statement: provider lanes can independently file duplicate follow-up issues for the same recurring repo-baseline blocker because `docs:freshness:maintain` emits machine-readable cohort ownership context, but `linear create-follow-up` does not consume a deterministic canonical-owner key before mutating Linear.
- Desired Outcome: recurring baseline debt follow-ups converge on one open same-team same-project canonical owner issue stamped with an exact key, while closed or canceled historical owners remain evidence only.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): complete CO-259 by bridging docs freshness maintenance cohort output into the Linear follow-up mutation path so future lanes reuse or update an exact stamped owner instead of creating duplicate follow-ups.
- Success criteria / acceptance: deterministic owner keys appear in maintenance output; `create-follow-up` can consume that key, reuse matching open owners, create and stamp a new owner when absent, and regression tests cover the Apr 19 duplicate cluster plus the closed-owner negative case.
- Constraints / non-goals: do not implement fuzzy duplicate detection, do not reopen closed owners solely by similar text, and do not broaden into same-attempt retry suppression owned by CO-186.

## Intent Checksum
- Exact user wording / phrases to preserve: `Reuse canonical owner issues before create-follow-up`, `baseline_cohort_id`, `stable tuple from machine output`, `Do not use fuzzy title/body matching`, `Done, Duplicate, and Canceled issues are evidence only`, `Apr 19 four-duplicate cluster`.
- Protected terms / exact artifact and surface names: `docs:freshness:maintain`, `linear create-follow-up`, `provider-worker`, `CO-253`, `CO-256`, `CO-257`, `CO-258`, `CO-254`, `CO-188`, `CO-187`, `CO-186`.
- Nearby wrong interpretations to reject: title similarity is not canonical reuse; closed `CO-254`-style owners are not active reuse targets; this lane does not repair docs freshness debt itself.

## Parity / Alignment Matrix
- Current truth: `docs:freshness:maintain` reports candidate cohort data and owner guidance, while `linear create-follow-up` always creates a new issue after validating structured inputs.
- Reference truth: recurring baseline owner lanes should be keyed by machine-readable cohort identity so separate parent/subagent lanes converge on the same open owner.
- Target truth / intended delta: maintenance reports carry deterministic `canonical_owner_key` values, and follow-up mutation accepts an exact key/marker that finds only open same-team same-project owners stamped with that marker before creating a new stamped issue.
- Explicitly out-of-scope differences: no fuzzy title/body duplicate search, no automatic reopening of terminal issues, no same-attempt retry suppression changes.

## Not Done If
- A follow-up can still create a duplicate when an open same-project owner already has the exact canonical-owner marker.
- Closed `Done`, `Duplicate`, or `Cancelled`/`Canceled` issues are reused as active owners.
- The owner key depends on prose, title matching, or mutable age-only data instead of stable machine output.
- Provider-worker guidance still tells lanes to always file a fresh follow-up for recurring baseline debt.

## Goals
- Add deterministic canonical-owner key and marker output to docs freshness maintenance candidate cohorts.
- Add canonical owner reuse/create behavior to the provider Linear follow-up helper.
- Preserve audit evidence and relation creation for both reused and newly-created owner paths.
- Update worker-facing guidance and tests for the new contract.

## Non-Goals
- Reopening terminal issues automatically.
- Redesigning Linear workflow states or project ownership.
- Broad duplicate detection outside exact canonical-owner markers.
- Changing same-attempt deterministic mutation suppression.

- Primary Success Metrics: focused regressions prove one owner for repeated Apr 19 follow-up attempts and no reuse of a closed owner.
- Guardrails / Error Budgets: keep helper failures fail-closed, preserve same-team same-project scope checks, and avoid extra Linear mutations when a reusable owner is found.

## User Experience
- Personas: provider workers and review operators triaging recurring baseline debt.
- User Journeys: a worker cites a canonical owner key from maintenance output, invokes `linear create-follow-up`, and either receives a reused owner or one newly-created stamped owner for later lanes.
