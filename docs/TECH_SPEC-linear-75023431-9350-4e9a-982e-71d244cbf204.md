## Architecture & Data
- Current behavior:
  - child-lane launch stores a `parent_snapshot` with parent base SHA and issue snapshot fields
  - accept claims the pending child lane, reads parent HEAD and current issue snapshot, then calls stale detection before patch/proof validation
  - stale detection currently treats raw `parent_snapshot.issue_updated_at !== currentIssue.issue_updated_at` as a hard stale reason
- Target behavior:
  - accept still reads parent HEAD and current issue snapshot before applying the patch
  - accept records the current issue snapshot used for the decision
  - unavailable live issue truth or backwards timestamp inversion returns an explicit retryable diagnostic instead of `provider_worker_child_lane_stale`
  - live forward `updated_at` movement and state/type drift remain fail-closed with `provider_worker_child_lane_stale`
