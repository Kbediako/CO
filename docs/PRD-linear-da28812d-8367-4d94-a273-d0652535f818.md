## Summary
- Problem Statement: CO's runtime capabilities are real, but the public release surface is still materially below release-ready quality. The repo still tracks `.runs/**` and `out/**` artifacts on the public branch, a tracked manifest exposes a non-expiring local `resume_token` plus machine-local paths and command metadata, the package front door still points users to non-packaged or contributor-only docs, provider onboarding is spread across internal docs and code instead of one shipped downstream contract, and many always-loaded docs, templates, and shipped skills still describe `codex-cli 0.117.0` even though the live CLI is `0.118.0`.
- Desired Outcome: before the next public release, CO exposes a minimal professional public surface with tracked runtime artifacts removed, blocking guards against future public leakage, packaged downstream-safe docs and provider onboarding, machine-checkable provider readiness and smoke guidance, and a fully refreshed `0.118.0` posture across README, guides, templates, and shipped skills.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): treat this as one coordinated release-prep lane rather than another narrow cleanup. Public repo/package exposure, downstream Linear and Telegram onboarding, public-vs-internal docs truthfulness, shipped skill/template quality, and the live Codex `0.118.0` posture all need to land together so CO looks professional and self-sufficient to npm users and public repo readers.
- Success criteria / acceptance:
  - no tracked `.runs/**` or `out/**` runtime artifacts remain on the intended public release surface unless explicitly allowlisted as first-class public artifacts
  - a blocking guard fails on tracked runtime artifacts and absolute machine-local paths in tracked public docs or packaged surfaces unless allowlisted
  - `README.md` is a short public front door and only points at stable packaged or public-safe docs
  - packaged docs and shipped help surfaces are self-consistent and downstream-safe
  - Linear and Telegram onboarding live in one shipped provider contract that covers once-per-machine vs once-per-repo setup, env vars, webhook secret, allowlists, `dispatch_pilot`, transport policy, and smoke flow
  - machine-checkable provider readiness or smoke surfaces report actionable failures
  - current CO posture, templates, and shipped skills reflect `codex-cli 0.118.0` and current `codex exec` / `codex login` behavior truthfully
- Constraints / non-goals:
  - do not preserve internal runtime artifacts or local-path-heavy mirrors on the public branch for symmetry
  - do not rewrite every historical planning artifact just to normalize wording
  - do not adopt additional `0.118.0` features that do not materially improve the shipped release surface
  - do not replace working provider/runtime behavior solely to tell a cleaner story

## Parity / Alignment Matrix
- Current truth:
  - `.gitignore` already ignores `.runs/` and `out/`, but the public branch still tracks a leaked `.runs/.../manifest.json` plus a large `out/**` tree
  - `resume_token` validation is local-manifest-based and non-expiring, so tracked manifests should not be public, but the token does not grant remote service access on its own
  - `README.md` still points users to `docs/README.md` and shipped skills that are not packaged as downstream public docs
  - `package.json.description` is currently wrong and renders as the setup GIF markdown
  - provider setup knowledge exists in code and older internal docs, but not as one shipped downstream contract
  - generic downstream setup exists today via `init codex`, `setup`, `doctor`, and `control-host --format json`, but provider-specific readiness/help is not first-class
  - the installed CLI is `codex-cli 0.118.0`, and local help now confirms `codex exec` prompt-plus-stdin support plus `codex login --device-auth`
- Reference truth:
  - a release-ready public surface should not track runtime artifacts or local machine paths
  - npm users should be able to start from the package README and packaged docs without depending on repo-internal guides
  - provider onboarding should be one clear contract with parity between Linear and Telegram
  - docs, templates, and shipped skills should reflect the current CLI behavior, not the previous release posture
- Target truth / intended delta:
  - current tracked runtime artifacts are removed from the public branch and blocked by deterministic guards
  - packaged public docs live in an explicit curated surface, while `docs/README.md` remains contributor-only
  - provider onboarding becomes a packaged public guide with matching scaffolding and readiness/help output
  - README, guides, templates, skills, and review guidance all move to the verified `0.118.0` contract
- Explicitly out-of-scope differences:
  - historical artifact scrubbing beyond the minimum evidence-backed release posture required for the public branch
  - broad control-runtime refactors unrelated to onboarding/help truthfulness
  - a full docs-site redesign

## Not Done If
- tracked `.runs/**` or `out/**` artifacts still exist on the branch without an explicit allowlist and guard
- public or packaged docs can still embed `/Users/...`-style machine-local paths without the guard failing
- package README or packaged help still points end users at non-packaged or contributor-only docs for core setup
- fresh downstream users still need internal CO specs or source spelunking to set up Linear or Telegram
- `0.117.0` or stale prompt/stdin guidance remains in active public docs, templates, or shipped skills

## Goals
- Remove tracked runtime artifacts from the public branch and add durable public-surface guards.
- Create a curated packaged public docs surface and keep contributor-only docs clearly internal.
- Productize Linear and Telegram onboarding for downstream users with packaged docs, scaffolding, and readiness/smoke surfaces.
- Update README, guides, templates, and shipped skills to the verified `codex-cli 0.118.0` contract.

## Non-Goals
- Rewriting archived or historical planning packets for aesthetics.
- Replacing current provider behavior purely because the docs surface is awkward.
- Treating the local `resume_token` leak as evidence of a compromised remote service credential.
- Expanding into unrelated status, cloud, or control-host refactors that are not required for release truthfulness.
