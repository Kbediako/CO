# TECH_SPEC - CO-580 centralize docs freshness lifecycle resolver and owner finalizer

## Summary
- Objective: centralize docs freshness lifecycle resolution and owner finalization so command surfaces agree on effective lifecycle and owner closeout behavior.
- Scope: docs packet and mirrors, shared lifecycle helper, `docs:freshness`, `docs:freshness:maintain`, `spec-guard`, archive-stub validation, provider merge closeout owner finalizer integration, tests, and guide/provider instructions.
- Constraints: no gate weakening, no blind review-date updates, no archive of packets with open local checklists, no broad provider-intake changes.

## Issue-Shaping Contract
- User-request translation carried forward: make the CO-579 recurrence-prevention state machine explicit, with terminal source issue state treated as evidence rather than local completion and owner finalization blocking terminal closeout while active candidates still resolve to that owner.
- Protected terms / exact artifact and surface names: `docs:freshness`, `docs:freshness:maintain`, `spec-guard`, `docs:check`, `archive-stub`, `terminal_pending_archive`, `preserved_historical_stub`, `canonical_owner_key`, `codex-orchestrator:canonical-owner-key`, owner finalizer, active-reviewed terminal source, local checklist obligations, `docs:freshness:lifecycle-resolver`.
- Nearby wrong interpretations to reject: weakening freshness gates, hiding terminal lifecycle debt, bulk-bumping `last_review`, archiving packets with open checklists, treating Linear terminal state as local completion, letting short-lived feature lanes close while still configured as owners, or using `terminal_pending_archive` as a non-blocking escape hatch.
- Explicit non-goals carried forward: no broad historical packet deletion, no cap/window expansion, no manual state-file cleanup, and no provider-intake behavior changes except owner closeout/finalizer integration.

## Technical Requirements
- Functional requirements:
  1. Extend or document `scripts/lib/docs-freshness-lifecycle.js` as the shared resolver boundary.
  2. Represent effective lifecycle states for active, `terminal_pending_archive`, archived, deprecated, `preserved_historical_stub`, and retained terminal packet history.
  3. Collect local open checklist obligations for indexed task-packet paths and keep terminal source rows active when obligations remain.
  4. Reject archived task-packet rows with open local checklist obligations.
  5. Make `spec-guard` skip terminal lifecycle only when the shared resolver proves terminal/archive readiness, not merely when source issue state is terminal.
  6. Reuse shared checklist/lifecycle helpers in archive-stub validation.
  7. Emit owner-finalizer evidence from `docs:freshness:maintain`, including active resolved owner issues, ignored terminal global owners when exact cohort owner precedence applies, and terminal-transition blockers.
  8. Update provider merge closeout to consume owner-finalizer evidence and fail closed when finalizer evidence is missing or blocking.
  9. Update docs/guides and provider-worker instructions so future agents route recurring blockers through resolver/finalizer output.
- Non-functional requirements:
  - keep lifecycle and owner evidence machine-readable
  - preserve strict fail-closed semantics
  - keep owner precedence deterministic and auditable
- Interfaces / contracts:

  - `docs:freshness` JSON report invalid/rolling/terminal lifecycle sections
  - `docs:freshness:maintain --format json` owner-finalizer section
