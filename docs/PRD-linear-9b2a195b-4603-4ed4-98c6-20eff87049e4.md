## Summary
- Problem Statement: A live provider-worker lane for `CO-31` can read Linear successfully, and control-host/operator surfaces already show that issue as actively running, but active-lane Linear mutations intermittently fail during the same attempt. The authoritative audit trail initially looked like generic `linear_request_failed` failures. Raw upstream inspection now shows the underlying condition is Linear hourly request-budget exhaustion surfaced as a GraphQL `RATELIMITED` error on HTTP `400`, combined with redundant preflight reads in `transition` and `upsert-workpad` that spend request budget after a successful `issue-context` call.
- Desired Outcome: Make the active provider-worker write-back path reliable when a live read has already succeeded by reusing recent issue context instead of immediately re-reading it, and surface rate-limit failures explicitly so the worker and operator truth surfaces stop treating them like opaque transport errors.
