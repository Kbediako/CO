## Problem
Docs freshness recurrence loops still require too much parent-lane diagnosis when stale docs/spec debt reappears after prior repair attempts. `docs:freshness:maintain` can identify maintenance candidates, but CO-431 requires it to verify live owner state by canonical owner key before deciding whether to create a new owner, update an existing owner, or replace a terminal owner.

This lane is not another manual freshness refresh. It closes the automation gap for three known recurrence shapes:
- `CO-428`: stale active-spec recurrence.
- `CO-429`: completed-lane registry residue.
- `CO-430`: terminal-owner replacement when the prior canonical owner is no longer open.

CO-188 and CO-323 must remain visible as escaped historical root-cause attempts. CO-431 should preserve their evidence and finish the canonical-owner automation they did not fully close.
