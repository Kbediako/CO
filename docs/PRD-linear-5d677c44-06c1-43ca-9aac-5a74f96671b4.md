## Problem
Docs freshness recurrence loops still require too much parent-lane diagnosis when stale docs/spec debt reappears after prior repair attempts. `docs:freshness:maintain` can identify maintenance candidates, but CO-431 requires it to verify live owner state by canonical owner key before deciding whether to create a new owner, update an existing owner, or replace a terminal owner.

This lane is not another manual freshness refresh. It closes the automation gap for three known recurrence shapes:
- `CO-428`: stale active-spec recurrence.
- `CO-429`: completed-lane registry residue.
- `CO-430`: terminal-owner replacement when the prior canonical owner is no longer open.

CO-188 and CO-323 must remain visible as escaped historical root-cause attempts. CO-431 should preserve their evidence and finish the canonical-owner automation they did not fully close.

## Goals
- Verify live owner state by exact canonical owner key before creating or updating a docs freshness owner issue.
- Route the three current recurrence shapes to the intended owner path: `CO-428` for stale active-spec debt, `CO-429` for completed-lane registry residue, and `CO-430` for terminal-owner replacement.
- Emit machine-readable create/update/noop/replacement evidence that parent lanes and reviewers can cite.
- Emit dry-run and no-token copyable owner bodies without mutating Linear.
- Run the same canonical-owner routing contract from scheduled maintenance and provider-worker preflight paths.
- Preserve fail-closed behavior for real current-diff, invalid registry, missing registry, uncatalogued docs, and current-task packet drift.

1. `docs:freshness:maintain` computes or consumes a deterministic `canonical_owner_key` and exact marker for docs freshness recurrence candidates.
2. Before mutation, the maintenance path verifies live same-team/same-project owner state by exact marker and classifies the owner as open, missing, or terminal.
3. `CO-428`, `CO-429`, and `CO-430` recurrence shapes are routed distinctly in the maintenance decision output.
4. Machine-readable reports include create/update/noop/replacement action evidence, owner identifiers when available, route reason, marker, and sample/copyable bodies.
5. Dry-run and no-token runs never mutate Linear and still emit copyable issue/comment bodies for the parent to apply manually.
6. Scheduled maintenance and provider-worker preflight paths run the canonical-owner routing contract.
7. CO-188 and CO-323 remain cited as historical root-cause attempts rather than active owners that make CO-431 unnecessary.
8. Focused tests cover open-owner update, missing-owner create, terminal-owner replacement, dry-run/no-token copyable output, and each CO-428/CO-429/CO-430 route.
