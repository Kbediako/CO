# PRD - CO: stop released provider claims from burning Linear issue-by-id polls

## Traceability
- Linear issue: `CO-160` / `95011595-52df-4ad8-9cb5-256e7eee5424`
- Linear URL: https://linear.app/asabeko/issue/CO-160/co-stop-released-provider-claims-from-burning-linear-issue-by-id-polls
- Related landed slices: `CO-159`, `CO-156`, `CO-144`
- Evidence issues: `CO-139`, `CO-96`

## Summary
- Problem Statement: after `CO-159` landed, normal deferred provider polls can still burn Linear request budget through `dispatch_source_issue_by_id` even when all retained claims are already locally `released`. The current local intake evidence shows 128 retained released claims, normal polls pass `trackedIssues: []` plus `deferFreshDiscovery: true`, and the released-claim path still falls through to per-claim direct issue reads instead of failing closed from cached local truth.
- Desired Outcome: deferred polls with retained released claims should perform zero direct issue-by-id reads for those claims, keep the claims locally visible without per-poll churn, and preserve a bounded reopen path when newer webhook or tracked-issue evidence says the issue is runnable again.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): finish `CO-160` by closing the remaining post-`CO-159` request-burn seam without weakening the dead-active handling already landed. The fix should stay narrow to retained released claims in the deferred poll path, prove the result with machine-checkable regression coverage, and keep bounded recovery for genuinely reopened work.
- Success criteria / acceptance:
  - a normal deferred poll with retained released claims and empty `trackedIssues` performs zero `resolveTrackedIssue` / `dispatch_source_issue_by_id` calls for those released claims
  - regression coverage exercises 128 retained released claims across at least two consecutive polls and proves no direct issue-by-id reads and no fresh discovery churn
  - already released inactive or non-mutable claims remain locally visible without per-poll `updated_at` churn
  - startup or recovery still has an explicit bounded path to revalidate or reopen a released claim when newer live evidence says the issue is runnable again
  - queued or in-progress release cleanup still works where cancellation or provider cleanup is genuinely pending
  - intake budget telemetry can show no per-poll burn from all-released retained claims
- Constraints / non-goals:
  - do not hide real `Ready` or `In Progress` work
  - do not weaken `CO-159` dead active claim handling
  - do not require a live Linear scan for every retained released claim
  - do not widen into unrelated GitHub throttling or `CO STATUS` rendering work

## Parity / Alignment Matrix
- Not applicable. This is a bounded provider poll fail-closed lane rather than a parity or alignment migration.
- Current truth:
  - `resolveTrackedIssuePollResolutionWithFallback(...)` only fails closed when `resolveProviderIssuePollFailClosedReason(...)` classifies the claim as cached non-runnable, review wait, or merge wait
  - `resolveProviderIssuePollFailClosedReason(...)` currently does not classify ordinary retained `released` claims, so deferred polls fall through to direct `resolveTrackedIssue(...)`
  - released-claim reopen seams already exist through `shouldReopenReleasedClaimAtCurrentTimestamp(...)`, `shouldReopenReleasedClaimOnRefresh(...)`, and `provider_issue_released_pending_reopen:*`
- Reference truth:
  - locally terminal released claims should stay visible and reusable without repeated live reads during ordinary deferred polls
  - only newer live evidence or explicit recovery paths should reopen released claims
  - deferred polls with no tracked issues should avoid fresh discovery churn when all retained claims are already released
- Target truth / intended delta:
  - deferred polls fail closed for retained released inactive or non-mutable claims from cached local truth
  - direct issue-by-id reads remain available for non-deferred refresh, pending-reopen recovery, or claim classes that still need live revalidation
  - existing release cleanup and reopen behavior remains intact
- Explicitly out-of-scope differences:
  - changing request-budget policy or reserve thresholds
  - replacing `CO-159` dead-active handling
  - changing unrelated `CO STATUS` or GitHub provider behavior

## Not Done If
- Deferred polls still call `dispatch_source_issue_by_id` for retained released claims when `trackedIssues` is empty.
- The only mitigation is still stopping or restarting the host manually.
- Released claims disappear or lose the bounded reopen path when newer evidence says the issue is runnable again.
- Released inactive or non-mutable claims keep churning `updated_at` or budget on every poll.

## Goals
- Keep deferred released-claim reconciliation local-first and fail closed.
- Preserve bounded reopen or revalidation behavior for genuinely runnable reopened issues.
- Prove the fix with focused regressions and machine-checkable validation artifacts.
- Keep release cleanup behavior truthful for queued or in-progress runs that still need cancellation or cleanup.

## Non-Goals
- Redesigning provider polling or request-budget accounting broadly.
- Hiding all released claims from intake state.
- Weakening review-handoff, merge-closeout, or dead-active-claim logic already landed.
- Fixing unrelated GitHub throttling, status UI, or archive-policy work.
