# PRD - CO: Harden Linear rate-limit handling with user-scoped budgeting, endpoint-aware buckets, and complexity-aware preflight

## Added by Bootstrap 2026-04-07

## Traceability
- Linear issue: `CO-106` / `95f1c334-e623-471e-a013-d7019feed423`
- Linear URL: https://linear.app/asabeko/issue/CO-106/harden-linear-rate-limit-handling-user-scoped-budgeting-endpoint-aware

## Summary
- Problem Statement: `CO-62` introduced shared Linear budget awareness, helper fail-fast behavior, and budget-shaped polling, but the current repo still mis-models several quota truths that matter under concurrent or mixed-endpoint load. Shared persistence is still keyed only by token fingerprint instead of Linear user scope, endpoint-specific request and complexity buckets are stored as one unkeyed pair rather than per endpoint identity, helper preflight only enforces explicit headroom for request counts, newer partial observations can clobber richer prior bucket data, the repo still does not preserve `x-ratelimit-endpoint-name` or `x-complexity`, and control-host polling still uses fixed pressure tiers without jitter or endpoint-specific degradation. The current behavior is therefore safer than pre-`CO-62`, but it still re-hits limits more often than necessary and still leaves concurrency or cross-endpoint truth gaps.
- Desired Outcome: make the shared Linear budgeting model truthful enough for multi-process and mixed-endpoint operation by aligning the canonical persisted budget to Linear user scope, preserving endpoint identity and per-request complexity cost, enforcing complexity-aware and reservation-aware preflight, merging partial observations safely, codifying cooldown precedence, and jittering or degrading polling when the relevant buckets are under pressure.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): finish the hardening pass left after `CO-62` so the shared Linear budget substrate reflects Linear’s real quota shape instead of a token-only approximation. The fix needs to stay bounded to rate-limit truth and scheduling, not broaden into a full provider-control-plane redesign, but it does need to cover the concrete remaining gaps the issue calls out: user scope, endpoint-aware buckets, complexity-aware headroom, safe merge behavior, explicit cooldown precedence, smarter polling degradation, and ideally shared reservations or smoothing across concurrent processes.
- Success criteria / acceptance:
  - shared budget persistence is no longer token-only; canonical state aligns to Linear user scope or records a precise, test-backed reconciliation
  - endpoint-specific request and complexity buckets are preserved and consulted by endpoint identity, using `x-ratelimit-endpoint-name` when available
  - preflight can fail before `fetch` on complexity headroom shortfall, not just request-count shortfall
  - partial newer observations preserve richer prior bucket or reset metadata instead of clobbering it wholesale
  - `x-complexity` is captured and surfaced when Linear sends it so operators and future preflight logic see actual request cost
  - cooldown precedence between `retry-after` and exhausted-bucket reset timestamps is explicit and test-locked
  - control-host polling is jittered and degrades more deliberately under low, exhausted, and endpoint-specific pressure
  - concurrent processes no longer rely only on reactive last-observation state; a bounded shared reservation or smoothing step is added, or a documented residual-risk follow-up is filed if that cannot land cleanly in scope
  - focused tests cover the hardened contract end to end
- Constraints / non-goals:
  - do not broaden into a full Linear provider architecture rewrite or unrelated workflow cleanup
  - do not require webhook-first migration before landing this slice
  - do not weaken truthful `linear_rate_limited` behavior in order to mask real exhaustion
  - do not silently invent endpoint identity when Linear does not provide it; use explicit fallback or preserve the unknown state truthfully

## Parity / Alignment Matrix
- Not applicable; this is a bounded truthfulness and concurrency-hardening lane, not a parity migration.
- Current truth:
  - shared Linear budget state already exists, but it is keyed by token fingerprint and stores only one endpoint request bucket plus one endpoint complexity bucket without endpoint identity
  - helper preflight only enforces `minimum_requests_remaining`; complexity buckets only block once they are already exhausted
  - `x-ratelimit-endpoint-name` and `x-complexity` are still dropped
  - polling intervals widen under pressure, but the policy is deterministic, tiered, and not endpoint-aware
  - concurrent callers still coordinate only through "read last observation, then request" rather than a reservation-aware smoothing step
- Reference truth:
  - Linear quota coordination should track the actual shared owner and the relevant endpoint bucket instead of a token-only or endpoint-agnostic approximation
  - low-budget callers should fail or defer before redundant requests, including on likely complexity shortfall, and concurrent callers should not stampede the same bucket
- Target truth / intended delta:
  - canonical budget persistence is user-scoped, with endpoint-aware bucket maps and explicit endpoint-name aliasing
  - preflight understands request and complexity floors, subtracts active reservations, and fails before `fetch` when the shared budget truth is already insufficient
  - observations merge bucket-wise, retain richer prior data, and expose an explicit cooldown precedence rule
  - polling intervals jitter and degrade according to the relevant budget pressure instead of using static low or constrained tiers only
- Explicitly out-of-scope differences:
  - full webhook migration
  - full provider-control-plane redesign
  - unrelated Linear attachment or review-loop cleanup

## Not Done If
- Shared budget persistence is still effectively token-only with no user-scope reconciliation path.
- Endpoint request or complexity buckets are still stored without endpoint identity.
- Complexity headroom still cannot fail preflight until the bucket is already exhausted.
- Newer partial observations can still erase richer prior bucket or reset metadata.
- Polling remains deterministic fixed-tier backoff with no jitter or endpoint-specific degradation.
- Concurrent processes still have no shared reservation or smoothing step and no explicit residual-risk documentation.

## Goals
- Align the canonical Linear budget key to the real shared user scope while keeping cold-start behavior fail-safe.
- Preserve `x-ratelimit-endpoint-name` and `x-complexity` and feed them into persisted state plus troubleshooting output.
- Add endpoint-aware request and complexity bucket maps with operation-to-endpoint aliasing.
- Extend preflight to enforce request and complexity headroom, ideally using recent endpoint cost observations when available.
- Merge partial observations safely and keep cooldown truth explicit.
- Add a bounded shared reservation or smoothing layer that reduces concurrent over-spend risk without turning the lane into a full distributed scheduler project.
- Jitter and degrade polling/backoff behavior under relevant pressure with focused tests.

## Non-Goals
- Full webhook-first provider migration.
- Generic caching of issue payload truth.
- Unrelated Linear workflow or dashboard redesign.
- Perfect quota prediction when Linear omits endpoint name or cost headers; the lane only needs truthful, bounded fallbacks.

## Technical Considerations
- Architectural Notes:
  - the existing `linearBudgetState.ts` helper is the right owner for most of this hardening; the repo should not fork the contract across dispatch, provider helpers, and control-host scheduling
  - provider helper issue-context and summary queries currently fetch only `viewer.organization.id`; they need `viewer.id` so user-scoped persistence can be learned outside dispatch-only paths
  - endpoint-aware buckets need a stable alias story so preflight can consult the right endpoint before a request; server-supplied endpoint name should be canonical when present
  - bounded reservation smoothing should stay lock-file-backed and piggyback on the existing atomic state writes instead of adding a new service
- Dependencies / Integrations:
  - Linear GraphQL response headers
  - shared lock-file and atomic-write helpers
  - control-host polling health persistence
  - provider helper audit / workpad / attachment flows that already rely on helper preflight
