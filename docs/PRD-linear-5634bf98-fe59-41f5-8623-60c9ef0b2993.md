## Summary
- Problem Statement: `docs:freshness:maintain` again points the global rolling freshness owner at terminal `CO-575` even though `CO-575` is `Done` / `completed`. After re-homing that owner, live validation also exposed the deeper freshness recurrence: stale registry rows for active task-packet mirrors can lag behind the authoritative `tasks/index.json` review date, older numeric task keys do not always alias PRD/ACTION mirrors, and one archived CO-300 classification finding remained marked `active`.
- Desired Outcome: re-home the global `docs:freshness:maintain` owner metadata to live `CO-579`, report baseline cohorts under their exact CO-568/CO-569 owners, treat newer task-index reviews as authoritative only for active indexed task-packet mirrors, support guarded historical numeric task-key aliases, reclassify the archived CO-300 finding row, preserve `CO-575` as terminal lineage, and keep freshness/spec policy strict while routing the larger lifecycle/finalizer state-machine work to CO-580.

## User Request Translation
- User intent / needs: restore a live same-project global owner for docs freshness debt so unrelated feature lanes do not inherit a terminal-owner blocker, while keeping stale/rolling debt visible and governed.
- Success criteria / acceptance:
  - `docs/docs-catalog.json` and `docs/guides/docs-freshness-cohorts.md` name `CO-579` as the live global owner for `docs:freshness:maintain`
  - `CO-575` remains visible as terminal lineage, not active configured owner
  - `CO-568` and `CO-569` remain live owners for their baseline cohort canonical keys
  - `docs:freshness` reports rolling cohorts under exact canonical owners, not the global fallback owner
  - indexed active task-packet mirrors use a newer `tasks/index.json` `last_review` with explicit `task_index_review_overrides` report metadata instead of remaining stale due to registry mirror drift
  - historical numeric task-key packets such as `1318-...` resolve PRD/ACTION aliases without blind registry date bumps
  - pre-expiry task specs are reclassified inactive `done` only after live Linear issue-context verifies their source issues are `Done/completed`, including the 2026-05-26 recurrence for CO-498, CO-442, CO-460, and CO-453
  - verified terminal packets become `retained_terminal_packet` only when local checklist obligations are closed; otherwise their packet rows remain active-reviewed with terminal source evidence
  - report-only finding rows may be retained as terminal history only when live Linear and PR evidence prove the source issue is complete, as with CO-364 / PR #643
  - `retained_terminal_packet` rows remain full packet history on `main` and cannot be converted into archive stubs by implementation-docs archive automation when task-index or registry task-status terminal evidence exists
  - a stray `retained_terminal_packet` status without terminal lifecycle evidence does not hide active or stale docs from archive handling
  - `docs:freshness:maintain --check` no longer reports `configured_owner_terminal` for the global owner
  - docs gates are green or fail only on separately routed, non-global-owner debt with live owner evidence
  - CO-580 exists as the linked Backlog issue for the shared lifecycle resolver and owner finalizer so CO-579 does not overclaim that larger refactor
- Constraints / non-goals:
  - no weakening of `docs:freshness`, `docs:freshness:maintain`, `docs:check`, or `spec-guard`
  - no blind `last_review` bumps or historical packet deletion
  - no implementation-docs archive path that stubs evidence-backed `retained_terminal_packet` rows by retention age or line threshold
  - no implementation-docs archive path that trusts `retained_terminal_packet` status alone for stray docs without terminal lifecycle evidence
  - no provider-intake, runtime freshness, review-wrapper, or `create-follow-up` behavior changes
  - no manual state-file cleanup as the fix
