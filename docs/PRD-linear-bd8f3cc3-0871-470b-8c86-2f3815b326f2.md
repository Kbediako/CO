## Summary
- Problem Statement: the shipped `co-status attach` path already keeps the live viewer on `primary scrollback`, but it still redraws each refresh as a full `CO STATUS` frame. Operators scrolling during live attach therefore see accumulated historical full-screen copies instead of one pinned live surface plus honest history. The same live view still has several operator-readability defects called out in the issue: long countdowns remain raw seconds or minute-only text, Codex and Linear rate-limit formatting is hard to scan, retry rows are vague, and the `EVENT` column still tends to surface generic text rather than the most meaningful current activity.
- Desired Outcome: keep `co-status attach` as a read-only live viewer for an already-running control-host, but change its live rendering semantics so the current `CO STATUS` surface updates in place on `primary scrollback` without stacked full-frame history. At the same time, improve the protected live operator surfaces so long countdowns, age/runtime values, rate-limit reset windows, retry rows, and event text are readable and truthful enough for active incident work. Launch-mode `co-status` alternate-screen behavior must remain distinct and unregressed.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): preserve live scrollable `CO STATUS` for the read-only attach flow, but stop treating "primary scrollback" as permission to append an entirely new full frame on each refresh. The live surface should stay pinned/current while history remains inspectable. The same lane must also make the visible countdowns, rate limits, stage/read-model truth, and event summaries trustworthy enough that operators can diagnose an active run without guessing what is current.
- Success criteria / acceptance:
  - `co-status attach` remains read-only and does not fall back to "just use alternate screen"
  - live attach scrolling no longer reveals stacked historical full-frame `CO STATUS` copies
  - live attach uses a bounded in-place region or equivalent pinned live-surface strategy in primary scrollback
  - launch-mode `co-status` alternate-screen behavior stays distinct and unregressed
  - `Next refresh`, `AGE / TURN`, retry countdowns, and rate-limit reset windows use higher-order duration formatting when values are large
  - Codex rate-limit rendering uses the requested percentage semantics and spacing, and Linear rate-limit rendering uses clearer request/complexity labels and spacing
  - the `EVENT` column and retry rows show more meaningful current activity than generic "Provider worker turn is active" or bare `attempt=1 in 0.000s`
  - final validation includes real screenshots from this device embedded directly in Linear, with explicit visual inspection and cleanup of any temporary proof sessions/windows
- Constraints / non-goals:
  - do not reopen the old paused duplication defect from `CO-67`
  - do not reopen the full `CO-83` telemetry restoration lane unless new evidence proves the remaining defect is in attach/view rendering
  - do not replace live attach with pause-first workflow or browser `/ui` redesign
  - do not treat docs/help updates alone as sufficient without changing live rendering behavior

## Parity / Alignment Matrix
- Current:
  - `co-status attach` keeps live output on `primary scrollback` by design.
  - each refresh still writes a full new `CO STATUS` frame into the primary buffer.
  - scrolling up reveals stacked historical full-frame copies.
  - several countdown and rate-limit surfaces still render in raw or hard-to-scan formats.
  - event and retry text remains too generic for live operator use.
- Reference:
  - operators should be able to keep one live `CO STATUS` surface visible while inspecting history without repeated duplicate full-frame accumulation.
  - current versus historical state should remain visually and operationally distinct.
  - live countdowns, runtime/age values, rate limits, stage labels, and event text should be readable enough for active debugging without guessing.
- Target:
  - live attach viewing preserves scrollability without full-frame accumulation.
  - attach mode remains read-only and operator-friendly.
  - the live surface updates in place or uses another bounded pinned-region strategy on primary scrollback.
  - countdowns, age/runtime values, and rate-limit reset windows use higher-order readable formatting.
  - backoff/event/stage text stays meaningful and does not preserve stale running truth after the underlying issue has already moved on.

## Not Done If
- Scrolling up during live `co-status attach` still reveals stacked historical full-frame `CO STATUS` copies.
- The shipped solution simply moves attach mode back to `alternate screen` and removes live scrolling.
- The viewer still rewrites the entire primary buffer frame-by-frame with no bounded in-place live region strategy.
- Long countdowns and age/runtime values still render only as raw seconds or minute-only text where that is misleading or hard to scan.
- Codex / Linear rate-limit surfaces still use the confusing percent/spacing output called out in the issue.
- Retry rows or event text remain so generic that operators still cannot tell what is actually being attempted.
- Validation stops at unit tests or screenshot capture without real-device images embedded directly in Linear and visually inspected.

## Technical Considerations
- Architectural Notes:
  - `startAttachedControlStatusDashboard(...)` already sets `liveSurfaceMode: 'primary'`, so the remaining scrollback defect is not a missing mode switch; it is the primary-surface write strategy in `controlStatusDashboard.ts`
  - the same file currently owns most of the visible countdown, rate-limit, retry, and event formatting seams
  - `compatibilityIssuePresenter.ts` and `selectedRunProjection.ts` already contain some stale-state suppression logic, so presenter widening should remain evidence-driven rather than assumed up front
  - the likely bounded fix is:
    - keep launch live mode on alternate screen
    - keep attach live mode on primary scrollback
    - change attach live redraw to update a pinned live block in place rather than appending a full new frame
    - tighten the dashboard formatting helpers for higher-order durations and clearer rate-limit/event/retry text
    - only patch read-model / presenter freshness if validation proves the stale `Merging` / retry projection is still real
