## Summary
`CO-202` closes the Ready reclaim gap for local provider-intake rows that retain a plain `provider_issue_released:not_active` reason with stale cached started-state metadata. A Ready/unstarted issue with no live same-issue worker must be eligible for fresh discovery after blockers clear or an operator moves it back to Ready, even when `provider-intake-state.json` still shows stale cached `issue_state=Blocked` and `issue_state_type=started`.

## User Request Translation
- Reclaim Ready/unstarted issues that are suppressed only by stale plain `released/not_active` provider-intake cache state.
- Treat stale cached non-active started-state metadata such as `Blocked` as recheckable when there is no live same-issue worker and the live issue is now Ready.
- Preserve the existing released-pending-reopen recovery path from `CO-193`.
- Preserve `CO-192` terminal stale-row pruning and `CO-189` live same-issue worker protection.
- Do not solve the incident by manually deleting provider-intake claims or increasing Linear request burn.

  - deleting the stale intake row as the product fix
  - treating stale cached `Blocked` as authoritative after the live issue returns to Ready
  - weakening terminal Done/cancelled cleanup or terminal stale-row pruning from `CO-192`
  - starting a duplicate worker when `CO-189` evidence shows a same-issue worker is genuinely live
  - broadening Linear polling or request burn to mask stale local state

## Acceptance Criteria
- A Ready/unstarted issue with no live same-issue worker is not suppressed solely because provider-intake still has a plain `released` / `provider_issue_released:not_active` row with stale cached `issue_state=Blocked` and `issue_state_type=started`.
- The fix covers plain `provider_issue_released:not_active`, not only `provider_issue_released_pending_reopen:*`.
- `fresh_discovery` can reclaim the eligible issue when live issue data confirms Ready/unstarted, no blockers, and no live worker.
- `CO-193` released-pending-reopen reclaim remains covered.
- `CO-192` terminal stale-row pruning and terminal Done/cancelled projection cleanup remain covered.
- `CO-189` live same-issue worker protection remains covered so duplicate provider workers are not launched.
- The solution retains provider-intake evidence and does not rely on manual local claim deletion.
- The solution does not broaden Linear polling or request burn.

## Non-Goals
- No manual deletion of `provider-intake-state.json` claims as the product fix.
- No broad Linear polling cadence increase.
- No change to terminal Done/cancelled projection cleanup.
- No weakening of active worker admission caps or same-issue duplicate protection.
- No status renderer redesign beyond preserving the required active/debug surfaces.

## Not Done If
- A Ready issue with no live same-issue worker remains suppressed only because provider-intake has stale plain `released/not_active` started-state metadata.
- The fix only handles `provider_issue_released_pending_reopen:*` and misses plain `provider_issue_released:not_active`.
- The fix deletes recovery evidence or weakens terminal stale-row pruning from `CO-192`.
- The fix can duplicate a genuinely live same-issue worker protected by `CO-189`.

## Metrics & Guardrails
- Primary success metrics:
  - focused provider handoff regression proves stale plain released/not_active `Blocked` cache is reclaimable after live Ready confirmation.
  - existing pending-reopen, terminal pruning, and live-worker regressions stay green.
  - delegation guard recognizes the same-issue child-lane manifest without blanket override.
- Guardrails:
  - terminal workflow states are not made reclaimable.
  - active and review-handoff cached states remain non-reclaimable.
  - same-issue live worker evidence vetoes reclaim.
  - stale provider-intake evidence remains inspectable.

- Operators can move a previously blocked issue back to Ready and expect normal provider pickup once blockers are gone, without manually editing local intake state.
- `CO STATUS` and debug surfaces continue to expose useful provider claim context without letting stale non-active cache suppress eligible Ready work.
