## Summary
- Problem Statement: CO is already close to Symphony parity for the local single-host path, but the optional distributed worker-host seam is still missing. `provider-linear-worker` metadata exposes terminal cleanup but no worker-host inventory, provider dispatch still reasons only about global or per-state agent capacity, control-host launch specs are still local-only, and provider proofs plus operator surfaces do not carry `worker_host`.
- Desired Outcome: add one bounded optional distributed worker-host layer that can select configured remote hosts, respect per-host capacity, launch eligible provider-worker runs over SSH, preserve truthful retry or recovery semantics, and surface `worker_host` in runtime or operator observability without replacing the single-host default.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): close the specific optional distributed worker-host parity gap, not by rewriting the runtime, but by adding the smallest real remote-execution path that can route bounded provider-worker work to configured SSH-backed hosts with explicit capacity and truthful `worker_host` observability.
- Success criteria / acceptance:
  - CO can target configured remote worker hosts through a bounded optional execution path
  - host inventory and per-host capacity selection are explicit and tested
  - `worker_host` is present in runtime or operator observability where relevant
  - failure recovery and ownership semantics remain truthful
  - the single-host default remains intact and documented
- Constraints / non-goals:
  - do not replace the single-host default path
  - do not absorb `CO-82`, `CO-83`, or `CO-88`
  - do not bundle BEAM, live-upgrade, or runtime-rewrite work into this lane
  - do not require every pipeline to support remote dispatch on day one

## Parity / Alignment Matrix
- Current truth:
  - `provider-linear-worker` workflow metadata exposes `terminal_cleanup` only
  - provider dispatch capacity is global or per-state only
  - control-host launch specs and provider-worker execution are local-only
  - provider proof, debug, and operator surfaces do not consistently carry `worker_host`
- Reference truth:
  - Symphony already models optional SSH worker hosts, per-host concurrency, preferred-host retry stickiness, and `worker_host` observability through the orchestrator, agent runner, workspace, SSH, and presenter seams
- Target truth / intended delta:
  - CO keeps the control-host authoritative while adding optional worker-host inventory plus selection
  - eligible worker runs can launch on configured remote hosts over bounded SSH transport
  - per-host capacity and chosen `worker_host` become visible in runtime or operator truth surfaces
  - retry or recovery remains truthful and does not silently reroute past side effects
- Explicitly out-of-scope differences:
  - broad app-server continuity or runtime hot-reload work
  - mandatory distributed execution
  - telemetry or debug-snapshot work already owned by `CO-82`, `CO-83`, or `CO-88`

## Not Done If
- The lane lands only config or docs without a real remote execution path.
- Capacity selection exists on paper but is not surfaced in runtime or operator evidence.
- `worker_host` is absent from the relevant proof or observability surfaces.
- The feature stops being optional or destabilizes the single-host default.

## Goals
- Add an explicit worker-host inventory or config contract for provider-worker execution.
- Add bounded host selection with per-host capacity and retry truthfulness.
- Route eligible launches to remote hosts over SSH while preserving current local defaults.
- Surface `worker_host` in provider proof, read-model, and operator surfaces where relevant.
- Keep the implementation focused on one real optional path plus tests and docs.

## Non-Goals
- Replacing the single-host execution model.
- Reopening `CO-82`, `CO-83`, or `CO-88`.
- Broad control-host or runtime architecture rewrites.
- Repo-wide telemetry cleanup or provider-debug-snapshot expansion.
- BEAM, live-upgrade, or rewrite debates.
