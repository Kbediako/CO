# PRD - CO: Prefer fresh source CLI execution in source checkouts after merge

## Added by Bootstrap 2026-04-09

- Linear URL: https://linear.app/asabeko/issue/CO-113/co-make-source-checkouts-prefer-fresh-cli-source-after-merge-without
- Related issues: `CO-25`, `CO-54`, `CO-100`

## Summary
- Problem Statement: merged TypeScript changes in a source checkout do not reliably reach the operator-facing CO CLI surfaces until `dist/` is rebuilt manually, and some source-checkout validation paths can still drift onto packaged `dist` assets or packaged config instead of the live repo-local source/config surface.
- Desired Outcome: a source checkout uses fresh source-backed execution by default after merge, packaged installs stay dist-only, and any source-checkout fallback to stale `dist` becomes explicit instead of silent.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): finish `CO-113` in this provider-worker workspace by making source checkouts authoritative for the main CO CLI/runtime surfaces, keeping packaged installs self-contained, and leaving auditable docs, tests, and validation evidence that manual `npm run build` is no longer the mitigation after merge.
- Success criteria / acceptance:
  - restarting a source checkout after merged TypeScript changes picks up the fresh source entrypoint without requiring a manual `npm run build`
  - `control-host`, provider child-stream, provider child-lane, delegation, and top-level CLI entrypoints all follow the same source-vs-dist resolution contract in a source checkout
  - temp-workspace or local validation paths do not silently fall back to packaged config or inherited provider snapshots when repo-local source/config should be authoritative
  - packaged installs still run compiled `dist` only and do not require repo-only source files or dev dependencies at runtime
  - any source-checkout fallback to `dist` is explicit and operator-visible
- Constraints / non-goals:
  - do not rely on local git hooks alone
  - do not add `postinstall` or `prepare` hooks for published installs
  - do not reopen unrelated merge-closeout, review-policy, or provider-fallback lanes
  - keep the change focused on source-authoritative execution and the named config-fallback seam

## Parity / Alignment Matrix
- Required for parity/alignment lanes; otherwise state `Not applicable`.
- Current truth:
  - `package.json` bins still target `dist/bin/codex-orchestrator.js`
  - `bin/codex-orchestrator.ts` already exists as the source entrypoint, but the checked-in npm/bin surface does not route through it
  - `providerLinearChildStreamShell`, `providerLinearChildLaneShell`, and provider-worker prompt guidance still prefer `dist/bin/codex-orchestrator.js`
  - `commandRunner` still special-cases the provider worker stage to `dist/orchestrator/src/cli/providerLinearWorkerRunner.js`
  - repo-local compatibility paths can still use packaged config fallback in source-checkout execution when that contract is not explicit
- Reference truth:
  - source checkouts should execute the repo-local TypeScript surface when the local TS runtime is available
  - packaged installs should execute only shipped JavaScript artifacts
- Target truth / intended delta:
  - checked-in bootstrap bin prefers source in source checkouts, otherwise runs packaged `dist`
  - provider child/delegation/control-host/helper surfaces reuse the same bootstrap or source-aware runner resolution
  - source-checkout config fallback becomes explicit/opt-in instead of ambient
- Explicitly out-of-scope differences:
  - automatic rebuild hooks
  - publish-time packaging policy beyond including the checked-in bootstrap bin
  - unrelated provider workflow ownership or merge-closeout behavior

## Not Done If
- The top-level CLI uses source but provider child streams, child lanes, delegation, or the provider worker stage still run stale `dist` after merge.
- Temp-workspace or local validation paths can still silently exercise packaged config or inherited provider snapshots instead of the repo-local checkout.
- Packaged installs become dependent on repo-only source files or dev tooling.
- The only mitigation remains "run `npm run build` after merge".

## Goals
- Add a checked-in bootstrap bin for `codex-orchestrator` / `codex-orch` that prefers the source entrypoint in a source checkout and falls back to compiled `dist` in packaged installs.
- Make the provider child/delegation/control-host surfaces use the same source-vs-dist contract.
- Eliminate silent source-checkout package-config fallback unless an explicit lane contract opts back in.
- Add focused coverage plus `pack:smoke` proof for source checkout execution and packaged-install execution.

## Non-Goals
- Replacing every internal runner with a new framework or launcher abstraction.
- Changing packaged-install publishing to ship TypeScript or `ts-node`.
- Broadly refactoring all config-loading semantics outside the named source-authority seam.
