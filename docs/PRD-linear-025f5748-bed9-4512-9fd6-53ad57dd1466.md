## Problem Statement

CO can keep issues in `Blocked` after all of their blockers have reached terminal states. When this happens, the issue is no longer blocked by live work, but it may still require an operator decision: unblock it, move it back into the queue, mark it as a duplicate, or attach it to a canonical owner. Today that state can stay hidden until a person manually inspects the blocker list.

The risk is not only stale queue state. The risk is unsafe automation: a naive fix could auto-close duplicates, weaken non-terminal blocker semantics, relaunch `Blocked issues` directly, or bypass the CO-259 canonical-owner reuse contract. CO-266 should instead add a read-only advisory that gives operators evidence and a recommended action while preserving all mutation decisions for existing workflows.

## Desired Outcome

- `Blocked issues` with only `terminal blockers` are surfaced as operator-visible advisory candidates.
- Each advisory record includes the required evidence fields: `issue id`, `live state`, `blocker list`, `canonical/duplicate hints`, and `recommended action`.
- Recommended actions classify candidates as `ready-to-unblock candidate`, `duplicate-cleanup candidate`, or an explicitly held read-only advisory when evidence is insufficient.
- Non-terminal blockers keep suppressing advisory eligibility.
- `operator-autopilot` can present the advisory, but the advisory remains read-only and does not transition, close, reopen, or relaunch issues.
- CO-259 canonical-owner reuse remains the only canonical owner creation/reuse path; CO-266 can supply `blocked-terminal-blocker-cleanup-advisory` as an exact owner key but must not replace that path.

## Acceptance Criteria

- The parent implementation identifies `Blocked issues` whose entire `blocker list` is terminal.
- Any issue with a non-terminal blocker is excluded or held with a clear non-terminal reason.
- Each advisory candidate emits `issue id`, `live state`, `blocker list`, `canonical/duplicate hints`, and `recommended action`.
- Candidate recommendations use the protected terms `ready-to-unblock candidate`, `duplicate-cleanup candidate`, and `read-only advisory`.

- CO-259 canonical-owner reuse remains authoritative for owner/follow-up reuse and receives the exact key `blocked-terminal-blocker-cleanup-advisory` when needed.

## Not Done If

- CO automatically closes, duplicates, unblocks, relaunches, or transitions an issue as part of this advisory.
- A `Blocked` issue with any non-terminal blocker can be surfaced as ready to unblock.
- Advisory output lacks any required evidence field: `issue id`, `live state`, `blocker list`, `canonical/duplicate hints`, or `recommended action`.
- The implementation replaces CO-259 canonical-owner reuse or uses fuzzy duplicate matching to select owners.
- Operators cannot distinguish `duplicate-cleanup candidate` from `ready-to-unblock candidate`.
- The advisory is only logged in an internal file and is not operator-visible.
