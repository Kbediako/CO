# PRD - CO: Add authoritative provider-worker progress and stall observability for self-repair and external debugging

## Traceability
- Linear issue: `CO-82` / `736c3631-341a-4250-875f-e8808ef2a31b`
- Linear URL: https://linear.app/asabeko/issue/CO-82/co-add-authoritative-provider-worker-progress-and-stall-observability
- Source issue: `CO-80`.

## Problem
- `CO-77` and `CO-78` proved that active provider-worker truth is still split across too many artifacts: provider intake, worker proof, worker audit, `runner.ndjson`, live Linear, live GitHub, and host process state.
- Appserver-backed workers can stay heartbeat-alive while the normal runner/event stream remains semantically quiet.
- Control-host claims say who owns an issue, but they still do not say clearly whether a lane is progressing, waiting on checks, attempting merge, stalled, or merely being refreshed.

## Outcome
- One authoritative per-issue debug snapshot exists on control-host read surfaces for active provider-worker lanes.
- Provider-worker proof carries structured progress and stall telemetry instead of only generic ownership and turn data.
- Appserver semantic progress is mirrored into the ordinary runner/event stream so liveness is not hidden behind proof-only sidecars.
- Control-host claim/read-model payloads expose progress semantics, freshness or rehydrate truth, and merge-closeout or attached-PR context in the same contract.
- Focused regressions prove the `CO-77` / `CO-78` classification can be answered from one surface.

## Protected Expectations
- This issue is an observability and debuggability contract, not another isolated logging tweak.
- One authoritative surface must answer whether a lane is active, waiting, stalled, merge-attempting, or completed.
- The emitted progress data must be structured enough for autonomous repair logic, not only operator forensics.
- The lane must complement `CO-79`, `CO-80`, and `CO-81` rather than re-implement their behavior fixes.

## Non-Goals
- Redesigning the HTTP dashboard UI.
- Rewriting every historical provider artifact format.
- Replacing the existing merge or recovery behavior already scoped in `CO-79`, `CO-80`, or `CO-81`.

## Done When
- [ ] Active provider-worker lanes emit structured progress and stall telemetry with a meaningful phase model, including merge-closeout.
- [ ] Appserver-backed worker progress is mirrored into the normal runner/event stream with semantic events.
- [ ] Control-host claim and read-model surfaces expose latest semantic progress time, progress kind, stall reason, and fresh-vs-rehydrated truth.
- [ ] `co-status --format json` or the equivalent control-host API exposes one authoritative per-issue debug snapshot with live Linear state, attached PR state, worker progress phase, last audit operation, last semantic progress time, stall classification, and recovery recommendation.
- [ ] Regression coverage proves the `CO-77` / `CO-78` class can be classified from one surface without manual artifact correlation.

## Parity Matrix

| Surface | Current | Target |
| --- | --- | --- |
| Provider-worker proof | Ownership-heavy, mostly generic turn snapshots | Structured semantic progress/stall contract with authoritative phase and timestamps |
| Runner/event stream | Heartbeats and generic command churn can hide real progress | Semantic progress updates appear in the normal runner/event stream |
| Control-host claims and read models | Ownership and retry truth are available, but progress semantics are thin | Fresh/rehydrated truth, progress kind, stall reason, and recovery recommendation are explicit |
| Active issue debugging | Requires manual correlation across multiple artifacts and live systems | One authoritative per-issue debug snapshot answers the active waiting or stalled question directly |
