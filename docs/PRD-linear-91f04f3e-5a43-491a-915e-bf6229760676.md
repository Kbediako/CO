## Summary
- Problem Statement: retained `provider-intake-state.json` rows with released/non-active claim state can keep re-entering provider refresh reconciliation as work to recheck or rewrite, causing refresh churn even when the row is only historical audit evidence and no current live worker, retry, promotion, merge-closeout, or reopened issue evidence exists.
- Desired Outcome: retained released provider-intake claims remain auditable, but passive released rows do not repeatedly churn refresh work or operator status. Refresh only revisits or mutates a retained released row when newer same-issue truth or current actionable evidence materially changes the row.

## User Request Translation
- User intent / needs: refresh the CO-549 docs-first packet before parent implementation so the implementation lane fixes retained released provider-intake claim refresh churn without deleting audit history or broadening provider-worker behavior.
- Success criteria / acceptance:
  - Passive retained released rows are idempotent across repeated refresh passes when no current actionable evidence exists.
  - The fix preserves retained provider-intake audit history, including claim reason, run metadata, closeout fields, and source-labeled terminal evidence.
  - Current actionable evidence still re-enters the appropriate refresh path, including active/reopened issue truth, active retry truth, current review promotion, current merge closeout, cancelable retained-run evidence, or bounded stale-proof revalidation.
  - Operator surfaces such as `co-status`, `/ui/data.json`, selected-run projection, and freshness-gauge evidence no longer treat passive retained released churn as live work.
- Constraints / non-goals:
  - This child lane is docs-only and must not edit source or tests.

  - No broad provider scheduler redesign, polling cadence increase, or docs-freshness scope change.
