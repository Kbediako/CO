# PRD - CO: Recover truthful intake and claim reattach after missing control-host manifest and expired shared-budget cooldown

## Added by Bootstrap 2026-04-04

- Related source issues: `CO-78`

## Summary
- Problem Statement: `CO-78` exposed a recovery seam where provider-worker continuation could not request a control-host refresh because the worker assumed `.runs/local-mcp/cli/control-host/manifest.json` existed, even when the live control-host had been launched through a path that left no manifest there. Under the same incident, shared Linear budget cooldown pressure left local intake truth pinned to an expired cooldown window until an operator manually replaced the stale host, and stale non-terminal run artifacts remained a risk during claim rehydrate.
- Desired Outcome: provider-worker refresh requests recover cleanly even when the control-host manifest lives somewhere else or is absent, shared-budget cooldown expiration no longer leaves local intake and claim truth frozen until a restart, rehydrate only revives genuinely live or explicitly resumable runs, and eligible live issues are reclaimed automatically after cooldown or restart without operator intervention.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): finish `CO-81` as a bounded provider/control-host recovery lane that keeps local claim truth and live issue reclamation honest after a host-path mismatch plus shared-budget cooldown pressure, using the repo’s docs-first, workpad, child-stream, and review workflow.
- Success criteria / acceptance:
  - missing local `control-host/manifest.json` no longer aborts provider-worker recovery or refresh requests
  - expired shared-budget cooldown does not keep local intake frozen until a human restarts the host
  - claim rehydrate only marks an issue active when the backing run is demonstrably alive or explicitly resumable/resumed
  - regression coverage exercises the `CO-78` class: missing local host manifest, shared-budget cooldown, host restart or recovery, and truthful post-cooldown reclaim
- Constraints / non-goals:
  - do not try to eliminate Linear rate limits entirely
  - do not replace the provider-worker lifecycle model wholesale
  - do not widen into unrelated `CO STATUS` or control-host redesign

## Not Done If
- a missing `control-host/manifest.json` can still abort provider refresh requests.
- local intake can remain frozen on an expired cooldown until a human restarts the host.
- rehydrated claims can still point at stale or dead runs as if they were active workers.
- post-cooldown or post-restart reclaim still depends on another operator intervention.

## Goals
- Make provider-worker control-host refresh recovery robust against missing or relocated control-host manifests.
- Keep shared-budget cooldown from pinning stale intake truth past its valid window.
- Preserve truthful active/resumable/completed classification during rehydrate and reclaim.
- Add focused automated coverage for the recovery and reclaim seams exposed in `CO-78`.

## Non-Goals
- Removing or bypassing Linear rate limits.
- Replacing the provider-worker lifecycle model.
- Broad `CO STATUS` redesign unrelated to provider recovery truth.
- Treating operator restart SOP as the normal recovery mechanism.
