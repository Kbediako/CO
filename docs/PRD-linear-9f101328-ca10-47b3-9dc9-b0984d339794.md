## Summary
`CO-219` closes the stale accepted-rehydration projection gap behind the observed `Agents: 4/3 max allowed` output. A retained provider-intake row can still be `accepted` with reason `provider_issue_rehydration_pending_revalidation` while the selected/discovered run manifest says `in_progress`, the local provider proof still says `owner_status=in_progress`, and the recorded local PID is already dead. `co-status` and `/ui/data.json` must stop counting that stale row as live running occupancy while still preserving genuinely live rows and true over-cap incidents.

## User Request Translation
- Keep `max_concurrent_agents=3` authoritative and stop stale accepted dead-proof rehydration rows from inflating `running/max_allowed`.
- Treat accepted pending-revalidation rows with dead local proof as non-running for active status projection without deleting retained audit artifacts.
- Preserve true live provider rows, true over-cap detection, and neighboring invariants from `CO-125`, `CO-162`, `CO-192`, and `CO-194`.
- Fix projection truth rather than hiding the symptom by raising the configured cap.

## Acceptance Criteria
- Reproduce the accepted pending-revalidation plus dead local proof plus manifest `in_progress` shape in focused regression coverage.
- `co-status` and `/ui/data.json` no longer count that stale row as running when the local proof is dead or stale.
- The stale PID and stale event are no longer rendered for that row on active running surfaces.
- True live active rows still render and count correctly, and true admission over-cap incidents remain detectable.
- Regression coverage distinguishes this shape from `CO-125`, `CO-162`, `CO-192`, and `CO-194`.

## Non-Goals
- No cap increase.
- No live-worker termination.
- No weakening of `CO-125` admission rules.
- No regression of accepted pending-revalidation behavior when proof and worker liveness are genuinely current.
- No broad CO STATUS renderer redesign.
- No destructive cleanup of retained `.runs` or provider-intake history as the primary fix.

## Not Done If
- An accepted pending-revalidation row with a dead local proof can still push `co-status` above `max_allowed`.
- The stale PID/event still appears on an active running row for that stale claim.
- The fix masks the issue by changing `max_concurrent_agents`.
- A real live over-cap incident is hidden instead of only the stale row being demoted.

## Metrics & Guardrails

## User Experience
- Operators no longer see `Agents: 4/3 max allowed` when only three live provider workers exist.
- The stale `CO-196`-style row stops appearing as a live running lane.
- Real live worker rows, including real over-cap conditions, remain visible and countable.

## Technical Considerations
- The likely owner is the active-running authority in `orchestrator/src/cli/control/controlRuntime.ts`, because that seam decides which compatibility sources are allowed into `projection.running`.
- `/ui/data.json` derives `counts.running` and running cards from the same projection, so the fix should stay in shared classification logic instead of renderer-only code.
- Neighboring stale-row coverage already exists for released terminal/no-live-worker paths; this lane should extend that pattern to accepted pending-revalidation rows with dead local proof.
- The retained issue/debug payload can remain visible if needed, but the active running row must be suppressed for the stale shape.
