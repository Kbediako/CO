## Summary
- Problem Statement: CO source-checkout runs and downstream packaged compatibility runs can still share fallback behavior closely enough that packaged fallback config or packaged fallback pipelines are treated as normal repo execution. That makes operator output, manifests, and CI unable to prove whether a run used authoritative repo-local CO policy or a downstream compatibility path.
- Desired Outcome: define a first-class repo-authoritative execution mode that requires repo-local config and canonical CO pipelines by default, while keeping packaged fallback config/pipelines available only through an explicit downstream compatibility mode or adapter.

## User Request Translation
- User intent / needs: create the docs-first packet for CO-385, so implementation can separate authoritative CO repo execution from downstream compatibility fallback without removing compatibility or overlapping nearby fallback-policy issues.
- Success criteria / acceptance:
  - define explicit repo-authoritative execution mode requiring repo-local config and canonical CO pipelines by default
  - move packaged fallback config/pipelines behind explicit downstream compatibility mode or adapter
  - operator output names active mode and why
  - CI exercises both modes separately
  - compatibility remains available only through explicit compatibility path, not silent repo default
- Constraints / non-goals:
  - do not remove downstream compatibility
  - do not change public packaged defaults without a compatibility migration path
  - do not duplicate CO-380 review-gate, CO-381 runtime fallback-policy, or CO-382 fallback-expiry policy
  - this child lane creates docs only and does not edit source or tests

## Not Done If
- Repo-local runs can silently use packaged fallback config as normal path.
- Compatibility fallback remains indistinguishable from authoritative repo execution in manifests/status/operator output.
- CI does not separately exercise repo-authoritative and downstream compatibility modes.
- The implementation removes downstream compatibility instead of gating it behind an explicit path.
- The implementation reopens CO-380, CO-381, or CO-382 policy scope instead of preserving their separate ownership.

## Goals
- Define the repo-authoritative mode contract for CO source-checkout and canonical CO pipeline execution.
- Define the downstream compatibility mode or adapter contract for packaged fallback config/pipelines.
- Require operator-visible and machine-readable output to name active mode and selection reason.
- Require focused CI coverage for both modes.

## Technical Considerations
- Architectural Notes:
  - Existing fallback language already calls packaged config a compatibility path; CO-385 turns that wording into an enforced mode split rather than a warning-only convention.
  - Implementation should choose one canonical authority-mode field and reuse it across resolver output, manifest/status output, and operator text.
  - Repo-authoritative mode should fail closed when required repo-local config or canonical CO pipelines are absent.
  - Downstream compatibility mode should be opt-in and should make packaged config/pipeline source visible.
  - 2026-04-26 implementation choice: use `--config-mode <repo-authoritative|downstream-compatibility>` and `CODEX_ORCHESTRATOR_CONFIG_MODE`; keep `--repo-config-required` / `CODEX_ORCHESTRATOR_REPO_CONFIG_REQUIRED` as legacy aliases.
  - 2026-04-26 implementation choice: expose machine-readable mode evidence as `config_resolution` in plan output and run manifests.
- Candidate implementation surfaces:
  - config and pipeline resolution around `codex.orchestrator.json`
  - manifest/status/operator output presenters
  - targeted CI and fixture tests for authoritative and compatibility modes
