## Summary
- Objective: land one release-prep hardening pass across public repo/package exposure, packaged downstream provider onboarding, public-vs-internal docs truthfulness, and the verified `codex-cli 0.118.0` posture.
- Scope:
  - docs-first packet for `CO-77`
  - removal of tracked public runtime artifacts plus deterministic guards
  - packaged public docs contract and README/package front-door cleanup
  - provider onboarding docs/scaffolding/readiness parity for Linear and Telegram
  - `0.118.0` posture refresh across active public docs, templates, and shipped skills
  - full validation, standalone review, elegance pass, and PR/review handoff
- Constraints:
  - preserve contributor-only docs as internal instead of pretending they are downstream docs
  - keep provider onboarding aligned to live env and policy contracts in code
  - prefer extending current docs/package/doctor surfaces instead of inventing independent one-off release checks

## Technical Requirements
- Functional requirements:
  - remove tracked `.runs/**` and `out/**` runtime artifacts from the public branch except explicit allowlisted public files
  - add a blocking guard that fails when tracked runtime artifacts reappear on the branch
  - add a blocking public-doc hygiene guard for absolute machine-local paths in tracked public docs, packaged docs, shipped skills, and seeded templates unless allowlisted
  - define a curated packaged public-doc surface distinct from contributor docs and make README link only to packaged or stable public-safe references for core setup
  - fix `package.json.description` and any misleading public metadata/help references
  - productize provider onboarding with one shipped contract covering:
    - once-per-machine steps
    - once-per-repo steps
    - Linear advisory binding and `CO_LINEAR_WEBHOOK_SECRET`
    - Telegram polling, chat allowlist, mutation and push flags
    - `dispatch_pilot` and transport policy expectations
    - smoke flow and expected readiness outcomes
  - add or extend scaffolding so fresh repos can generate provider examples/defaults from shipped assets instead of freehand copying internal docs
  - expose machine-checkable provider readiness or smoke output with actionable failure reasons through shipped CLI surfaces
  - update active posture/docs/templates/skills from `0.117.0` to `0.118.0` where appropriate
  - re-audit and update stale guidance around `codex exec` prompt-plus-stdin behavior and `codex login --device-auth`
  - leave durable automation for:
    - tarball doc/link validation
    - shipped-skill reference checking
    - stale-version and stale-example linting
    - provider docs/help consistency checks
    - top-level public-doc sprawl control
- Non-functional requirements:
  - deterministic local checks only for release guards
  - package-surface validation must use real `npm pack` output rather than source-tree assumptions alone
  - documentation rules must remain auditable and allowlist-driven, not heuristic-only
- Interfaces / contracts:
  - `scripts/docs-hygiene.ts`
  - `scripts/docs-freshness.mjs`
  - `scripts/pack-smoke.mjs`
  - `scripts/lib/npm-pack.js`
  - `README.md`
  - curated packaged docs under `docs/public/**` or equivalent shipped namespace
  - `docs/README.md` as contributor guide only
  - `package.json`
  - `skills/**`
  - `templates/codex/**`
  - `orchestrator/src/cli/setupBootstrapShell.ts`
  - `orchestrator/src/cli/init.ts`
  - `orchestrator/src/cli/doctorCliShell.ts`

## Architecture & Data
- Architecture / design adjustments:
  - extend the existing docs catalog/truthfulness gate from `CO-75` to distinguish packaged public docs from internal repo guides and to block public local-path leakage
  - add a public-surface guard that inspects tracked git paths for runtime artifacts and only allows documented exceptions
  - extend pack/tarball validation so README and shipped docs can only reference files that exist in the packed artifact or safe public URLs
  - introduce one packaged provider guide plus provider example/default templates reused by `init codex` or related shipped scaffolding
  - extend `doctor` or another shipped CLI help/readiness surface with provider-specific readiness output rather than burying provider setup in internal docs
- Data model changes / migrations:
  - new packaged public docs and/or provider example templates
  - docs catalog updates for packaged public-doc classes and public-path allowlists
  - possible new doctor JSON section(s) for provider readiness parity
  - no migration of historical internal packets beyond removing public-branch runtime artifacts
- External dependencies / integrations:
  - git tracked-path inspection
  - `npm pack` tarball inspection
  - existing docs gates and pack smoke
