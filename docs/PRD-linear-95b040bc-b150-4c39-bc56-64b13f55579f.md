# PRD - CO: Reduce steady-state Linear request burn across control-host and provider-worker flows

## Added by Bootstrap 2026-04-01

## Summary
- Problem Statement: CO still spends Linear requests faster than the shared hourly bucket refills during ordinary control-host plus provider-worker operation. The main drivers are the fixed `15s` control-host tracked-issue poll, the narrow `10s` direct-mutation trust window for `issue-context` reuse, provider-worker tracked-issue rereads after every turn, and multi-step helpers such as `create-follow-up` that continue issuing sequential Linear GraphQL calls even when the latest known bucket is already exhausted. The current repo surfaces explicit `linear_rate_limited` errors in a few narrow seams from `CO-33` and `CO-34`, but it does not yet persist a shared reset window or apply shared budget awareness across the major callers that reuse the same token.
- Desired Outcome: reduce avoidable steady-state Linear request burn without weakening issue truthfulness by introducing shared token-scoped budget awareness, adaptive control-host polling, truthful fail-fast behavior for helper flows when the latest known budget is already exhausted or clearly insufficient, and reviewer-visible proof of the active reset or suppression state.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): keep the existing `CO-33` and `CO-34` fixes intact, but solve the broader budget problem they left behind. The right owner is not better wording alone and not a generic provider rewrite. The lane should reduce normal Linear pressure across control-host polling, provider-worker rereads, and helper mutation flows, while preserving truthful issue state and explicit visibility into reset-window behavior.
- Success criteria / acceptance:
  - control-host polling no longer burns the shared Linear token on a rigid `15s` cadence regardless of budget pressure
  - provider-worker and helper flows respect a shared reset or cooldown window after explicit `linear_rate_limited` responses instead of re-hitting the exhausted bucket from other processes
  - multi-step helper flows stop early when the latest known request budget is already exhausted or clearly insufficient for the planned sequence
  - worker proof plus control-host read surfaces expose the relevant Linear rate-limit or reset metadata and any active suppression state
  - validation shows lower avoidable request pressure together with truthful behavior when the upstream bucket is genuinely exhausted
- Constraints / non-goals:
  - do not replace Linear as the provider or change auth ownership
  - do not introduce a broad stale cache that weakens issue truth guarantees
  - do not reopen unrelated workpad formatting or terminal UI issues
  - do not hide real exhaustion behind silent retries or best-effort loops

## Parity / Alignment Matrix
- Not applicable; this is a bounded request-budget and observability lane, not a parity migration.
- Current truth:
  - control-host tracked-issue refresh is scheduled on a fixed `15s` loop in `controlServerPublicLifecycle.ts`
  - helper-side direct mutation reuse trusts cached `issue-context` data for only `10s` in `providerLinearWorkflowFacade.ts`
  - provider-worker rereads the tracked issue live after each turn and only retries once when a short reset window is available
  - explicit `linear_rate_limited` metadata is surfaced in narrow seams, but the reset state is not persisted or shared across callers that reuse the same token
- Reference truth:
  - ordinary CO operation should not burn Linear budget unnecessarily when no meaningful issue truth changed
  - when the shared bucket is exhausted, later callers should fail or defer based on the known reset window instead of rediscovering the same exhaustion through more live requests
- Target truth / intended delta:
  - a shared Linear budget state records relevant headers from success and rate-limited GraphQL responses and exposes active cooldown or suppression decisions
  - control-host polling adapts to that state and schedules a truthful next poll instead of blindly rearming every `15s`
  - helper mutation flows and provider-worker rereads consult the shared state before spending additional avoidable requests
- Explicitly out-of-scope differences:
  - provider replacement or auth redesign
  - generic stale caching of full issue payloads beyond the already bounded mutation-context reuse
  - unrelated dashboard redesign or workpad formatting changes

## Not Done If
- CO can still spend requests faster than the Linear bucket refills during ordinary control-host plus provider-worker operation.
- Control-host polling remains fixed and budget-blind when request pressure is already high.
- Helper flows still spend multiple live Linear requests when the latest known budget is already exhausted or clearly insufficient.
- The only new behavior is post-exhaustion cooldown handling with no steady-state request reduction.

## Goals
- Persist token-scoped Linear rate-limit observations from the major callers that share the same API token.
- Use that shared state to reduce avoidable steady-state request volume, especially in the control-host poll loop and helper mutation entrypoints.
- Preserve explicit failure truth when the upstream bucket is genuinely exhausted.
- Surface the shared budget or suppression state in machine-checkable worker and control-host read models.

## Non-Goals
- Redesigning the broader Linear provider architecture.
- Hiding exhausted buckets behind silent retries or generic fallback mutations.
- Reopening unrelated UI, workpad, or terminal-output polish.
