# PRD - CO: Distinguish Missing And Not-Found Codex Cloud Environments

## Summary
- Problem Statement: `codex-orchestrator doctor --cloud-preflight` can prove that `CODEX_CLOUD_ENV_ID` is present, but that is weaker than proving the configured Codex Cloud environment is usable. A required cloud canary can still fail when `codex cloud exec` reports `environment ... not found`, and CO needs that to be classified separately from the absent-env `missing_environment` fallback path.
- Desired Outcome: CO-358 gives parent implementation a concise contract for separating missing cloud environment configuration from configured-but-inaccessible or not-found environment ids, while preserving required cloud canary and promotion gates.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): make cloud preflight and cloud-canary evidence precise enough that operators can tell whether cloud evidence is blocked because no `CODEX_CLOUD_ENV_ID` is configured, or because the configured id cannot be seen by `codex cloud list` and would fail `codex cloud exec`.
- Success criteria / acceptance:
  - `codex-orchestrator doctor --cloud-preflight` reports `missing_environment` only when no environment id is configured.
  - A configured environment id that produces `environment ... not found` through the cloud execution path is classified as a distinct required-cloud blocker, not as `missing_environment`.
  - Required cloud canary output preserves the distinction between absent env id fallback and inaccessible/not-found env id failure.
  - Promotion gates remain blocked unless required cloud canary evidence and the required fallback contract pass, or an owner-approved waiver records the exact blocker.
- Constraints / non-goals: do not relax required cloud evidence, bypass cloud canary failures, rename cloud modes, change model defaults, or treat local fallback as proof of cloud execution.

## Parity / Alignment Matrix
- Current truth: cloud preflight checks env-id presence, Codex CLI availability, and remote branch existence. A present but invalid or inaccessible env id can still reach `codex cloud exec` and fail with `environment ... not found`, which blocks required cloud canary evidence but is easy to conflate with missing config.
- Reference truth: CO version promotion gates require current required cloud canary evidence and required fallback evidence. Missing env fallback uses `missing_environment`; not-found env id is a configured-cloud failure.
- Target truth / intended delta: missing configuration remains `missing_environment`; configured-but-not-found or inaccessible ids are reported as a separate cloud-preflight/canary blocker with guidance to repair `CODEX_CLOUD_ENV_ID` or cloud account/environment access.
- Explicitly out-of-scope differences: no cloud-mode rename, no canary bypass, no model default changes, no provider-worker supervision redesign, and no broad Codex version promotion.

## Not Done If
- `environment ... not found` is still reported as `missing_environment`.
- Required cloud canary can pass or be skipped solely because fallback canary behavior works.
- A present `CODEX_CLOUD_ENV_ID` lets `codex-orchestrator doctor --cloud-preflight` look fully ready while the same id is known to be rejected by `codex cloud list` / `codex cloud exec`.
- Promotion gates can be cleared without a clean required cloud canary artifact or explicit owner-approved blocker/waiver.
