# PRD - CO-535 Docs Freshness Repo Gate Visibility

## Summary
- Problem Statement: `co-status` can report `repo_gates.docs_freshness_maintain=null` even when a fresh local or task-scoped docs freshness maintenance report contains a real `docs_freshness_maintain` repo gate.
- Desired Outcome: The operator dashboard and JSON status surface fresh maintenance evidence from valid local/task-scoped reports, distinguish missing, stale, and invalid report evidence, and never hide a blocking docs freshness gate behind a missing scheduled-report path.

## User Request Translation
- User intent / needs: The operator wants docs freshness issues fixed at the root cause rather than patched over with date bumps, cap changes, or manual copying. The status monitor must show the live maintenance truth that operators should act on.
- Success criteria / acceptance: `co-status --format json` finds fresh `docs-freshness-maintenance.json` evidence from valid local/task-scoped output paths, preserves the normalized repo-gate fields, and emits explicit degraded evidence when no usable report exists.
- Constraints / non-goals: Do not weaken `docs:freshness` or `docs:freshness:maintain`, do not change CO-522 ownership, do not absorb CO-530/CO-536 historical cohort cleanup, and do not make scheduled workflow output the only accepted status source.

## Intent Checksum
- Protected terms: `co-status`, `repo_gates.docs_freshness_maintain`, `docs_freshness_maintain`, `out/local/docs-freshness-maintenance.json`, `out/docs-truthfulness-maintenance/docs-freshness-maintenance.json`, `source_path`, `generated_at`, `CO-522`, `docs-freshness:co-status-repo-gate-visibility:v1`.
- Nearby wrong interpretations to reject: Do not create another docs freshness owner issue. Do not fix this by copying local reports into the scheduled path. Do not treat a missing report as clean. Do not widen freshness policy or rolling caps.

## Not Done If
- `co-status --format json` can still show `repo_gates.docs_freshness_maintain=null` while a fresh valid local/task-scoped maintenance report exists.
- A stale scheduled report can mask fresher local/task evidence.
- Missing, stale, or invalid report states collapse to silent `null` with no evidence.
- The fix weakens strict docs freshness gates, widens caps/windows, or changes CO-522 maintenance ownership.

## Goals
- Make report-source discovery deterministic across explicit, task-scoped, local, and scheduled maintenance report paths.
- Preserve normalized repo-gate fields including severity, owner, spec guard, capacity, action count, blocking flags, `source_path`, and `generated_at`.
- Surface missing/stale/invalid report evidence explicitly so operators can tell whether the dashboard is clean, degraded, or actionable.
- Add focused regression coverage for scheduled/local/task precedence and stale masking.

## Non-Goals
- No docs freshness cap/window change.
- No historical packet reclassification in this lane.
- No CO-522 owner change.
- No duplicate Linear issue for docs freshness prevention.

## Fallback / Refactor Decision
- Applies to fallback, compatibility, legacy, stale, cached, break-glass, or minor-seam behavior? Yes.
- Decision: remove the silent scheduled-path-only fallback. Missing scheduled reports must not cause the repo-gate surface to disappear when fresher valid reports exist.
- Large-refactor check: A bounded consolidation inside the existing repo-gate reader is acceptable because the authority split is one function plus its projection callers; a broader status-source abstraction is deferred unless additional repo gates need the same discovery semantics.
