# PRD - CO: add recurring autonomous repo-wide stewardship audit and freshness upkeep beyond markdown docs surfaces

## Added by Bootstrap 2026-04-09

## Traceability
- Linear issue: `CO-124` / `d43b6785-88d6-442b-a34e-2ad19d4f723a`
- Linear URL: https://linear.app/asabeko/issue/CO-124/co-add-recurring-autonomous-repo-wide-stewardship-audit-and-freshness

## Summary
- Problem Statement: CO currently has truthful README and docs checks plus a docs freshness registry, but the active upkeep contract is still markdown-centric. `scripts/lib/docs-helpers.js` inventories markdown roots plus `README.md` / `AGENTS.md`, `scripts/docs-freshness.mjs` only reasons about that inventory, and the weekly automation only publishes docs drift. Tracked repo surfaces under `orchestrator/`, `packages/`, `scripts/`, `.github/workflows/`, configs, tests, fixtures, `reference/**`, `archives/**`, `patterns/**`, and `prompt-snippets/**` are not part of a recurring whole-repo stewardship decision system.
- Desired Outcome: add a recurring repo-wide stewardship audit over tracked files, classify each tracked surface into one machine-checkable decision (`validate`, `update`, `delete`, or `retain_with_rationale`), keep root/front-door docs under the existing truthfulness checks instead of file age, publish reviewable JSON and markdown audit artifacts, and split any meaningful historical-residue cleanup into a bounded follow-up instead of silently broadening this lane.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): extend CO from docs-only freshness upkeep into a deterministic repo-wide stewardship system that inventories every tracked file class, keeps README/public onboarding truthfulness tied to the current packaged CLI and runtime posture, and turns detected upkeep work into reviewable diffs or follow-up issues rather than silent churn.
- Success criteria / acceptance:
  - a repo-wide tracked-file audit path exists
  - each tracked surface receives a machine-checkable stewardship decision
  - root/front-door surfaces are included without relying on file age alone
  - code, docs, configs, tasks, skills, tests, fixtures, references, archives, and assets are explicitly covered
  - recurring automation runs the upkeep lane and publishes reviewable artifacts
  - the new lane composes with `CO-88`, `CO-75`, and `CO-102` instead of reopening them
  - the audit proves it can flag at least one non-doc-root surface that needs action
- Constraints / non-goals:
  - do not collapse this lane into the one-time cleanup owned by `CO-88`
  - do not reopen `CO-75` or `CO-102` as umbrella docs owners
  - do not silently auto-edit the entire repo
  - do not broaden into unrelated provider telemetry, runtime continuity, or CO STATUS work

## Parity / Alignment Matrix
- Current truth:
  - `collectDocFiles(...)` only scans markdown doc roots plus root `README.md` / `AGENTS.md`
  - `docs:freshness` only reasons about that markdown/doc inventory
  - the weekly automation only publishes docs drift artifacts
  - non-doc tracked surfaces have no shared recurring stewardship decision contract
- Reference truth:
  - CO should have a recurring tracked-file stewardship lane that covers all major repo surface classes and produces reviewable upkeep output
  - front-door docs should continue using the existing truthfulness checks for CLI/runtime/skill posture rather than raw file age
- Target truth / intended delta:
  - tracked-file inventory comes from `git ls-files`
  - a stewardship catalog classifies every tracked surface into an explicit decision
  - weekly automation publishes repo-stewardship output alongside docs checks
  - historical residue clusters outside this issue land as linked follow-up work instead of silent broad edits
- Explicitly out-of-scope differences:
  - one-time bulk cleanup of all flagged historical residue
  - changes to standalone-review or provider-worker merge policy
  - unrelated runtime or UI behavior work

## Not Done If
- the new audit still only covers markdown/doc roots
- tracked non-doc surfaces have no machine-checkable stewardship decision
- recurring automation still only runs markdown-centric upkeep
- the lane claims repo-wide stewardship without detecting any non-doc-root action candidate

## Goals
- Add a tracked-file stewardship catalog and audit command over `git ls-files`.
- Reuse the existing docs truthfulness system for README/public onboarding surfaces instead of inventing a second front-door policy.
- Publish JSON and markdown stewardship artifacts that future agents can act on deterministically.
- Introduce a recurring automation lane for repo stewardship.
- Split any real out-of-scope historical-residue cluster into a same-project follow-up with explicit rationale.

## Non-Goals
- Performing all repo cleanup found by the audit in this lane.
- Weakening or replacing `docs:check` / `docs:freshness`.
- Broad task/doc mirror reduction work already owned elsewhere.
