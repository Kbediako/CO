## Summary
- Problem Statement: `CO-35` shipped parent-owned same-issue child lanes plus machine-checked file-scope acceptance, but phase scope is still only descriptive. The current parent launch path rejects pure phase-only lanes up front, and parent acceptance also fails phase-only records closed because there is no explicit repo contract that turns declared phases into bounded file or codepath ownership.
- Desired Outcome: phase-scoped child lanes remain bounded helpers, and the parent can machine-verify their patches with the same fail-closed bar as file-scoped lanes before `git apply`.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): finish the missing trust-boundary slice from `CO-35` instead of widening the runtime. Keep the existing parent-owned child-lane model, but add a deterministic phase-to-path contract so declared phases are no longer advisory-only and the parent can accept or reject phase-scoped patches without ad hoc judgment.
- Success criteria / acceptance:
  - declared child-lane phases map through one documented machine-checkable contract to bounded repo path selectors
  - launch, proof, ledger, and parent acceptance all carry the same resolved contract data
  - the parent accepts valid phase-scoped patches, rejects out-of-phase patches, and rejects stale or tampered scope data before `git apply`
  - existing file-scoped acceptance remains intact
- Constraints / non-goals:
  - keep `provider-linear-worker` as the only parent authority for issue lifecycle mutations
  - do not weaken child-lane trust boundaries by treating phases as reviewer-only hints
  - do not widen into generic scheduler ownership, queueing, or broad code-ownership policy work

## Parity / Alignment Matrix
- Current truth:
  - child-lane scope stores raw `files` and `phases`, but there is no persisted machine-checkable mapping from phases to allowed repo paths
  - phase-only launches fail closed in `providerLinearChildLaneShell`
  - parent acceptance rejects phase-only records because it can only compare patch paths against explicit file scope
- Reference truth:
  - file-scoped child lanes already have machine-checkable acceptance because the parent inspects patch paths before `git apply`
- Target truth / intended delta:
  - a declared phase resolves to deterministic repo path selectors, those selectors are persisted in proof and ledger data, and the parent recomputes and verifies them before accepting a phase-scoped patch
- Explicitly out-of-scope differences:
  - generic code-owner routing
  - queueing or scheduler semantics for overlapping lanes
  - remote app-server authority changes

## Not Done If
- Phase-only lanes still fail closed only because there is no contract data to verify.
- The parent trusts persisted phase selector data without recomputing it from the declared phase contract.
- Out-of-phase, stale, or tampered phase-scoped lane data can still reach `git apply`.

## Goals
- Define a repo-local phase contract that maps supported child-lane phases to bounded repo path selectors.
- Persist that contract through child-lane proof and parent ledger data.
- Extend launch and acceptance so phase-only lanes can run and be accepted when the resolved scope is valid.
- Add focused regressions for valid acceptance, out-of-phase rejection, and stale or tampered contract data.

## Non-Goals
- Replacing explicit file-scope ownership with a generic phase-only model everywhere.
- Broadening child-lane permissions beyond parent-reviewed patch artifacts.
- Introducing filesystem-glob heuristics that depend on the current checkout state instead of a stable contract.
- Refactoring unrelated provider-worker or observability behavior.

## Metrics & Guardrails
- Primary Success Metrics:
  - phase-only child-lane launch and acceptance succeed when the patch stays inside the declared phase contract

- Guardrails / Error Budgets:

  - fail closed on unknown phase names or missing contract metadata
