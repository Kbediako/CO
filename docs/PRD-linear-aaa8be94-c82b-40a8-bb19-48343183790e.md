# PRD - CO: Refactor docs-hygiene spark policy guard into parser-backed fixture harness

- Linear issue: `CO-191` / `aaa8be94-c82b-40a8-bb19-48343183790e`
- Source issue: `CO-183` / `df69fabe-63c2-4b98-a226-9c37892b4f9d`
- Problem Statement: CO-183 preserved the immediate `explorer_fast` / `gpt-5.3-codex-spark` file/codebase-search-only policy with targeted regressions, but the docs-hygiene guard is now difficult to audit because policy wording is encoded in clause/window heuristics and inline test phrases.
- Desired Outcome: preserve CO-183's accepted production behavior while moving spark-policy classification into a focused parser-backed classifier and a structured fixture harness with stable ids, expected verdicts, and failure reasons.

## User Request Translation (Context Anchor)
- User intent / needs: replace the regex-heavy spark-policy model-posture check with a small deterministic classifier and fixture-driven evaluator so future wording changes are reviewed by editing structured cases instead of growing ad hoc heuristics.
- Success criteria / acceptance:
  - existing CO-183 behavior is preserved for the current docs and existing tests
  - classifier concepts are explicit: marker, clause, assertion window, scope qualifier, forbidden usage, restrictive non-use, and neutral reference
  - spark-policy phrase coverage lives in structured fixtures with stable ids and expected reasons
  - `npm run docs:check` still uses the same production rule as the fixture harness
  - focused docs-hygiene tests, docs:check, and scoped guardrails pass
- Constraints / non-goals:
  - do not change the spark policy itself
  - do not weaken file/codebase-search-only restrictions for `explorer_fast`
  - do not reopen Codex CLI 0.120 adoption posture
  - do not touch docs:freshness or CO-184/CO-188 freshness maintenance
  - do not introduce a broad natural-language parser for all docs hygiene rules

## Parity / Alignment Matrix
- Current truth:
  - docs-hygiene model posture checks allow the current primary `gpt-5.4` posture, the `explorer_fast` `gpt-5.3-codex-spark` exception, and known unsupported-review warning forms
  - the spark exception rule depends on line-level detection and inline regression phrases
  - adding new allowed, forbidden, or neutral cases requires editing test bodies and guard heuristics together
- Reference truth:
  - CO-183 final behavior defines the accepted file/codebase-search-only spark exception policy
  - `docs/guides/codex-version-policy.md` remains the current posture source
  - existing `tests/docs-hygiene.spec.ts` cases encode the current regression floor
- Target truth / intended delta:
  - the production model-posture rule delegates spark-policy classification to a focused parser-backed classifier
  - fixtures encode stable ids, categories, expected verdicts, and expected reasons
  - table-driven tests report fixture id, expected verdict, actual verdict, and reason on mismatch
  - `docs:check` and tests share the same classifier path
- Explicitly out-of-scope differences:
  - no Codex CLI adoption posture changes
  - no docs:freshness maintenance work
  - no weakening of `explorer_fast` restrictions
  - no semantic parser beyond this bounded spark-policy classifier

## Not Done If
- The spark policy classifier remains an expanding regex/window heuristic with the main phrase corpus embedded in a single test body.
- Future allowed/forbidden/neutral cases cannot be added as structured fixtures with stable ids and failure reasons.
- The refactor changes CO-183's final accepted production behavior without explicit fixture evidence and migration notes.
- The `docs:check` production rule diverges from the fixture/evaluator path.
- The issue broadens into unrelated docs:freshness, Codex 0.120 adoption, or general docs-language parsing.
