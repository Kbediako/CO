## Not Done If
- A provider worker can still call `child-lane --phases classification,analysis` after the prompt/helper already knows only `docs,implementation,tests` are supported.
- A workpad/temp parent change can still make a read-only child-lane launch fail without actionable preflight guidance or sequencing.
- A deterministic `linear_follow_up_parity_matrix_missing` failure can be retried in the same attempt without adding a parity matrix or changing to a non-parity issue classification.
- A zero-byte read-only child-lane patch is treated as successful evidence without a useful summary/artifact or explicit parent-owned fallback path.
- The fix is only a manual runbook note and lacks prompt/helper and regression/eval coverage.

## Goals
- Preflight supported child-lane phases and classification/analysis alternatives before workers issue unsupported helper calls.
- Make parent-dirty and scope-conflict launch feedback actionable before retries.
- Suppress same-attempt duplicate follow-up retries after deterministic parity-matrix failures unless inputs change correctly.
- Classify zero-byte read-only child-lane output truthfully and require useful parent evidence.
- Add focused regression/eval coverage based on the `CO-184` trace.

## Non-Goals
- Do not weaken phase-scoped child-lane validation.
- Do not silently add unsupported child-lane phases unless the phase-to-path contract, docs, and tests are updated together.
- Do not remove the parity matrix requirement for parity/alignment follow-up issues.
- Do not broaden `CO-184` or `CO-183` implementation scope.
- Do not redesign the scheduler, active-worker accounting, or Linear dispatch lifecycle beyond the prompt/helper retry friction needed here.
- Do not mask zero-byte child-lane outputs as useful proof.
- Do not mutate Linear, update registries, or implement code in this docs-only child lane.
