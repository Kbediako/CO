# PRD - CO: enforce provider max-concurrency across webhook, retry, and resume admission paths
## Added by Bootstrap 2026-04-10

## Traceability
- Linear issue: `CO-125` / `e122d053-40f2-4246-9648-81c9001715f1`
- Linear URL: https://linear.app/asabeko/issue/CO-125/co-enforce-provider-max-concurrency-across-webhook-retry-and-resume

## Summary
- Problem Statement: CO currently enforces provider max-concurrency only inside the poll/refresh dispatch loop. Direct webhook acceptance and queued retry dispatch can still reach the shared start/resume launch helpers without the same global admission check, so non-poll paths can over-admit provider-worker runs even when the host is already at or above the configured cap. Current live host evidence also shows that the active claims involved in this incident still point at real `in_progress` manifests, so this is a true admission-control gap rather than a stale historical-manifest inflation bug.
- Desired Outcome: every provider-worker admission path that can start or resume work consults one shared provider capacity gate seeded from the same authoritative active-claim and active-run truth already used to determine what is actually running. If the host is already over cap after restart or rehydration, CO should preserve existing live work, refuse only new admissions, and recover naturally once active count drops.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): fix the real provider admission bug without drifting into generic `CO STATUS` cleanup. The shared launch boundary needs to enforce max concurrency for poll, webhook/direct accepted issues, queued retry starts, and queued retry resumes, while keeping the live-running-count truth anchored to actual active claims and manifests.
- Success criteria / acceptance:
  - poll dispatch, webhook/direct acceptance, queued retry start, and queued retry resume all pass through one shared provider capacity gate before launch
  - the shared gate uses authoritative active-claim and active-run truth rather than only the current poll budget loop
  - focused regressions prove non-poll paths cannot over-admit beyond cap
  - already-overcommitted hosts are not forced back under cap by canceling or killing live runs; only new admissions are blocked
  - regression coverage keeps this fix distinct from historical-manifest inflation work by proving the active-claim seam still maps to real `in_progress` manifests
- Constraints / non-goals:
  - do not broaden this into generic `CO STATUS` freshness, runtime ticking, or event-text work
  - do not reopen `CO-67` or `CO-107`, which already own denominator/current-activity truth
  - do not kill or cancel active provider workers just to restore cap compliance after the fact
  - do not reopen unrelated queue-shepherding, review-promotion, or control-host supervision lanes

## Parity / Alignment Matrix
- Current truth:
  - `createProviderPollDispatchBudget(...)` is instantiated only inside `runRefreshCycle(...)`
  - `handleAcceptedTrackedIssue(...)` delegates directly to `processTrackedIssueCandidate(...)`
  - `processTrackedIssueCandidate(...)` can launch work via `launchStartForTrackedIssue(...)` without a shared global capacity gate in front of it
  - `dispatchQueuedProviderRetry(...)` can call `launchResumeForRun(...)` and `launchStartForTrackedIssue(...)` outside the poll-budget path
  - `discoverProviderIssueRuns(...)` plus persisted claims already provide the closest authoritative view of active provider work
- Reference truth:
  - all provider-worker admission paths should use one shared concurrency contract
  - authoritative active-claim and active-run truth should remain the source of occupancy
  - hosts that restart over cap should retain live work and recover by attrition rather than forceful cancellation
- Target truth / intended delta:
  - start and resume launch helpers enforce one shared provider-admission gate regardless of whether the source is poll, webhook, or queued retry
  - the poll loop reuses the same occupancy truth instead of carrying a separate global-only gate
  - focused regressions cover webhook/direct start, queued retry start, queued retry resume, and already-over-cap restart behavior
- Explicitly out-of-scope differences:
  - stale historical-manifest denominator cleanup
  - status dashboard/runtime/event presentation changes
  - unrelated provider supervision or merge/review workflow changes

## Not Done If
- Webhook/direct accepted issues can still start provider workers beyond the configured cap.
- Queued retry start or queued retry resume can still bypass the same shared cap.
- The fix changes active-running counts by weakening authoritative claim/run truth instead of enforcing admission.
- The host kills or cancels already-running provider workers when it boots above cap.

## Goals
- Move provider concurrency enforcement to the shared start/resume admission boundary.
- Keep occupancy seeded from authoritative active claims and discovered active runs.
- Preserve live work on already-over-cap hosts while blocking only new admissions.
- Add focused regression coverage for the exact non-poll admission seams involved in the issue.

## Non-Goals
- Generic `CO STATUS` denominator or freshness cleanup.
- Runtime ticking, event-text, or dashboard presentation work.
- Forced cancellation of already-running provider workers.
- Reopening unrelated queue or supervision lanes.

  - the gate should refuse new start/resume attempts when occupancy is already at or above cap, but it must not mutate existing running claims into canceled or released states
