## Summary
`CO-292` closes a metadata freshness gap for retained `provider-intake-state.json` rows that remain `state=released` with `reason=provider_issue_released:not_active`. When live Linear truth for the same issue is newer, the retained row must refresh `issue_state`, `issue_state_type`, and `issue_updated_at` even if the issue remains non-active, such as `Blocked -> Rework`.

## User Request Translation
- Refresh retained released/not_active claim metadata, not claim activity or admission state.
- A retained `provider-intake-state.json` row with `state=released` and `reason=provider_issue_released:not_active` must update `issue_state`, `issue_state_type`, and `issue_updated_at` from newer live same-issue truth.
- The refresh must handle live non-active transitions such as `Blocked -> Rework`; remaining non-active is not a reason to keep stale cached metadata.
- Dependent blocker snapshots and retained claim metadata should agree when they describe the same issue at a newer `issue_updated_at`.
- Keep the implementation in `providerIssueHandoff.ts` or its existing helper path narrow enough to preserve adjacent CO issue contracts.

## Acceptance Criteria
- A retained `provider-intake-state.json` row with `state=released` and `reason=provider_issue_released:not_active` updates `issue_state`, `issue_state_type`, and `issue_updated_at` when newer live same-issue Linear truth is available.
- The row remains `released/not_active`; it is not promoted to an active claim or Ready admission path.
- `Blocked -> Rework` and other non-active state changes are treated as metadata refresh candidates.
- Dependent blocker snapshots and retained claim metadata no longer disagree on the same issue when the snapshot carries newer live truth.
- The solution does not delete retained rows or convert the fix into provider-intake cleanup.
- The solution does not add unbounded direct issue-by-id reads over retained released residue.
- Adjacent behaviors for `CO-202`, `CO-212`, `CO-248`, `CO-41`, `CO-64`, and `CO-276` remain separable.

## Non-Goals
- No active-claim metadata refresh.
- No Ready reclaim/admission change.
- No refresh-stuck restart recovery or lifecycle watchdog change.
- No destructive cleanup, pruning, or rewriting of retained released rows beyond the three metadata fields.
- No broad Linear polling cadence increase or retained-residue issue-by-id sweep.
- No implementation or test edits by this docs child lane.

## Not Done If
- A retained `provider_issue_released:not_active` row can keep stale `issue_state`, `issue_state_type`, or `issue_updated_at` after newer live same-issue truth is known.
- A `Blocked -> Rework` transition remains visible only in dependent blocker snapshots while `provider-intake-state.json` still reports the older Blocked metadata.
- The fix reopens the issue for Ready admission, active-claim handling, or restart recovery instead of metadata freshness.
- The fix relies on deleting retained rows or sweeping every retained released issue with direct issue-by-id reads.

## Metrics & Guardrails
- Primary success metric: focused parent-owned regression proves a retained released/not_active row refreshes the three metadata fields from newer same-issue live truth while retaining `state=released` and `reason=provider_issue_released:not_active`.
- Guardrails: preserve retained claim audit history, request-budget discipline, active-worker safety, Ready reclaim boundaries, and control-host lifecycle diagnostics.

## User Experience
- Operators inspecting `provider-intake-state.json` should see current non-active Linear metadata for retained released/not_active rows instead of stale claim-state residue.
- Dependent blocker context should no longer contradict retained claim metadata for the same issue.
