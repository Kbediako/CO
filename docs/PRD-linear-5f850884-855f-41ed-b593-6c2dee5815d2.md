## Summary
- Problem Statement: the current run contract has no durable `source 0` anchor, so adjacent memory consumers still derive context from separate seams. `schemas/manifest.json` and the generated manifest types expose no `memory` block, `scripts/run-review.ts` plus `scripts/lib/review-prompt-context.ts` build review context without a shared pointer-based anchor, `orchestrator/src/cli/services/orchestratorCloudPromptBuilder.ts` builds cloud prompts without one, provider-worker and child-lane prompts each have their own local context shape, and `orchestrator/src/cli/rlm/context.ts` already owns a pointer-based context-object substrate that is not yet surfaced through the shared run contract.
- Desired Outcome: add an additive, schema-backed `memory.source_0` run-contract field and one shared read path so reviewer, cloud, provider-worker, child-lane, and RLM consumers can all read the same pointer-based, traceable anchor instead of flattening or re-deriving it per surface.

## Parity / Alignment Matrix
- Current truth:
  - the shared manifest schema and generated types expose no `memory` or `source_0` field
  - reviewer, cloud, provider-worker, and child-lane consumers each build their own nearby context without a shared manifest-backed anchor
  - RLM already has a pointer-based, copyable context-object substrate (`source.txt`, `index.json`, `ctx:<object>#chunk:<chunk>`) but it is not connected to the run contract
  - the current planner is a deterministic stage planner and does not consume task memory, so the immediate gap is in actual post-bootstrap memory consumers rather than planner selection
- Reference truth:
  - the existing RLM context-object format is already pointer-based, traceable, and copy-friendly across run directories
  - provider-worker and child-lane flows already expect run-scoped manifests and artifacts to be the durable contract boundary
- Target truth / intended delta:
  - each run manifest carries an additive `memory.source_0` descriptor that points at a traceable context object
  - child or delegated runs inherit the same anchor semantically, preserving object identity and local readability
  - reviewer, cloud, provider-worker, child-lane, and RLM read the same descriptor through a shared helper instead of re-deriving bespoke anchor shapes
- Explicitly out-of-scope differences:
  - retrieval indexing, scoring, or experience reuse policy
  - resident app-server continuity and restart semantics
  - provider-worker telemetry or distributed worker-host routing parity
  - broad `0303` or run-memory-controller expansion

## Not Done If
- `source 0` exists in only one consumer surface.
- `source 0` is only a file path or flattened prose blob.

- the lane broadens into telemetry or runtime continuity work already owned elsewhere.

## Goals
- Add a shared run-contract field for `source 0` with schema and generated type backing.
- Reuse the existing pointer-based context-object substrate rather than inventing a second storage format.
- Provide one shared reader path for reviewer, cloud, provider-worker, child-lane, and RLM surfaces.
- Keep the payload bounded, pointer-based, and traceable back to its originating run metadata.

## Non-Goals
- Building the full run-memory controller or retrieval policy.
- Adding block memory.
- Reopening `CO-82`, `CO-83`, `CO-89`, or `CO-90`.
- Reopening `0303` as the umbrella owner.
- Converting the anchor into prompt-only prose.
