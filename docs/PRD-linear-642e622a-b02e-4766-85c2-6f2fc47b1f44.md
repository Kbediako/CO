## Summary
- Problem Statement: `operator-autopilot` already surfaces `local_rollout` entries as `ProviderOperatorAutopilotPendingActionRecord` objects when merged closeout truth indicates a post-merge local rollout follow-up is still pending. The current record shape is only a pending reminder. Operators do not have a durable acknowledge/clear/dismiss lifecycle that distinguishes "seen and accepted", "completed", and "intentionally skipped", so a pending rollout can either keep resurfacing without operator state or disappear only because config or stale merged-closeout truth changes.
- Desired Outcome: define and implement an auditable `local_rollout` lifecycle for `operator-autopilot` pending actions. Each `ProviderOperatorAutopilotPendingActionRecord` can be acknowledged, cleared, or dismissed through a bounded operator control path, with durable audit evidence and observability that survive refresh cycles and restarts.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): make `operator-autopilot` post-merge `local_rollout` follow-up actionable instead of a one-way pending list. Operators need to acknowledge a rollout reminder, clear it after local rollout work is done, or dismiss it when intentionally not applicable, without corrupting merge-closeout truth or hiding current pending work.
- Success criteria / acceptance:
  - `local_rollout` pending actions have an explicit lifecycle with at least pending, acknowledged, cleared, and dismissed states.
  - The lifecycle is keyed to the rollout action identity so a clear or dismiss decision does not accidentally apply to a different issue or a later merge closeout.
  - Acknowledge records operator intent without removing the action from unresolved rollout follow-up truth.
  - Clear records completed local rollout work and prevents the same rollout action from reappearing on subsequent refreshes.
  - Dismiss records an intentional non-rollout decision with a reason and prevents the same rollout action from reappearing unless a new rollout identity is created.
  - Every acknowledge/clear/dismiss transition is durable in operator-autopilot audit output and visible through the control/observability read model.
  - Stale config-disable or no-live-merged-closeout cleanup remains separate from explicit operator clear/dismiss decisions.
- Constraints / non-goals:
  - Do not automate the actual local rollout, rebuild, restart, or deployment in this lane.
  - Do not mutate Linear state as part of acknowledging, clearing, or dismissing `local_rollout`.
  - Do not weaken merge-closeout truth, shared-root proof, or provider-intake claim preservation.
  - Do not redesign backlog promotion, review-handoff rework, merge shepherding, or the whole operator-autopilot policy.
  - This child lane owns docs only; implementation, Linear state, workpad, validation, and PR lifecycle remain parent-owned.

## Parity / Alignment Matrix
- Current truth:
  - `collectPendingActions(...)` emits `ProviderOperatorAutopilotPendingActionRecord` records for merged claims when `post_merge_rollout.enabled` is true.
  - `resolveEffectivePendingActions(...)` keeps pending actions when rollout is enabled and drops them when rollout is disabled.
  - Existing tests cover stale pending rollout cleanup when rollout is disabled or no live merged closeout remains.
  - There is no explicit operator lifecycle state for acknowledge, clear, or dismiss.
- Reference truth:
  - Post-merge local rollout follow-up should behave like an operator action queue, not only a generated reminder list.
  - Operator decisions should be durable, auditable, and distinguish "seen", "done", and "not applicable".
  - Historical merge-closeout evidence should remain intact even when an operator clears or dismisses the rollout reminder.
- Target truth / intended delta:
  - `local_rollout` records carry or resolve against lifecycle state that can be pending, acknowledged, cleared, or dismissed.
  - Lifecycle transitions are explicit operator actions with actor/source, timestamp, reason where applicable, and stable action identity.
  - Cleared and dismissed rollout actions remain audit-visible but no longer appear as unresolved pending actions for the same rollout identity.
  - Observability surfaces expose enough lifecycle state for operators and reviewers to see why a rollout reminder is pending, acknowledged, cleared, or dismissed.
- Explicitly out-of-scope differences:
  - local rollout execution automation
  - Linear issue transition changes
  - queue ordering or promotion policy changes
  - merge-closeout controller changes unrelated to identifying the rollout action
  - a broad dashboard redesign

## Not Done If
- `ProviderOperatorAutopilotPendingActionRecord` still only supports generated pending reminders with no acknowledge/clear/dismiss lifecycle.
- Acknowledging a `local_rollout` silently removes it from unresolved follow-up.
- Clearing or dismissing a `local_rollout` lacks durable audit evidence, actor/source metadata, or a reason when one is required.
- The same cleared or dismissed rollout action reappears on the next refresh without a new rollout identity.
- Explicit operator clear/dismiss decisions are conflated with config-disable cleanup or stale merged-claim cleanup.
- Implementation mutates Linear state, rewrites merge-closeout history, or automates rollout execution under this issue.

## Goals
- Define a stable lifecycle contract for `operator-autopilot` `local_rollout` actions.
- Preserve `ProviderOperatorAutopilotPendingActionRecord` as the protected action surface while extending or pairing it with lifecycle metadata.
- Add durable audit and observability for acknowledge, clear, and dismiss transitions.
- Keep pending-action collection driven by current merged closeout truth while allowing operator decisions to resolve specific rollout identities.

## Non-Goals
- Running local rollout commands automatically.
- Restarting control-host or provider-worker processes as part of this lifecycle.
- Moving Linear issues between workflow states.
- Hiding or deleting merge-closeout claim evidence.
- Reworking `operator-autopilot` backlog or review-handoff behavior.
- Adding broad new operator workflow engine semantics.

## Metrics & Guardrails
- Primary Success Metrics:
  - an operator can acknowledge a pending `local_rollout` and still see it as unresolved but acknowledged.
  - an operator can clear a completed `local_rollout` and the same action does not reappear on refresh.
  - an operator can dismiss an intentionally skipped `local_rollout` with a reason and the same action does not reappear on refresh.
  - audit JSONL and read-model output expose lifecycle transitions without needing a live chat transcript.
- Guardrails / Error Budgets:
  - fail closed when action identity is ambiguous.
  - preserve append-only audit evidence.
  - do not consume Linear request budget for lifecycle-only decisions.
  - keep stale generated-action cleanup distinct from explicit operator lifecycle decisions.

## User Experience
- Personas:
  - operator returning after merge to decide whether local rollout work is needed.
  - reviewer auditing why a merged lane still has or no longer has pending rollout work.
- User Journeys:
  - operator sees a pending `local_rollout`, acknowledges it, and later still sees the action as acknowledged and unresolved.
  - operator completes the local rollout, clears the action, and future refreshes no longer present it as pending for that merge.
  - operator determines no local rollout is needed, dismisses the action with a reason, and audit output preserves that decision.
  - reviewer inspects `provider-operator-autopilot.jsonl` and control read-model output to confirm the lifecycle transition and identity.

## Technical Considerations
- Architectural Notes:
  - lifecycle state should live near `providerOperatorAutopilot.ts` action identity and audit handling, not in Linear issue state.
  - cleared and dismissed records should be retained as audit history while excluded from unresolved `pending_actions`.
  - acknowledge should be idempotent for the same action identity and should not suppress the unresolved action.
  - clear and dismiss should fail closed when the requested action cannot be matched to a current or known rollout identity.
- Dependencies / Integrations:
  - `orchestrator/src/cli/control/providerOperatorAutopilot.ts`
  - `orchestrator/src/cli/control/providerIssueHandoff.ts`
  - `orchestrator/src/cli/control/observabilityReadModel.ts`
  - `provider-operator-autopilot.jsonl`
  - focused tests in `orchestrator/tests/ProviderOperatorAutopilot.test.ts` and `orchestrator/tests/ProviderIssueHandoff.test.ts`
