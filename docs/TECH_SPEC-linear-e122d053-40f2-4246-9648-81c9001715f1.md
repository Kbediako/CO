## Summary
- Objective: enforce one authoritative provider max-concurrency gate across every start/resume admission path instead of only the poll loop.
- Scope:
  - register the docs-first packet and workpad for `CO-125`
  - add a shared admission-capacity helper at the provider start/resume launch boundary
  - cover webhook/direct acceptance, queued retry start, queued retry resume, and poll dispatch
  - keep active occupancy sourced from authoritative active claims and discovered runs
  - add focused regression coverage for non-poll admission and over-cap restart behavior
- Constraints:
  - do not broaden into generic `CO STATUS` denominator or projection work
  - do not kill already-running provider workers
  - keep the change bounded to `providerIssueHandoff.ts` and direct tests unless a smaller adjacent seam is required

## Technical Requirements
- Functional requirements:
  - every call that can start or resume provider-worker work must pass through one shared global-capacity check before launch
  - the shared gate must use authoritative active-claim and active-run truth, including discovered `in_progress` runs and active claims that already own work
  - poll dispatch must remain state-aware but cannot be the only place where global capacity is enforced
  - if the host is already over cap during restart or rehydration, new admissions must be blocked while existing running work continues unchanged
- Non-functional requirements (performance, reliability, security):
  - occupancy calculation must stay repo-local and deterministic
  - the gate must fail closed for new admissions without corrupting existing claim/run metadata
  - regression coverage must distinguish this bug from stale-historical-manifest inflation
- Interfaces / contracts:
  - `orchestrator/src/cli/control/providerIssueHandoff.ts`
  - `orchestrator/tests/ProviderIssueHandoff.test.ts`
  - `createProviderPollDispatchBudget(...)`
  - `processTrackedIssueCandidate(...)`
  - `launchStartForTrackedIssue(...)`
  - `launchResumeForRun(...)`
  - `dispatchQueuedProviderRetry(...)`

## Architecture & Data
- Architecture / design adjustments:
  - introduce a shared provider admission helper that computes current occupancy from discovered attachable runs and persisted claims, then decides whether a new start/resume may proceed
  - route both `launchStartForTrackedIssue(...)` and `launchResumeForRun(...)` through that helper so webhook/direct, retry, and poll paths all converge on one gate
  - preserve the poll-only state-limit logic where useful, but remove any gap where non-poll paths can bypass the global cap
- Data model changes / migrations:
  - none expected; the lane should reuse existing claim and discovered-run truth
- External dependencies / integrations:
  - provider-worker manifest and proof discovery in `.runs/**`
  - existing claim persistence in `provider-intake-state.json`

## Validation Plan
- Tests / checks:
  - audited `linear child-stream --pipeline docs-review`
  - focused `ProviderIssueHandoff` regressions for webhook/direct over-admission, queued retry start blocking, queued retry resume blocking, and already-over-cap restart behavior
  - full repo validation floor after implementation if the final diff is non-trivial
- Rollout verification:
  - confirm non-poll paths cannot exceed the configured provider cap
  - confirm active live work remains untouched when the host starts above cap
  - confirm active-claim coverage still maps to real `in_progress` manifests rather than relying on stale history inflation
- Monitoring / alerts:
  - rely on provider-worker proof/manifests plus focused tests and review evidence

## Open Questions
- Whether state-specific poll ceilings should remain as a thin wrapper around the shared occupancy helper or collapse into the same helper outright. The implementation should pick the smallest approach that prevents future bypasses.
