# PRD - CO: Rebaseline Marketplace Docs and Pack-Smoke for Codex 0.125

## Summary
- Problem Statement: Codex CLI `0.125.0` exposes the marketplace flow under `codex plugin marketplace ...`, including remote marketplace sources and marketplace upgrade/remove commands, while CO still has public docs and downstream smoke surfaces that describe or depend on the older top-level `codex marketplace add` command.
- Desired Outcome: rebaseline public docs, version policy, and downstream pack-smoke behavior so current Codex users see `codex plugin marketplace add`, `upgrade`, and `remove` guidance, while release-facing smoke pins are promoted only when supported by canary evidence.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): complete CO-355 as the Codex `0.125.0` marketplace rebaseline lane. The repo should stop telling current users to run `codex marketplace add`, should document the new plugin marketplace command surface and remote/local source behavior, and should make `pack:smoke` understand the `0.125.0` command layout without silently dropping older supported marketplace pins.
- Success criteria / acceptance:
  - current public docs recommend `codex plugin marketplace add` instead of top-level `codex marketplace add`
  - README/downstream docs mention `codex plugin marketplace upgrade` and `remove`
  - remote owner/repo, HTTPS, SSH, and local marketplace source guidance is preserved
  - pack-smoke marketplace detection/execution works with the `0.125.0` plugin marketplace surface and preserves older top-level fallback
  - release-facing smoke pin promotion is backed by current canary evidence or explicitly held with rationale
  - version policy records the `0.125.0` marketplace decision and compatibility boundary
  - the packet validates CO-355 against the CO-351/CO-352 `gpt-5.5` / app-server direction by avoiding any conservative model/runtime posture reset
- Constraints / non-goals: do not promote workflow pins without evidence, do not weaken pack-smoke coverage, and do not use CO-355 to settle the separate CO-351/CO-352 model/runtime posture lanes. CO-355 must remain compatible with that direction by deferring to the current version-policy posture instead of hard-coding a contradictory model or app-server hold.

## Parity / Alignment Matrix

| Surface | Current truth | Reference truth | Target truth |
| --- | --- | --- | --- |
| Public marketplace docs | README and downstream setup still show `codex marketplace add ...`. | Local Codex `0.125.0` exposes `codex plugin marketplace add`, `upgrade`, and `remove`; top-level `codex marketplace add` fails. | Current docs use plugin marketplace commands and preserve source/launcher guidance. |
| Pack-smoke marketplace coverage | `scripts/pack-smoke.mjs` requires top-level `codex marketplace add`. | Codex `0.125.0` moves marketplace commands under `plugin marketplace`; older supported smoke pins may still expose top-level commands. | Pack-smoke detects the current plugin command and keeps a legacy fallback for older pins. |
| Release-facing smoke pins | Core, release, and backstop workflows are pinned to older marketplace-capable Codex. | `0.125.0` is npm latest locally and supports plugin marketplace commands. | Pins move only if current pack-smoke canary evidence passes; otherwise policy records the hold. |
| CO-351/CO-352 model/runtime direction | PR handoff requires validation against the requested `gpt-5.5` / app-server direction. | Version policy already treats `gpt-5.5` / `xhigh` as current when available and keeps app-server as the normal local runtime path. | CO-355 does not hard-code a contradictory conservative posture; it scopes changes to marketplace smoke and defers model/runtime posture to the owning lanes. |

Explicitly out-of-scope differences: implementing CO-351/CO-352 model/runtime changes inside this marketplace lane, cloud-canary active-target promotion, plugin marketplace product design beyond CO install docs, and broad workflow refactors.

## Not Done If
- Public docs still recommend top-level `codex marketplace add` for current Codex.
- Pack-smoke is unpinned or promoted without canary evidence.
- Remote install, marketplace upgrade, or remove support is ignored.
- Older supported marketplace pins regress without an intentional policy decision.
- Version policy omits the `0.125.0` marketplace compatibility boundary.

## Goals
- Rebaseline user-facing marketplace install docs for Codex `0.125.0`.
- Keep pack-smoke marketplace proof explicit and compatible across current and older supported command surfaces.
- Record the release-facing pin decision with direct canary evidence.
- Keep implementation scoped to marketplace docs, pack-smoke behavior/tests, and version-policy/workflow pin posture.

## Non-Goals
- No standalone implementation of CO-351/CO-352 model/runtime changes inside this marketplace lane; CO-355 only validates that its docs and policy wording do not contradict that direction.
- No removal of npm as the baseline CO CLI install path.
- No unrelated plugin manifest redesign.
- No weakening of docs/review/pack-smoke validation gates.

## Stakeholders
- Product: downstream CO users installing through Codex plugin marketplace.
- Engineering: CO maintainers responsible for docs, pack-smoke, release workflows, and Codex version policy.
- Design: not applicable.

## Metrics & Guardrails
- Primary Success Metrics: no stale top-level marketplace guidance for current Codex; focused pack-smoke tests prove both current plugin and legacy command detection; final `npm run pack:smoke` runs with the recorded pin decision.
- Guardrails / Error Budgets: fail closed on missing marketplace coverage, preserve older fallback behavior, and keep canary evidence attached to any release-facing pin change.

## User Experience
- Personas:
  - downstream user installing CO from npm and enabling the Codex plugin
  - maintainer reviewing whether release smoke can run on `@openai/codex@0.125.0`
  - reviewer checking that current docs match the live Codex CLI command surface
- User Journeys:
  - User reads README/downstream setup and runs `codex plugin marketplace add <source>`.
  - User later runs `codex plugin marketplace upgrade` or `remove` without hunting through CLI help.
  - Maintainer runs `npm run pack:smoke` and sees marketplace coverage use the supported command surface.

## Technical Considerations
- Architectural Notes: pack-smoke should choose the newest supported marketplace command when present and keep explicit fallback behavior for older CLI pins. Docs should keep npm as baseline and marketplace as additive plugin registration.
- Dependencies / Integrations: local `/opt/homebrew/bin/codex` `0.125.0`, npm `@openai/codex` metadata, GitHub workflow pins, pack-smoke temp marketplace source, Codex config marketplace entries, and docs freshness/spec guards.
