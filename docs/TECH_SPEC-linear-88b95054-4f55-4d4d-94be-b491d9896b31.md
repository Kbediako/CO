## Summary
- Objective: keep persisted provider claim metadata truthful during active provider-worker runs by refreshing tracked-issue state metadata whenever live Linear has newer eligible data.
- Scope:
  - docs-first registration for the current provider-worker lane
  - audited child docs-review for packet evidence
  - bounded `providerIssueHandoff.ts` changes for active-run refresh or rehydration
  - focused provider handoff regressions plus the required validation floor
- Constraints:
  - preserve ownership and lifecycle behavior for active runs
  - do not change eligibility, rate-limit behavior, or poll cadence
  - keep operator truth fixes rooted in persisted claim data instead of read-side-only presentation

## Technical Requirements
- Functional requirements:
  - active eligible claims must refresh `issue_state`, `issue_state_type`, and `issue_updated_at` from the freshest live tracked issue during refresh or rehydration
  - the persisted claim must stay `running` when the run is still active and eligible
  - terminal-proof and manifest-authority safeguards must keep behaving as they do today
  - focused regressions must cover `Ready` to `In Progress` drift and prove lifecycle ownership remains intact
- Non-functional requirements:
  - keep the diff narrow and audit-friendly
  - avoid adding redundant read-side compensations if persisted truth can be fixed at the source
  - preserve deterministic provider-worker validation behavior
- Interfaces / contracts:
  - persisted claim contract in `orchestrator/src/cli/control/providerIntakeState.ts`
  - provider rehydrate or refresh lifecycle in `orchestrator/src/cli/control/providerIssueHandoff.ts`
  - operator truth surfaces backed by `provider-intake-state.json`

## Architecture & Data
- Architecture / design adjustments:
  - reuse existing tracked-issue metadata merge helpers where possible instead of inventing a second active-run metadata path
  - inspect both startup rehydration and later refresh handling to ensure `provider_issue_rehydrated_active_run` cannot preserve stale state fields in one path while another path is fixed
  - preserve run-identity, manifest-authority, and proof-authority rules already exercised by nearby terminal-path tests
- Data model changes / migrations:
  - no schema change expected; the goal is to populate existing claim fields more truthfully
  - docs-first mirrors and task registry entries for this lane
- External dependencies / integrations:
