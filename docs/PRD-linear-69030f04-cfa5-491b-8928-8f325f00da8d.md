## Summary
- Problem Statement: `CO-55` added pause, compact inspect, snapshot export, and the dedicated `co-status` launch surface, but the live dashboard still redraws in the primary terminal buffer. After a few live refreshes, pausing leaves a readable current frame but scrollback still contains multiple full historical `CO STATUS` frames stacked on top of each other. That makes paused inspection truthful only in the narrowest sense; it is not a clean operator inspection path.
- Desired Outcome: preserve the same `CO STATUS` terminal dashboard for routine live monitoring, but make paused inspection scrollback-clean by defining explicit live-vs-paused terminal semantics. The narrow target is live rendering that does not pollute the primary scrollback path and a paused state that yields one clean inspect snapshot the operator can review and scroll around without depending on noisy historical redraws.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): keep `CO STATUS` as the normal terminal monitor, but stop treating stacked redraw history as the paused inspection path. The operator should be able to pause, inspect the top of the frame and surrounding terminal history cleanly, and then resume live monitoring without switching to `/ui`, without merely slowing redraws, and without reducing the dashboard into a weaker surface.
- Success criteria / acceptance:
  - `CO STATUS` has explicit live and paused inspection semantics instead of relying on primary-buffer redraw history
  - paused inspection yields a clean, truthful snapshot path in the terminal after the operator presses pause
  - repeated live redraws no longer stack full `CO STATUS` frames in the primary terminal scrollback during normal monitoring
  - pause, compact inspect, and snapshot export continue working on the same dashboard surface
  - tests and proof explicitly cover paused inspection semantics, not only live refresh behavior
- Constraints / non-goals:
  - preserve `CO STATUS` as the same terminal dashboard surface
  - do not reopen the broader Symphony parity lane from `CO-44`
  - do not rebuild the operator dataset or presenter just to change terminal semantics
  - do not move normal inspection to the browser `/ui` surface

## Technical Considerations
- Architectural Notes:
  - the narrow owner remains `orchestrator/src/cli/control/controlStatusDashboard.ts`
  - the current defect is caused by interactive writes always using `ANSI_CLEAR_HOME` in the primary buffer
  - the smallest truthful approach is:
    - render live interactive refreshes in the terminal alternate screen
    - on pause, exit alternate screen and write one clean static dashboard snapshot into the primary buffer
    - while paused, keep explicit redraws in-place in the primary buffer instead of appending fresh full-frame history
    - on resume, re-enter the alternate screen and continue normal live monitoring
  - non-TTY and JSON surfaces must remain unchanged
- Dependencies / Integrations:
  - `orchestrator/src/cli/control/controlStatusDashboard.ts`
  - `orchestrator/tests/ControlStatusDashboard.test.ts`
