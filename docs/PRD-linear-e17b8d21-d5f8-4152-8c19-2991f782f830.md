## Summary
- Problem Statement: duplicate cleanup or terminal cleanup can free a provider slot, but the control-host can still miss the immediate follow-on admission because `provider_intake`, `selected_claim`, or fresh-discovery slot budgeting stays pinned to pre-cleanup truth. The result is a free-capacity window where `fresh_discovery` does not backfill newly eligible `Ready` work until some unrelated later refresh or operator action.
- Desired Outcome: once duplicate cleanup or terminal cleanup makes a claim non-capacity-consuming, the control-host should refresh provider-intake/admission truth in time for bounded `fresh_discovery` to backfill the newly free slot, or surface a specific machine-checkable reason why no admission happened.

## User Request Translation
- User intent / needs: create the docs-first packet for `CO-317` only and keep the seam narrow: provider intake, fresh discovery, and admission-budget or `selected_claim` refresh behavior after duplicate cleanup or terminal cleanup frees capacity.
- Success criteria / acceptance:
  - duplicate-cleanup or terminal-cleanup outcomes that genuinely free capacity no longer leave `fresh_discovery` starved behind stale pre-cleanup intake truth
  - parent implementation can prove whether the missed admission comes from stale admission-budget accounting, stale `selected_claim` refresh, or both
  - bounded `fresh_discovery` backfills a newly free slot for current truthful `Ready` work without waiting for unrelated relaunch or queue churn
  - real active workers, real queued workers, and still-current cleanup/action-required rows remain capacity-consuming when they should
  - the issue stays distinct from generic Linear queue redesign, cleanup advisory generation, or cleanup hook implementation work
