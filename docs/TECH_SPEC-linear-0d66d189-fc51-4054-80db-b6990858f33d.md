## Summary
- Objective: prevent duplicate same-task control-host processes from competing on the same provider-intake state.
- Scope:
  - docs-first registration for `CO-152`
  - same-task owner lock acquisition before provider refresh polling
  - active duplicate fail-closed behavior
  - deterministic stale-owner cleanup
  - operator-visible owner metadata and duplicate diagnostics
  - focused tests plus required validation and review handoff
- Constraints:
  - preserve active provider-worker safety
  - do not clear provider-intake claims as part of duplicate detection
  - keep API budget/backoff and launchd replacement out of scope

## Issue-Shaping Contract
- User-request translation carried forward: add ownership/lock detection so only one `control-host --task local-mcp --run control-host --pipeline provider-linear-worker` owns a repo/task/run/pipeline provider-intake state at a time, with diagnostics that explain duplicates distinctly from cooldown or API exhaustion.
- Protected terms / exact artifact and surface names: `control-host`, `local-mcp`, `provider-linear-worker`, `provider-intake-state.json`, `provider_poll_lifecycle_stuck`, `provider_refresh_lifecycle_stuck`, `co-status`.
- Nearby wrong interpretations to reject: manual `pkill` as the durable fix, clearing provider-intake claims, weakening provider rehydration, or relabeling GitHub/Linear budget issues as duplicate ownership.
- Explicit non-goals carried forward: broad process killing, launchd removal, GitHub API backoff, provider claim corruption.

## Parity / Alignment Matrix
- Not applicable as a formal parity lane.
- Current truth: startup lacks a same-task ownership guard and can reach provider refresh scheduling while another host owns the same run directory.
- Reference truth: a host must own durable metadata before polling; duplicates must fail closed against active owner identity.
- Target truth / intended delta: atomic owner lock plus diagnostic metadata before polling, stale cleanup when owner pid is inactive, and no active-worker or claim mutation side effects.
- Explicitly out-of-scope differences: Linear/GitHub budget policy and broad supervision replacement.

## Readiness Gate
- Not done if:
  - two same-task hosts can poll concurrently
  - duplicate startup does not fail before provider refresh polling
  - diagnostics do not identify the owner
  - detection kills provider workers or wipes claims
- Pre-implementation issue-quality review evidence:
  - 2026-04-11: parent worker self-review confirmed the issue is not still plausibly narrower than the user request. The implementation scope covers startup ownership, stale cleanup, diagnostics, and active-worker safety while excluding CO-151 GitHub budget work.
- Safeguard ownership split:
  - parent lane owns the single overlapping startup/provider-intake/status contract; child-lane implementation was not launched in this turn because write scopes would collide.

## Technical Requirements
- Functional requirements:
  - acquire lock before ready instance startup and provider refresh startup trigger
  - reject active duplicate owner with structured metadata
  - reclaim stale lock only after pid liveness check proves inactive owner
  - release current lock on close only if token still matches
  - publish latest owner or duplicate/stale diagnostics to local artifact and compatible runtime/status metadata
- Non-functional requirements:
  - fail closed when metadata is unreadable or liveness is ambiguous
  - keep behavior deterministic on local macOS and CI fixtures
  - keep backward-compatible provider-intake and polling payloads
- Interfaces / contracts:
  - `control-host-owner.json` or equivalent local owner artifact
  - `ControlServerPublicLifecycleState` owner release handle
  - provider polling/runtime metadata additions

## Architecture & Data
- Architecture / design adjustments:
  - introduce a focused `controlHostOwnership` helper
  - wire owner acquire/release around public lifecycle startup/shutdown
  - enrich duplicate/stale diagnostics without changing provider claim state
- Data model changes / migrations:
  - add a local owner artifact in the run directory
  - optionally add additive owner metadata fields under persisted `polling`
  - no migration required for existing `provider-intake-state.json` claims
- External dependencies / integrations:
  - process liveness through `process.kill(pid, 0)` or injectable equivalent for tests
  - filesystem atomic create/remove primitives
