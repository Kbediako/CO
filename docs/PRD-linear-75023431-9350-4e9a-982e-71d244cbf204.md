## Summary
- Problem Statement: provider child-lane acceptance can treat stale parent run-manifest `issue_updated_at` as if it were current issue truth. CO-486 and CO-493 show accept comparing a newer child-lane snapshot against an older fallback timestamp and invalidating with `provider_worker_child_lane_stale`, even though backwards timestamp inversion is not legitimate Linear freshness movement.
- Desired Outcome: `linear child-lane --action accept` refreshes and records the live current issue snapshot at accept time, never uses parent run-manifest `issue_updated_at` as current truth, leaves unavailable or backwards current snapshots retryable with an explicit diagnostic, and still invalidates true stale cases such as parent HEAD movement, live `updated_at` moving forward, or issue state changes.

## User Request Translation
- User intent / needs: define the CO-494 implementation contract so provider child-lane patch acceptance can use current issue truth instead of rejecting already-correct child output only because the issue timestamp moved during parent-owned workpad or Linear updates.
- Success criteria / acceptance:
  - Reproduce the CO-486 and CO-493 backwards timestamp inversion examples where the child snapshot is newer than the fallback/current timestamp.
  - Update accept-time logic so unavailable or backwards current issue truth returns an explicit retryable diagnostic instead of invalidating the child lane as stale.
  - Record accept-time current issue snapshot source and unavailable reason alongside the launch-time parent snapshot.
  - Preserve fail-closed behavior for parent HEAD movement, live `updated_at` moving forward, issue state changes, patch scope violations, proof mismatch, dirty parent edits, and missing patch/proof artifacts.
  - Add focused tests around accept-time current issue snapshot handling and preserved stale rejection.
- Constraints / non-goals:
  - no child-lane authority to mutate Linear, workpads, PRs, or parent lifecycle state
  - no bypass of `provider-linear-child-lane.patch` / `provider-linear-child-lane-proof.json` validation
  - no acceptance after real issue state/type or issue identity changes
  - no broad provider-worker or control-host redesign
  - no reliance on manual `git apply` recovery as the normal success path
