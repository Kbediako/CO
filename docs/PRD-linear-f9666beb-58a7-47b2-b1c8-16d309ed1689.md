## Summary
- Problem Statement: released provider-intake claims can retain stale blocker metadata and overwrite a live issue back to `Blocked` even after every referenced blocker is terminal. The concrete incident is `CO-295`, which was manually moved from `Ready` to `In Progress` at `2026-04-22T04:58:34.614Z`, then showed live as `Blocked` again by `2026-04-22T05:01:21.766Z` while `.runs/local-mcp/cli/control-host/provider-intake-state.json` still had a released row with `reason=provider_issue_released:not_active`, `run_id=2026-04-21T16-08-48-577Z-c085bbbf`, and `issue_blocked_by=[CO-300 Done]`.
- Desired Outcome: queue truth stays stable after reclaim. Terminal blocker metadata from a released claim is dropped, neutralized, or explicitly ignored before reconcile/poll logic can force a live issue back to `Blocked`, while released-claim provenance and fail-closed provider behavior remain auditable.

## User Request Translation
- Preserve the issue as a stale released blocker metadata lane, not a general `CO-295` PR attachment ownership lane.
- Treat `CO-300 Done` as terminal blocker evidence that must not keep `CO-295` blocked.
- Prevent the stale released row from silently overwriting live Linear state after a manual reclaim.
- Preserve provider provenance guardrails; do not hide drift by deleting unrelated released-claim fields or by adding another manual state flip.
- If automatic repair cannot land immediately, surface explicit stale-blocker advisory evidence instead of silently forcing `Blocked`.

1. Released/not-active claim refresh or reconcile logic drops, neutralizes, or explicitly ignores blocker links whose referenced issues are terminal (`Done` / `Cancelled`).

3. Add focused regression coverage for the `CO-295` / `CO-300 Done` stale-blocker overwrite shape.
