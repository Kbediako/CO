## Summary
- Problem Statement: Symphony requires workflow startup to fail closed on missing or invalid configuration, but later reload failures must keep the service running on the last known good workflow while logging and surfacing the error. CO still records that parity seam as missing, and the current provider-control path resolves repo-local `codex.orchestrator.json` directly at launch time without a persisted last-known-good fallback.
- Desired Outcome: Add a control-host managed workflow/config store for the provider launch path so startup still fails on missing or invalid repo configuration, later reload failures preserve the last known good effective provider workflow/config, operators can see the current error state until it is corrected, and a later valid edit replaces the cached snapshot and clears the error.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): Complete Linear issue `CO-6` by closing the specific Symphony parity gap around workflow reload safety, using the actual CO workflow seam that exists today instead of inventing an unrelated runtime layer.
- Success criteria / acceptance:
  - invalid or missing workflow config at startup still prevents the control-host provider workflow from booting
  - after startup, a bad repo-config edit does not take down the running service or break provider dispatch; CO keeps using the last known good effective workflow/config
  - reload failures are logged and exposed through operator-visible control-host observability until a good config is restored
  - a later valid edit replaces the last known good snapshot and clears the sticky error state
  - focused tests cover startup failure, good-to-bad retention, and bad-to-good recovery
- Constraints / non-goals:
  - keep scope bounded to workflow reload safety and observability for the control-host provider path
  - do not mix this with unrelated provider feature work, issue workflow changes, or broader runtime refactors
  - delegation must be recorded as an explicit override in this worker run because subagent spawning is unavailable in-session

## Goals
- Match the Symphony startup-fail-closed and runtime-last-known-good reload contract in the corresponding CO control-host/provider seam.
- Persist and reuse a last known good effective provider workflow/config snapshot after successful startup.
- Surface reload failures clearly in logs and observability until corrected.
- Keep the implementation small, auditable, and test-backed.

## Non-Goals
- Adding a new user-facing workflow DSL or a CO-local `WORKFLOW.md` abstraction.
- Redesigning provider issue handoff, queue semantics, or PR shepherding behavior.
- Broad changes to repo-config parsing outside the fallback seam needed for provider launches.
