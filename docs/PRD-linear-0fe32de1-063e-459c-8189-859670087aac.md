## Summary
- Problem Statement: `create-follow-up` can successfully create a Linear follow-up issue, then fail during description verification with `linear_follow_up_description_update_incomplete` before relation creation and packet traceability repair. Same-attempt retry can then create another follow-up because the partial success is not reconciled from the already-created issue id.
- Desired Outcome: once Linear returns a created follow-up id, later retry or reconciliation paths must reuse that issue, repair/verifiably complete missing description, labels, relations, packet traceability, and audit evidence, and never blindly create a duplicate follow-up for the same source failure.

## User Request Translation
- Preserve the CO-527 issue contract: fix the duplicate-follow-up root cause for partial create success, especially the `linear_follow_up_description_update_incomplete` path.
- Treat `follow_up_issue_id` and issue URL returned after creation as authoritative evidence that the mutation already happened.
- Keep fail-closed verification for labels, description, relation creation, packet mirrors, and traceability; do not turn a partial success into an unverified success.

## Acceptance Criteria
1. A follow-up creation that returns `linear_follow_up_description_update_incomplete` after receiving a Linear issue id does not blindly create a duplicate follow-up on same-attempt retry.
2. Retry/reconciliation reuses the created `follow_up_issue_id` from run/audit evidence or deterministic signature and repairs/verifies description, labels, relations, packet traceability, and audit evidence before retrying creation.
3. Non-canonical create-follow-up paths preserve relation creation and packet traceability even when Linear normalizes Markdown differently.
4. Existing canonical-owner retry suppression and label-resolution behavior remain compatible.
5. Focused tests cover partial-success id capture, retry suppression/reuse, relation/traceability continuation, and the existing canonical-owner retry paths.

## Not Done If
- `linear_follow_up_description_update_incomplete` with a created issue id can still produce a second follow-up for the same source issue and signature.
- The helper treats Linear Markdown normalization as success without repairing/verifying relation and traceability requirements.
- Existing canonical-owner follow-up behavior regresses.
- The fix hides failure evidence from manifests, workpads, or provider-worker retry state.
