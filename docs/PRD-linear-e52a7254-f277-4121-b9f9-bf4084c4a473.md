## Summary
- Problem Statement: CO now carries the authoritative runtime and retry data that earlier Symphony parity slices required, but the operator-facing surface is still thinner than the current Symphony Elixir reference. This packet opened when `/ui` still used the generic selected-run monitor dataset; current CO routes `/ui` through `operatorDashboardPresenter.ts`, so keep this PRD as issue-open context rather than live architecture guidance.
- Desired Outcome: Add a richer read-only CO operator observability surface aligned with Symphony's optional status/dashboard contract and current Elixir information density, while keeping UI/API layers thin over shared presenter/read-model logic and avoiding a new backend-truth rewrite.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): finish Linear issue `CO-7` by delivering a truthful richer operator dashboard surface for CO, grounded in the real local Symphony references rather than overstating the work as cloning one specific screen.
- Success criteria / acceptance:
  - show running sessions and retry queue from authoritative runtime state
  - show aggregate token totals, runtime totals, latest rate-limit data, and refresh/poll health
  - show per-issue details including issue identifier, lifecycle state, session/thread id, turn count, recent agent activity, workspace path/host, retry status, and last error when available
  - keep UI/API layers thin over a shared presenter/read-model layer instead of duplicating orchestration logic
  - if a refresh trigger is exposed, keep it operational only and not correctness-bearing
- Constraints / non-goals:
  - this is optional presentation parity, not another backend-truth lane
  - do not describe the work as cloning one specific Symphony screen
  - keep the implementation read-only and stop coding once the issue reaches the team review state
  - this worker run cannot spawn subagents in-session, so delegation-required guard steps need explicit override evidence

## Goals
- Deliver a richer dedicated operator dashboard on the current `/ui` surface.
- Reuse the existing compatibility projection and issue payload seams instead of rebuilding orchestration logic in the frontend.
- Expose refresh/poll health in a way that matches the current CO control-host lifecycle and remains operational only.
- Preserve the current read-only `/api/v1/*` contract while improving information density and operator drilldown.

## Non-Goals
- Reopening the already-landed backend-authoritative runtime, retry, or poll-dispatch parity slices.
- Building a new mutable control plane or adding correctness-bearing UI actions.
- Requiring terminal and web parity in the same packet; this lane can be web-first.
- Replacing the existing compatibility issue payload contract with a different API family.
