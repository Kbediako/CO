## Summary
- Problem Statement: the original live `CO-102` incident described a provider-worker issue that stayed pinned in `In Progress` while the attached PR was already green because `provider-linear-worker` kept timing out when it asked the control-host refresh path to recover late-turn truth. Current archived evidence is more precise: the source run eventually self-recovered to `In Review`, `Merging`, and `Done`, but it did so while refresh-request timeouts continued for long stretches, which means the remaining seam is the refresh request contract under pressure rather than a permanently wedged terminal state.
- Desired Outcome: make late-turn control-host refresh requests truthful and resilient under repeated timeout bursts so active provider-worker claims can recover and hand off cleanly without indefinite `In Progress` pinning, while still preserving explicit stuck/restart-required truth for genuinely stuck control-host lifecycles and keeping historical attached-PR disambiguation on the already-landed bounded path.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): close the late-turn recovery seam exposed by `CO-102` by ensuring the control-host refresh path does not leave an otherwise healthy provider-worker claim stuck in `In Progress` just because refresh requests time out while the host is already busy completing handoff work around a green PR. Preserve the exact surfaces `provider-linear-worker`, `control-host refresh`, `provider_refresh_lifecycle_stuck`, `In Progress`, and stale historical attached PRs.

  - do not solve this only by increasing the worker-side timeout if the real problem is a slow refresh acknowledgement contract

  - `providerLinearWorkerRunner.ts` currently expects `/api/v1/refresh` to return before a 15s client timeout, even when the control-host already has pending refresh work
  - `controlServerPublicLifecycle.ts` can queue/coalesce refresh work behind an active provider poll/refresh chain, but the current caller-facing contract is still slow enough to produce repeated provider-worker timeout noise
  - `CO-104` already tightened same-repo attached-PR disambiguation for reopened and historical merge-closeout paths
- Reference truth:
  - refresh requests should acknowledge quickly when the control-host already knows a refresh is queued or running, because that is sufficient to preserve late-turn recovery progress
  - stuck/restart-required truth should only represent genuinely unhealthy control-host lifecycle states, not busy-but-healthy queued work
  - historical attached-PR ambiguity should continue to use the existing truthful selector rather than a second heuristic
- Target truth / intended delta:
  - busy-but-healthy refresh requests return prompt accepted/queued/coalesced truth instead of timing out repeatedly
  - active provider-worker claims can recover and reach the truthful next workflow state without indefinite `In Progress` pinning
  - explicitly stuck control-host lifecycles still surface `provider_refresh_lifecycle_stuck` / `restart_required`
  - stale historical attached PRs remain disambiguated on the existing bounded path
- Explicitly out-of-scope differences:
  - review-to-`Merging` promotion policy beyond preserving the existing handoff path
  - merge-closeout redesign after an issue is already in `Merging`
  - generic dashboard, UI, or event-feed cleanup
  - launchd/root-host supervision changes

## Not Done If
- A green PR can still leave a live provider-worker issue pinned in `In Progress` while refresh requests keep timing out.
- Busy-but-healthy queued refresh work still waits long enough to look like a failed recovery request to the provider worker.
- `provider_refresh_lifecycle_stuck` / `restart_required` still oscillate during late-turn timeout bursts without separating actual stuck states from queued healthy work.
- Historical attached PRs can still poison the recovered late-turn path before normal handoff or closeout takes over.

- Make the control-host refresh acknowledgement contract resilient under late-turn pressure.
- Preserve truthful active-claim recovery semantics for provider-worker issues with green or nearly green PRs.
- Preserve explicit stuck/restart-required truth only for genuinely unhealthy control-host lifecycles.
- Reuse existing attached-PR disambiguation rather than reopening adjacent closeout lanes.

## Non-Goals

- Re-implementing `CO-116` or `CO-104`.
- Broad Linear polling-budget redesign unrelated to the late-turn refresh seam.
- Treating the source issue’s now-completed terminal state as the thing that still needs fixing.
