## Summary
- Problem Statement: completed-run rehydrate can preserve an older `merge_closeout.status=action_required` record from `Merging` even after newer tracked-issue truth shows the same Linear issue back in active work such as `In Progress`. That stale record keeps the claim pinned at `handoff_failed` / `provider_issue_merge_closeout_action_required` and blocks autonomous pickup until someone edits local intake state by hand.
- Desired Outcome: newer non-`Merging` issue truth must invalidate or safely supersede stale merge-closeout failure residue so completed-run rehydrate and later poll recovery can relaunch the issue normally, while truly current `Merging` closeout behavior stays unchanged.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): repair the exact stale merge-closeout failure mode from the `CO-120` incident by making rehydrate trust newer tracked issue truth over older `merge_closeout` residue when the issue has already left `Merging`. The fix belongs in provider intake and completed-run rehydrate handling, not in manual recovery procedures or unrelated merge-closeout lanes.
- Success criteria / acceptance:
  - the `CO-120` shape is reproducible from local artifacts: newer issue truth in `In Progress` plus older `merge_closeout.status=action_required` / `merge_state_status=BEHIND`
  - completed-run rehydrate no longer leaves the claim in `handoff_failed` / `provider_issue_merge_closeout_action_required` when newer tracked-issue truth is no longer `Merging`
  - stale merge-closeout state is cleared or superseded so normal pickup works without manual state-file edits
  - truly current `Merging` merge-closeout behavior remains authoritative and unchanged
  - focused regressions cover completed-run rehydrate plus poll / rehydrate recovery for this stale-record shape
