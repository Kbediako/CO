## Summary
- Problem Statement: `docs:freshness`, `docs:freshness:maintain`, `spec-guard`, `docs:check`, archive-stub validation, provider merge closeout, and status surfaces can still infer different lifecycle or owner truth from registry status, task-index status, terminal Linear source state, and local checklist readiness. CO-579 cleared the immediate terminal owner recurrence but left the larger resolver/finalizer contract explicit as CO-580.
- Desired Outcome: a shared docs freshness lifecycle resolver and owner-finalizer contract makes effective lifecycle and owner closeout decisions deterministic, keeps terminal source issue state as evidence rather than local completion when local checklists remain open, and blocks terminal closeout for configured owners while active freshness candidates still resolve to them.

## User Request Translation
- User intent / needs: prevent the recurring docs freshness owner swap loop by centralizing lifecycle/owner authority instead of relying on policy prose or manual `last_review` changes.
- Success criteria / acceptance:
  - a shared lifecycle resolver or documented smallest equivalent module boundary is used across freshness, maintain, spec guard, docs check or archive-stub validation, and provider owner closeout
  - effective lifecycle states distinguish active, terminal pending archive, archived, deprecated, and preserved historical stubs
  - terminal Linear state never hides open local checklist obligations
  - configured global and exact cohort owners cannot close terminal while active candidates still resolve to them
  - regressions cover terminal source plus open checklist, archived plus open checklist, active-reviewed terminal source rows, task-index/registry/spec disagreement, exact cohort owner precedence, and simulated owner-terminal closeout
- Constraints / non-goals:
  - no weakening `docs:freshness`, `docs:freshness:maintain`, `docs:check`, `spec-guard`, or archive-stub checks
  - no broad historical packet deletion
  - no cap/window expansion
  - no manual state-file cleanup as the fix
  - no provider-intake behavior changes except owner closeout/finalizer integration needed to prevent terminal owner recurrence

## Not Done If
- Any docs freshness command can infer a different lifecycle/owner for the same path.
- A terminal Linear issue can remain the configured global owner while active candidates still resolve to it.
- Terminal source state is confused with local archive readiness.
- Active-reviewed terminal-source packets require ad hoc notes instead of structured lifecycle evidence.
- Owner finalization lacks regression coverage that simulates the owner issue becoming terminal.
