# PRD - CO: Clear Codex 0.121 Cloud Canary Promotion Gate

## Summary
- Problem Statement: CO-207 was reopened after the required Codex CLI `0.121.0` cloud canary became runnable with `CODEX_CLOUD_ENV_ID` present. The required cloud execution command now passes, but the required fallback command still exits `1` even when the fallback manifest and run summary prove the expected local MCP fallback contract.
- Desired Outcome: CO-207 can clear the promotion gate only when both required commands produce current evidence. The fallback wrapper should pass the documented fallback contract after its existing assertions prove expected fallback behavior, without weakening cloud preflight or marketplace packaging boundaries.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): make the CO-196 marketplace packaging blocker machine-readable and actionable by producing current Codex CLI `0.121.0` cloud canary evidence or recording a precise blocker/waiver.
- Success criteria / acceptance: CO-207 remains related as `blocks` CO-196, the required cloud execution command passes, the required fallback command passes or records an explicit owner-approved waiver/blocker, CO-196 remains blocked while CO-207 is open, and local intake exposes CO-207 in `CO-196.issue_blocked_by`.
- Constraints / non-goals: do not implement CO-196 marketplace descriptors, do not weaken cloud preflight/sandbox/delegation/authority guardrails, and do not switch provider workers to app-server supervision.

- Exact user wording / phrases to preserve: `CO-207`, `CO-196`, Codex CLI `0.121.0`, `CODEX_CLOUD_ENV_ID=<env-id> CODEX_CLOUD_CANARY_REQUIRED=1 npm run ci:cloud-canary`, `CODEX_CLOUD_ENV_ID=<env-id> CODEX_CLOUD_CANARY_REQUIRED=1 CLOUD_CANARY_EXPECT_FALLBACK=1 npm run ci:cloud-canary`, `issue_blocked_by`, `blocks`.
- Protected terms / exact artifact and surface names: `scripts/cloud-canary-ci.mjs`, `.runs/<task>/cli/<runId>/manifest.json`, `run-summary.json`, `cloud_execution`, `cloud_fallback`, `mode_requested=cloud`, `mode_used=mcp`, `missing_environment`, `Failure class: configuration`, `CODEX_CLOUD_CANARY_REQUIRED`, `CLOUD_CANARY_EXPECT_FALLBACK`.
- Nearby wrong interpretations to reject: treating required cloud execution success alone as enough for promotion, closing CO-207 while fallback still exits failed without waiver, implementing marketplace descriptors here, or suppressing real configuration/credential/connectivity failures outside the expected fallback contract.

## Parity / Alignment Matrix
- Required for parity/alignment lanes; otherwise state `Not applicable`.
- Current truth: required cloud execution can pass with a configured cloud environment, but fallback mode clears `CODEX_CLOUD_ENV_ID`, records a successful local MCP fallback manifest, and then reclassifies the expected `missing_environment` signal as a fatal required-mode configuration failure.
- Reference truth: the documented fallback command is a required promotion gate and should prove fallback behavior through manifest and run-summary evidence.
- Target truth / intended delta: when `CLOUD_CANARY_EXPECT_FALLBACK=1`, the wrapper passes only after existing assertions prove `manifest.status=succeeded`, no `cloud_execution`, `cloud_fallback.mode_requested=cloud`, `cloud_fallback.mode_used=mcp`, a non-empty fallback reason, a `missing_environment` issue, and matching run-summary fields.
- Explicitly out-of-scope differences: no marketplace descriptor changes, no broader cloud runtime redesign, no relaxed preflight outside the explicit expected fallback assertion path, no credential/profile rotation changes.

## Not Done If
- CO-196 remains Blocked without CO-207 appearing as a machine-readable non-completed blocker.
- The fallback canary command still exits `1` after producing an otherwise valid expected fallback manifest.
- The implementation hides genuine configuration, credential, connectivity, or execution failures for non-fallback runs.
- Codex `0.121.0` is promoted without both canary artifacts or an owner-approved waiver.
