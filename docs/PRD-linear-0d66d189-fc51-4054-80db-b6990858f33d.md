## Summary
- Problem Statement: a second same-task `control-host --task local-mcp --run control-host --pipeline provider-linear-worker` process was observed alongside the launchd-supervised host while both pointed at the same CO root, run directory, logs, and provider-intake state. That shape can produce competing provider refreshers, ambiguous `provider_poll_lifecycle_stuck` / `provider_refresh_lifecycle_stuck` evidence, wasted Linear/GitHub request budget, and provider-worker refresh-request timeouts.
- Desired Outcome: make a same-task control-host ownership boundary explicit and fail closed before duplicate provider refresh polling can start, while preserving active provider-worker claims and leaving operator-visible owner metadata for diagnosis and recovery.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): add durable same-task ownership detection for the local control-host provider intake path so only one process owns a repo/task/run/pipeline intake state at a time. Startup should reject or coordinate with an existing owner before polling begins, stale owner metadata should be safe to clean up, and diagnostics should distinguish duplicate ownership from provider cooldown, GitHub throttling, and Linear budget exhaustion.
- Success criteria / acceptance:
  - duplicate same-task host startup fails closed or cleanly hands off before provider refresh polling starts
  - owner identity and lock metadata are visible in a local diagnostic artifact and eligible status surfaces
  - stale lock cleanup is deterministic and does not kill active workers
  - active owner protection is covered by tests or deterministic harnesses
  - stuck lifecycle and refresh-request diagnostics include duplicate-owner context when it is relevant
- Constraints / non-goals:
  - do not implement broad process killing or `pkill` as the durable fix
  - do not remove launchd supervision or require manual restart as the normal path
  - do not widen into GitHub API budget/backoff, which is tracked by `CO-151`
  - do not weaken provider rehydration, claim reconciliation, or active-claim safety

## Parity / Alignment Matrix
- Not applicable as a formal parity lane. Guardrail comparison for this control-host safety lane:
- Current truth:
  - startup loads provider-intake state and starts the ready instance without a same-task ownership lock
  - the provider refresh startup trigger can begin polling after startup even when another same-task process already owns the same run directory
  - persisted polling health can report stuck or restart-required without distinguishing duplicate host ownership from API budget or lifecycle wedge
- Reference truth:
  - a single repo/task/run/pipeline owner should publish a durable owner identity before provider refresh polling begins
  - a duplicate process should fail closed against active owner metadata and should only reclaim stale metadata when the recorded owner is demonstrably inactive
- Target truth / intended delta:
  - ownership is acquired before provider refresh coordination and released on orderly shutdown
  - active-owner duplicates fail with structured diagnostics and stale-owner cleanup is deterministic
  - owner metadata is visible in a local diagnostic artifact and, where useful, in persisted provider polling diagnostics
  - active provider workers and provider-intake claims remain untouched by the duplicate-host protection path
- Explicitly out-of-scope differences:
  - GitHub API budget/backoff behavior
  - Linear request-budget policy
  - launchd service removal or broad host process cleanup

## Not Done If
- Two same-task `control-host --task local-mcp --run control-host` processes can still poll the same provider-intake state concurrently.
- The only mitigation is a manual `pkill` or restart runbook.
- Linear/GitHub budget exhaustion is hidden behind duplicate-host wording.
- Duplicate detection can corrupt, discard, or orphan active provider claims.

## Goals
- Add an atomic same-task owner lock for the control-host provider-intake state.
- Fail closed on active duplicate startup before provider refresh polling starts.
- Clean up stale owner metadata only when the owner process is no longer alive.
- Surface owner metadata and duplicate/stale diagnostics through local artifacts and compatible status reads.
- Add focused tests for duplicate startup, stale cleanup, and active-worker safety.

## Non-Goals
- Process killing as normal remediation.
- Replacing launchd supervision.
- GitHub API budget/backoff work.
- Provider claim cleanup, release, or rehydration weakening.

## Metrics & Guardrails
- Primary Success Metrics:
  - duplicate same-task startup yields a classified duplicate-owner error before refresh polling starts
  - stale lock fixtures are reclaimed without touching provider-intake claims

## Technical Considerations
- Architectural Notes:
  - acquire ownership in the control-host public lifecycle before ready-instance startup and provider refresh scheduling
  - store metadata in the run directory next to `provider-intake-state.json` so operator diagnostics survive process exit
  - use process liveness only to distinguish active owner from stale owner; never terminate the owner or provider workers
  - add polling/status metadata additively so existing provider-intake and compatibility projections remain backward compatible
