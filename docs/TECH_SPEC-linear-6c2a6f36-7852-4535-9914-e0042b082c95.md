## Required Behavior
- Released pending-reopen claims with fresh started Linear truth and an attachable same-issue `in_progress` provider run rehydrate to `running` with `provider_issue_rehydrated_active_run`.
- Ordinary `released` / `provider_issue_released:not_active` claims with fresh started Linear truth and an attachable same-issue `in_progress` provider run rehydrate to `running` with `provider_issue_rehydrated_active_run`.
- Ordinary released/not_active claims do not rehydrate from cached started metadata alone; they require fresh started issue truth so no-live or terminal retained-release optimizations remain intact.
- Rehydrate persists active run identity, manifest path, worker host when available, retry cleanup fields, and fresh issue metadata when the existing resolver can safely provide it.
- The pre-refresh cheap rehydrate pass must not attach a released active run from stale cached started metadata before fresh non-started Linear truth can be applied.
- In-flight release cancellation blocks active-run reattachment until the cancel attempt drains.
- Fresh Ready/Todo/terminal truth keeps the claim released, refreshes issue metadata, and avoids CO STATUS active classification based on stale `In Progress`.
- CO STATUS compatibility projection counts started released-pending-reopen and ordinary released/not_active workers during the intake catch-up window so `counts.running` matches displayed running rows and admission occupancy.

## Non-Goals
- No schema migration, scheduler redesign, Linear workflow mutation, broad polling increase, or duplicate worker launch.
- Do not count terminal/historical retained released claims as active.

## Validation
- Focused serialization tests cover started rehydrate, fresh terminal/non-started truth, refresh metadata fallthrough, and cancel-in-flight reattach blocking.
- Runtime projection test covers the released pending-reopen started worker row.
- Rework regression covers the ordinary released/not_active started worker row and provider-intake rehydrate path.
