## Summary
- Problem Statement: `CO-168` fixed delegation transport/setup startup truth and bounded cleanup for orphaned `delegate-server` chains that are no longer rooted in a live `codex` parent. That still leaves a narrower gap: long-lived top-level Codex parent sessions can accumulate `delegate-server` children that remain parented and therefore stay classified as active forever, even after the delegated work they once served is gone. Current doctor/cleanup surfaces only show aggregate active/stale counts, stale PIDs, and stale RSS, so operators cannot tell which child belongs to an active issue-worker/review/child-lane process tree versus a long-lived parent-session leftover.
- Desired Outcome: CO exposes enough per-process observability to separate active issue-worker/review/child-lane ownership from long-lived parent-session leftovers, reports parent PID/cwd/RSS/associated manifest context, and provides bounded recovery guidance without regressing `CO-168` protections for truly active delegated work.

## Technical Considerations
- Current repo truth:
  - `orchestrator/src/cli/utils/delegationMcpHealth.ts` builds process truth from `ps -ax -o pid=,ppid=,etime=,rss=,args=` and only returns aggregate active/stale counts plus stale RSS
  - `isDelegateServerRootedInCodex(...)` currently treats any `delegate-server` with a live `codex` ancestor as active
  - `cleanupStaleDelegateServerProcesses(...)` only operates on `stalePids`
  - `orchestrator/src/cli/doctor.ts` serializes only aggregate process stats, not per-process parent/manifest context
  - current run manifests already expose `workspace_path`, `artifact_root`, `manifest_path`, and scheduler `instance_id` / `capability` data that the parent can reuse for process correlation
- Dependencies / Integrations:
  - `orchestrator/src/cli/utils/delegationMcpHealth.ts`
  - `orchestrator/src/cli/doctor.ts`
  - `orchestrator/src/cli/delegationCliShell.ts`
  - `.runs/**/manifest.json` and runner logs under `.runs/**/runner.ndjson`

## Open Questions
- Should `idle` stay advisory-only, with `stale` as the only cleanup-applicable class? Current preference: yes, because the issue requires safe observability first and bounded cleanup second.
- What exact evidence should promote `idle` to `stale` for still-parented children: age alone, missing/terminal manifest association, absent active issue-worker/review/child-lane tree, or a combination? Current preference: require age plus missing or terminal active-work evidence.
