## Summary
`CO-189` fixes an accounting gap where a provider claim can remain `released` / `provider_issue_released_pending_reopen:*` even though the same Linear issue has a live provider-worker run, fresh proof, and active child-lane/process evidence. The product outcome is truthful provider-worker accounting: `provider-intake-state.json`, `co-status --format json`, `counts.running`, attach mode, and provider admission all agree with live started-worker reality before new workers are admitted.

Rework update 2026-04-16: the same accounting contract also applies to ordinary retained `released` / `provider_issue_released:not_active` claims when fresh Linear truth says the issue is started and local same-issue provider-worker evidence is live. The original pending-reopen path is not broad enough for live CO-195 / CO-198 style workers.

## User Request Translation
- Rehydrate live same-issue provider workers before preserving released/not-active claim state.
- Keep the existing no-burn retained-release protections for terminal or no-live-worker claims from `CO-160`, `CO-162`, `CO-182`, and `CO-73`.
- Do not repair the mismatch by killing workers, launching duplicates, widening Linear polling, or masking the discrepancy with display-only text.

## Intent Checksum
- Protected terms and surfaces: `provider-intake-state.json`, `claims[].state=released`, `provider_issue_released_pending_reopen`, `provider_issue_released:not_active`, `provider_issue_rehydrated_active_run`, `provider-linear-worker-proof.json`, `co-status --format json`, `counts.running`, `provider_debug_snapshot.claim`, live provider worker process tree, child-lane process tree, `CO-185`.
- Reject nearby wrong interpretations: always preserve released claims, count terminal released issues again, duplicate-launch a provider worker, cancel the live worker, or hide the mismatch in status text while admission/accounting stays wrong.

## Parity / Alignment Matrix

| Surface | Current | Target |

## Acceptance Criteria
- Provider-intake reconciliation checks same-issue live evidence before preserving a `released` / not-active claim: current run manifest, fresh provider proof, and active process/child-lane evidence where available.
- Ordinary `released` / `provider_issue_released:not_active` claims rehydrate to running when the same issue has an attachable live `in_progress` provider run and fresh started Linear truth.
- Recovered claims persist `state: "running"`, `reason: "provider_issue_rehydrated_active_run"`, active `task_id`, `run_id`, `run_manifest_path`, and `worker_host` where available.
- `co-status --format json` and attach mode include live started provider workers during both released-pending-reopen and ordinary released/not_active catch-up windows.
- Admission/max-allowed accounting cannot over-admit because a live worker is hidden behind a retained released claim.
- Fresh non-started or terminal Linear truth prevents stale cached `In Progress` metadata from reattaching a released active run.
- In-flight release cancellation prevents immediate active-run reattachment until cancellation is no longer pending.
- Regression tests cover the `CO-185` mismatch and preserve terminal/historical no-burn behavior.

## Non-Goals
- No provider scheduler redesign, workflow-state relabeling, broader fresh-discovery policy change, or Linear mutation.
- No manual restart/runbook-only fix.
- No active worker kill/cancel/cleanup shortcut for a worker that should be counted.
- No expansion into `CO-185` helper preflight implementation details.

## Not Done If
- A started provider worker with live proof/process evidence disappears from CO STATUS or active-worker accounting.
- A retained released pending-reopen or ordinary released/not_active claim can remain released while fresh started truth and a same-issue active run are available.
- Terminal Done/Duplicate/Cancelled or historical non-provider rows re-enter active counts.
- Duplicate `launcher.start` or `launcher.resume` is invoked for an already-live same-issue worker.
- The fix increases Linear issue-by-id reads as the primary reconciliation mechanism.
