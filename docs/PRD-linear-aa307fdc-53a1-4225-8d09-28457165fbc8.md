# PRD - CO STATUS: stop synthetic linear-* task-id fallback rows from rendering as running issues

## Added by Bootstrap 2026-04-10

## Traceability
- Linear issue: `CO-146` / `aa307fdc-53a1-4225-8d09-28457165fbc8`
- Linear URL: https://linear.app/asabeko/issue/CO-146/co-status-stop-synthetic-linear-task-id-fallback-rows-from-rendering
- Related lane: `CO-112` / `4a7e540f-b47d-4fa3-a083-e6e9047e68a5`

## Summary
- Problem Statement: `CO STATUS` can render a synthetic running issue row whose visible identifier is only a provider-worker fallback task id such as `linear-0b49c08c-...`. The fallback leaks from projection because manifest snapshots still promote `task_id` into `issue_identifier` / `issue_id` when no authoritative issue identity exists, selected or discovered in-progress sources can then be treated as running issues, and the operator dataset forwards that fallback identifier as if it were a real issue row.
- Desired Outcome: retain underlying run and manifest diagnostics, but suppress or rebind fallback-only `linear-*` identities before they reach the running issue list so operators only see canonical issue rows.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): fix the identity seam, not the table styling. Provider-worker manifests that lack canonical issue identity should either bind to the real issue before presentation or stay out of the running issue table entirely, while legitimate provider rows and non-provider rows with real identity continue to render normally.
- Success criteria / acceptance:
  - `CO STATUS` no longer renders synthetic running ids like `linear-...` when a manifest lacks authoritative issue identity
  - a selected or discovered in-progress manifest without canonical issue identity is either suppressed from the running issue table or rebound to the correct issue before presentation
  - real provider-worker issue rows still render correctly
  - regression coverage proves the fallback `task_id` row does not appear while the real issue row still does
  - debug or observability paths still preserve access to the underlying task id, run id, and manifest when needed
- Constraints / non-goals:
  - do not reopen `CO-112` EVENT provenance work
  - do not reopen stale retry-state duplication or broader claim-state cleanup
  - do not broaden this into a STATUS dashboard redesign

## Parity / Alignment Matrix
- Current truth:
  - `selectedRunProjection.ts` falls back `issue_identifier` and `issue_id` to `task_id` / `run_id`
  - selected or discovered in-progress sources can still be admitted as running activity before canonical issue identity is established
  - compatibility projection and operator dataset can therefore expose a fallback `linear-*` identifier as a running issue row
- Reference truth:
  - running issue presentation should key off canonical issue identity, not provider-worker task ids
  - fallback task ids may remain available for debugging, but they should not masquerade as issue identifiers
- Target truth / intended delta:
  - fallback-only synthetic `linear-*` identities are rebound via tracked issue or intake claim when possible
  - otherwise they are excluded from running issue presentation
  - canonical issue rows remain visible and dedupe correctly
- Explicitly out-of-scope differences:
  - EVENT provenance ranking
  - retry-state or claim-state redesign
  - table layout or general UI redesign

## Not Done If
- `CO STATUS` still shows a synthetic `linear-*` row as a running issue when no canonical issue identity exists.
- A real provider-worker issue row disappears because the fallback row was suppressed without rebinding to the real issue.
- Operators lose access to underlying run or manifest diagnostics that are still useful for debugging.

## Goals
- Stop fallback-only `linear-*` task ids from rendering as running issue identifiers.
- Rebind fallback-only sources to canonical issue identity when tracked or claim-backed identity is available.
- Preserve legitimate provider-worker and explicit-identity non-provider rows.
- Add focused regression coverage for selected and discovered fallback shapes.

## Non-Goals
- Reworking EVENT/message provenance.
- Reworking retry or claim-state semantics.
- Redesigning the operator dashboard.
