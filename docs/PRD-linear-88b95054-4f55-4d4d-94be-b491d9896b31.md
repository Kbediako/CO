# PRD - CO: Refresh tracked issue metadata for active provider claims

## Summary
- Problem Statement: while `CO-63` was actively running, `.runs/local-mcp/cli/control-host/provider-intake-state.json` kept the issue cached as `Ready` even after live Linear and the worker audit had already moved it to `In Progress`. That leaves persisted provider-claim truth stale during normal active-run refresh or rehydration.
- Desired Outcome: keep persisted provider-claim metadata truthful for active runs by refreshing `issue_state`, `issue_state_type`, and `issue_updated_at` from the freshest eligible tracked issue without changing pickup eligibility, lifecycle ownership, or review-state policy.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): finish `CO-64` in the current provider-worker workspace by fixing the active-run rehydration or refresh seam that preserves stale tracked-issue metadata in `provider-intake-state.json`, so operator truth stays aligned with live Linear during normal work.
- Success criteria / acceptance:
  - persisted provider claims update `issue_state`, `issue_state_type`, and `issue_updated_at` while an active run is still in progress
  - an issue that moves from `Ready` to `In Progress` no longer remains cached locally as `Ready`
  - focused regression coverage proves lifecycle ownership stays intact while metadata refreshes
  - operator-facing reads backed by the persisted claim no longer diverge from live Linear for this drift class
- Constraints / non-goals:
  - do not change which issues are eligible for pickup
  - do not change worker transition policy, review-state semantics, or rate-limit or poll cadence
  - do not patch only read-side presentation while leaving persisted claim truth stale underneath

## Parity / Alignment Matrix
- Current truth: active-run rehydration can keep a persisted claim in `state: running` while still carrying stale tracked-issue metadata such as `issue_state: Ready`, `issue_state_type: unstarted`, and an older `issue_updated_at`, even after live Linear has already moved the issue to `In Progress`.
- Reference truth: live Linear issue reads and worker audit metadata already reflect the newer state and timestamp for the same active claim.
- Target truth / intended delta: when a tracked issue remains eligible and its run remains active, the persisted claim stays `running` but refreshes its tracked-issue metadata to the freshest live state, type, and update timestamp.
- Explicitly out-of-scope differences: pickup eligibility, lifecycle ownership, review handoff semantics, rate-limit or poll cadence changes, and read-side presentation-only fixes.

## Not Done If
- A running claim can still show stale `issue_state`, `issue_state_type`, or `issue_updated_at` after the live issue has transitioned.
- `provider-intake-state.json` can still disagree with live Linear for an already-running claim during ordinary refresh or rehydration.
- The fix patches only one active-run path while another path still preserves stale tracked-issue fields.
- CO STATUS or other operator reads remain misleading because the persisted claim truth is still stale underneath them.

## Goals
- Identify the exact active-run refresh or rehydration seam that preserves stale tracked-issue metadata.
- Refresh tracked-issue metadata for active eligible claims without changing lifecycle ownership or release behavior.
- Add focused regression coverage around `Ready` to `In Progress` drift and active-run persistence.
- Leave behind manifest-backed docs, validation, and review evidence for later handoff.

## Non-Goals
- Reopening broader control-host pickup or refresh-stall recovery work.
- Changing issue eligibility rules or worker state-transition policy.
- Adjusting Linear polling frequency, rate-limit behavior, or unrelated child-lane adoption.
- Fixing only read-side operator surfaces while leaving persisted claim truth wrong.
