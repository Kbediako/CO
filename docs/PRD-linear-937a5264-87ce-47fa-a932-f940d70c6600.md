## Summary
- Problem Statement: `co-status --format json` can report a `.running[]` row whose `issue_id`, `task_id`, and `run_id` belong to one provider worker while `workspace_path` comes from a neighboring selected or last active provider row. The observed 2026-05-02 mismatch involved concurrent `CO-456` and `CO-474` workers where the `CO-474` row was rehydrated through `provider_issue_rehydrated_active_run` fallback/projection data but carried the `CO-456` workspace path.
- Desired Outcome: every running status row keeps row identity and row-local workspace/proof data coupled across JSON/API/UI projection surfaces, including fallback/rehydrated rows. Existing fallback provenance must remain visible, but it must not let selected-run or neighboring-row workspace data override row-local authority.
