# ACTION PLAN - CO-535 Docs Freshness Repo Gate Visibility

## Summary
- Goal: Prevent docs freshness status blindness by making `co-status` read fresh local/task-scoped maintenance reports and expose degraded evidence when no usable report exists.
- Scope: Docs packet, repo-gate reader, projection/UI tests, and report search-order guidance.
- Assumptions: `scripts/docs-freshness-maintain.mjs` remains the authority that generates maintenance decisions and normalized `repo_gate` payloads.

## Issue Readiness Gate
- Intent checksum / protected terms carried forward: `co-status`, `repo_gates.docs_freshness_maintain`, `docs_freshness_maintain`, `out/local/docs-freshness-maintenance.json`, `out/docs-truthfulness-maintenance/docs-freshness-maintenance.json`, `source_path`, `generated_at`, `CO-522`.
- Not done if: fresh local/task-scoped report evidence remains hidden, stale scheduled reports mask fresher evidence, or degraded states return silent `null`.
- Pre-implementation issue-quality review: CO-535 is a status-source root-cause lane, not another stale-doc cleanup lane. CO-530 and CO-536 cleared terminal historical debt; this issue prevents the operator dashboard from hiding future maintenance evidence.
- Fallback / refactor decision: remove scheduled-path-only fallback behavior; missing scheduled reports must not imply no repo gate.
- Durable retention evidence: Not applicable; the target is removal of the silent fallback.
- Large-refactor check: Keep implementation bounded to the existing repo-gate reader and projection tests unless additional repo gates require shared discovery abstractions.

## Milestones & Sequencing
1. Create and register the CO-535 packet and workpad evidence.
2. Update the repo-gate reader to discover, rank, and normalize candidate reports with degraded evidence.
3. Add focused tests and docs guidance, then run validation gates.
4. Open PR, verify Codex/CodeRabbit review signals, drain checks, and complete Linear handoff.

## Dependencies
- Existing `scripts/docs-freshness-maintain.mjs` report shape and `repo_gate` payload.
- Existing `orchestrator/src/cli/control/docsFreshnessRepoGate.ts` normalization.
- Existing status projection/UI surfaces that already carry `repo_gates`.

## Validation
- Checks / tests: focused `DocsFreshnessRepoGate` tests, selected projection/UI tests, `spec-guard`, `build`, `docs:check`, `docs:freshness`, `docs-freshness-maintain`, and manual `co-status` proof.
- Rollback plan: Revert repo-gate reader/projection changes and packet updates; the status surface returns to scheduled-path-only behavior, which is the known defect.

## Risks & Mitigations
- Risk: Selecting stale local output over fresh scheduled output. Mitigation: compare `generated_at` and report freshness explicitly.
- Risk: Treating missing evidence as clean. Mitigation: degraded evidence status is explicit and test-covered.
- Risk: Repo-gate status inflates provider WIP. Mitigation: preserve `provider_wip_impact=excluded_repo_gate` and projection tests.
