# PRD - CO: Add a Codex 0.121 plugin marketplace distribution path for Codex Orchestrator

## Summary
- Problem Statement: CO currently ships an npm install path and an MCP stdio entrypoint (`codex-orchestrator mcp serve`), but it does not ship the Codex marketplace descriptors that Codex CLI `0.121.0` expects for repo-local or Git-backed marketplace distribution.
- Desired Outcome: add the smallest marketplace/plugin distribution path that lets Codex discover and register CO through the marketplace flow while preserving the existing npm install path and default delegation/MCP guardrails.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): make CO installable through the new Codex marketplace flow without replacing npm packaging, without inventing a new runtime seam, and with an isolated smoke proving the marketplace path can discover and register the existing CO MCP surface.
- Success criteria / acceptance:
  - confirm the supported descriptor shape from local `codex marketplace add --help` and official Codex plugin/marketplace docs
  - add the minimal plugin and marketplace descriptors needed for repo-local / Git-backed marketplace distribution
  - keep the marketplace entry wired to the existing `codex-orchestrator mcp serve` surface
  - avoid secrets, postinstall hooks, private run artifacts, or workspace-only paths
  - add a temp-home smoke showing the marketplace source can be added and the CO plugin can be discovered/registered
  - document npm versus marketplace install, rollback/removal, and support posture
- Constraints / non-goals: do not replace npm packaging, do not add hosted marketplace infrastructure, do not broaden authority/delegation boundaries, and do not use this lane to revisit Codex version-policy promotion logic.

## Parity / Alignment Matrix
- Current truth: CO ships npm package metadata, docs, templates, skills, and the `codex-orchestrator mcp serve` CLI surface, but no `.agents/plugins/marketplace.json` and no packaged `plugins/codex-orchestrator/.codex-plugin/plugin.json` or `plugins/codex-orchestrator/.mcp.json`.
- Reference truth: Codex plugin docs require a plugin root with a descriptor file; in CO's layout that means `plugins/codex-orchestrator/.codex-plugin/plugin.json` plus optional `plugins/codex-orchestrator/.mcp.json`. Repo and local marketplaces are driven by `marketplace.json`; local `codex marketplace add --help` accepts local marketplace roots and Git-backed sources.
- Target truth / intended delta: the repo and packed npm tarball include a minimal marketplace root plus plugin descriptor that register the existing CO MCP surface, and downstream docs explain when to use npm versus the marketplace path.
- Explicitly out-of-scope differences: hosted marketplace infrastructure, plugin UIs/apps, new MCP server behavior, authority changes, or Codex version-policy promotion changes outside what this lane needs to describe honestly.

## Not Done If
- The change only adds descriptor files without a smoke proving Codex can use them.
- The plugin descriptor points at workspace-only paths or private artifacts.
- The marketplace path bypasses or changes the existing `mcp serve` contract.
- The npm tarball still omits the new marketplace/plugin descriptors.
- Docs fail to explain removal/rollback or leave the support posture ambiguous.

## Goals
- Add a repo-root marketplace source for CO that points at `./plugins/codex-orchestrator`.
- Add a minimal plugin descriptor that reuses the existing CLI entrypoint.
- Keep npm as a supported install path.
- Keep the shipped package free of secrets, hooks, and workspace-only paths.

## Non-Goals
- Replacing npm packaging.
- Shipping hosted marketplace infrastructure.
- Adding app metadata, widget surfaces, or new MCP authority.
- Refactoring unrelated package or delegation code.

## Stakeholders
- Product: downstream CO users choosing between npm and marketplace install paths.
- Engineering: CLI/package maintainers, MCP/delegation maintainers, and reviewers checking downstream tarball safety.
- Design: not applicable.

## Metrics & Guardrails
- Primary Success Metrics: marketplace descriptors are present in the repo and tarball, temp-home smoke proves the marketplace flow can register CO, and downstream docs describe both install paths plus rollback.
- Guardrails / Error Budgets: zero secrets in shipped descriptors, zero new authority surface, zero workspace-local paths, zero postinstall hooks, and zero regressions to the existing npm path.

## User Experience
- Personas:
  - downstream user installing CO through npm
  - downstream user on a Codex CLI with marketplace support
  - reviewer validating that marketplace install is additive and reversible
- User Journeys:
  - user installs CO through npm exactly as before
  - user adds the CO marketplace source, discovers the plugin, and registers the existing MCP surface
  - user removes the marketplace source or plugin and falls back to npm docs without ambiguity

## Technical Considerations
- Architectural Notes: the new marketplace distribution path should be additive, repo-rooted, package the plugin under `plugins/codex-orchestrator`, and use the same `codex-orchestrator mcp serve` stdio contract already documented in templates and downstream setup. Because Codex caches only the plugin source directory, the packaged launcher should resolve the recorded marketplace source root from `~/.codex/config.toml` and exec the packaged CO CLI there instead of assuming the full package contents are copied into the plugin cache.
- Dependencies / Integrations: local Codex CLI `marketplace` help surface, official Codex plugin docs, npm pack contents, downstream docs, and temp-home smoke artifacts.
