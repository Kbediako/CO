## Summary
- Objective: prevent duplicate Linear follow-up creation when a previous `create-follow-up` attempt already created an issue but returned `linear_follow_up_description_update_incomplete`.
- Scope: `create-follow-up` workflow facade/shell behavior, provider-worker retry classification, focused tests, docs-first packet, and validation evidence.
- Constraints: preserve fail-closed verification, relation creation, label handling, packet traceability, and existing canonical-owner retry suppression.

## Issue-Shaping Contract
- User-request translation carried forward: created follow-up ids must be durable partial-success evidence and same-attempt retries must repair/reuse them instead of creating duplicates.
- Protected terms / exact artifact and surface names: `linear_follow_up_description_update_incomplete`, `follow_up_issue_id`, `create-follow-up`, `relation creation`, `packet traceability`, `canonical owner evidence`, `providerLinearWorkflowFacade`.
- Nearby wrong interpretations to reject: do not mark partial success complete without verification; do not skip relations; do not create another issue as a repair strategy; do not weaken CO-509/CO-513 safeguards.
- Explicit non-goals carried forward: no broad retry architecture rewrite, no unrelated label contract changes, no release-intake or docs-freshness scope.

## Parity / Alignment Matrix
- Current truth: a created follow-up id can be returned in a failure payload, but duplicate-prevention is incomplete for `linear_follow_up_description_update_incomplete`.
- Reference truth: existing canonical-owner and retry-suppression paths avoid repeating deterministic mutations once authoritative identity evidence exists.
- Target truth / intended delta: non-canonical partial-success failures reuse the created issue id and continue repair/verification before any new create call is allowed.
- Explicitly out-of-scope differences: global retry scheduler behavior and unrelated follow-up admission policy.

## Readiness Gate
- Not done if: duplicate follow-ups can still be created from the same partial-success failure, or relation/traceability repair is skipped.
- Pre-implementation issue-quality review evidence: live CO-527 issue-context, initial workpad, and `linear parallelization --decision stay_serial --reason overlapping_scope` are recorded.
- Safeguard ownership split: parent owns docs, tests, and implementation because the failure contract spans the same code/test surface; model-backed subagents are quota-blocked in this run.

## Technical Requirements
- Functional requirements:
  - Preserve `follow_up_issue_id` and URL when issue creation succeeds but later verification fails.
  - Classify the partial-success failure as duplicate-unsafe for blind retry.
  - Add or extend reconciliation to re-read the created follow-up, repair/verify description, labels, relations, packet traceability, and audit evidence.
  - Prevent create mutation from running again when the same attempt already has a created follow-up id.
  - Keep existing canonical-owner and label-resolution retry paths working.
- Non-functional requirements:
  - Fail closed when the created issue cannot be re-read or verified.
  - Keep audit output explicit enough for workpad and provider-worker retry state.
  - Avoid broad refactors outside the create-follow-up workflow.
- Interfaces / contracts:
  - `codex-orchestrator linear create-follow-up`
  - provider-worker follow-up result payloads containing `follow_up_issue_id`
  - provider Linear workflow facade result/error codes
