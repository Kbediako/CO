## Summary
- Problem Statement: recent successful GitHub `Core Lane` jobs still spend roughly `21.5m` to `22.1m` wall time in aggregate, with the `Test` step averaging about `20.9m`. The hot tail is concentrated in `tests/run-review.spec.ts` and `tests/cli-command-surface.spec.ts`, where many assertions repeatedly cold-boot TypeScript entrypoints through `node --loader ts-node/esm`.
- Desired Outcome: cut the hot test-step tail into the low teens by redesigning the two subprocess-heavy suites so pure contract assertions run in-process, a bounded subprocess smoke matrix still proves end-to-end entrypoint behavior, and the resulting timing improvement is recorded without dropping build/lint/spec/docs/pack-smoke guardrails.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): keep the current Core Lane guardrail shape, but materially reduce wall time by focusing on the two cited hot suites instead of weakening unrelated validation. The new test design should preserve confidence while removing unnecessary process startup cost.
- Success criteria / acceptance:
  - `tests/run-review.spec.ts` and `tests/cli-command-surface.spec.ts` remain the primary optimization targets
  - pure CLI-surface and review-wrapper contract assertions move to faster in-process coverage where a real subprocess is not required
  - a smaller subprocess smoke matrix remains for end-to-end command-surface and review-wrapper confidence
  - any remaining subprocess cases avoid repeated cold `ts-node/esm` startup where a lighter shared harness or built entrypoint preserves the same confidence
  - before/after timings are recorded for the two hot suites and the resulting full `Test` step improvement
- Constraints / non-goals:
  - do not cut Core Lane by removing validation categories that still matter
  - do not reopen older hang-truth or teardown-stall lanes unless fresh evidence proves the slowdown is actually a hang regression
  - do not turn this into a general CI cleanup umbrella unrelated to the two cited suites

## Technical Considerations
- Architectural Notes:
  - `bin/codex-orchestrator.ts` and `scripts/run-review.ts` currently centralize behavior behind unexported `main()` entrypoints, which makes pure contract assertions expensive because tests re-enter them through subprocess boot
  - the fastest honest path is likely a hybrid harness: export/import-friendly entrypoint seams for pure contracts, then keep a reduced subprocess matrix for true end-to-end launch semantics
  - remaining subprocess smoke should prefer a shared built entrypoint or another lighter harness over repeated `ts-node/esm` cold boot when the same contract can be preserved
- Dependencies / Integrations:
  - `bin/codex-orchestrator.ts`
  - `scripts/run-review.ts`
  - `tests/cli-command-surface.spec.ts`
  - `tests/run-review.spec.ts`
  - any extracted test helpers needed to support in-process execution or shared subprocess harnesses
