## Architecture & Data
- Architecture / design adjustments:
  - build a maintenance decision layer on top of `runDocsFreshness(...)` report data and current git diff/task-id context
  - model debt classifications as data, not console strings
  - compute policy capacity status from configured rolling policy plus detected candidate cohorts
  - classify changed-path blockers by comparing current branch changes to freshness/spec failures and task packet paths
  - fail closed when no git base is available to prove the current branch diff is clean
  - keep owner action as an explicit recommended action when automation cannot safely create/update Linear
- Data model changes / migrations:
  - add a JSON decision schema for docs freshness maintenance artifacts
  - optionally add owner issue metadata in catalog policy only when it remains bounded and reviewed
- External dependencies / integrations:

  - Linear owner issue helper if owner creation/update is implemented directly
  - GitHub Actions for scheduled/preflight maintenance
