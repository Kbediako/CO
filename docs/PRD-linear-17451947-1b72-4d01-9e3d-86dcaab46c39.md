## Summary
- Problem Statement: CO-351 adopted the Codex CLI `0.125.0` app-server seam only for guarded local/control-host proof usage, while provider workers still intentionally rely on `codex exec` / `codex exec resume` supervision. CO-360 exists to prove whether provider-worker supervision can safely move to the app-server path before any default replacement.
- Desired Outcome: create an evidence gate that runs a configured `provider-linear-worker` canary under `runtime_mode=appserver`, proves sticky environments and persisted-turn resume/fork/pagination behavior, and keeps fail-closed `codex exec` / `codex exec resume` fallback with machine-readable reasons until parity is proven.

## User Request Translation
- User intent / needs: create the docs-first packet for CO-360 so implementation can prove provider-worker Codex `0.125.0` app-server supervision before replacing the existing exec/resume provider-worker control path.
- Success criteria / acceptance:
  - run a configured `provider-linear-worker` canary under `runtime_mode=appserver` with real turns and proof artifacts
  - prove sticky environment behavior with a real configured environment id, or record the exact blocker and retain fallback
  - prove real turn-backed pagination/resume/fork behavior on persisted turns, or record the exact blocker and retain fallback
  - preserve fail-closed fallback to `codex exec` / `codex exec resume` with machine-readable fallback reasons
  - surface selected runtime, app-server thread/turn ids, sticky env id, and resume/fork outcomes in manifests/status proof
  - add regression coverage for provider-supervision runtime selection and truth preservation
  - do not remove JSONL/session-log truth until parity is proven
- Constraints / non-goals:
  - do not claim blanket provider-worker app-server migration before the proof lane passes
  - do not make CO-358 a hard blocker unless cloud preflight is required for the exact canary path
  - this child lane does not touch implementation, test, Linear state, workpad, PR lifecycle, or full validation suites

## Not Done If
- The canary is not a configured `provider-linear-worker` run under `runtime_mode=appserver`.
- The canary uses synthetic turns where the issue requires real turns and persisted-turn proof.
- Sticky environment behavior lacks a real configured environment id and no exact blocker/fallback reason is recorded.
- Pagination/resume/fork behavior is not proven on persisted turns and no exact blocker/fallback reason is recorded.
- Fallback to `codex exec` / `codex exec resume` is removed, weakened, or represented only in prose.
- Manifests/status proof omit selected runtime, app-server thread/turn ids, sticky env id, resume/fork outcomes, or machine-readable fallback reasons.
- JSONL/session-log truth is removed before parity is proven.
- CO-358 is treated as a hard blocker without an exact canary-path cloud preflight dependency.

## Goals
- Establish a provider-worker app-server supervision proof gate for Codex CLI `0.125.0`.
- Preserve current exec/resume fallback until the proof lane passes.
- Record exact blockers when app-server parity cannot be proven.
- Require manifest/status proof that downstream reviewers can audit without trusting narrative summaries.
- Add regression coverage for runtime selection and truth preservation.

## Non-Goals
- No blanket provider-worker app-server migration before the proof lane passes.
- No removal of `codex exec` / `codex exec resume` fallback.
- No removal of JSONL/session-log truth before parity is proven.
- No cloud-preflight dependency on CO-358 unless the exact canary path requires it.
- No unrelated provider-worker, cloud, review-wrapper, or control-host refactor.
- No implementation or test changes in this docs-packet child lane.
