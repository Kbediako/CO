## Problem / Outcome
During `CO-392`, manual `codex-orchestrator start provider-linear-worker` launches could bypass root control-host semantics, producing wrong cwd/artifact truth or null manifest/provider-intake provenance. Operators need `control-host recover|relaunch|nudge --issue-id <id>` from the shared root. It must use root control-host state, deterministic issue worktree cwd, root `.runs` / `out`, manifest `provider_launch_source=control-host`, `provider_control_host_task_id`, `provider_control_host_run_id`, provider-intake `launch_source=control-host`, and non-null launch-token/intake reservation truth.

## Protected Contract
Preserve exact surfaces: `root control-host relaunch/recover/nudge`, `specific Linear issue`, `matching control-host provenance`, `actionable command hint`, `deterministic issue worktree cwd`, `provider-intake-state.json`, `CO-392`, `CO-244`, and `CO-189`. Reject broadening `CO-392`, redoing `CO-244`, weakening duplicate-worker/admission caps, hand-editing intake state, promoting direct issue-worktree starts, or requiring operators to memorize provenance env vars.
