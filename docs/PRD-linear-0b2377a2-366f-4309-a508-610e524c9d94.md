## Summary
- Problem Statement: During `CO-392` recovery, manual `codex-orchestrator start provider-linear-worker` launches could create provider-worker run truth outside root control-host launch semantics. One shape used the wrong workspace path and source-routed through another issue worktree; another direct issue-worktree launch fixed cwd/artifact routing but still missed control-host provenance, leaving same-issue child-lane helpers to fail later. The supported path is root control-host relaunch/recover/nudge for a specific Linear issue, but there is no operator-facing command or fail-closed direct-start guard that makes unsafe starts hard to misuse.
- Desired Outcome: Operators have one supported root control-host relaunch/recover/nudge path for a specific Linear issue. That path uses root control-host state, preserves deterministic issue worktree cwd, routes artifacts to root `.runs` / `out`, records manifest `provider_launch_source=control-host`, `provider_control_host_task_id`, `provider_control_host_run_id`, provider-intake `launch_source=control-host`, and non-null launch-token/intake reservation truth. Direct `codex-orchestrator start provider-linear-worker` outside matching control-host provenance fails closed with an actionable command hint before confusing run truth is created.
