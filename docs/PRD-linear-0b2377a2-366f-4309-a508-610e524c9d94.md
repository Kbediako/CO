## Problem
During `CO-392` recovery, direct `codex-orchestrator start provider-linear-worker` launches could bypass root control-host launch semantics. One launch recorded the wrong workspace path; another fixed cwd/artifact routing but still lacked control-host provenance. Provider-intake could show active worker truth while `launch_source` or launch-token truth was missing, and same-issue child lanes failed later.

## Desired Outcome
Operators use one supported root `control-host recover|relaunch|nudge --issue-id <id>` path for a specific Linear issue. The command uses root control-host state, deterministic issue worktree cwd, root `.runs` / `out` artifact routing, manifest `provider_launch_source=control-host`, `provider_control_host_task_id`, `provider_control_host_run_id`, provider-intake `launch_source=control-host`, and non-null launch-token/intake reservation truth. Direct provider-worker starts without matching control-host provenance fail closed with an actionable hint.

## Protected Contract
- Preserve exact surfaces: `root control-host relaunch/recover/nudge`, `specific Linear issue`, `codex-orchestrator start provider-linear-worker`, `matching control-host provenance`, `actionable command hint`, `deterministic issue worktree cwd`, root `.runs` / `out`, `provider_launch_source=control-host`, `provider_control_host_task_id`, `provider_control_host_run_id`, provider-intake `launch_source=control-host`, launch-token/intake reservation truth, `provider-intake-state.json`, `CO-392`, `CO-244`, and `CO-189`.
- Reject wrong interpretations: not `CO-392` stale pending-reopen work, not `CO-244` manifest-field introduction, not permission to weaken `CO-189` duplicate-worker protection or provider admission caps, not a manual `provider-intake-state.json` repair, and not a recommendation to use direct issue-worktree starts.

## Acceptance Criteria
- Supported root control-host recovery path exists for a specific Linear issue.
- Supported path uses root control-host state and deterministic issue worktree cwd.
- Supported path preserves root artifacts, manifest provenance, provider-intake `launch_source=control-host`, and launch-token/reservation truth.
- Direct `start provider-linear-worker` without matching provenance fails closed with a clear command hint.
- Regressions cover direct-start failure, supported recovery provenance, stale claim release on gone/out-of-scope issues, and child-lane compatibility.
- `CO-244`, `CO-392`, and `CO-189` adjacent invariants remain green.

## Not Done If
- A direct provider-worker start can still create wrong-root artifacts or null provenance without an actionable failure.
- Supported recovery loses root artifact routing, deterministic issue worktree cwd, manifest provenance, provider-intake launch source, or launch token truth.
- Child lanes still fail after supported recovery because provenance is incomplete.
- The fix broadens `CO-392`, redoes `CO-244`, weakens duplicate-worker/admission protections, or relies on hand-editing intake state.

## Fallback / Seam Decision
This lane touches fallback/seam behavior. Retain root control-host launch authority as a non-expiring supported contract because it owns admission, launch tokens, provenance, and run truth. Remove the unsafe direct-start fallback by failing closed when matching provenance is absent.
