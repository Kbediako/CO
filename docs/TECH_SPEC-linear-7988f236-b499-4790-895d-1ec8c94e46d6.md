## Summary
- Objective: Make `co-status` surface fresh `docs_freshness_maintain` repo-gate evidence from valid local/task-scoped maintenance reports instead of silently returning null when the scheduled report path is absent.
- Scope: `orchestrator/src/cli/control/docsFreshnessRepoGate.ts`, control/status projection consumers as needed, focused tests, and docs guidance for report search order.
- Constraints: Preserve strict docs freshness behavior, CO-522 ownership, provider WIP exclusion, and current control-host queue semantics.

## Issue-Shaping Contract
- User-request translation carried forward: Fix the root-cause visibility gap so future docs freshness blockers are visible to operators rather than hidden by stale/missing status-source assumptions.
- Protected terms / exact artifact and surface names: `co-status`, `repo_gates.docs_freshness_maintain`, `docs_freshness_maintain`, `out/local/docs-freshness-maintenance.json`, `out/docs-truthfulness-maintenance/docs-freshness-maintenance.json`, `source_path`, `generated_at`, `CO-522`.
- Nearby wrong interpretations to reject: stale-doc cleanup, cap widening, scheduled-output-only guidance, duplicate Linear issue creation, or changing provider WIP accounting.
- Explicit non-goals carried forward: no CO-530/CO-536 scope, no CO-522 owner re-home, no docs freshness policy relaxation.

## Technical Requirements
- Candidate report discovery must include:
  - explicit `reportPath` when supplied
  - `out/docs-truthfulness-maintenance/docs-freshness-maintenance.json`
  - `out/local/docs-freshness-maintenance.json`
  - active task-scoped output from `MCP_RUNNER_TASK_ID`, `CODEX_ORCHESTRATOR_TASK_ID`, `TASK`, or provider-intake claims
- Candidate selection must prefer the freshest valid report by `generated_at`; a stale scheduled report must not mask fresher local/task evidence.
- Candidate discovery must not scan arbitrary `out/*/docs-freshness-maintenance.json` paths, because unrelated historical task reports must not mask current missing/stale/invalid evidence.
- Freshness is explicit: reports older than `CODEX_DOCS_FRESHNESS_REPO_GATE_MAX_AGE_MS` (default 24 hours) are stale evidence, not current repo-gate truth.
- The normalized payload must preserve `severity`, `freshness_decision`, owner fields, spec guard, capacity, `next_expiry`, action count, blocking flags, `provider_wip_impact=excluded_repo_gate`, `sample_paths`, `source_path`, and `generated_at`.
- Missing, stale, and invalid report states must produce explicit evidence instead of silent `null`.
- Status projection/UI data must keep repo-gate evidence separate from provider-worker WIP.

## Validation Plan
- Focused repo-gate reader tests for scheduled missing with fresh local report, stale scheduled versus fresh local report, stale blocking versus fresh clean/warning report, task-scoped report precedence, and missing/invalid/stale evidence.
- Projection/UI tests proving `repo_gates.docs_freshness_maintain` is present and excluded from provider WIP.
- `node scripts/spec-guard.mjs --dry-run`
- `npm run build`
- `npm run docs:check`
- `npm run docs:freshness`
- `node scripts/docs-freshness-maintain.mjs --check --format json --warn`
- Manual `co-status --format json` proof with non-null `repo_gates.docs_freshness_maintain.source_path`.
