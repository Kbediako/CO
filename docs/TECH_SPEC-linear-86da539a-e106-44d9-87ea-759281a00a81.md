# TECH_SPEC - CO-395 review-wrapper fallback expiry

This mirror points to the canonical task spec at `tasks/specs/linear-86da539a-e106-44d9-87ea-759281a00a81.md`.

## Implementation Summary
- Preserve CO-395 as a docs-first `review wrapper` fallback expiry lane.
- Parent implementation should remove fallback paths that cannot preserve explicit scope, expire fallback metadata that remains temporary, and justify retaining fallback paths only when they are durable safety contracts with focused validation.
- Keep scope bounded to the existing standalone-review wrapper seams and direct test coverage.
- Do not weaken manifest-backed standalone review, `FORCE_CODEX_REVIEW` provider handoff, unrelated PR ready-review behavior, or telemetry semantics.

## Protected Surfaces
- scoped prompt retry
- synthesized title fallback
- generated fallback notes
- bounded-review fallback contracts
- review telemetry classification

## Fallback Expiry Decisions

| Fallback | Decision | Parent implementation guidance |
| --- | --- | --- |
| Retry without explicit scope after scoped prompt transport failure | remove fallback | Delete or fail closed if the retry cannot carry the same `--base`, `--commit`, or `--uncommitted` scope. |
| Synthesized title fallback / artifact-only scoped prompt fallback metadata | expire fallback if retained | Keep only with explicit expiry rationale and truthful launch-context telemetry. |
| Generated fallback notes | justify retaining fallback | Retain only as a bounded no-empty-context guard with generated-note markers and tests. |
| Command-intent bounded retry | justify retaining fallback | Retain as a durable bounded-review fallback contract because it preserves scope and fail-closed behavior. |
| Review telemetry classification | justify retaining fallback | Retain as durable audit classification for clean success, bounded success, failed boundary, prompt delivery, and reviewer-visible transport. |

## Validation Contract
- Child lane:
  - docs packet and registry mirror patch artifact only
  - JSON parse checks for edited registries
  - `node scripts/spec-guard.mjs --dry-run`
  - focused review-wrapper test coverage for direct-success and retry/fallback contracts
  - no full repo validation suites
- Parent lane:
  - docs-review before implementation
  - focused implementation tests for any touched wrapper seams
  - normal parent-owned validation, PR lifecycle, and Linear/GitHub integration
