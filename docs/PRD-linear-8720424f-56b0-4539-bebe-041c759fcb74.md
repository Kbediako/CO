## Summary
- Problem Statement: retained claims and released claims can still dominate a provider refresh cycle even when active provider-worker claims remain below `max_concurrent_agents`. If retained/released reconciliation spends the cycle on stale or released residue, `fresh_discovery` may not run, so unrelated `Ready` issue admission is starved even though global capacity remains. Operators can then misread the stall as a Linear label/filter problem, while the real issue is the refresh lifecycle getting stuck or over-focused on retained/released reconciliation.
- Desired Outcome: bound retained/released reconciliation so it cannot starve fair `fresh_discovery` under remaining capacity, preserve the `CO-160` / `CO-161` no-burn guarantees, and expose enough stuck phase/count diagnostics in `provider-intake-state.json` or CO STATUS for operators to distinguish refresh-starvation from eligibility or cooldown.

- Success criteria / acceptance:
  - per-refresh retained/released claim reconciliation is bounded and cannot consume the whole cycle while capacity remains

  - `dispatch_source_issue_by_id` is not reintroduced for retained released claims in the `CO-160` / `CO-161` all-released no-burn cases

## Parity / Alignment Matrix
- Current truth:
  - retained released claims already have local-first no-burn behavior from `CO-160`
  - `CO-161` separated released-only cached skip handling from unrelated `fresh_discovery` suppression
  - retained/released residue can still be present in `provider-intake-state.json` while active claims are below the provider concurrency cap
  - refresh health can surface `provider_refresh_lifecycle_stuck` / `restart_required`, but the operator-facing stuck reason lacks enough phase/count detail to identify whether admission was blocked by reconciliation, capacity, budget, or discovery
- Reference truth:
  - active/running claims consume provider capacity; released residue should not consume global admission capacity
  - retained released claims should stay local-first and zero-direct-read unless a bounded, explicit newer-evidence reopen path applies
  - `fresh_discovery` should admit unrelated `Ready` work when capacity remains and no budget/cooldown guard blocks the fetch
  - stale refresh diagnostics should identify the phase and counts that explain why admission did or did not happen
- Target truth / intended delta:
  - retained/released reconciliation gets a per-refresh bound or ordering rule that yields to fair `fresh_discovery` when global capacity remains
  - unrelated `Ready` issues discovered by `fresh_discovery` can be claimed without duplicating retained active claims or reviving released residue
  - the `CO-160` / `CO-161` no-burn behavior remains intact for retained released all-released cases
  - `provider-intake-state.json` or CO STATUS includes stuck phase/count diagnostics such as active count, max capacity, retained count, released count, direct-read count, and `fresh_discovery` attempted/skipped reason
- Explicitly out-of-scope differences:
  - label/filter repair
  - broad scheduler redesign
  - request-budget threshold redesign
  - killing or forcibly releasing active provider-worker claims
  - changing review/merge handoff behavior for adjacent issues

## Not Done If
- A refresh with capacity remaining can still spend the cycle on retained/released residue and skip unrelated `Ready` issue admission.
- `fresh_discovery` is suppressed solely because retained released claims were classified through local cached no-burn paths.
- Retained released claims regain `dispatch_source_issue_by_id` reads in the all-released deferred-poll or startup no-burn cases protected by `CO-160` / `CO-161`.
- A fresh unrelated `Ready` issue is not claimed in the two-active-claims plus one-free-slot regression shape.
- Stuck diagnostics still only say `provider_refresh_lifecycle_stuck` without enough phase/count evidence to distinguish reconciliation starvation from Linear eligibility/cooldown.
- The implementation solves the issue by changing labels, widening scheduler policy, or killing active provider workers.

## Goals
- Keep fresh `Ready` issue admission fair under remaining provider capacity.
- Bound retained/released claim reconciliation within each refresh cycle.
- Preserve local-first no-burn behavior for retained released claims.
- Expose stuck refresh phase/count diagnostics in durable operator-facing state.
- Preserve active provider workers and adjacent `CO-180` work while recovering admission.

## Non-Goals
- Redesigning provider scheduling, cap semantics, or Linear workflow policy.
- Changing Linear labels, project filters, or issue triage rules.
- Weakening shared-budget, cooldown, or `restart_required` safeguards.
- Reintroducing direct issue-by-id reads for locally terminal released claims.
- Editing implementation source or tests from this child docs lane.
- Running Linear mutation helpers from this child docs lane.
