# ACTION_PLAN - CO-356 archive automation Core Lane checks

## Summary
- Goal: make archive automation PRs produce the required PR-visible `Core Lane` context instead of leaving workflow-created PR heads with approval-required zero-job runs or detached dispatch-only success.
- Scope: docs-first packet, archive base workflow, core lane dispatch trigger, and focused workflow tests.
- Assumptions: branch protection requires the `Core Lane` job context only, and `workflow_dispatch` can be used as the safe explicit rerun path from the archive workflow.

## Issue Readiness Gate
- Intent checksum / protected terms carried forward: `automation/tasks-archive`, `Core Lane`, `action_required`, `PR #637`, `ARCHIVE_AUTOMERGE_TOKEN`, `.github/workflows/archive-automation-base.yml`, `.github/workflows/core-lane.yml`.
- Not done if: future archive PRs still miss required checks, PR `#637` is merged without green PR-visible `Core Lane`, dispatch failures are ignored, detached Core Lane success is not reflected on the PR head, or branch protection is weakened.
- Pre-implementation issue-quality review: parent verified live Linear state, PR attachment, no PR comments/reviews/review threads, no current checks, repeated historical `action_required` runs, and branch protection requiring `Core Lane`.

## Milestones & Sequencing
1. Create docs-first packet, registry row, task checklist, and workpad update.
2. Add `workflow_dispatch` support to `core-lane.yml` with archive PR audit inputs.
3. Add an archive-base post-PR step that dispatches `core-lane.yml` on the created/updated archive PR branch and fails closed on dispatch failure.
4. Wait for the dispatched run and publish a commit status named `Core Lane` on the archive PR head that mirrors success/failure.
5. Add focused workflow contract tests.
6. Run focused tests and docs/spec validation.
7. Refresh workpad, run standalone review and elegance review, update/open PR, attach PR, and run ready-review before review handoff.

## Dependencies
- GitHub Actions branch protection requiring `Core Lane`.
- GitHub CLI availability in hosted runners.
- Existing archive automation reusable workflow.

## Validation
- Checks / tests:
  - `npm run test -- tests/archive-automation-workflow.spec.ts`
  - `node scripts/spec-guard.mjs --dry-run`
  - `npm run docs:check`
  - broader repo gates before handoff as applicable
- Rollback plan: revert workflow and docs/test packet changes; archive PRs return to approval-required `pull_request` runs and must not be merged until a separate green required check path exists.

## Risks & Mitigations
- Risk: a dispatched Core Lane run could be treated as less PR-contextual than a native `pull_request` run.
- Mitigation: dispatch the same `core-lane.yml` job on the PR head branch, wait for the exact run, and publish the terminal result as the required `Core Lane` status context while preserving native PR-triggered workflows for non-automation PRs.
- Risk: dispatch silently fails and the PR remains blocked.
- Mitigation: make the dispatch step fail the archive workflow instead of hiding the problem.
- Risk: invalid auto-merge token still prevents automatic merge after checks pass.
- Mitigation: keep auto-merge token handling separate and record current `401 Bad credentials` as operational evidence; manual merge still requires green branch protection.
