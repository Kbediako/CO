## Technical Requirements
- Functional requirements:
  1. Parent implementation defines an explicit authority order for `co-status --format json` active-failure classification.
  2. A failed `provider manifest` / `outer provider manifest` with `Run provider linear worker` exit code `1` remains visible as diagnostic metadata.
  3. Terminal `worker proof` with `owner_phase=ended` and `owner_status=succeeded`, PR merged, Linear terminal closeout / `Done`, `retrying=0`, and no retry metadata reconciles the outer wrapper failure so it does not appear as current failed work.
  4. Missing, failed, stale, or non-terminal worker proof does not get reconciled as success.
  5. Active retry state or retry metadata keeps current failed/retrying work visible.
  6. Active failed workers remain visible in `co-status --format json`.
  7. Registry mirrors point to the CO-563 packet without changing unrelated task rows.
- Non-functional requirements:
  - keep the status-classification change narrowly scoped and auditable
  - preserve operator diagnostics for wrapper failure root-cause analysis
  - avoid broad changes to provider-worker lifecycle, Linear state, PR lifecycle, or retry orchestration
- Interfaces / contracts:
  - `co-status --format json` is the user-facing machine-readable status surface
  - provider manifest status is wrapper-level diagnostic evidence
  - `worker proof` is worker lifecycle authority when terminal and corroborated
  - PR merged, Linear `Done`, and inactive retry state are corroborating gates for reconciliation
