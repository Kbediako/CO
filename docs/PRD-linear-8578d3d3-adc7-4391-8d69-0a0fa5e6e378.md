## Summary
- Problem Statement: `operator-autopilot` already promotes the first safe `Backlog` head to `Ready` and already threads `expectedStateName`, `expectedStateType`, and `expectedUpdatedAt` into the shared Linear transition helper. However, if an operator manually demotes that same issue from `Ready` back to `Backlog`, the next autopilot cycle can infer that the issue is once again the first safe backlog head because backlog promotion currently evaluates live tracked-issue and claim truth only and does not preserve a bounded manual-demotion hold keyed to the newer `Backlog` snapshot. At the same time, autopilot action and observability payloads flatten transition output down to a narrow transition record, so the parent lane cannot currently show the relevant `issue_updated_at`, hold reason, or whether any force path was used when explaining a promotion or hold.
- Desired Outcome: prevent `Backlog` to `Ready` re-promotion churn after a manual `Ready` to `Backlog` demotion, while preserving normal safe backlog-head promotion when no manual-demotion signal exists, keeping backlog-promotion expected-state metadata at the shared transition-helper seam, and surfacing `issue updated_at`, hold reason, and force-path usage through observability.

## Goals
- Prevent immediate `Backlog -> Ready` re-promotion churn after a manual `Ready -> Backlog` demotion.
- Keep the hold bounded to a specific issue snapshot so future legitimate backlog evaluation remains possible.
- Preserve existing backlog-promotion expected-state inputs at the transition-helper seam instead of weakening them.
- Surface enough transition/hold metadata for operators to explain what happened from stored result and observability output.
- Keep the lane strictly narrower than scheduler redesign, generic transition-CAS work, or refresh-stall recovery.

- The next autopilot cycle still re-promotes the same issue immediately after a manual `Ready -> Backlog` demotion.
- The fix blocks normal safe backlog-head promotion when no manual-demotion signal exists.

## Technical Considerations
- Architectural Notes:
  - `maybeRunBacklogPromotion(...)` already passes `expectedStateName`, `expectedStateType`, and `expectedUpdatedAt` into `transitionProviderLinearIssueState(...)`.
  - `transitionProviderLinearIssueState(...)` already emits `transition_guard` with expected-state and force fields, so `CO-216` should keep using that shared seam rather than re-implementing generic transition logic.
  - `mapTransitionRecord(...)` currently narrows that output down to operator-facing state/update fields and force-path usage for the autopilot result path.
  - `runProviderOperatorAutopilot(...)` currently evaluates backlog promotion from live tracked issues and existing claims; `previous_result` is not currently used to suppress re-promotion after a newer manual demotion.
  - Prefer a bounded signal derived from existing autopilot result and live `updated_at` truth over a broad new backlog lifecycle subsystem.
