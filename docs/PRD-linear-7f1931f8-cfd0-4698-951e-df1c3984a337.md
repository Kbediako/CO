# PRD - CO STATUS: bound provider proof session-log refresh cost

## Summary
- Problem Statement: `CO-98` restored truthful provider-worker runtime telemetry for the root control-host read path, but repeated in-progress proof refreshes still rebuild session-log hydration from byte `0` for the same active `rollout-*.jsonl` file. On long-running workers, repeated `CO STATUS` refreshes therefore re-read the entire session log and repeat `O(file-size)` I/O and allocation work even though the rendered values remain correct.
- Desired Outcome: repeated in-progress proof refreshes for the same provider-worker session become incremental or equivalently bounded while preserving the authoritative runtime telemetry that `CO-98` restored for `Tokens`, `SESSION`, `Throughput`, and Codex `5-hour` / `weekly` rate-limit rendering.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): keep the live root control-host telemetry path truthful, but stop paying full-file reread cost on every in-progress provider-proof refresh. The fix has to stay on the authoritative proof/projection seam, not by slowing refresh cadence or accepting telemetry loss.
- Success criteria / acceptance:
  - repeated in-progress proof refreshes for the same active worker avoid rereading the same session log from byte `0`
  - the bounded refresh still preserves truthful `Tokens`, `SESSION`, `Throughput`, and Codex `5-hour` / `weekly` telemetry on root `CO STATUS`
  - focused tests cover repeated refreshes against a growing `rollout-*.jsonl` log and prove correctness is unchanged
- Constraints / non-goals:
  - no refresh-cadence-only mitigation
  - no heuristic tail read that can miss real telemetry updates
  - no regression of the truthful runtime telemetry restored under `CO-98`
  - no broad provider-proof schema churn unrelated to bounded session-log hydration

## Parity / Alignment Matrix
- Current truth:
  - `selectedRunProjection` refreshes in-progress provider proofs through `refreshProviderLinearWorkerProofSnapshot(...)`
  - session-log hydration reconstructs a fresh tail state at byte `0` on each refresh, so the same active `rollout-*.jsonl` file is reread repeatedly until all required telemetry fields are present
  - rendered telemetry remains correct today, but refresh cost scales with file size for long-running workers
- Reference truth:
  - the `CO-98` authoritative telemetry fix means root `CO STATUS` should continue reading truthful provider-worker runtime data from the proof/projection path
- Target truth / intended delta:
  - repeated in-progress proof refreshes reuse persisted hydration progress or an equivalent bounded strategy for the same session-log file, while still surfacing authoritative runtime telemetry updates promptly
- Explicitly out-of-scope differences:
  - browser or `/ui` redesign
  - synthetic or inferred telemetry placeholders
  - schema churn outside the bounded hydration state required for this optimization

## Not Done If
- `refreshProviderLinearWorkerProofSnapshot` still rereads the same session log from byte `0` on repeated in-progress refreshes for the same active worker.
- A long-running worker with a large `rollout-*.jsonl` file still causes repeated full-file reads during root control-host status refreshes.
- The optimization drops or delays truthful runtime telemetry that `CO-98` restored for `Tokens`, `SESSION`, `Throughput`, or Codex `5-hour` / `weekly` rate-limit segments.

## Goals
- Keep the root `CO STATUS` telemetry path authoritative.
- Bound repeated session-log hydration work for the same active provider-worker session.
- Fail closed to safe rereads when the session-log file rotates, truncates, or otherwise stops matching the persisted hydration state.
- Add focused regressions that lock both bounded rereads and telemetry correctness.

## Non-Goals
- Browser or `/ui` dashboard redesign.
- Synthetic or inferred telemetry placeholders.
- Broad provider-proof schema churn unrelated to bounded session-log refreshes.
- Changes that trade away telemetry correctness for lower I/O.

- Architectural Notes:

  - `refreshProviderLinearWorkerProofSnapshot(...)` currently rebuilds session-log hydration from scratch each time
  - the fix should persist only the minimum bounded hydration state needed to resume safely for the same `rollout-*.jsonl` file
  - when the active session-log path changes or becomes inconsistent, the implementation must fall back to a safe full reread
