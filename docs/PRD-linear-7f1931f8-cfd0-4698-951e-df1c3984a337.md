# PRD - CO STATUS: bound provider proof session-log refresh cost

## Traceability
- Linear issue: `CO-99` / `7f1931f8-cfd0-4698-951e-df1c3984a337`
- Linear URL: https://linear.app/asabeko/issue/CO-99/co-status-bound-provider-proof-session-log-refresh-cost
- Source issue: `CO-98` / `bbc5ad99-0806-4b13-a8fc-0b49b0e8a9bc`

## Summary
- Problem Statement: `CO-98` restored truthful provider-worker runtime telemetry for the root control-host read path, but repeated in-progress proof refreshes still rebuild session-log hydration from byte `0` for the same active `rollout-*.jsonl` file. On long-running workers, repeated `CO STATUS` refreshes therefore re-read the entire session log and repeat `O(file-size)` I/O and allocation work even though the rendered values remain correct.
- Desired Outcome: repeated in-progress proof refreshes for the same provider-worker session become incremental or equivalently bounded while preserving the authoritative runtime telemetry that `CO-98` restored for `Tokens`, `SESSION`, `Throughput`, and Codex `5-hour` / `weekly` rate-limit rendering.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): keep the live root control-host telemetry path truthful, but stop paying full-file reread cost on every in-progress provider-proof refresh. The fix has to stay on the authoritative proof/projection seam, not by slowing refresh cadence or accepting telemetry loss.
- Success criteria / acceptance: repeated in-progress proof refreshes stop rereading the same session log from byte `0`, still preserve truthful `Tokens`, `SESSION`, `Throughput`, and Codex `5-hour` / `weekly` telemetry on root `CO STATUS`, and are locked by focused growing-log regressions.
- Constraints / non-goals: no cadence-only mitigation, no heuristic tail read that can miss real telemetry updates, no regression of the truthful runtime telemetry restored under `CO-98`, and no broad provider-proof schema churn unrelated to bounded session-log hydration.

## Intent Checksum
- Exact wording / protected terms to preserve: `CO STATUS`, `provider-linear-worker-proof.json`, `refreshProviderLinearWorkerProofSnapshot`, `selectedRunProjection`, `rollout-*.jsonl`, authoritative runtime telemetry, incremental or bounded refresh hydration, `Tokens`, `SESSION`, `Throughput`, `5-hour`, `weekly`.
- Nearby wrong interpretations to reject:
  - "this is fixed by slowing down refresh cadence without changing reread behavior"
  - "a heuristic tail read is acceptable if it can miss real telemetry updates"
  - "performance work may regress truthful `Tokens`, `SESSION`, `Throughput`, or Codex `5-hour` / `weekly` rendering"

## Parity / Alignment Matrix
- Current truth: `selectedRunProjection` refreshes in-progress provider proofs through `refreshProviderLinearWorkerProofSnapshot(...)`, and session-log hydration reconstructs a fresh tail state at byte `0` on each refresh even though rendered telemetry remains correct.
- Reference truth: the `CO-98` authoritative telemetry fix means root `CO STATUS` should continue reading truthful provider-worker runtime data from the proof/projection path.
- Target truth / intended delta: repeated in-progress proof refreshes reuse persisted hydration progress or an equivalent bounded strategy for the same session-log file while still surfacing authoritative runtime telemetry promptly.
- Explicitly out-of-scope differences: browser or `/ui` redesign, synthetic or inferred telemetry placeholders, and schema churn outside the bounded hydration state required for this optimization.

## Not Done If
- `refreshProviderLinearWorkerProofSnapshot` still rereads the same session log from byte `0` on repeated in-progress refreshes for the same active worker.
- A long-running worker with a large `rollout-*.jsonl` file still causes repeated full-file reads during root control-host status refreshes.
- The optimization drops or delays truthful runtime telemetry that `CO-98` restored for `Tokens`, `SESSION`, `Throughput`, or Codex `5-hour` / `weekly` rate-limit segments.
