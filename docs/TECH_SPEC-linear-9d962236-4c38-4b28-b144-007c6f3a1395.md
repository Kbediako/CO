## Summary
- Objective: require a bounded automated-feedback drain between the latest PR update and the provider worker's `In Review` transition.
- Scope:
  - mode-aware extension of the shipped PR quiet-window monitor so pre-review handoff can reuse the current GitHub signal surface
  - provider-worker prompt + repo-local Linear skill updates to require the new drain before `In Review`
  - focused regressions for no-feedback, actionable-feedback-handled, and bounded-timeout behavior
- Constraints:
  - preserve the current merge-monitor contract for `pr resolve-merge`
  - keep the wait bounded and exit-code driven
  - record the current Symphony baseline explicitly; current Elixir workflow has a strong PR feedback sweep before `Human Review`, but no separate bounded quiet-window drain is documented there today
  - record delegation override explicitly because this worker run cannot spawn subagents

## Technical Requirements
- Functional requirements:
  - expose a shipped pre-review PR monitor mode that waits for a quiet window after the latest clean state without requiring human-review approval
  - require green required checks before the pre-review drain succeeds
  - keep draft PRs, do-not-merge labels, `CHANGES_REQUESTED`, unresolved review threads, actionable bot inline feedback, failing required checks, and active bot rereview signals as action-required blockers
  - do not treat GitHub `REVIEW_REQUIRED` as blocking for the pre-review drain
  - preserve a bounded timeout path so workers do not poll indefinitely before `In Review`
  - update provider-worker instructions so the new drain is mandatory after opening or updating a PR and before transition to `In Review`
- Non-functional requirements:
  - keep CLI help, exit codes, and monitor logs script-friendly and consistent with the current `pr` command family
  - reuse the current GitHub polling/query surfaces instead of introducing duplicate API logic
  - keep the change minimal and auditable across docs, worker prompt, and tests
- Interfaces / contracts:
  - PR monitor core: `scripts/lib/pr-watch-merge.js`
  - PR CLI surface: `orchestrator/src/cli/prCliShell.ts`
  - worker prompt contract: `orchestrator/src/cli/providerLinearWorkerRunner.ts`
  - repo-local workflow contract: `skills/linear/SKILL.md`

## Architecture & Data
- Architecture / design adjustments:
  - extract a mode-aware policy inside the existing PR monitor rather than creating a separate polling implementation
  - add a dedicated `pr` subcommand for pre-review handoff that reuses the current quiet-window loop with review-handoff-specific blocking rules
  - update worker/skill text to call the new subcommand before `In Review`
- Data model changes / migrations:
  - no persisted schema changes expected
  - monitor output may need mode-aware wording so logs and help text reflect review-handoff readiness rather than merge readiness
- External dependencies / integrations:
  - GitHub CLI (`gh`) for checks, comments, review threads, and review metadata
  - Linear issue workpad/state transitions remain worker-owned via the existing helper surface
  - Symphony baseline files noted above remain reference-only inputs for contract alignment
