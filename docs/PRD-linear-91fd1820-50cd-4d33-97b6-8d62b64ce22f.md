## Summary

CO-574 owns the repeated control-host/provider-admission failure seen while CO-575 and CO-574 were promoted into active work: explicit `control-host nudge` calls produced accepted provider claims with no `launch_token`, no `run_id`, and no `run_manifest_path`, then returned `provider_issue_start_blocked:max_concurrency` or `provider_worker_recover_no_launch_evidence` instead of a durable launch, retry, release, or truthful pending acknowledgement.

The product requirement is a deterministic recovery contract for provider-worker admission. Operators must be able to tell whether a recovery is queued, capacity-blocked, launched, failed, or released without cycling Linear states or manually editing provider-intake state.

The `/ui/machine-status.json` endpoint is part of that operator contract. It must return bounded current or explicitly degraded JSON even if refresh/reconcile state is in flight, rather than stalling long enough to trigger same-endpoint timeout churn.

## User Intent

- Fix the root cause of repeated orchestration failures instead of patching around them with Backlog/In Progress churn.
- Preserve the shared-root posture: clean latest `main`, no stale branch workers, no direct `provider-intake-state.json` edits.
- Keep WIP governed and do not weaken docs freshness, provider-worker state semantics, or review gates.
- Use GPT-5.5/xhigh-backed review/research evidence and GPT Pro advisory for repeated issues.

## Scope

- Provider-worker recovery acknowledgement semantics for accepted pending-revalidation claims without launch evidence.
- Provider admission/recovery behavior for manifestless claims that are executable but have no launch identity.
- Control-host machine-status/probe timeout behavior only where it directly amplifies stale owner, endpoint stalls, or admission churn.
- Focused regression coverage for the CO-574/CO-575 incident shape.

## Non-Goals

- No broad provider-worker rewrite.
- No manual provider-intake cleanup as the fix.
- No WIP cap removal or "launch anyway" bypass.
- No docs freshness owner re-home work; CO-575 remains the docs freshness lane and is blocked by this control-host issue.
- No timeout-only masking of unhealthy control-host probes.

## Acceptance Criteria

- Explicit recover/nudge for an accepted pending-revalidation claim with no launch evidence returns queued/pending or a completed handoff result, not a terminal `provider_worker_recover_no_launch_evidence` observation.
- Manifestless pending-revalidation claims preserve a stable age anchor so repeated rehydrate/probe churn cannot keep them fresh forever.
- `/ui/machine-status.json` returns a bounded current dataset or explicit `machine_status_degraded` fail-closed JSON when the read path stalls.
- Capacity-blocked recovery records truthful durable state and does not silently refresh into accepted/no-run without retry or recovery evidence.
- Control-host status/probe failures with active workers are classified as degraded or fail-closed without restart churn that strands worker ownership.
- Regression tests cover the no-run accepted acknowledgement, stale-age preservation, capacity-blocked retry/pending semantics, and active-worker probe timeout classification.
