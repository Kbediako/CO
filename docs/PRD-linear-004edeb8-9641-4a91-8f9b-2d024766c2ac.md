## Summary
`CO-208` closes a diagnostics/reporting gap exposed by `CO-207`: cloud connector/auth drift can currently look like generic cloud setup, cloud denial, or provider runtime failure, even when the actionable condition is `missing_github_connector_link` with the safe user-facing signal `GitHub connection not found for user`. The desired outcome is a machine-readable diagnostic contract across `scripts/cloud-canary-ci.mjs`, cloud failure diagnostics, and provider-worker proof so cloud-canary gates can fail closed with the right cause and remediation.

This lane does not repair a live GitHub connector, provision cloud auth, mutate Linear, or change cloud admission policy. Parent implementation owns source/test work and any live verification after this docs packet is accepted.

## User Request Translation
- Create the docs-first packet and registry mirrors for `CO-208`.
- Frame the issue as diagnostic classification and reporting drift exposed by `CO-207`, not as live connector repair.
- Preserve exact protected terms: `missing_github_connector_link`, `GitHub connection not found for user`, `CODEX_CLOUD_ENV_ID`, cloud-canary gates, and provider-worker proof.
- Define acceptance criteria and validation for machine-readable diagnostic value across `scripts/cloud-canary-ci.mjs`, cloud failure diagnostics, and provider-worker proof.
- Make explicit that configured `CODEX_CLOUD_ENV_ID` is not sufficient readiness proof when connector/auth state is missing or stale.
- Exclude tokens, cookies, emails, org ids, connector secrets, and raw account identifiers from docs, fixtures, logs, and proof artifacts.

## Intent Checksum
- Protected terms and surfaces: `missing_github_connector_link`, `GitHub connection not found for user`, `CODEX_CLOUD_ENV_ID`, cloud-canary gates, provider-worker proof, `scripts/cloud-canary-ci.mjs`, cloud failure diagnostics, cloud required/fallback canaries, manifest/run-summary diagnostics, and provider worker closeout evidence.
- Nearby wrong interpretations to reject: live GitHub connector repair, OAuth re-link automation, token refresh, secret discovery, broad cloud runtime redesign, generic `CODEX_CLOUD_ENV_ID` readiness, Codex version promotion, and treating all cloud failures as environment configuration.
- Exact diagnostic split required: missing environment, missing connector link, cloud denial, quota/auth/rate-limit, and provider/runtime failure must remain separable.

## Acceptance Criteria
- Docs packet and registry mirrors exist for the listed `CO-208` files.
- Parent implementation classifies `missing_github_connector_link` distinctly from missing `CODEX_CLOUD_ENV_ID`, cloud denial, quota/auth/rate-limit, and provider/runtime failure.
- The safe signal `GitHub connection not found for user` is preserved as redacted diagnostic detail when it appears.
- `scripts/cloud-canary-ci.mjs` and cloud failure diagnostics produce stable machine-readable output that cloud-canary gates and parent/provider closeout can consume.

- Required/fallback cloud canary outcomes stay fail-closed when connector/auth drift blocks execution.
- Fixtures or focused tests prove the connector-missing class and contrast it with missing env, cloud denial, and provider runtime failures.
