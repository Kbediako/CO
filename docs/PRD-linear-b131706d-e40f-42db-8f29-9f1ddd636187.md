# PRD - CO-392 live-started released-pending-reopen refresh

## Added by Bootstrap 2026-04-26

## Summary
- Problem Statement: a no-run provider intake claim for `CO-379` can remain `released` with reason `provider_issue_released_pending_reopen:provider_issue_released:not_active` and stale cached `Ready/unstarted` metadata even after live Linear truth advances to `In Progress/started`.
- Desired Outcome: an accepted and drained control-host refresh must refresh/reclassify/reclaim that stale claim, launch or requeue through the normal admission path when eligible, or record a concrete recovery reason instead of silently preserving stale released truth.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): fix the control-host/provider-intake refresh path so stale released-pending-reopen claims whose live Linear issue has already moved into started work cannot hide from admission/status while capacity is available.
- Success criteria / acceptance: add the CO-379-shaped regression, update claim metadata from live `In Progress/started`, recover via the normal admission path or actionable diagnostics, preserve CO-193 and CO-189 safeguards, and keep CO STATUS active/running/admission truth aligned after recovery.
- Constraints / non-goals: do not hand-edit `provider-intake-state.json`, weaken provider admission caps, duplicate same-issue workers, broaden into CO-391 projection timeouts, or reintroduce unbounded issue-by-id polling for retained released claims.

## Not Done If
- A no-run `released-pending-reopen` claim can still keep stale `Ready/unstarted` metadata after live Linear moves the same issue to `In Progress/started`.
- `POST /api/v1/refresh` can drain successfully while leaving the claim stale with no diagnostic or recovery reason.
- `CO STATUS` omits a live started issue from active/running/admission truth while provider capacity remains available.
- The fix only masks display output without correcting provider intake refresh/reclaim behavior.
- The fix duplicates a same-issue worker or weakens existing CO-193 / CO-189 safety coverage.

## Goals
- Refresh stale no-run released-pending-reopen metadata when live Linear truth is newer and started.
- Reclassify or reclaim the same issue through existing provider admission paths when no duplicate-worker veto exists.
- Surface a deterministic recovery reason when the claim cannot be safely launched.
- Keep CO STATUS provider-intake summary and selected projection consistent with the repaired intake truth.

## Non-Goals
- No provider cap relaxation.
- No manual state-file cleanup as the product fix.
- No Linear workflow mutation to move issues backward.
- No broad CO-391 status timeout or stale terminal suppression work.
- No unbounded issue-by-id polling for all retained released claims.
