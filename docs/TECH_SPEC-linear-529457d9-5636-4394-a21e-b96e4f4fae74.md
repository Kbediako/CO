## Summary
- Objective: harden provider-worker and control-host recovery so missing control-host manifests, shared-budget cooldown pressure, and restart-time rehydrate do not corrupt local claim truth or block automatic reclaim.
- Scope:
  - durable control-host discovery for provider-worker refresh requests
  - cooldown recovery that resumes truthful intake after suppression expires
  - truthful rehydrate and reclaim of active or resumable runs only
  - focused regression coverage for the `CO-78` recovery class
- Constraints:
  - keep the provider lifecycle model intact
  - preserve fail-closed behavior for untrusted endpoints and ambiguous liveness
  - avoid unrelated control-host or status-surface redesign

## Technical Requirements
- Functional requirements:
  - provider-worker refresh requests must no longer require one exact control-host manifest path in order to recover
  - shared-budget cooldown expiration must allow local issue reconciliation to resume without manual host replacement
  - claim rehydrate must mark claims active only when the run is demonstrably alive, or resumable only when a legitimate resume path exists
  - eligible live issues must be reclaimable automatically after cooldown or restart
  - regression coverage must cover missing-manifest refresh, shared cooldown, restart or rehydrate, and truthful reclaim
- Non-functional requirements (performance, reliability, security):
  - retain allowed-bind-host validation and endpoint auth checks for control-host refresh requests
  - avoid fabricating active state when run liveness is ambiguous
  - keep recovery deterministic and auditable through existing manifests and intake-state artifacts
- Interfaces / contracts:
  - provider-worker refresh requester: `orchestrator/src/cli/providerLinearWorkerRunner.ts`
  - claim rehydrate lifecycle: `orchestrator/src/cli/control/providerIssueHandoff.ts`
  - shared cooldown persistence and suppression: `orchestrator/src/cli/control/linearBudgetState.ts`
  - tracked-issue resolution: `orchestrator/src/cli/control/linearDispatchSource.ts`

## Architecture & Data
- Architecture / design adjustments:
  - prefer durable control-host discovery from persisted locator, workspace path, or endpoint/auth evidence over a single canonical manifest-path assumption
  - keep rehydrate classification tied to discovered live/resumable runs instead of stale non-terminal manifests
  - ensure cooldown clearing leads back into the normal refresh or reconciliation path without manual host replacement
- Data model changes / migrations:
  - no schema or migration changes intended
  - existing `provider-intake-state.json`, run manifests, proof sidecars, and shared Linear budget state remain the authority inputs
- External dependencies / integrations:
  - local control-host endpoint/auth files
  - provider-worker proof snapshots
  - shared Linear budget status file and live issue reads

## Parity / Alignment Matrix
