## Summary
`CO-212` closes the remaining Ready reclaim gap for plain `released` / `provider_issue_released:not_active` claims that still retain completed blocker history. A cached CO-196-like claim can keep `state=released`, `reason=provider_issue_released:not_active`, cached `issue_state=Blocked`, stale `issue_updated_at`, and `issue_blocked_by=[CO-207 Done/completed]` even after the live tracked issue is `Ready` / `unstarted` with newer `updated_at=2026-04-17T03:33:27.936Z` and no non-terminal blockers. The desired outcome is a minimal reclaim fix through the normal control-host path: completed blockers stay preserved for audit, but they no longer suppress eligibility once current live issue truth says the issue is runnable again.

## Acceptance Criteria
- Add a regression with cached CO-196-like claim: `state=released`, `reason=provider_issue_released:not_active`, cached `issue_state=Blocked`, stale `issue_updated_at`, and `issue_blocked_by=[CO-207 Done/completed]`.
- Add a retained-terminal-run variant where the old `provider-linear-worker` manifest is `succeeded`, started after the Ready transition, and has no `issue_updated_at`, proving fresh Ready truth still reclaims.
- Live tracked issue is `Ready` / `unstarted`, newer `updated_at=2026-04-17T03:33:27.936Z`, and has no non-terminal blockers.
- Refresh/fresh discovery persists fresh Ready metadata and launches/reclaims through the normal control-host path without manual worker start.
- Completed blockers remain preserved for audit but do not suppress eligibility.
- Focused tests cover provider handoff and CO STATUS/runtime projection.

## Non-Goals
- No manual worker start as the fix.
- No deletion of `provider-intake-state.json` history or `issue_blocked_by` audit evidence.
- No broad rewrite of blocker semantics, workflow-state naming, or scheduler behavior.
- No status-only/UI-only repair that leaves reclaim suppressed.
- No child-lane code or test edits.

## Not Done If
- A cached plain `released` / `provider_issue_released:not_active` claim still suppresses a live `Ready` / `unstarted` issue only because its retained blockers are already completed.
- Live issue truth is newer and blocker-free, but refresh/fresh discovery still does not reclaim through the normal control-host path.
- Completed blockers disappear from audit history instead of remaining preserved.
- The fix depends on manual worker start, manual state cleanup, or broad workflow-state changes.
- Focused provider handoff and CO STATUS/runtime projection coverage do not exist for the issue shape.

## Metrics & Guardrails
- Primary success metrics:
  - the cached completed-blocker claim shape is reclaimable through normal control-host flow once live issue truth is Ready/unstarted and newer
  - completed blockers remain available in audit surfaces
  - focused provider handoff and runtime/status projection tests cover the shape
- Guardrails:
  - non-terminal blockers must remain blocking
  - audit history must not be deleted or silently rewritten away
  - no manual worker launch is required to satisfy the fix
  - status coverage must stay consistent with handoff/reclaim truth

## User Experience
- Operators can move an issue back to `Ready` and expect normal control-host reclaim when the only remaining blocker evidence is already completed.
- Audit readers can still see which blocker previously held the issue without treating that history as current suppression.
- Parent maintainers can validate reclaim truth and `CO STATUS` truth from the same focused regression slice.

## Technical Considerations
- The minimal fix must stay within the protected seams: `providerIssueHandoff.ts` for reclaim/persist logic, `providerLinearWorkflowStates.ts` for workflow/blocker classification, `provider-intake-state.json` for retained audit history, and `co-status --format json` runtime/status projection coverage.
- Completed blocker history should be treated as audit evidence after live issue truth is refreshed, not as an eligibility veto by itself.
- Fresh Ready metadata should be persisted before reclaim/launch so later status/debug surfaces reflect the corrected truth.
- The runtime/status coverage should prove the reclaim shape without widening into a general workflow-state redesign.
