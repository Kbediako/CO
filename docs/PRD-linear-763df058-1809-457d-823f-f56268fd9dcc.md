## Summary
- Problem Statement: current CO provider-worker launches inherit the repo-wide Codex baseline reasoning posture instead of selecting reasoning per issue or per run. The shipped baseline remains explicit and global: `codexDefaultsSetup.ts` writes `model_reasoning_effort = "xhigh"`, `AGENTS.md` says CO defaults to `xhigh`, and `providerLinearWorkerRunner.ts` launches the worker by resolving runtime mode plus the ordinary Codex command without any issue-level reasoning contract. At the same time, the live CO Linear team still reports `issueEstimationType = "notUsed"`, `linearDispatchSource.ts` does not read or model estimate/risk in tracked-issue resolution, and current provider-worker token telemetry is opportunistic proof hydration rather than a scheduler-truth input.
- Desired Outcome: complete `CO-17` as a docs-first evaluation lane that answers two concrete questions truthfully: dynamic provider-worker reasoning selection is not worth shipping now, but it is worth preserving as a bounded future experiment once truthful token telemetry and explicit issue/run override metadata exist. The packet should keep docs-first authoring on the highest reasoning setting, reject Linear estimate as a sole or current routing signal, require manifest-backed per-run overrides for any future pilot, and avoid changing dispatch ordering in this lane.
