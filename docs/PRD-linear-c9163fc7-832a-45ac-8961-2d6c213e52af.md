## Summary
`CO-203` is the safety follow-up to `CO-202`: a plain `provider_issue_released:not_active` claim can retain historical `run_id` / `run_manifest_path` while `resolveProviderReleaseRun(...)` returns `null` because the retained run record disappeared or the retained manifest is unreadable. Today that unresolved retained identity keeps the issue excluded from `fresh_discovery` even when the old worker is no longer live. This lane should reclaim only after machine-checkable evidence proves there is no live same-issue worker and no release cancel pending.

## User Request Translation
- Define the evidence contract for the missing/disappeared retained-run shape instead of treating every `run === null` released claim as reclaimable.
- Allow `fresh_discovery` to reclaim eligible plain `provider_issue_released:not_active` rows only after no-live-worker proof passes.
- Preserve `CO-189` duplicate-worker protection, unreadable-manifest occupancy blocking, and pending release-cancel blocking.
- Keep pending-reopen missing-manifest rows excluded.
- Do not solve the incident by manually deleting `provider-intake-state.json` rows or by using global capacity as a proxy for same-issue liveness.

## Acceptance Criteria
- Define the evidence contract for proving a retained run identity with missing or unreadable manifest has no live same-issue worker.
- Keep unreadable-manifest live proof or equivalent worker-host evidence as a hard fresh-discovery blocker.
- Allow fresh discovery for eligible plain `provider_issue_released:not_active` rows only after the no-live-worker proof passes.
- Preserve pending-reopen missing-manifest exclusion.
- Add focused regressions for both safe reclaim and duplicate-worker prevention.

## Non-Goals
- No broad Linear polling cadence or request-burn change.
- No terminal Done/cancelled projection cleanup change.
- No weakening of CO-189 active same-issue worker protection.
- No deletion of provider-intake recovery evidence.
- No reopening of the already-landed `CO-202` stale cached Blocked reclaim scope.

## Not Done If
- A retained `run_id` / `run_manifest_path` with a missing manifest can launch a duplicate worker while an old same-issue process is still live.
- A missing or disappeared run record remains permanently suppressed even after machine-checkable no-live-worker evidence exists.
- Pending-reopen missing-manifest rows are treated as equivalent to plain `provider_issue_released:not_active` rows.
- The fix depends on manually deleting `provider-intake-state.json` entries.
