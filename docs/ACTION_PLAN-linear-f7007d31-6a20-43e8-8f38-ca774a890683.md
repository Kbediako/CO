# ACTION_PLAN - CO-572 current-endpoint status-machine split

## Summary
- Goal: remove the dashboard-projection dependency from machine health so `co-status --format json` and supervision can report fresh zero-WIP health while preserving fail-closed behavior for real failures.
- Scope: docs-first packet, cheap machine status contract, co-status degraded current-endpoint behavior, supervisor probe decoupling, progress-field propagation, dashboard degraded classification, focused regression tests, validation, review, and PR handoff.
- Assumptions:
  - current shared root stays on latest `main`
  - no active GitHub PRs or started/review WIP lanes are currently competing for capacity
  - GPT Pro/local advisory evidence is design input; deterministic local tests and live artifacts remain authoritative

## Issue Readiness Gate
- Intent checksum / protected terms carried forward: `co-status --format json`, `/ui/data.json`, `same-endpoint current-endpoint timeout`, `probe_failed`, `probe_timeout`, `provider-intake-state.json`, `polling.restart_required=false`, `polling.stuck=false`, `active_claims=[]`, `progress_updated_at`, `machine status endpoint`, `readCompatibilityProjection()`.
- Not done if:
  - zero-WIP fresh provider polling plus same-endpoint dashboard timeout still exits nonzero
  - dashboard-only slowness can still trigger supervisor restart/quarantine
  - degraded status hides auth, stale source, dead endpoint, true stuck, active no-progress, or failed run evidence
  - dashboard partial/error output lacks explicit degraded metadata
- Pre-implementation issue-quality review:
  - CO-572 is the right owner because CO-571 is merged and fixed the provider progress-stall path, while the recurrence is now status/probe coupling to heavy `/ui/data.json`
  - CO-407/CO-376/CO-336 are historical adjacent owners but do not cover zero-WIP same-endpoint timeout with fresh polling
  - Linear labels and related issue links were verified after creation
- Fallback / refactor decision:
  - Yes, this touches fallback/seam behavior.
  - Remove fallback: `/ui/data.json` as machine health authority.
  - Remove fallback: active-claim-only degraded current-endpoint fallback.
  - Justify retaining fallback: source-labeled degraded machine status when cheap status proves freshness.
  - Remove fallback: supervisor dependence on heavy dashboard.
  - Expire fallback: unbounded dashboard response behavior if not fully fixed in first PR.
- Durable retention evidence: source-labeled degraded machine status is retained as a supported operator mode because it exposes partial machine truth and the dashboard failure reason without fabricating a full UI dataset.
- Large-refactor check: the right refactor boundary is a status-machine split. A timeout bump or another minor branch is rejected. If the work must split, PR 1 must deliver machine status plus `co-status` plus supervisor decoupling; dashboard optimization may follow under CO-572.
