## Summary
`CO-217` hardens the post-`CO-196` release posture so `npm run pack:smoke` and the relevant CI/release workflows cannot silently miss Codex marketplace install/register coverage. `CO-196` is treated as the local validation proof this issue builds on, not something to replace. The new policy is: marketplace smoke is mandatory by default, and any skip must be an explicit env/flag opt-out with a documented reason and allowed-use boundary.

## User Request Translation
- After `CO-196` has landed, make sure `npm run pack:smoke` exercises Codex marketplace install/register coverage instead of only proving older packaged CLI/skill behavior.
- Ensure CI and release workflow posture cannot report green marketplace readiness when marketplace coverage did not actually run.
- Document the required decision: marketplace smoke is mandatory by default, and may be skipped only through an explicit env/flag opt-out with documented allowed use and evidence.
- Preserve `Codex 0.121+ marketplace support` as the compatibility boundary for this coverage.
- Keep this lane docs-first only; parent owns implementation, focused validation, workflow edits, Linear state, workpad, PR, and merge.

## Acceptance Criteria
- `npm run pack:smoke` includes Codex marketplace install/register coverage by default in the packaged downstream simulation after `CO-196`.
- Marketplace smoke may be skipped only with an explicit env/flag opt-out and documented allowed-use reason; silent omission is not accepted.
- Relevant CI/release workflows fail or report non-green/waived evidence when marketplace coverage does not run, instead of reporting generic pack-smoke success as marketplace proof.
- Release-facing workflows do not use the opt-out to claim marketplace coverage; any emergency or unsupported-version skip remains visibly non-coverage evidence.
- The implementation preserves the local validation proof added by `CO-196` and does not replace it with weaker pack-smoke-only proof.
- The hardening remains limited to marketplace pack-smoke and workflow posture; no broad release-pipeline, guardrail, delegation, or runtime-mode refactor is required.

## Non-Goals
- No broad release-pipeline refactor.
- No guardrail, delegation, or runtime-mode behavior changes.
- No replacement of the local validation proof that `CO-196` adds.
- No Codex CLI adoption/promotion decision beyond the protected `Codex 0.121+ marketplace support` boundary.
- No implementation, test, package, or workflow edits in this docs-only child lane.

## Not Done If
- `npm run pack:smoke` can pass without marketplace install/register coverage and without an explicit allowed opt-out.
- A CI/release lane can silently skip marketplace smoke and still present its result as full marketplace coverage.
- The opt-out lacks a machine-checkable reason or documented allowed-use boundary.
- The fix broadens into release orchestration, guardrail/delegation/runtime changes, or Codex version adoption.
- The parent solution replaces or weakens the `CO-196` local validation proof.

## Metrics & Guardrails
- Primary success metrics:
  - pack-smoke output or artifacts make marketplace install/register execution visible
  - CI/release workflows make marketplace smoke mandatory by default or visibly waived
  - skip evidence includes the chosen explicit env/flag and reason
- Guardrails:
  - preserve `CO-196` as the local validation reference
  - keep release workflow changes narrow and evidence-driven
  - avoid hiding unsupported Codex version behavior behind green marketplace coverage

## User Experience
- Release operators can trust `npm run pack:smoke` and CI/release summaries to say whether marketplace install/register coverage actually ran.
- Reviewers see an explicit marketplace smoke result or an explicit allowed skip, not an ambiguous generic pack-smoke pass.
- Downstream users keep the stronger `CO-196` local validation proof plus package-level smoke coverage for marketplace install/register behavior.

## Technical Considerations
- The parent implementation likely touches the existing pack-smoke harness and whichever CI/release workflows currently gate downstream package publication or backstop pack smoke.
- The marketplace smoke should run from the packaged downstream simulation, not a source-checkout-only path, so it proves the npm package actually carries enough marketplace support.
- If the parent introduces an opt-out, it should require reason text and produce machine-readable evidence in logs/artifacts.
