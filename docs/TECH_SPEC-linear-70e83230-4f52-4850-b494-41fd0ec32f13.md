## Summary
- Objective: add one guarded resident-session restart seam above the existing worker-lifetime continuation loop, so restart-safe completed-turn boundaries can drain, restart, and resume the same logical resident thread/session lineage truthfully.
- Scope:
  - derive a restart-safe resident continuity envelope from the prior provider proof when the previous worker ended at an allowed completed-turn boundary
  - pass that continuity envelope through the control-host launch seam into the next `provider-linear-worker`
  - let the restarted worker resume the prior resident thread on its first turn and preserve logical continuity metadata in proof/read-model state
  - keep unsafe exits fresh-start or fail-closed
  - document restart-boundary reload semantics without claiming hot upgrade
- Constraints:
  - no mid-turn hot reload or in-memory state preservation
  - no remote command-authority expansion
  - no unrelated telemetry or distributed-worker work

## Implementation Boundary
- Continuity source:
  - derive resident continuity only from a fresh provider proof at restart-safe completed-turn boundaries
  - preserve fail-closed behavior when the previous proof is stale, malformed, or ended unsafely
- Control seam:
  - extend the provider launch handoff with additive resident continuity metadata
  - keep the start/resume split truthful; this lane does not redefine top-level orchestrator `resume(runId)`
- Worker behavior:
  - on a guarded resident resume, the next provider-worker uses the prior `thread_id` on its first turn instead of starting a brand-new resident thread
  - preserve a stable logical session identifier and monotonic logical turn count across guarded worker restarts
- Observability:
  - surface resident continuity metadata in proof/read-model state so operators can distinguish fresh starts from guarded resumes

## Design
- Guarded resident continuity contract:
  - source run id
  - source thread id
  - source turn count
  - restart count
  - boundary reason
  - stable logical session id
- Safe boundary rule:
  - initial implementation only trusts explicit completed-turn boundaries proven by provider proof
  - unsafe exits remain fresh-start or retry-only and do not claim resident continuity
- Reload truth:
  - config or policy changes take effect on the next supervised worker restart boundary
  - docs must state that this is rolling restart/resume, not live hot upgrade
