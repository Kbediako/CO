## Summary
`CO-204` fixes misleading `CO STATUS` progress projection after a parent invalidates a failed child lane. Historical child-lane evidence must remain available in `provider-linear-worker-child-lanes.json` and proof artifacts, but an unchanged old child-lane `summary` must not become current progress merely because the parent wrote a newer `decision_at`.

## User Request Translation
- Stop `co-status --format json` / `provider-linear-worker-progress` from presenting stale failed child-lane summaries as current worker progress after `decision=invalidated`, `decision=rejected`, or `decision=accepted`.
- Keep active or pending child-lane summaries visible when they are still current work.
- Preserve historical child-lane records and proof artifacts.
- Keep the fix scoped to status projection, not the separate `spec-guard command not found` child-lane failure.

## Parity / Alignment Matrix

| Surface | Current | Target |
| --- | --- | --- |
| Invalidated child-lane summary | A failed lane keeps its old `summary`, and `decision_at` can make that old summary look fresh. | Disposed lanes remain historical records but do not provide active `child_lane_summary` progress candidates. |
| Child summary freshness | Candidate ranking uses `latest(decision_at, launched_at)`, conflating child output and parent disposition. | Current child-lane summary candidates use child-lane output proxy timing and do not use parent disposition timestamps as message freshness. |
| Replacement lane | A replacement child lane eventually outranks the old failed summary once visible. | Pending or replacement child lanes dominate disposed stale summaries immediately when they are current. |
| Active/pending child failures | Failed but still pending child lanes need truthful operator visibility. | Active or pending child-lane summaries remain visible until the parent disposition supersedes them. |
| Historical evidence | Proof artifacts retain all lane records. | Proof artifacts continue to retain all lane records; only current progress projection changes. |

## Acceptance Criteria
- `deriveProviderLinearWorkerProgressSnapshot(...)` or its projection helpers distinguish child-lane output timestamps from parent disposition timestamps.
- Invalidated, rejected, and accepted child-lane summaries do not rank as current `child_lane_summary` progress unless a future explicit disposition-history surface renders them separately.
- A newer pending or replacement child lane dominates an older invalidated failed lane in the projected progress snapshot.
- Focused tests cover CO-204's source CO-203 artifact sequence: failed child lane, parent invalidation, stale failure no longer selected, and replacement or current worker progress selected.
- Existing useful child-lane summary behavior for active or pending child lanes remains intact.

## Non-Goals
- No fix for the original `provider-linear-child-lane` `spec-guard command not found` failure.
- No change to CO-202 or CO-203 provider-intake reclaim behavior.
- No change to Linear issue lifecycle mutation rules for child lanes.
- No broad CO STATUS dashboard layout or polling redesign.
- No deletion of historical child-lane records from proof artifacts.

## Not Done If
- Invalidating a failed child lane still causes `CO STATUS` or `co-status --format json` to present the old child failure as current worker progress.
- An older invalidated child lane can outrank a newer replacement child lane or parent progress message solely because `decision_at` is newer than the stale child summary.
- The fix conflates this status projection bug with the separate `spec-guard command not found` seam.
- Historical child-lane evidence disappears instead of being separated from current progress.
- There is no focused regression for failed child lane, parent invalidation, replacement lane, and current progress projection.
