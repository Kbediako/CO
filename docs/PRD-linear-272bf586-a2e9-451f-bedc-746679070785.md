## Summary
- Problem Statement: `CO-26` shipped a live terminal `CO STATUS` surface, but it only reused the redraw loop and shared dataset seam. The operator-visible frame is still materially thinner than Symphony's `StatusDashboard`: plain-text headings instead of terminal chrome, no width-aware running table, a different backoff layout, and a different summary/header structure.
- Desired Outcome: turn `CO STATUS` into a real Symphony-parity terminal dashboard for operators while keeping CO's existing observability truth path. This is a parity-correction lane for terminal presentation and behavior, not a new backend observability lane.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): re-audit the actual local Symphony terminal dashboard implementation and snapshot fixtures, compare them against the current CO STATUS implementation and the earlier CO-7 / CO-26 packet, then close the remaining operator-facing terminal parity gaps intentionally.
- Success criteria / acceptance:
  - `CO STATUS` can be described honestly as terminal parity with Symphony's operator dashboard rather than a thinner imitation
  - the terminal frame uses Symphony-style chrome and section structure
  - the running panel is a width-aware table with headers, separators, and scan-friendly columns
  - the backoff queue is sorted and uses operator-friendly relative due-time plus sanitized error display
  - the top summary/header carries the Symphony information architecture closely enough for operator scanning
  - narrow terminals degrade intentionally instead of relying on generic truncation alone
  - focused tests include snapshot-style or equivalent full-frame regression coverage
  - any justified remaining deviation is recorded explicitly before parity is claimed
- Constraints / non-goals:
  - preserve the shared CO observability truth path when possible
  - keep the operator-facing title exactly `CO STATUS`, even though Symphony's reference title is `SYMPHONY STATUS`
  - do not reopen `/ui`, provider ownership, retry scheduling, or another observability-read-model lane unless the parity work proves a smaller additive seam is missing
  - do not let "same dataset" excuse a materially different terminal UX

## Goals
- Upgrade `CO STATUS` from a plain-text report into a Symphony-style terminal dashboard frame.
- Mirror Symphony's header, running, and backoff scanning model as closely as CO's current runtime data allows.
- Make terminal-width behavior explicit and truthful.
- Add focused full-frame regression coverage so future drift from the intended parity is detectable.

## Non-Goals
- Reworking the already-landed CO `/ui` operator dashboard.
- Creating a second terminal-only observability truth path.
- Adding control actions, prompts, or mutable terminal affordances.
- Broadening the lane into unrelated control-host lifecycle or provider workflow changes.
