## Summary
- Problem Statement: `codex-orchestrator linear create-follow-up` is the governed path for filing or reusing same-project follow-up issues, but the follow-up label contract is not yet enforced. A provider worker can create a structurally valid follow-up while omitting lifecycle, priority, area, or type labels, or broadening into CO-400 label projection work to discover labels indirectly.
- Desired Outcome: `create-follow-up` derives the required follow-up labels from the live source issue labels, applies them with live Linear label ids during create or canonical-owner reuse, fails closed when the live source labels cannot satisfy the policy or the target issue omits requested labels, and emits immediate traceability without expanding into CO-400 label projection.

## User Request Translation
- User intent / needs: give provider workers a narrow, auditable way to label follow-up issues at creation/reuse time from live source issue labels and live Linear label ids, while keeping current-state label projection owned by CO-400.
- Success criteria / acceptance:
  - `codex-orchestrator linear create-follow-up` has a documented and tested label assignment contract based on live source issue labels and Linear label ids, not free-form label names or cached projection.
  - The helper derives `Lifecycle: Implementation`, at least one `Priority:*`, at least one `Area:*`, and one type label (`Bug`, `Improvement`, or `Feature`) from the live source issue labels before treating a follow-up mutation as successful.
  - Newly created follow-up issues receive the derived labels through `labelIds` when live source labels satisfy the policy.
  - Reused follow-up issues receive missing derived labels through `addedLabelIds`; they must not silently report success while derived labels are absent.
  - CLI JSON exposes resulting follow-up labels, and label-assignment failures include requested, observed, and missing label evidence.
  - Missing source label definitions or incomplete create/update responses fail closed with clear recovery details.
  - Focused tests cover facade mutation behavior, created and reused follow-up paths, missing-label definition handling, and label failure evidence.
  - No CO-400 issue-context label projection, cache label projection, or generic Linear label synchronization behavior is changed.
- Constraints / non-goals:
  - File only this docs-first packet in the child lane; parent owns implementation and Linear/GitHub lifecycle.
  - Do not broaden into CO-400 label projection or cached label current-state authority.
  - Do not infer labels from stale provider state, workpad text, old issue-context snapshots, fuzzy matching, or hard-coded ids.
  - Do not mutate Linear label definitions, projects, states, or unrelated issue metadata.
  - Do not weaken existing `create-follow-up` requirements for title, description, intent checksum, non-goals, `Not Done If`, acceptance criteria, related links, parity matrices, canonical-owner reuse, or blocker linkage.
