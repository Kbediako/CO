# PRD: CO-417 recover stale provider-worker proof locks during active appserver runs

## Added by Provider Worker 2026-04-28

## Summary
- Problem Statement: Active appserver provider-worker runs can keep heartbeating in the manifest while `provider-linear-worker-proof.json` stops advancing behind a retained `provider-linear-worker-proof.json.lock`, causing repeated `Failed to acquire provider-linear-worker proof lock after 50 attempts` warnings and stale read-model truth.
- Desired Outcome: Provider proof-lock writes carry enough owner/age diagnostics to classify stale locks, recover or fail closed without corrupting active writers, and keep live proof/read-model updates moving for active appserver workers.

## User Request Translation
- User intent / needs: Repair the distinct active-run stale proof-lock lifecycle gap observed after CO-403, using CO-415 and CO-400-style artifacts as replay evidence, without manual lock deletion or hiding proof persistence failures.
- Success criteria / acceptance: Reproduce or fixture the active appserver stale-lock shape; add lock owner/age diagnostics; implement safe stale-lock recovery or actionable fail-closed classification; add regression coverage for proof updates after recovery/classification; validate with focused lifecycle tests and replay/fixture evidence.
- Constraints / non-goals: Do not weaken provider proof/read-model requirements, do not make proof writes best-effort silent failures, do not remove lock protection without concurrent-writer safety, and do not broaden into unrelated control-host UI timeout or provider-intake freshness work.

## Intent Checksum
- Protected terms / exact artifact and surface names: `provider-linear-worker-proof.json`, `provider-linear-worker-proof.json.lock`, `Failed to acquire provider-linear-worker proof lock after 50 attempts`, appserver provider worker, manifest heartbeat, CO-415, CO-403, CO-400, stale lock, live proof update.
- Nearby wrong interpretations to reject: Deleting proof locks manually as the fix; treating a fresh manifest heartbeat as proof that frozen live-proof state is harmless; reopening CO-403 without checking for a distinct active-run lock lifecycle gap; hiding proof update failures from runner stderr.

## Parity / Alignment Matrix

| Surface | Current truth | Reference truth | Target truth |
| --- | --- | --- | --- |
| Active appserver manifest | Manifest can keep `status=in_progress` with fresh `heartbeat_at`. | Manifest heartbeat is necessary but not sufficient proof of provider read-model progress. | Manifest heartbeat and provider proof freshness remain separately inspectable. |
| Provider proof sidecar | `provider-linear-worker-proof.json` can freeze behind a stale lock. | Live proof updates should advance during active turns or fail closed with actionable diagnostics. | Stale lock recovery/classification allows the proof sidecar to advance safely or emit a precise owner/actionable failure. |

## Not Done If
- Active appserver workers can still freeze `provider-linear-worker-proof.json` behind a stale lock while manifest heartbeats continue.
- The lock file lacks enough owner/age metadata for safe recovery or diagnosis.
- Stale proof-lock recurrence is only manually cleaned up instead of recovered or failed closed by code.
- Runner stderr continues to spam proof-lock failures without an actionable diagnostic owner.

## Goals
- Fixture the CO-415 active-run stale proof-lock shape where manifest heartbeat continues while proof writes fail after repeated lock attempts.
- Add metadata or equivalent diagnostics for `provider-linear-worker-proof.json.lock` ownership and age.
- Implement conservative stale-lock recovery or fail-closed classification that preserves concurrent-writer safety.
- Keep proof persistence failures visible and actionable.
- Validate the fix with focused lifecycle tests and replay/fixture evidence for CO-415/CO-400-style artifacts.

## Non-Goals
- Do not weaken provider-worker proof/read-model requirements.
- Do not make proof writes best-effort silent failures.
- Do not remove lock protection without preserving concurrent-writer safety.
- Do not broaden into control-host UI timeout, provider-intake freshness, queue capacity, or unrelated current-state authority work.

## Technical Considerations
- CO-403 added stale lock timeout/keepalive support, but this issue targets the fresh active appserver live-proof path where the current proof lock retry configuration may not pass a stale threshold or owner diagnostics.
- The narrow likely seams are `orchestrator/src/persistence/lockFile.ts` and proof persistence in `orchestrator/src/cli/providerLinearWorkerRunner.ts`.
- The implementation should preserve existing child-stream and child-lane ledger lock behavior unless source audit proves the same lifecycle gap applies there.

## Fallback / Refactor Decision
- Applies to fallback, compatibility, legacy, stale, cached, break-glass, or minor-seam behavior? Yes.
- Decision: justify retaining stale proof-lock recovery as a durable lock-file safety contract, not a temporary fallback. The retained behavior must be gated by owner/age evidence, live-lock keepalive, and focused concurrent-writer tests.
- Large-refactor check: A broad lock subsystem rewrite is not required if the existing `acquireLockWithRetry` seam can carry owner diagnostics and stale classification safely for provider proof writes. Broader lock API cleanup remains out of scope unless needed for writer safety.
