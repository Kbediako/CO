## Summary
- Problem Statement: after `CO-210`, STATUS/proof projection can hydrate the real child-lane manifest and proof before the parent `provider-linear-child-lane` shell successfully replaces the original `launching-*` reservation row in `provider-linear-worker-child-lanes.json`. That is correct for current progress truth, but `accept` / `reject` / `invalidate` still target the latest pending ledger row, so a missed reservation replacement can leave parent-owned decisions blocked on a stale placeholder even when stronger same-parent, same-issue child-run evidence already exists.
- Desired Outcome: add a bounded reconciliation path so parent-owned decision flows can repair or upgrade a stale `launching-*` reservation row when stronger same-lineage child proof/manifest evidence already exists. The repair must remain metadata-safe, artifact-safe, fail closed on mismatches, and preserve truthful non-acceptance for zero-byte or missing-patch child lanes.

## Goals
- Keep `CO-218` narrowly scoped to stale pending-ledger repair in the parent-owned child-lane decision flow.
- Allow parent-owned decision paths to reconcile stale `launching-*` reservations using stronger same-lineage child proof/manifest evidence.
- Preserve artifact-root anchoring, proof lineage checks, and fail-closed behavior on malformed or mismatched evidence.
- Keep zero-byte and missing-patch child lanes non-accepted and truthful after reconciliation.
- Add focused regression coverage for the stale-reservation repair path and its fail-closed boundaries.

## Non-Goals
- Changing `CO-210` STATUS/progress hydration or selected-run projection semantics.
- Weakening stale Linear `updated_at` invalidation.
- Auto-accepting child lanes without explicit parent-owned `accept` flow.
- Redesigning child-lane launch admission, cap accounting, or scheduler policy.
- Deleting or compacting historical child-lane ledger/proof records.
- Broad refactors outside the child-lane decision seam.

## Not Done If
- Parent `accept` / `reject` / `invalidate` can still fail on a stale `launching-*` placeholder even when matching same-parent, same-issue child proof/manifest evidence exists.
- The reconciliation path can claim unrelated child runs by stream name, mtime, or artifact recency without matching the same parent/issue lineage.
- The fix changes stale Linear `updated_at` invalidation, child-lane decision authority, or `CO-210` progress hydration semantics.
- The fix only changes STATUS display while leaving the pending ledger unusable for parent-owned decisions.
