## Summary
- Problem Statement: `providerLinearWorkflowFacade.ts` currently hydrates issue PR attachment snapshots through `hydrateIssuePullRequestAttachments(...)` one candidate at a time. Issues with several GitHub PR attachments therefore pay serial `gh pr view --json state,mergedAt,updatedAt` latency even though each snapshot lookup is independent.
- Desired Outcome: add bounded-concurrency PR snapshot hydration for issue-context attachment snapshot reads so multiple independent PR snapshots can resolve concurrently under a small explicit cap, while preserving the current `pull_request_attachments` semantics for `current`, `historical`, `conflicting`, and `unknown`.

## Acceptance Criteria
- `hydrateIssuePullRequestAttachments(...)` resolves PR attachment snapshots with bounded concurrency under an explicit cap rather than serially awaiting every candidate.
- The cap is local and deterministic; no unbounded fanout over all attached PRs is introduced.
- Duplicate GitHub PR URLs are still deduped before any snapshot lookup is scheduled.
- For the same attachment list and the same resolver success/failure outcomes, `pull_request_attachments.current`, `.historical`, `.conflicting`, and `.unknown` match the pre-change classification.
- Resolver failures and invalid snapshots still classify only the affected candidate as `unknown`; they do not fail the full issue-context call.
- `fetchIssueContextPullRequestSnapshot(...)` keeps the same `gh pr view --json state,mergedAt,updatedAt` contract unless parent evidence proves a narrower internal helper refactor is required.
- Focused parent tests prove the concurrency cap and semantic parity without depending on real GitHub or Linear network calls.

## Not Done If
- PR snapshot hydration remains fully serial for multiple unique PR attachments.
- The implementation uses unbounded fanout or schedules every attachment without an explicit cap.
- Any same-input/same-resolver-output case changes `current`, `historical`, `conflicting`, or `unknown` classification.
- A resolver failure causes the full issue-context read to fail instead of preserving per-attachment `unknown` behavior.
- The repair drifts into `CO-220` PR-selection correctness, merge-closeout readiness gates, attachment cleanup, or semantics changes.
