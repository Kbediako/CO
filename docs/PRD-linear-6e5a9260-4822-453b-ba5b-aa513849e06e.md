## Summary
- Problem Statement: active provider-worker and appserver-backed `CO STATUS` still renders several operator-facing fields from sparse proof or compatibility fallbacks instead of authoritative live runtime telemetry. As a result, header `Tokens` and `Throughput`, row `TOKENS` and `SESSION`, `AGE / TURN`, `STAGE`, `EVENT`, and Codex rate-limit visibility can stay empty, generic, or misleading during real active usage.
- Desired Outcome: `CO STATUS` becomes truthful end to end for active provider-worker usage. Live runtime token, session, turn, event, and Codex usage-window telemetry flows into the worker proof and read-model; terminal `CO STATUS` then renders authoritative values for stage, event, tokens, throughput, session, turn lineage, and Codex `5-hour` and `weekly` windows, with focused regression coverage and real-device screenshot proof embedded directly in Linear.

## Parity / Alignment Matrix
- Current truth:
  - `ProviderLinearWorkerProof` already has fields for session, turn count, event, message, tokens, and `rate_limits`, but active operator truth still degrades before reaching the status surface
  - `STAGE` is projected from raw run status plus pause/input overlays rather than the operator-visible workflow phase
  - `EVENT` selection can still collapse to low-signal generic activity
  - rate-limit rendering understands the older generic bucket shape but not the richer Codex usage-window semantics called out by the issue
- Reference truth:
  - Symphony carries live session id, turn count, token totals, app-server PID, and meaningful last-message/event semantics through the running snapshot
  - Symphony humanizes rate-limit update events and summarizes usage windows instead of exposing raw low-signal payloads
  - Symphony’s running row stage is the operator-facing running-state label while event/message stay distinct and useful
- Target truth / intended delta:
  - provider-worker proof and control read-model carry authoritative runtime telemetry through to terminal `CO STATUS`
  - running rows and header lines render truthful or explicit unavailable values
  - stage/event semantics become operator-useful and distinct
  - Codex rate-limit rendering includes the authoritative `5-hour` and `weekly` windows when present
- Explicitly out-of-scope differences:
  - cloning Symphony output pixel-for-pixel
  - redesigning the terminal frame layout
  - unrelated provider-worker workflow changes outside the telemetry/projection seam

## Not Done If
- active runs still show `n/a`, `-`, empty fields, or hard-zero throughput when authoritative runtime telemetry exists
- `STAGE` still renders raw `in_progress` while the operator-visible issue phase is later (`In Review`, `Merging`, and similar)
- `EVENT` still falls back to generic text like `item completed` when more meaningful runtime activity exists
- Codex `5-hour` and `weekly` usage/rate-limit data is still missing or mislabeled
- the proof/read-model remains telemetry-starved and the renderer is only guessing
- Symphony mapping remains hand-wavy instead of explicit
- screenshot proof is missing or not embedded directly in Linear
