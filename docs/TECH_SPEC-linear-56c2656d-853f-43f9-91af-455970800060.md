## Summary
- Objective: make archive automation PRs produce the branch-protection-required `Core Lane` context even when the PR branch is created or updated by GitHub Actions.
- Scope: `.github/workflows/archive-automation-base.yml`, `.github/workflows/core-lane.yml`, docs-first packet, and focused workflow contract tests.
- Constraints: preserve branch protection, keep archive content behavior unchanged, and fail visibly if Core Lane dispatch cannot be requested.

## Issue-Shaping Contract
- User-request translation carried forward: fix recurring missing required checks on `automation/tasks-archive` archive PRs, not just PR `#637`.
- Protected terms / exact artifact and surface names: `PR #637`, `automation/tasks-archive`, `Core Lane`, `action_required`, `.github/workflows/archive-automation-base.yml`, `.github/workflows/core-lane.yml`, `ARCHIVE_AUTOMERGE_TOKEN`.

- Explicit non-goals carried forward: no archive eligibility rewrite, no docs freshness debt cleanup, no branch protection change.

## Parity / Alignment Matrix
- Current truth: the archive workflow uses `peter-evans/create-pull-request` with `github.token`; resulting `pull_request` runs are approval-required. The Apr 25 archive automation run `24920815600` dispatched Core Lane run `24920823983`, which succeeded on head `a1e20f23bca84c1d35e31dbbb190be6a70eaf3be`, but PR `#637` still reports `statusCheckRollup=[]` and `mergeStateStatus=BLOCKED`.
- Reference truth: GitHub documents that `workflow_dispatch` is an exception that can create workflow runs from workflow-authenticated `GITHUB_TOKEN` actions.
- Target truth / intended delta: archive automation dispatches `core-lane.yml` on the archive PR branch after creating or updating the PR, waits for that exact run, and mirrors its terminal conclusion into the branch-protection-required PR-visible `Core Lane` status context.
- Explicitly out-of-scope differences: requiring a valid PAT to create the PR, changing PR authoring identity, or changing required check names.

## Technical Requirements
- Functional requirements:
  - add `workflow_dispatch` support to `.github/workflows/core-lane.yml`
  - preserve existing `pull_request` and `push` triggers for `main`
  - after archive PR creation/update, dispatch `core-lane.yml` on the configured `inputs.pr_branch`
  - pass PR number, PR branch, and head SHA as dispatch inputs for auditability
  - require and validate dispatch PR inputs so a no-context manual dispatch cannot produce a green required check
  - reject archive dispatches whose supplied head SHA does not match the dispatched ref SHA
  - reject archive dispatches whose supplied PR number does not resolve to an open `main` PR on the expected archive head branch, repository, ref, and head SHA
  - publish a pending commit status context named `Core Lane` on the archive PR head before dispatch
  - wait for the exact dispatched `core-lane.yml` `workflow_dispatch` run on the archive PR head SHA
  - publish `Core Lane=success` only when that dispatched run concludes `success`
  - publish `Core Lane=failure` or `Core Lane=error` and fail the archive workflow if dispatch, run discovery, or the dispatched run fails
  - keep existing auto-merge behavior separate from Core Lane dispatch
- Non-functional requirements: no additional checked-in dependencies; no secret values logged; no branch-protection changes.
- Interfaces / contracts:
  - `core-lane.yml` requires `workflow_dispatch` inputs `archive_pr_number`, `archive_pr_branch`, and `archive_pr_head_sha`
  - archive base uses `GH_TOKEN: ${{ github.token }}` for dispatch because `workflow_dispatch` is an allowed recursion exception
  - archive base requires `statuses: write` to publish the PR-visible required context
  - required branch-protection context remains `Core Lane`

## Architecture & Data
- Architecture / design adjustments: keep the explicit workflow-to-workflow dispatch after the create-pull-request action, then bridge the detached dispatch result back to branch protection with a commit status on the archive PR head SHA.
- Data model changes / migrations: none.
- External dependencies / integrations: GitHub Actions, GitHub CLI on hosted runners, `peter-evans/create-pull-request`.
