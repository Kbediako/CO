## Summary
- Objective: make archive automation PRs produce the branch-protection-required `Core Lane` check even when the PR branch is created or updated by GitHub Actions.
- Scope: `.github/workflows/archive-automation-base.yml`, `.github/workflows/core-lane.yml`, docs-first packet, and focused workflow contract tests.
- Constraints: preserve branch protection, keep archive content behavior unchanged, and fail visibly if Core Lane dispatch cannot be requested.

## Issue-Shaping Contract
- User-request translation carried forward: fix recurring missing required checks on `automation/tasks-archive` archive PRs, not just PR `#637`.
- Protected terms / exact artifact and surface names: `PR #637`, `automation/tasks-archive`, `Core Lane`, `action_required`, `.github/workflows/archive-automation-base.yml`, `.github/workflows/core-lane.yml`, `ARCHIVE_AUTOMERGE_TOKEN`.
- Nearby wrong interpretations to reject: branch-protection bypass, one-off manual merge, Cloud Canary-only fix, broad archive policy redesign, or secret rotation as a checked-in change.
- Explicit non-goals carried forward: no archive eligibility rewrite, no docs freshness debt cleanup, no branch protection change.

## Parity / Alignment Matrix
- Current truth: the archive workflow uses `peter-evans/create-pull-request` with `github.token`; resulting `pull_request` runs are approval-required and current PR `#637` has no reported checks on head `9dc56bb50fe2f2f7f0fe9fbc4bf60d619ba97bde`.
- Reference truth: GitHub documents that `workflow_dispatch` is an exception that can create workflow runs from workflow-authenticated `GITHUB_TOKEN` actions.
- Target truth / intended delta: archive automation dispatches `core-lane.yml` on the archive PR branch after creating or updating the PR, giving branch protection a real `Core Lane` check to evaluate.
- Explicitly out-of-scope differences: requiring a valid PAT to create the PR, changing PR authoring identity, or changing required check names.

## Readiness Gate
- Not done if: `Core Lane` is still missing from archive PR heads; dispatch errors are ignored; required checks are bypassed; PR `#637` is merged before a green required check; or future archive PRs still need a manual one-off workaround.
- Pre-implementation issue-quality review evidence: parent read Linear issue context, moved `Ready` to `In Progress`, recorded the workpad and parallelization decision, swept PR `#637` feedback, confirmed zero comments/reviews/review threads, confirmed no reported checks, and inspected recent action-required runs.
- Safeguard ownership split: parent owns workflow/docs/tests because the fix is a single bounded workflow contract; no safe child lane exists before the workflow seam is narrowed.

## Technical Requirements
- Functional requirements:
  - add `workflow_dispatch` support to `.github/workflows/core-lane.yml`
  - preserve existing `pull_request` and `push` triggers for `main`
  - after archive PR creation/update, dispatch `core-lane.yml` on `steps.cpr.outputs.pull-request-branch`
  - pass PR number and head SHA as dispatch inputs for auditability
  - fail the archive workflow if the dispatch request fails
  - keep existing auto-merge behavior separate from Core Lane dispatch
- Non-functional requirements: no additional checked-in dependencies; no secret values logged; no branch-protection changes.
- Interfaces / contracts:
  - `core-lane.yml` accepts optional `workflow_dispatch` inputs `archive_pr_number` and `archive_pr_head_sha`
  - archive base uses `GH_TOKEN: ${{ github.token }}` for dispatch because `workflow_dispatch` is an allowed recursion exception
  - required branch-protection context remains the job name `Core Lane`

## Architecture & Data
- Architecture / design adjustments: add an explicit workflow-to-workflow dispatch after the create-pull-request action instead of relying on approval-gated `pull_request` recursion.
- Data model changes / migrations: none.
- External dependencies / integrations: GitHub Actions, GitHub CLI on hosted runners, `peter-evans/create-pull-request`.
