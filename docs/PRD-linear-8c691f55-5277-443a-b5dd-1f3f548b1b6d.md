# PRD - CO: stop startup recovery sweeps from burning issue-by-id reads for retained released claims

## Traceability
- Linear issue: `CO-162` / `8c691f55-5277-443a-b5dd-1f3f548b1b6d`
- Linear URL: https://linear.app/asabeko/issue/CO-162/co-stop-startup-recovery-sweeps-from-burning-issue-by-id-reads-for
- Related landed slices: `CO-160`, `CO-159`, `CO-156`, `CO-144`
- Related follow-up: `CO-161`
- Evidence PR / head: PR `#453` (open), branch `linear/co-162-startup-recovery-sweep-no-id-burn`

## Summary
- Problem Statement: after `CO-160` landed, ordinary deferred polls now fail closed for retained released claims, but the control-host startup full recovery sweep still burns Linear request budget through `dispatch_source_issue_by_id`. The first startup refresh schedules `mode: recovery_sweep`, fetches tracked issues once, then calls `providerIssueHandoff.poll(...)` without enabling the same released-claim fail-closed path. Retained released claims that are absent from the recovery-sweep tracked-issue map therefore fall through to per-claim direct issue-by-id reads.
- Desired Outcome: the first startup recovery sweep should perform zero direct issue-by-id reads for locally terminal retained released claims, still process runnable tracked issues returned by the recovery sweep, preserve the already-landed deferred-poll no-burn behavior from `CO-160`, and leave explicit non-deferred reopen or active-run recovery paths intact.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): finish `CO-162` by closing the remaining post-`CO-160` startup request-burn seam without disabling recovery sweeps or hiding real runnable work. The implementation should stay narrow to the released-claim poll fail-closed admission seam, prove the result with focused regressions plus guarded restart evidence, and keep bounded recovery when the sweep or an explicit revalidation path returns genuinely runnable work.
- Success criteria / acceptance:
  - startup full recovery sweep with 128 retained released `not_active` or `not_mutable` claims performs zero `resolveTrackedIssue` / `dispatch_source_issue_by_id` reads for those claims
  - recovery sweep still processes tracked issues returned by the sweep and does not hide runnable work
  - normal deferred-poll behavior from `CO-160` remains intact: zero direct reads and zero fresh-discovery churn in the all-released retained-claim case
  - regression coverage distinguishes deferred poll, startup recovery sweep, non-deferred explicit refresh or reopen, and active-run recovery
  - a guarded restart proof shows no new `dispatch_source_issue_by_id` entries in `request_burn_history` after the first startup recovery sweep
- Constraints / non-goals:
  - do not disable recovery sweeps entirely
  - do not weaken the `CO-160` deferred-poll no-burn contract
  - do not hide real `Ready` or `In Progress` work returned by the recovery sweep
  - do not broaden into generic scheduler redesign, GitHub throttling, or `CO STATUS` rendering work

- Current truth:
  - ordinary deferred polls call `providerIssueHandoff.poll(...)` with `deferFreshDiscovery: true`
  - startup full recovery sweep fetches tracked issues with `mode: recovery_sweep`, then calls `providerIssueHandoff.poll(...)` without the released-claim fail-closed flag path
  - `resolveTrackedIssuePollResolutionWithFallback(...)` only uses the released-claim cached fail-closed helper when poll callers explicitly allow it
- Reference truth:
  - locally terminal retained released claims should not require per-claim live issue-by-id reads during startup recovery
  - recovery sweep should still act on runnable tracked issues it already fetched from live Linear
  - explicit non-deferred or newer-evidence recovery should remain available for genuinely reopened work
- Target truth / intended delta:
  - startup recovery-sweep polls share the released-claim cached fail-closed seam already used by deferred polls
  - recovery-sweep returned tracked issues remain eligible for normal owned/start or resume handling
  - non-deferred reopen and active-run recovery keep their existing live revalidation behavior
- Explicitly out-of-scope differences:
  - generic fresh-discovery policy redesign

## Not Done If
- Restarting the control host still causes retained released claims to burn `dispatch_source_issue_by_id` requests.
- The fix only works for deferred polls and leaves startup recovery sweeps unguarded.
- Recovery sweeps stop finding genuinely runnable tracked issues returned by the sweep.
- The operational workaround remains manual stop or restart of the host.

## Technical Considerations
- Architectural Notes:
  - the code seam is split between `orchestrator/src/cli/control/controlServerPublicLifecycle.ts` and `orchestrator/src/cli/control/providerIssueHandoff.ts`
  - the smallest truthful fix is likely to let startup recovery-sweep polls opt into the released-claim fail-closed helper without conflating that with unrelated fresh-discovery policy
  - the tracked-issue map returned by `mode: recovery_sweep` should remain authoritative for runnable work already discovered during startup
- Dependencies / Integrations:
  - `orchestrator/src/cli/control/controlServerPublicLifecycle.ts`
  - `orchestrator/src/cli/control/providerIssueHandoff.ts`
  - `orchestrator/tests/ProviderIssueHandoff.test.ts`

## Decision
- Use the dedicated `allowPollFailClosed` poll input for startup recovery-sweep callers, but keep it scoped to the released-claim cached fail-closed helper. Deferred polls remain the only path that enables the broader cached fail-closed reasons tied to `deferFreshDiscovery`.
