# PRD - CO-380 authoritative review gate

## Summary
- Problem Statement: Review wrapper fallback paths have become routine success behavior. Generated `NOTES`, handoff-only non-interactive output, and manual fallback language make it too easy for a failed or incomplete review to look like a completed review.
- Desired Outcome: CO owns one authoritative review gate for implementation handoff. The gate has explicit inputs, terminal telemetry verdicts, and only documented break-glass fallback paths with owner, expiry, and evidence.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): Replace routine review-wrapper fallback posture with a repo-owned review contract that either runs to a classified terminal verdict or fails with explicit waiver evidence.
- Success criteria / acceptance:
  - Inventory `codex-orchestrator review`, `npm run review`, scoped review, non-interactive, provider-worker, docs-review, and implementation-gate review paths.
  - Define one authoritative review gate with explicit inputs, outputs, telemetry, and terminal verdict rules.
  - Remove or demote routine fallback branches that preserve old behavior.
  - Keep only documented break-glass fallbacks with owner, expiry, and audit evidence.
  - Update tests so fallback activation is exceptional, not normal success.
  - Validate build, lint, review-wrapper tests, docs gates, and pack smoke when downstream review/package paths move.
- Constraints / non-goals: Do not broaden into review-agent model policy, GitHub review-thread handling, PR merge automation, or docs-relevance advisory behavior beyond preserving it as explicitly advisory.

## Intent Checksum
- Exact user wording / phrases to preserve: `authoritative review gate`, `routine review-wrapper fallbacks`, `handoff-only`, `generated fallback notes`, `explicit waiver artifact`, `break-glass`.
- Protected terms / exact artifact and surface names: `codex-orchestrator review`, `npm run review`, `review/telemetry.json`, `docs-review`, `implementation-gate`, `provider-linear-worker`, `CODEX_REVIEW_NON_INTERACTIVE`, `FORCE_CODEX_REVIEW`.
- Nearby wrong interpretations to reject:
  - Treating a printed non-interactive handoff prompt as gate success.
  - Treating generated `NOTES` as normal authoritative input.
  - Silently converting a failed review into manual review without a waiver artifact.
  - Weakening `docs-relevance-advisory`; it remains prompt-only and advisory by design.

## Parity / Alignment Matrix
- Current truth: `resolveReviewNotes` can synthesize fallback `NOTES`; non-interactive wrapper paths can print a handoff prompt when not forced; review pipelines set force flags inconsistently enough that the contract is implicit rather than authoritative.
- Reference truth: `docs-review`, `implementation-gate`, and provider-worker handoff review lanes should execute the wrapper-led review under an explicit authoritative gate and classify the result through telemetry.
- Target truth / intended delta: Gate lanes require authored `NOTES` or an explicit break-glass waiver, disallow prompt-only success, and classify `clean-success` / `bounded-success` as success while preserving `failed-boundary` and `failed-other` as failed terminal verdicts.
- Explicitly out-of-scope differences: Advisory docs relevance prompt generation, external GitHub PR feedback semantics, and broad review prompt redesign are not part of CO-380.

## Not Done If
- The primary path still emits handoff-only or generated fallback notes as a normal success condition.
- A failed authoritative review can silently become a manual review without an explicit waiver artifact.

## Goals
- Make the authoritative review gate the repo-owned path for implementation handoff review.
- Keep break-glass fallbacks auditable with owner, expiry, reason, and evidence.
- Update tests and docs to make fallback activation exceptional.

## Non-Goals
- No change to review model selection.
- No change to PR ready-review drain semantics.
- No new dependency on external review systems.

## Stakeholders
- Product: CO operator workflow.
- Engineering: Codex orchestrator maintainers.
- Design: Not applicable.

## Metrics & Guardrails
- Primary Success Metrics: Missing `NOTES` under the authoritative gate fails unless a waiver is supplied; non-interactive handoff-only output cannot satisfy the gate; review telemetry has a terminal classified verdict.
- Guardrails / Error Budgets: Break-glass paths must expose owner, expiry, reason, and evidence in review-visible context and telemetry or command output.

## User Experience
- Personas: Provider worker and top-level CO operator.
- User Journeys: A worker reaches review handoff, runs the authoritative gate, receives either terminal success telemetry or a clear failed verdict requiring waiver/rework.

## Technical Considerations
- Architectural Notes: Keep the change at the review wrapper contract layer and existing pipeline env configuration.
- Dependencies / Integrations: `scripts/run-review.ts`, `scripts/lib/review-prompt-context.ts`, `scripts/lib/review-non-interactive-handoff.ts`, `codex.orchestrator.json`, and review wrapper tests.
