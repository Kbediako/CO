# PRD - CO: Re-audit Codex CLI 0.122.0 Posture and Align Explicit Pins

## Summary
- Problem Statement: at issue intake, the checked-out CO policy still named Codex CLI `0.118.0` as the active compatibility/adoption target and `0.121.0` as the latest audited stable candidate, while upstream stable and the local official CLI had already moved to `0.122.0`. At the same time, `.github/workflows/core-lane.yml`, `.github/workflows/release.yml`, and `.github/workflows/pack-smoke-backstop.yml` still installed `@openai/codex@0.121.0`, and `.github/workflows/cloud-canary.yml` still installed unpinned latest. CO-269 therefore needed an explicit posture-and-pins packet so policy docs, cloud-gate assumptions, and workflow pin policy could converge on a single evidence-backed split outcome.
- Desired Outcome: produce the CO-269 docs-first packet so the parent lane can re-audit `0.122.0`, preserve the exact predecessor constraints from CO-183, CO-195, CO-199, and CO-207, record a single hold-versus-promote decision, and align workflow pin policy only after that decision is evidence-backed.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): treat CO-269 as the `codex 0122 posture and pins` lane, not a blind version bump. The parent lane needed a bounded docs packet that kept the predecessor `CO-183` / `CO-195` / `CO-199` / `CO-207` contract intact while it decided whether CO should promote from the current documented posture to Codex CLI `0.122.0`, whether `core-lane` / `release` / `pack-smoke-backstop` pins should move, and how `cloud-canary` should be classified in the final workflow policy.
- Success criteria / acceptance:
  - the packet explicitly links CO-269 to CO-183, CO-195, CO-199, and CO-207
  - current/reference/target posture truth is captured for `0.118.0`, `0.121.0`, and `0.122.0`
  - explicit `@openai/codex` pin surfaces are identified as parent-owned follow-up surfaces
  - required runtime/cloud gate semantics from CO-195 and CO-207 are preserved
  - sandbox/security carry-forward from CO-199 is preserved
  - workflow pin policy is explicit for `.github/workflows/core-lane.yml`, `.github/workflows/release.yml`, `.github/workflows/pack-smoke-backstop.yml`, and `.github/workflows/cloud-canary.yml`
  - non-goals reject blind pin bumps, guardrail weakening, and unrelated runtime redesign
- Constraints / non-goals: this packet lane must keep the issue scoped to posture evidence, version-policy docs, workflow/test pin classification, and exact blocker reporting; do not broaden into marketplace rebaseline, provider-worker app-server migration, remote-control redesign, tool-search/image-generation/plan-mode follow-ups without evidence, or guardrail weakening.
