## Summary
- Problem Statement: `CO-70` already restored the truthful baseline for `CI=1 npm run test`: the suite exits cleanly when the late files are allowed to finish. The remaining defect is operator visibility. After earlier green file output, the suite can go quiet for a long tail while `tests/run-review.spec.ts` and then `tests/cli-command-surface.spec.ts` are still running. Earlier operators manually killed that quiet tail and misdiagnosed it as a lingering-process exit failure. The same visibility gap also affected standard truthy CI runs such as GitHub Actions `CI=true`.
- Desired Outcome: keep the clean terminal exit baseline untouched, but add a bounded runtime signal that makes the still-running late files legible during `CI=1 npm run test` and standard truthy CI runs so operators can distinguish an active long file from a genuinely hung Vitest process.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): preserve the truthful `CO-70` clean-exit result, reproduce the current late-suite profile on the current tree, and add visibility only. The fix must not be another timeout, auto-kill, skip, or waiver. It should show that the suite is still alive while the long late files are in flight.
- Success criteria / acceptance:
  - current-tree evidence reproduces the late-suite timing/profile that made the run look stalled
  - `CI=1 npm run test` and standard truthy CI full-suite runs emit a clear pre-exit signal naming the still-running late files instead of staying plausibly idle
  - the signal appears before terminal completion and preserves the existing pass/fail semantics
  - the full suite still exits cleanly after the visibility improvement
- Constraints / non-goals:
  - do not reopen `CO-70` as if process-exit cleanup were still broken
  - do not solve the problem with blind timeouts, auto-kill behavior, skips, or weakened coverage
  - keep the change bounded to Vitest/test-surface observability rather than broad test-performance work

## Technical Considerations
- Architectural Notes:
  - the narrow missing seam is the top-level Vitest reporting surface, not the test semantics themselves
  - local probing on the current tree shows Vitest reporter hooks expose file-level `run` and terminal updates, which is sufficient for a small active-file progress reporter
  - the smallest truthful design is a reporter that tracks collected file ids, notices file `run` updates, and emits bounded "still running" lines only for files that stay active past a threshold
  - the reporter should integrate through `vitest.config.core.ts` so `CI=1 npm run test` and standard truthy CI lanes inherit the visibility improvement automatically
- Dependencies / Integrations:
  - `vitest.config.core.ts`
  - a new reporter helper under `scripts/lib/`
  - focused reporter regression coverage under `tests/`
