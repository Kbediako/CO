## Summary
- Objective: define a truthful, evidence-gated issue-parent architecture for multiple same-issue worker lanes that preserves the current single external issue owner and explicitly stays downstream of `CO-13` plus the `0.117/main` canary.
- Scope:
  - docs-first comparison against current Symphony, `CO-13`, and the `0.117/main` canary
  - explicit inheritance of the already-landed or already-tracked parity slices that `CO-14` sits above
  - explicit authority, lineage, observability, workspace, and conflict rules for a future issue-parent owner
  - activation criteria and staged follow-on slices for eventual implementation
  - documentation and checklist updates only; no runtime or workflow behavior changes in this lane
- Constraints:
  - preserve one authoritative issue claim and running owner
  - preserve one externally authoritative workspace, workpad, thread, and handoff state machine
  - serialize all Linear lifecycle mutations through the parent owner
  - preserve per-issue workspace confinement and manifest lineage
  - keep `delegation-guard` intact
  - keep the default provider-worker supervision seam on `codex exec` / `codex exec resume`

## Technical Requirements
- Functional requirements:
  - compare the future issue-parent design against the following:
    - current Symphony single-owner behavior
    - `CO-13` bounded provider-worker child streams
    - the `0.117/main` adoption-canary conclusions
    - current bounded parity slices for intake, continuation, runtime snapshot, retry ownership, and operational workflow truth
  - define the future issue-parent as the only actor allowed to:
    - transition Linear issue state
    - mutate the single active workpad comment
    - attach PRs
    - declare review, rework, merging, and done handoffs
  - define subordinate child lanes as internal helpers only:
    - each child lane must carry `<parent-task-id>-<stream>` task identity plus `parent_run_id`
    - each child lane must have declared purpose and scope before launch
    - child lanes must not mutate Linear state, workpad, or PR metadata directly
  - define the workspace and branch model truthfully:
    - the externally authoritative issue workspace remains parent-owned
    - child lanes may use internal lane-local worktrees, branches, or patch artifacts only within that per-issue workspace boundary
    - parent integrates accepted child outputs back into the authoritative branch and workspace state
  - define conflict rules for parallel same-issue work:
    - parallel child lanes require disjoint file or phase ownership
    - overlapping scopes must be serialized or explicitly re-scoped by the parent
    - stale child outputs must be rejected or rerun when issue state, review feedback, or base branch changes invalidate them
  - define the observability contract:
    - selected and compatibility views remain parent-centric
    - child lanes surface as additive nested lineage records, not peer scheduler-owned issue runs
    - the design should extend the current provider proof plus child-stream ledger pattern rather than inventing a separate truth source
  - define the activation gate:
    - implementation stays deferred unless `CO-13` or the current single-worker seam shows evidence-backed throughput or quality bottlenecks around review, merge, workpad, or supervision that bounded child streams cannot solve
- Non-functional requirements (performance, reliability, security):
  - keep the design audit-friendly and grounded in manifest-backed lineage
  - preserve control-host ownership and provider-intake truthfulness
  - keep remote app-server command and watcher surfaces out of the default authority model until a later explicit guardrail lane

## Architecture & Data
- Architecture / design adjustments:
  - Current baseline:
    - control-host owns issue claim and dispatch
    - one `provider-linear-worker` owns the issue workspace, workpad, and handoff lifecycle
    - `CO-13` child streams are bounded helpers and are intentionally excluded from scheduler-owned provider-run discovery
  - Future issue-parent design:
    - keep the current provider issue owner as the external issue parent rather than introducing a second external owner
    - allow the issue parent to coordinate multiple subordinate lanes inside the same issue boundary
    - treat subordinate lanes as internal execution units whose outputs are accepted or rejected by the parent, not as peer issue owners
  - Proposed lineage model:
    - authoritative parent identity remains the issue task id and parent run id
    - child lane identity uses `<parent-task-id>-<stream>`
    - every child lane records `pipeline_id`, `task_id`, `run_id`, `manifest_path`, `purpose`, `declared_scope`, `status`, and timestamps
    - parent proof artifact remains the external truth source; child-lane lineage is nested underneath it
  - Proposed observability model:
    - extend the current selected and compatibility payloads with additive `child_lanes` summaries when implementation eventually lands
    - preserve one authoritative selected issue row for the parent owner
    - keep retry and workspace truth attached to the parent row while child rows remain subordinate detail
  - Proposed staged epic slices:
    1. evidence and docs gate only
    2. parent or child lineage schema plus read-side observability extension
    3. parent-controlled lane ownership and cancellation contract
    4. optional parallel coding-lane activation only after conflict and merge discipline are proven
- Data model changes / migrations:
  - none in this lane
  - future implementation should extend the existing provider worker proof plus child-stream ledger rather than replacing them
