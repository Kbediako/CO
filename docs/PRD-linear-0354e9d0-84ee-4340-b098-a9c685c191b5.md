## User Request Translation
Operators need `control-host recover|relaunch|nudge --issue-id <id>` to acknowledge accepted provider-worker work inside the request budget. The CLI must not report `control-host provider-worker nudge request timed out after 25000ms` after `provider_launch_source=control-host` side effects already started, and accepted pending-revalidation rows with null launch/run identity must not consume active capacity.

## Evidence
- CO-403 nudge timed out after `25000ms`, while manifest, proof, and Linear audit artifacts showed provider launch and `Ready -> In Progress`.
- Source pointers identify `controlHostProviderWorkerRecoverCliShell.ts` request timeout handling and `observabilityApiController.ts` delayed `202` acknowledgement.
- CO-405 added the accepted-without-run variant: `state=accepted`, `reason=provider_issue_rehydration_pending_revalidation`, `run_id=null`, `run_manifest_path=null`, `launch_started_at=null`, `launch_token=null`, later followed by false `provider_issue_start_blocked:max_concurrency`.

## Acceptance Criteria
- Accepted recover/relaunch/nudge requests return truthful acknowledged JSON within the request budget.
- Asynchronous launch or resume returns machine-checkable `queued` or `starting` metadata.
- Retrying an ambiguous acknowledgement does not duplicate provider-worker launches or intake truth.
- Accepted pending-revalidation rows without launch/run identity remain visible but do not count as running/starting capacity.
- Regressions cover timeout-after-launch, retry idempotency, and accepted-without-run capacity.

## Non-Goals
- Do not reopen CO-393 direct-start guardrails.
- Do not fold in CO-403 stale proof-lock or child-lane retry recovery unless implementation proves shared root.
- Do not solve this by only increasing `--request-timeout-ms` or manually editing `provider-intake-state.json`.

## Parity Matrix
| Variant | Current Truth | Target Truth | Out Of Scope |
| --- | --- | --- | --- |
| Timeout after successful launch | CLI can time out while launch/proof/audit side effects succeed. | API/CLI returns accepted `queued` or `starting` truth inside the request budget. | CO-393 and CO-403 behavior outside acknowledgement truth. |
| Accepted without run | Pending revalidation can have null run/manifest/launch/token and be counted as active. | It remains pending truth only and does not occupy running/starting capacity. | Broad provider concurrency redesign. |

## Not Done If
- Hard timeout still surfaces after accepted side effects.
- API `202` waits for full recover/relaunch/nudge completion.
- Retry can create duplicate launch or duplicate intake truth.
- Accepted pending-revalidation rows without launch/run identity consume capacity.
