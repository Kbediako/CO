## Summary
- Problem Statement: after `CO-571` fixed provider-refresh progress stall classification, `co-status --format json` can still fail on the current same endpoint because the machine health read is coupled to the heavy `/ui/data.json` dashboard projection. Live evidence shows fresh provider polling, `polling.stuck=false`, `polling.restart_required=false`, no active claims, and a direct `/ui/data.json` success followed by `co-status` same-endpoint timeout failure. The current degraded fallback refuses zero-WIP healthy cases because it expects active provider-intake evidence.
- Desired Outcome: split cheap machine health from dashboard projection so `co-status --format json` and control-host supervision can prove a fresh quiescent host without hiding real dead, stale, stuck, failed, auth, or active no-progress states. `/ui/data.json` remains dashboard data and becomes bounded/degraded explicitly instead of consuming the machine-health budget.

## Not Done If
- Plain `co-status --format json` still depends on `/ui/data.json` or exits nonzero while machine evidence proves fresh zero-WIP health.
- Supervisor still reports restart-required, `probe_failed`, or `probe_timeout` solely because the dashboard projection is slow under fresh zero-WIP health.
- The fix treats all current-endpoint timeouts as healthy.
- `polling.restart_required=true`, `polling.stuck=true`, stale source, auth failure, or failed run evidence is hidden by degraded status.
- `/ui/data.json` remains the only machine health read path.
- The implementation only raises client/server timeouts.
- The dashboard endpoint returns partial/degraded data without explicit degraded metadata.

## Goals
- Add a cheap authenticated machine status contract that avoids `readCompatibilityProjection()` and exposes bounded source, polling, progress, and provider-intake counts.
- Make plain `co-status --format json` and supervisor probing use cheap machine status; keep dashboard-only behavior explicit behind dashboard flags.
- Preserve zero-WIP fresh quiescent status while keeping real failures fail-closed.
- Bound `/ui/data.json` dashboard work or at least classify its timeout/reset as dashboard degraded, not machine health failure.
- Add regression tests for zero-WIP same-endpoint timeout, connection reset, active fresh claim, stale source, auth failure, true dead endpoint, real provider stuck, and active no-progress refresh.

## Fallback / Refactor Decision
- Applies to fallback, compatibility, legacy, stale, cached, break-glass, or minor-seam behavior? `Yes`.
- Decision summary:
  - `remove fallback`: using `/ui/data.json` dashboard projection as machine health authority.
  - `remove fallback`: active-claim-only degraded fallback for current-endpoint timeouts.
  - `justify retaining fallback`: source-labeled local degraded partial status as a supported diagnostic path when cheap status proves fresh health and dashboard projection is degraded.
  - `remove fallback`: supervisor probe dependency on the heavy dashboard projection.
- Large-refactor check: the policy favors a larger consolidation because status authority is split across live HTTP, persisted provider-intake, dashboard projection, freshness gauge, and supervision. CO-572 is that consolidation at the status-machine boundary. If the diff must split, the first PR must still land cheap machine status, `co-status`, and supervisor decoupling; dashboard optimization may follow under the same issue only if the first PR already stops the recurrence without hiding failures.
- Minor-seam decision: acceptable only for the single source-labeled degraded machine status diagnostic contract; no timeout-only or dashboard-authority seam is retained.
