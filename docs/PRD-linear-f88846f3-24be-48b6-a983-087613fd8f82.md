# PRD - CO-426 Provider Validation Audit Isolation
## Request / Problem
Provider-worker validation must not let synthetic Linear CLI fixture audit rows leak into a live `CODEX_PROVIDER_LINEAR_AUDIT_PATH`. CO-405 showed `co-status` surfacing failed `create-follow-up` rows for fixture issue id `lin-issue-1` from validation as if they were live provider activity, even though the real CO-405 follow-up had already succeeded as CO-419.
## Protected Terms
`CODEX_PROVIDER_LINEAR_AUDIT_PATH`, `provider-linear-worker-linear-audit.jsonl`, `lin-issue-1`, `LinearCliShell.test.ts`, `create-follow-up`, `linear_follow_up_parity_matrix_missing`, `co-status`, live provider-worker audit.
## Scope / Non-Goals
Make provider-worker validation hermetic so test fixture audit writes cannot contaminate the live provider-worker audit path, while preserving real helper audit rows for actual provider operations. Do not weaken parity-matrix guards, remove real provider helper audit logging, patch CO-405 review-command-intent behavior, or hide audit rows in `co-status` without fixing or explicitly classifying the synthetic source.
## Not Done If
- Fixture issue ids such as `lin-issue-1` can still be appended to a live `CODEX_PROVIDER_LINEAR_AUDIT_PATH`.
- `LinearCliShell.test.ts` mutating cases still rely on inherited provider-worker audit env for fixture operations.
- The regression only passes when `CODEX_PROVIDER_LINEAR_AUDIT_PATH` is absent.
- Real provider helper audit rows for actual issue ids are suppressed or removed.
- `co-status` still presents mismatched fixture rows as live provider operations for unrelated issues.
## Current / Reference / Target Parity Matrix
| Surface | Current | Reference | Target |
| --- | --- | --- | --- |
| Provider-worker validation | Fixture tests can inherit live provider audit env. | Validation must be hermetic and cannot mutate live worker artifacts. | Fixture validation uses temp or mocked audit paths and leaves live audit files unchanged. |
| `CODEX_PROVIDER_LINEAR_AUDIT_PATH` regression | Setting the env can capture synthetic `lin-issue-1` rows. | A live audit path belongs only to actual provider operations. | A focused regression proves fixture execution leaves the configured live path unchanged. |
| Real helper audit logging | Real CO-405 follow-up succeeded as CO-419 before synthetic rows appeared. | Actual provider helper calls remain auditable. | Actual issue ids still append normal provider audit records. |
| `co-status` handling | Synthetic fixture rows can look like latest live provider activity. | Status surfaces should distinguish live operations from stale or mismatched artifact rows. | `co-status` no longer receives synthetic rows or classifies/ignores mismatched fixture rows. |
## Acceptance Criteria
- [ ] Validation under a provider-worker environment cannot append fixture issue ids such as `lin-issue-1` to the live audit path.
- [ ] `LinearCliShell` mutating tests use temp or mocked audit paths, including parity-matrix-missing.
- [ ] A regression proves validation leaves the live audit unchanged when `CODEX_PROVIDER_LINEAR_AUDIT_PATH` is set.
- [ ] Real provider helper calls still append audit records for actual issue ids.
- [ ] `co-status` no longer receives, or explicitly ignores/classifies, mismatched fixture rows.
