## Summary
- Problem Statement: CO now reaches truthful review handoff and deterministic `Merging` closeout, but both seams still stop when GitHub reports a green PR as `mergeStateStatus=BEHIND` or `DIRTY`. `pr-watch-merge.js` classifies those states as action-required, review-handoff promotion refuses them, and merge closeout will not autonomously refresh or recover the branch, so green-but-dirty PRs still require a manual operator step.
- Desired Outcome: add one bounded autonomous merge-recovery path that refreshes `BEHIND` branches, attempts a bounded GitHub-side recovery for `DIRTY` branches, and moves the issue into truthful `Rework` with exact conflict metadata when repair cannot proceed automatically.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): close the remaining operator bottleneck for green-but-dirty PRs by teaching the shared PR watcher and provider merge lifecycle to request branch refresh automatically, re-evaluate readiness after the refresh, and stop leaving conflicting PRs parked in `In Review` or `Merging` without an explicit recovery or rework path.
- Success criteria / acceptance:
  - a green `In Review` PR with `mergeStateStatus=BEHIND` no longer waits for manual author action; CO requests a bounded branch refresh and reevaluates the snapshot automatically
  - a green `Merging` PR with `mergeStateStatus=BEHIND` self-recovers through the same bounded refresh path
  - a `DIRTY` / conflicting PR does not stay indefinitely in `In Review`; CO either recovers it through the bounded branch-update path or moves the issue into explicit `Rework`
  - exact blocker reasons and exact recovery attempts are persisted in a machine-checkable surface
  - the solution stays within the current merge-closeout and review-handoff contracts instead of reopening unrelated closeout-truth lanes
  - focused tests cover `BEHIND` refresh, `DIRTY` recovery or rework fallback, resumed watching after refresh, and failure-to-recover paths
- Constraints / non-goals:
  - do not redesign GitHub merge queue behavior
  - do not weaken review, required-check, or merge-closeout safety gates
  - do not reopen unrelated `CO-116`, `CO-104`, `CO-111`, or `CO-138` scope beyond direct reuse
  - do not hide conflict repair behind generic waiting text or silent retries

## Parity / Alignment Matrix
- Current truth:
  - `pr-watch-merge.js` exits with action-required blockers for `merge_state=BEHIND` and `merge_state=DIRTY`
  - `runProviderReviewHandoffPromotion(...)` refuses `BEHIND` and `DIRTY` review handoffs without a recovery attempt
  - `runProviderDeterministicMergeCloseout(...)` refuses `BEHIND` and `DIRTY` `Merging` closeout without a recovery attempt
  - no shared machine-checkable record exists for a branch-refresh or conflict-recovery attempt
- Reference truth:
  - green-but-behind PRs should use a bounded branch-update primitive before surfacing manual action
  - conflicting PRs should move into explicit rework if bounded autonomous recovery cannot proceed
  - review-handoff and merge-closeout should reuse one recovery contract rather than inventing separate heuristics
- Target truth / intended delta:
  - the watcher and provider lifecycle reuse one bounded branch-update recovery path
  - successful branch refresh returns the PR to ordinary readiness watching
  - failed conflict recovery records exact attempt metadata and moves the issue into `Rework` when the failure is conflict-shaped
  - operator-facing proof retains exact blocker and recovery-attempt truth
- Explicitly out-of-scope differences:
  - a generic local merge-conflict resolver
  - merge-queue redesign
  - unrelated shared-root reconciliation or historical attached-PR selection changes

## Not Done If
- Green `BEHIND` PRs still need manual author refresh before review promotion or merge closeout can continue.
- Conflicting PRs still sit indefinitely in `In Review` or `Merging` without a truthful `Rework` move.
- Recovery attempts are only visible in transient logs and not persisted in provider proof.
- The implementation weakens the current merge-ready classifier instead of adding bounded recovery around it.

## Goals
- Add one bounded branch-refresh primitive for `BEHIND` and `DIRTY` PR states.
- Reuse that primitive in both review-handoff promotion and deterministic merge closeout.
- Persist exact recovery attempts and exact blocker truth in provider records.
- Move conflict-shaped failures into truthful `Rework` instead of leaving them in review handoff or merge closeout limbo.

## Non-Goals
- Solving arbitrary merge conflicts by editing files locally and pushing them automatically.
- Broad GitHub CLI redesign.
- Reworking shared-root reconciliation, attached-PR disambiguation, or unrelated Linear cooldown policy.
- Weakening `pr ready-review`, required checks, or review blockers.

## Metrics & Guardrails
- Primary Success Metrics:
  - green `BEHIND` handoffs and `Merging` issues self-refresh through the bounded recovery path
  - conflicting PRs enter `Rework` with explicit conflict metadata instead of waiting silently
  - provider proof and operator projection show the last recovery attempt and blocker reason
- Guardrails / Error Budgets:
  - preserve the current merge-ready blocker classifier after recovery
  - keep recovery non-interactive and bounded to GitHub-side branch update
  - fail closed on repo mismatch, issue-context failures, or unclassified recovery-command failures

## User Experience
- Personas:
  - operator watching review handoff or merge closeout progress in provider proof surfaces
  - worker or reviewer debugging why a green PR still moved into `Rework`
- User Journeys:
  - a green `In Review` PR is `BEHIND`; CO requests a branch refresh, re-reads the snapshot, and then promotes or keeps watching truthfully
  - a green `Merging` PR is `BEHIND`; CO requests the same branch refresh and resumes merge closeout once the PR is clean again
  - a `DIRTY` PR fails bounded recovery with a conflict-style GitHub error; CO records the attempt and transitions the issue to `Rework` with exact next-action truth
