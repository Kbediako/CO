## Summary
`CO-197` adopts the Codex 0.121 MCP Apps metadata that is directly relevant to the delegation MCP server's `initialize` and `tools/list` surfaces. The target is better app-server/operator UX through supported MCP tool fields, especially tool titles and `annotations`, without adding capabilities, loosening input schemas, enabling new tools, or changing authorization. Existing mode visibility, dynamic bridge attestation, confirm-nonce injection, delegation-token rejection, and secret-boundary protections remain authoritative.

## User Request Translation
- Capture official Codex 0.121 MCP/App deltas that affect tool metadata and app-server presentation.
- Audit current delegation MCP `initialize` and `tools/list` behavior.
- Add only supported metadata fields to existing delegation tool descriptors.
- Mark mutating and destructive tools truthfully using MCP annotations where supported.
- Do not add per-tool parallel-call fields; Codex 0.121 parallel MCP execution is a server configuration opt-in.
- Preserve `full`, `question_only`, and `status_only` visibility; preserve bridge attestation, secret rejection, and confirmation nonce guards.
- Prove through tests that tool visibility and guard behavior are unchanged and that the metadata shape is accepted by a Codex 0.121/MCP app-server-compatible client.

## Intent Checksum
- Protected terms and surfaces: `Codex 0.121`, `MCP Apps`, `tools/list`, `initialize`, `annotations`, `readOnlyHint`, `destructiveHint`, `idempotentHint`, `openWorldHint`, `supports_parallel_tool_calls`, `delegate.status`, `delegate.question.poll`, `delegate.cancel`, `github.merge`, `coordinator.status`, `dynamic_tool_bridge_attestation`, `confirm_nonce`, `delegation_token`, `status_only`, `question_only`, `full`.
- Nearby wrong interpretations to reject: no new delegation tools, no dynamic bridge policy change, no provider-worker runtime migration, no marketplace distribution, no blanket `supports_parallel_tool_calls` default, no invented per-tool parallel flag, no schema widening, and no metadata-as-authorization substitution.

## Official Source Notes
- Codex 0.121 release notes include MCP Apps tool calls, namespaced MCP registration, parallel-call opt-in, and sandbox-state metadata for MCP servers.
- Codex MCP config docs define `supports_parallel_tool_calls = true` as a server-level setting and warn to enable it only for servers whose tools are safe to run concurrently.
- Codex config docs expose app/connectors policy keys such as `features.apps`, `apps._default.destructive_enabled`, `apps._default.open_world_enabled`, and per-tool app approval/enabled overrides.
- OpenAI app/tool approval docs treat tools without `readOnlyHint` as write actions and use destructive/open-world hints for app approval posture.
- The MCP schema defines tool `title` and `annotations` with `readOnlyHint`, `destructiveHint`, `idempotentHint`, and `openWorldHint` as client hints, not trusted authorization.

## Acceptance Criteria
- Official Codex 0.121/MCP Apps metadata deltas are captured in the PRD/spec/action plan.
- Current delegation MCP `initialize` and `tools/list` behavior is audited.
- Existing tool input schemas remain unchanged.
- Supported MCP `title` and `annotations` fields are added to delegation tool descriptors.
- Mutating tools carry `readOnlyHint: false`; destructive/confirmation-required tools carry `destructiveHint: true`.
- True read-only/status tools carry `readOnlyHint: true`; `delegate.question.poll` remains write-classified because existing expiry fallback can mutate child run state.
- No unsupported per-tool parallel-call metadata is added.
- Mode visibility, bridge attestation, secret rejection, and confirm-nonce protections are covered by unchanged guard tests.
- A Codex 0.121/MCP app-server smoke asserts metadata-bearing tools list cleanly and only use supported descriptor/annotation fields.

## Non-Goals
- No new delegation capabilities.
- No dynamic bridge policy changes.
- No provider-worker runtime migration.
- No marketplace/plugin distribution changes.
- No app widget/component `_meta` fields.
- No server-level parallel MCP opt-in in CO config.

## Not Done If
- Tool schemas become broader or accept new authority-bearing parameters.
- A tool becomes visible in a mode where it was previously hidden.
- `delegate.question.poll` is advertised as read-only while expiry fallback can still mutate child run state.
- Destructive or confirmation-required tools are left indistinguishable from additive writes.
- Metadata hints are described or implemented as authorization controls.
- Any bridge attestation, secret rejection, or nonce guard test regresses.
