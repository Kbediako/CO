## Summary
- Problem Statement: CO-local provider workers are intended to inherit the current ChatGPT-auth `gpt-5.5` / `xhigh` posture when no `--model` override is passed, but worker proof and read-model artifacts do not record the resolved runtime model slug or the confidence/source for that claim.
- Desired Outcome: provider-worker proof, session-log hydration, CO STATUS, and read-model surfaces expose structured resolved-model provenance so operators can distinguish runtime-reported `gpt-5.5`, config-derived inference, explicit `--model` override, portable `gpt-5.4` fallback, and unknown/degraded evidence.

## User Request Translation
- User intent / needs:
  - stop requiring operators to infer provider-worker model truth from command construction and `~/.codex/config.toml`
  - preserve the current local `gpt-5.5` / `xhigh` posture when available
  - keep `gpt-5.4` as portable fallback only, not the routine local worker target
  - record resolved model provenance in `provider-linear-worker-proof.json`
  - record the same truth or degraded reason in `provider-linear-worker-session-log-hydration.json`
  - expose provenance through `CO STATUS` and read-model output
  - keep binary provenance and auth provenance in their own lanes (`CO-450` and `CO-451`)
- Success criteria / acceptance:
  - no-override provider-worker runs can record inherited `gpt-5.5` truthfully, with source and confidence
  - explicit model overrides, when supported, are recorded distinctly
  - missing runtime model evidence writes a degraded/unknown provenance object instead of pretending certainty
  - `model_reasoning_effort` is captured with the same source/confidence discipline
  - focused tests cover inherited `gpt-5.5`, explicit override, and missing-runtime-model degradation
- Constraints / non-goals:
  - no automatic worker model migration
  - no packaged/default template flip from `gpt-5.4` to `gpt-5.5`
  - no binary provenance work beyond the `CO-450` link
  - no auth provenance work beyond the `CO-451` link
  - no broad lifecycle refactor
  - no docs-freshness or spec-guard baseline repair

## Technical Considerations
- Architectural Notes:
  - Prefer runtime-reported model and reasoning effort when Codex JSON emits them.
  - Otherwise record source/confidence, such as `config_default`, `command_override`, or `unknown`.
  - Missing runtime model evidence is a provenance degradation, not a launch failure.
  - `gpt-5.4` remains the portable fallback context, while local appserver ChatGPT-auth workers should truthfully expose `gpt-5.5` when inherited.
- Dependencies / Integrations:
  - provider-worker proof writer
  - session-log hydration
  - CO STATUS projection
  - read-model output
  - Codex JSON event parsing from `codex exec --json` and `codex exec resume --json`
