## Summary
- Problem Statement: `CO-106` landed the substrate hardening that made shared Linear budgeting truthful, user-scoped, reservation-aware, complexity-aware, and cooldown-preferential, but the provider and control-host paths still burn through the request bucket under low-headroom single-issue operation because polling and helper rereads keep consuming requests before the bucket is actually exhausted.
- Desired Outcome: preserve the current truthful fail-fast substrate while reducing practical request-bucket burn through earlier request-pressure slowdown, reset-aware polling, and more aggressive reuse of already-known authoritative issue/context truth when a fresh scoped cache is already available.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): keep the `CO-106` hardening intact, but stop ordinary low-concurrency work from chewing through the request bucket just because tracked-issue polling and helper rereads keep firing until the bucket reaches `0`. The system should slow earlier, spend fewer redundant reads, and still tell the truth when the request bucket is actually rate-limited.
- Success criteria / acceptance:
  - a reproduced low-concurrency single-issue scenario exhausts the request bucket less often than current `main`
  - request pressure starts slowing polling and helper rereads before `requests.remaining` reaches `0`
  - `providerLinearWorkflowFacade.ts` reuses already-known authoritative issue/context truth when that truth is still sufficiently fresh for the immediate read path
  - the current `CO-106` hardening remains intact: user-scoped budgeting, reservation-aware preflight, complexity-aware checks, safe observation merge, and truthful fail-fast behavior
  - focused regressions preserve the distinction between request-bucket exhaustion and complexity-bucket exhaustion
- Constraints / non-goals:
  - do not reopen `CO-106` substrate hardening
  - do not reopen `CO-108` next-refresh display truth
  - do not broaden into webhook migration or provider-control-plane redesign
  - do not weaken truthful `linear_rate_limited` fail-fast behavior

## Not Done If
- A low-concurrency single-issue workload still predictably burns through the request bucket with little or no earlier slowdown.
- Repeated helper rereads still drive request exhaustion even when local authoritative issue/context truth is already available.
- The only change is observability wording with no measurable request-pressure reduction.

## Goals
- Slow tracked-issue polling earlier when the request bucket is running ahead of its reset horizon.
- Reuse authoritative cached issue/context truth more aggressively on read-only helper paths when request headroom is already degraded.
- Preserve and document the distinction between request exhaustion and complexity exhaustion.
- Add focused regression coverage and artifact-backed proof around request-burn reduction.

## Non-Goals
- Reopening the `CO-106` substrate contract.
- Reworking control-host/provider orchestration architecture.
- Relaxing `linear_rate_limited` fail-fast semantics.
- Treating request exhaustion with high remaining complexity as a correctness bug by itself.
