## Summary
- Problem Statement: retrieval-stage experience selection is still plain top-K by raw reward sum. `orchestrator/src/persistence/ExperienceStore.ts` ranks candidates by `gtScore + relativeRank`, prompt-pack manifests only declare `experienceSlots`, and run manifests only expose the final injected snippet strings. That leaves no explicit competitive scoring stage, no anti-dominance normalization, no controller-declared retrieval policy, and no inspectable explanation for why repeated-source candidates were penalized or excluded.
- Desired Outcome: add a bounded retrieval-policy slice that defines literal retrieval-stage `competitive scoring`, applies inspectable `anti-dominance normalization` using existing experience provenance, and exposes the policy plus selection diagnostics through controller-readable prompt-pack and manifest surfaces without broadening into TF-GRPO training, telemetry, or runtime continuity work.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): land the bounded sibling autonomy-stream slice that makes retrieval-stage memory selection explicitly policy-driven and diversity-aware so one repetitive source does not dominate the fixed-model memory injections. The policy must be declared through controller-readable config and must expose why candidates were selected, penalized, or dropped.
- Success criteria / acceptance:
  - retrieval-stage scoring is explicitly defined and implemented
  - dominance penalties or diversity controls are inspectable
  - tests cover repeated-source suppression and ranking changes
  - the policy is controller-consumable rather than hidden in store-local heuristics
- Constraints / non-goals:
  - this is not model training, not TF-GRPO reward persistence, and not a new umbrella
  - do not duplicate `CO-82`, `CO-83`, `CO-89`, or `CO-90`
  - do not reopen `0303`
  - keep the slice bounded to retrieval policy, manifest surfaces, and focused tests

## Parity / Alignment Matrix
- Current truth:
  - `ExperienceStore.fetchTop(...)` uses raw `gtScore + relativeRank` ordering with only a `minReward` threshold
  - prompt-pack manifests expose `experienceSlots` but no retrieval-policy contract
  - run manifests expose selected snippet strings but no scoring policy, source-penalty explanation, or diagnostics path
  - stored experience provenance already exists via `runId`, `groupId`, `taskId`, `stampSignature`, and `manifestPath`
- Reference truth:
  - existing prompt-pack manifests are already the controller-owned config seam for experience injection
  - existing experience records already carry enough provenance to derive repeated-source groups without inventing a separate memory backend
- Target truth / intended delta:
  - prompt-pack manifests declare an explicit retrieval policy for experience selection
  - retrieval runs a competitive scorer that combines weighted reward terms with anti-dominance normalization
  - selected runs persist manifest-visible diagnostics that explain selected vs penalized candidates and repeated-source suppression
- Explicitly out-of-scope differences:
  - changing TF-GRPO rewarder semantics
  - adding a new memory backend or model
  - broader runtime observability or worker-host parity

## Not Done If
- selection is still plain top-K reward or recency.
- there is no visibility into why a source was excluded or penalized.
- repeated-source suppression and ranking changes are untested.
- the issue broadens into unrelated telemetry or runtime continuity work.

## Goals
- Declare retrieval-stage policy on the controller-owned prompt-pack surface.
- Implement competitive scoring with anti-dominance normalization over existing experience provenance.
- Persist bounded selection diagnostics so operators and controllers can inspect scoring and exclusion reasons.
- Keep prompt consumers backward-compatible with existing string snippet injection.
