## User Request Translation

Normalize the shipped MCP client configs so client launchers do not receive the whole command line as one string. The current root `mcp-client.json` and `templates/codex/mcp-client.json` encode MCP server startup as `"command": "codex-orchestrator mcp serve"`. The target shape is a split executable and argument vector: `"command": "codex-orchestrator"` plus `"args": ["mcp", "serve"]`.

The plugin-local `plugins/codex-orchestrator/.mcp.json` is a protected reference surface because it already uses split `command` plus `args` with `node`, `./launcher.mjs`, `mcp`, and `serve`. CO-271 should align the shipped root/template client configs with the same structured launch contract without changing plugin launcher semantics.

## Problem Statement

Many MCP clients treat `command` as an executable path and `args` as the argument vector. A config that stores `codex-orchestrator mcp serve` as the executable can fail in clients that do not invoke a shell or split strings. That makes the shipped root and template MCP configs less portable than the plugin `.mcp.json` reference shape.

This is a packaging/config normalization issue, not a runtime MCP server behavior issue.

## Desired Outcome

- `mcp-client.json` uses `command` for only the executable and `args` for `mcp serve`.
- `templates/codex/mcp-client.json` uses the same split `command` plus `args` shape.
- `plugins/codex-orchestrator/.mcp.json` remains split and is not regressed to a single command string.
- Existing MCP server names are preserved unless parent implementation evidence proves a rename is required.
- The actual server invocation remains equivalent to `codex-orchestrator mcp serve`.
- The change is validated as JSON plus config-shape parity, with downstream package smoke retained by the parent lane because shipped templates are downstream-facing.
