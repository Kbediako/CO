## Problem
After a provider-worker attempt is reset into live `Rework`, the current issue truth can already show `Rework`, the retired PR can already be closed, and no new PR may be attached yet. Even so, `co-status`, intake-derived projection, and `issue-context` can still project the retired attempt as current by surfacing stale rehydrated `running` / `Merging` state and stale closed attachment `#509`.

## Contract
- Preserve exact surfaces: `CO STATUS`, intake-derived projection, `issue-context`, `Rework`, stale rehydrated `running`, stale rehydrated `Merging`, closed PR attachment, `#509`, `no attached PR yet`.
- Reject these narrower interpretations: display-only cleanup, attachment-filter-only cleanup, or historical-artifact deletion.
- Keep this lane bounded away from `CO-210`, `CO-214`, and `CO-219`.

## Parity Matrix
- Current: live issue can be `Rework` while stale rehydrated `running` / `Merging` and closed `#509` still look current.
- Reference: a reset attempt with no new PR should read as `Rework` plus `no attached PR yet`, with stale evidence still inspectable.
- Target: all status/projection surfaces agree on current `Rework` truth, stale `#509` becomes historical-only, and real live `Merging` plus real current PRs still project correctly.

## Acceptance Criteria
- Reproduce the retired-attempt reset shape in focused tests.
- Stop projecting the retired attempt as current `running` / `Merging` after live reset to `Rework`.
- Stop presenting closed stale attachment `#509` as the current PR when the reset attempt has no new PR yet.
- Keep historical evidence inspectable and keep real live `Merging` / current PR truth intact.

## Non-Goals / Not Done If
