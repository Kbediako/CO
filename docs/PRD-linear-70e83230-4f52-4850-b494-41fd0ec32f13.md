## Summary
- Problem Statement: CO already has one important resident-session building block from `1312`: within a single `provider-linear-worker` lifetime, the worker can keep one live thread/session lineage alive across normal successful turns. The remaining gap is the outer runtime/control seam. Control-host refresh and rehydrate logic still reason mostly in terms of run manifests, retry queues, and fresh worker starts, so bounded restart continuity remains under-specified. After a supervised restart boundary such as `max_turns_reached_issue_still_active`, CO can relaunch provider work, but it does not yet expose an explicit guarded resident-session contract for drain, restart, and resume on the same logical resident thread/session lineage where that is safe.
- Desired Outcome: add one truthful, guarded resident-session control seam that makes restart-safe drain/restart/resume continuity explicit at completed-turn boundaries, preserves stable logical thread/session lineage where possible, surfaces continuity state in runtime proof/read models, and documents the exact boundary: no mid-turn hot reload and no claim of BEAM-style in-memory live upgrade.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): strengthen the existing Node/TypeScript provider-worker supervision path, so CO can keep longer-lived resident app-server style sessions under clearer restart control without rewriting the runtime or overclaiming hot upgrade. The new seam should sit above the current worker-lifetime thread/session reuse and define how drain-safe restarts resume the same logical resident session, how unsafe failures fall back, and what operators can trust from runtime proof/state.
- Success criteria / acceptance:
  - one bounded runtime/control seam owns resident-session restart continuity intentionally instead of leaving it implicit in refresh or retry helpers
  - completed-turn drain/restart/resume behavior is deterministic, tested, and visible in proof or read-model state
  - stable logical thread/session continuity persists across guarded supervised restarts when the previous boundary is restart-safe
  - config or policy reload expectations are documented truthfully as restart-boundary behavior, not mid-turn hot reload
  - the lane stays inside the existing Node/TypeScript architecture and does not reopen CO-82, CO-83, CO-88, distributed worker scheduling, or a language rewrite
- Constraints / non-goals:
  - no Elixir / BEAM rewrite
  - no claim of true in-place live code upgrade with preserved in-memory process state
  - no distributed or SSH worker-host scheduling
  - no duplication of existing telemetry or observability cleanup lanes
  - no widening into remote `thread/shellCommand` or watcher authority surfaces by default

## Parity / Alignment Matrix
- Current truth:
  - `1312` already keeps one worker-lifetime live thread/session lineage inside a single `provider-linear-worker` run
  - `providerIssueHandoff.ts` can rehydrate active runs and queue retries for completed ones, but continuation after a completed worker boundary still falls back to run-level refresh or a fresh worker launch contract rather than an explicit resident-session restart contract
  - provider proof already records `thread_id`, `latest_session_id`, `turn_count`, `owner_phase`, `owner_status`, and `end_reason`, but it does not define a stable resident logical-session contract across supervised worker restarts
  - config or policy changes are effectively picked up on later launches, but the boundary is implicit rather than documented or surfaced as a guarded continuity rule
- Reference truth:
  - Symphony-style long-lived ownership informed this lane: one authoritative issue owner, truthful thread/session lineage, and restart-safe continuation only at explicit safe boundaries
  - the reference does not require BEAM-style hot code upgrade; it requires truthful restart continuity semantics
- Target truth / intended delta:
  - when a provider worker exits at a restart-safe completed-turn boundary, the next supervised worker can resume the same logical resident thread/session lineage using an explicit guarded continuity contract
  - proof and debug surfaces show whether the current worker is a fresh resident start or a guarded resident resume, plus the restart count and source boundary
  - unsafe exits remain fail-closed and do not pretend they can hot-resume the same resident state
- Explicitly out-of-scope differences:
  - distributed worker-host orchestration
  - mid-turn hot reload or in-memory state preservation
  - unrelated telemetry cleanup or UI work

## Not Done If
- Resident-session ownership is still implicit or fragmented across restart helpers.
- Restart/resume continuity remains undefined or best-effort after a completed-turn restart-safe boundary.
- Docs imply true hot upgrade when the implementation only supports drain/restart/resume.
- The lane broadens into a runtime rewrite or unrelated parity work.

## Goals
- Define one guarded resident continuity contract above the existing worker-lifetime session reuse.
- Resume the same logical resident thread/session lineage across restart-safe supervised worker restarts when prior proof permits it.
- Surface resident continuity state in proof/read-model data, so operators can distinguish fresh starts from guarded resumes.
- Keep reload truth explicit: new policy/config applies on restart boundaries only.
- Add focused tests around completed-turn drain or restart continuity and fail-closed unsafe boundaries.

## Non-Goals
- Elixir or BEAM migration.
- Mid-turn hot reload or preserved in-memory live upgrade.
- Remote app-server command surface expansion.
- CO-82, CO-83, CO-88, or distributed worker-host scope.
