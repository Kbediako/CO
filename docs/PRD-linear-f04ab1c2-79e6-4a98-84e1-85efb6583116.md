## Summary
- Problem Statement: control-host status projections still carry legacy proof/status fallback behavior across CLI, API, and UI read models. CO-398 must apply CO-382 `fallback expiry` discipline before implementation so operators do not keep adding another `minor seam` whenever status authority disagrees.
- Desired Outcome: create the CO-398 docs-first packet and mirrors, then have the parent implementation inventory each control-host status fallback path and choose `remove fallback`, `expire fallback`, or `justify retaining fallback` with owner, trigger, review date, maximum lifetime, removal condition, and focused projection validation.

## User Request Translation
- User intent / needs: stop normalizing fallback-heavy control-host status projections. The implementation must treat `control-host status surfaces` as a governed high-churn surface and explicitly decide which legacy proof/status fallback paths are removed, which temporarily expire, and which are durable audit contracts.
- Success criteria / acceptance:
  - PRD, canonical TECH_SPEC, TECH_SPEC mirror, ACTION_PLAN, task checklist, and `.agent` mirror exist for `linear-f04ab1c2-79e6-4a98-84e1-85efb6583116`

  - packet preserves the protected terms and exact surfaces named by the Linear issue
  - packet includes a CO-382 fallback decision inventory for control-host status projection paths
  - parent implementation covers CLI/API/UI status projection behavior without weakening `/ui/data.json` truth or source labels
  - focused validation covers the touched control-host projection surfaces
- Constraints / non-goals:
  - no control-host implementation in this packet lane
  - no provider workflow, review-wrapper, runtime routing, docs freshness, supervision, launchctl, or provider worker lifecycle redesign
  - no removal of audit-visible legacy proof without focused CLI/API/UI regression coverage
