# PRD - CO workflow: automate docs:freshness baseline upkeep and diff-local gating

## Added by Bootstrap 2026-04-15

## Summary
- Problem Statement: `docs:freshness` remains a truthful repo-wide gate, but date-boundary historical task/report cohorts still turn unrelated provider-worker lanes red. Recent CO-183, CO-184, and CO-175 evidence shows agents can spend product-lane validation time diagnosing historical packet debt that should have an autonomous owner path before review handoff.
- Desired Outcome: add an autonomous docs freshness maintenance path and provider-worker gate contract that separates current-diff/task-packet blockers from owned, eligible, in-window repo-wide historical debt while preserving fail-closed behavior for real drift.

## User Request Translation (Context Anchor)
- User intent / needs (in your own words): implement CO-188 as more than another date bump. The repo needs a non-interactive machine-readable decision that classifies docs freshness failures, creates or updates one owner path for historical cohorts, and lets provider-worker review gates pass only when unrelated owned rolling debt is safe and the current diff/task packet is clean.
- Success criteria / acceptance:
  - a maintenance command or workflow emits deterministic JSON with `freshness_decision`, owner/action fields, diff-local blockers, policy capacity/window status, and samples
  - owned, eligible, in-window historical debt remains visible but no longer blocks unrelated clean provider-worker lanes
  - public/active guidance, missing/invalid/uncatalogued docs, expired or over-budget rolling debt, and current-task packet drift remain hard blockers
  - CO-175 and CO-184 recurrence shapes are covered by tests/evals
- Constraints / non-goals:
  - do not weaken `docs:freshness`, `spec-guard`, `docs-review`, or `implementation-gate`
  - do not use warning-only validation as the normal provider-worker answer
  - do not silently bump `last_review`
  - do not raise `max_entries=300` or the window just to pass this issue

## Parity / Alignment Matrix
- Current truth:
  - `docs:freshness` already distinguishes blocking stale entries from configured rolling cohort entries in reports.
  - `spec-guard` already uses the same rolling cohort catalog for stale active specs.
  - `docs-review` and `implementation-gate` run blocking freshness checks, so unrelated provider-worker lanes still stop on repo-wide historical debt that is not autonomously owned or routed.
  - CO-175 added one static rolling cohort policy with `max_entries=300`; CO-184 showed the Apr 15 `173` row cohort would exceed capacity if added beside the existing `221`.
- Reference truth:
  - CO-175 preserved Apr 14 rolling debt visibility under owner issue `CO-175`.
  - CO-184 classified the Apr 15 over-cap shape and refreshed reviewed rows instead of expanding policy caps.
- Target truth / intended delta:
  - add a maintenance decision layer above existing freshness/spec data so future lanes can cite machine-readable ownership, capacity, expiration, and diff-local blocker evidence.
  - add a non-interactive scheduled/preflight path that opens or updates one owner/action path for eligible historical cohorts before unrelated provider-worker review gates trip.
  - teach provider-worker gate behavior to consume the decision instead of parsing console text or requiring manual cohort diagnosis.
- Explicitly out-of-scope differences:
  - changing the underlying freshness cadence semantics for active docs
  - broad archiving of historical packets without a separate review/archive rationale
  - replacing Linear owner issue workflows with a new external tracker

## Not Done If
- Agents still need to manually diagnose or date-bump historical `docs:freshness` cohorts after UTC rollover.
- Provider workers still block unrelated review handoff on owned, eligible, repo-wide historical debt while their current diff/task packet is clean.
- Gates pass when public docs, active guides, missing registry rows, invalid rows, uncatalogued docs, expired rolling cohorts, over-budget rolling cohorts, or current-task packet drift are present.
- CO-175/CO-184 rolling and refresh provenance becomes less visible in reports or owner issues.
- The implementation lacks a deterministic machine-readable decision that future agents can cite.

## Goals
- Add a docs freshness maintenance command/workflow that runs non-interactively and emits stable JSON.
- Preserve existing fail-closed checks for real drift.
- Add deterministic owner/action evidence for historical cohort debt.
- Add provider-worker gate support for diff-local pass/fail decisions.
- Cover recurrence fixtures for Apr 14 and Apr 15 baseline shapes.

## Non-Goals
- Disable or weaken `docs:freshness`, `spec-guard`, `docs-review`, or `implementation-gate`.
- Convert provider-worker validation to blanket warn-only freshness checks.
- Raise rolling cohort caps/window to make unrelated lanes pass.
- Silently bump `last_review` without a review, archive, reclassification, or owner decision.
- Move public docs, active guides, agent policy, shipped skills/templates, missing rows, invalid rows, or uncatalogued docs into historical deferral.

## Stakeholders
- Product: CO operators and provider-worker authors who need unrelated feature lanes to hand off without manual baseline repair.
- Engineering: maintainers of docs truthfulness, provider-worker validation, orchestration pipelines, and Linear owner issue automation.
- Design: not applicable.

## Metrics & Guardrails
- Primary Success Metrics:
  - maintenance decision JSON is deterministic and includes owner, candidate cohort, blocker, capacity/window, expiration, recommended action, and sample path fields
  - provider-worker gates can pass with only owned in-window repo-wide debt and a clean current diff/task packet
  - missing/invalid/uncatalogued/public/active/current-task drift still blocks
- Guardrails / Error Budgets:
  - no silent `last_review` updates
  - no cap/window expansion for convenience
  - no duplicate owner issues for the same detected cohort/batch
  - no parsing of human console output where JSON decisions are available

## User Experience
- Personas:
  - provider-worker author preparing a review handoff
  - CO maintainer triaging date-boundary freshness debt
  - reviewer auditing why a lane passed despite rolling debt
- User Journeys:
  - A scheduled/preflight maintenance command detects a historical cohort, records whether it is owned, in-window, and within policy capacity, and opens or updates a single owner/action path.
  - A provider-worker gate consumes the same decision and allows unrelated clean work to proceed only when the historical debt is safe and visible.
  - A true blocker such as uncatalogued public docs or current task packet drift still fails fast with sample paths.

## Technical Considerations
- Architectural Notes:
  - Reuse `runDocsFreshness(...)` report data rather than duplicating registry parsing.
  - Keep `spec-guard` as a fail-closed freshness consumer for active specs; any maintenance decision must not override spec failures for current-diff specs.
  - Prefer one JSON decision contract that workflow automation and provider-worker gates can consume.
- Dependencies / Integrations:
  - `scripts/docs-freshness.mjs`
  - `scripts/spec-guard.mjs`
  - `docs/docs-catalog.json`
  - `docs/docs-freshness-registry.json`

  - `codex.orchestrator.json`
  - provider-worker validation/handoff code
  - `.github/workflows/docs-truthfulness-weekly.yml`
