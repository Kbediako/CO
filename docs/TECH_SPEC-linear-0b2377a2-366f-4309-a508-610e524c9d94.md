# TECH_SPEC - CO: add supported provider-worker relaunch/recover command and guard direct starts

## Added by Docs Child Lane 2026-04-26

## Context

`CO-393` is the follow-up lane for making provider-worker relaunch/recover/nudge safe and operator-supported. During `CO-392` recovery, direct `codex-orchestrator start provider-linear-worker` launches could create confusing provider-worker truth: wrong workspace path, split root/worktree artifacts, null manifest provenance, or provider-intake `state=running` with missing `launch_source`. Same-issue child-lane helpers then failed later because the control-host provenance tuple was absent.

The required contract is narrow: supported operator relaunch/recover/nudge must go through root control-host for a specific Linear issue, while direct `codex-orchestrator start provider-linear-worker` outside matching control-host provenance fails closed with an actionable command hint.

The issue-shaping contract requires exact preservation of:

- `root control-host relaunch/recover/nudge`
- `specific Linear issue`
- `codex-orchestrator start provider-linear-worker`
- `matching control-host provenance`
- `actionable command hint`
- `deterministic issue worktree cwd`
- root `.runs` / `out` routing
- manifest `provider_launch_source=control-host`
- `provider_control_host_task_id`
- `provider_control_host_run_id`
- provider-intake `launch_source=control-host`
- non-null launch-token/intake reservation truth
- duplicate-worker protection/provider admission caps
- `CO-392` stale pending-reopen refresh logic remains out of scope
- existing root control-host launcher remains in place

## Requirements

1. Add an operator-supported root control-host relaunch/recover/nudge path for a specific Linear issue.
2. Supported relaunch/recover must use root control-host state and existing provider admission/reservation semantics.
3. Supported relaunch/recover must launch or resume work in the deterministic issue worktree cwd while routing artifacts to root `.runs` / `out`.
4. Supported relaunch/recover must persist manifest `provider_launch_source=control-host`, `provider_control_host_task_id`, and `provider_control_host_run_id`.
5. Supported relaunch/recover must persist provider-intake `launch_source=control-host` plus non-null launch-token/intake reservation truth.
6. Direct `codex-orchestrator start provider-linear-worker` outside matching control-host provenance must fail closed before creating provider-worker run truth.
7. Direct-start failure must include an actionable command hint for the supported root control-host relaunch/recover/nudge path.
8. Same-issue child-lane helpers must work after supported relaunch/recover because the manifest and provider-intake provenance are complete.
9. Preserve duplicate-worker protection/provider admission caps and do not bypass active worker safety.
10. Preserve `CO-392` stale pending-reopen refresh/reclaim scope, `CO-244` provenance behavior, and the existing root control-host launcher.
