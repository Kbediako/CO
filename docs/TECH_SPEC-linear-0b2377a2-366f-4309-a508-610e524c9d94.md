# TECH_SPEC - CO-393 control-host provider-worker recovery

## Context
`CO-393` adds an operator-supported recovery path for a specific Linear provider worker and guards unsafe direct starts. The source issue is anchored at `ctx:sha256:69a80e9d4a8a919410857a6ff81972d38b23e6ec4a7988ddfa79bb23dd6505e1#chunk:c000001`, with payload `../../.runs/linear-0b2377a2-366f-4309-a508-610e524c9d94/cli/2026-04-26T17-26-50-546Z-360e85a4/memory/source-0/source.txt`.

## Requirements
1. Add `control-host recover|relaunch|nudge --issue-id <id>` under the root control-host surface.
2. Resolve the target issue through root control-host state and reuse existing provider admission/reservation/launch paths.
3. Launch or resume in the deterministic issue worktree while routing artifacts to root `.runs` / `out`.
4. Persist manifest `provider_launch_source=control-host`, `provider_control_host_task_id`, and `provider_control_host_run_id`.
5. Persist provider-intake `launch_source=control-host` and non-null launch-token/intake reservation truth.
6. Fail closed for `codex-orchestrator start provider-linear-worker` unless matching control-host provenance and launch token are present.
7. Include an actionable hint for the supported recovery command.
8. Preserve child-lane compatibility and adjacent `CO-244`, `CO-392`, and `CO-189` invariants.

## Protected Issue Contract
- Exact terms: `root control-host relaunch/recover/nudge`, `specific Linear issue`, `codex-orchestrator start provider-linear-worker`, `matching control-host provenance`, `actionable command hint`, `deterministic issue worktree cwd`, root `.runs` / `out`, `provider_launch_source=control-host`, `provider_control_host_task_id`, `provider_control_host_run_id`, provider-intake `launch_source=control-host`, launch-token/intake reservation truth, `provider-intake-state.json`.
- Non-goals: no `CO-392` stale pending-reopen expansion, no `CO-244` schema introduction, no duplicate-worker/admission weakening, no manual intake-state repair, no promotion of direct issue-worktree starts.
- Micro-task path is unavailable because correctness depends on exact protected surfaces, parity/alignment, launch provenance, and fallback/seam behavior.

## Design
- `startCliRequestShell` performs the direct-start guard before `provider-linear-worker` dispatch.
- `controlHostProviderWorkerRecoverCliShell` attaches to the running control host and POSTs issue/action JSON to `/api/v1/provider-worker/recover`.
- `observabilityApiController` exposes the authenticated compatibility API endpoint.
- `controlAuthenticatedRouteHandoff` delegates API requests to `ProviderIssueHandoffService.recoverIssue`.
- `providerIssueHandoff.recoverIssue` resolves the specific issue, then reuses `processTrackedIssueCandidate` for existing admission, duplicate-worker protection, launch-token generation, and claim updates.
- Release resolutions from direct issue lookup release any stale existing claim so inactive/out-of-scope issues do not keep active admission truth.

## Interfaces
- CLI: `codex-orchestrator control-host recover|relaunch|nudge --issue-id <linear-issue-id> [--format json] [--request-timeout-ms <n>]`.
- API: `POST /api/v1/provider-worker/recover` with `issue_id`/`issueId` and optional `action` of `recover`, `relaunch`, or `nudge`.
- Direct-start failure: throws before run creation and points to `codex-orchestrator control-host recover --issue-id <id> --format json`.
- Recovery result summarizes claims without exposing raw launch tokens; it reports `launch_token_present`.

## Validation Plan
- Unit regressions for direct-start guard allow/deny behavior.
- API regression for authenticated provider-worker recovery request/response.
- Provider handoff regressions for successful recovery launch-token/provenance and release of stale claims when direct issue lookup resolves as gone.
- Existing child-lane provenance compatibility regression remains green.
- Full gates: delegation guard, spec guard, build, lint, test, docs check/freshness, repo stewardship, diff budget, pack smoke, standalone review, elegance pass, PR ready-review.

## Fallback / Refactor Decision
| Surface | Decision | Rationale |
| --- | --- | --- |
| Root control-host launch authority | justify retaining fallback | It is the intended non-expiring operator authority for admission, tokens, provenance, and run truth. |
| Direct provider-worker start without matching provenance | remove fallback | It is the unsafe seam that created confusing run truth; it now fails closed. |

No migration is required; this is an additive command/API plus a guard around the existing start path.
