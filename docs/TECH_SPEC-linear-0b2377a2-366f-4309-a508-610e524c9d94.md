## Requirements
1. Resolve the target issue through root control-host state and reuse provider admission, duplicate-worker protection, launch-token generation, and launch/resume paths.
2. Preserve deterministic issue worktree cwd with root `.runs` / `out`, manifest control-host provenance, provider-intake `launch_source=control-host`, and non-null launch-token/reservation truth.
3. Fail closed for direct provider-worker starts unless matching control-host provenance and launch token are present, with an actionable recovery hint.
4. Preserve child-lane compatibility and adjacent `CO-244`, `CO-392`, and `CO-189` invariants.

## Protected Contract
Keep exact terms: `root control-host relaunch/recover/nudge`, `specific Linear issue`, `matching control-host provenance`, `deterministic issue worktree cwd`, `provider-intake-state.json`, `provider_launch_source=control-host`, `provider_control_host_task_id`, `provider_control_host_run_id`, provider-intake `launch_source=control-host`, and launch-token/intake reservation truth. Do not broaden `CO-392`, redo `CO-244`, weaken `CO-189`, hand-edit intake state, or promote direct starts. Micro-task path is unavailable because this is a parity/fallback provenance lane.

## Design
- `startCliRequestShell` blocks `provider-linear-worker` direct starts unless env provenance and the root provider-intake reservation match the requested issue/identifier/token/task/run.
- `controlHostProviderWorkerRecoverCliShell` attaches to the running control host and POSTs to `/api/v1/provider-worker/recover`.
- `observabilityApiController` and authenticated route handoff expose the API.
- `ProviderIssueHandoffService.recoverIssue` resolves the issue, reuses `processTrackedIssueCandidate`, and releases stale existing claims when direct lookup says gone/out-of-scope.
- Recovery output summarizes `launch_token_present` without exposing raw tokens.
