# Technical Specification

## Context
`providerLinearWorkflowFacade.ts` owns the provider Linear issue-context helper. During `getProviderLinearIssueContext(...)`, the facade reads Linear issue context and then calls `hydrateIssuePullRequestAttachments(...)`. That hydration path extracts unique GitHub PR attachment URLs and resolves each PR snapshot through `resolvePullRequestSnapshot(...)`, whose default implementation is `fetchIssueContextPullRequestSnapshot(...)` using `gh pr view --json state,mergedAt,updatedAt`.

The current implementation awaits each snapshot in a serial loop. This preserves simple per-candidate fallback behavior but makes issue-context hydration slower when an issue has several independent PR attachments. `CO-230` is the bounded scheduling lane for that loop. It should improve call shape and latency without changing which PR is `current`, `historical`, `conflicting`, or `unknown`.

This issue is explicitly not the `CO-220` correctness lane. `CO-220` and later merge-closeout packets own stale closed-unmerged prior-attempt PR selection behavior. `CO-230` must keep that semantic boundary intact.

## Requirements
1. Keep GitHub PR URL parsing and duplicate suppression before snapshot scheduling.
2. Resolve unique PR snapshots through bounded concurrency with an explicit cap.
3. Avoid unbounded fanout even when an issue has many PR attachments.
4. Preserve the injected resolver contract used by `getProviderLinearIssueContext(...)` tests.
5. Preserve `fetchIssueContextPullRequestSnapshot(...)` mapping for `state`, `mergedAt`, and `updatedAt`.
6. Preserve per-candidate fallback: thrown resolver calls and unmappable snapshots classify only that attachment as `unknown`.
7. Preserve `classifyIssuePullRequestAttachments(...)` behavior for the same resolved/unknown candidate set.
8. Preserve the `pull_request_attachments` output shape and `provider-linear-issue-context-cache.json` compatibility.
9. Keep implementation bounded to `orchestrator/src/cli/control/providerLinearWorkflowFacade.ts` unless parent evidence proves a tiny local helper file is necessary.

## Design
- Replace the serial snapshot loop inside `hydrateIssuePullRequestAttachments(...)` with a small bounded scheduler.
- The scheduler should accept the deduped candidate list and a cap, run at most that many resolver calls at once, and return per-candidate outcomes.
- Successful resolver output should still flow through `mapIssuePullRequestAttachmentSnapshot(...)`.
- Invalid snapshots and thrown resolver calls should still add only that candidate attachment to `unknown`.
- The resulting `resolved` and `unknown` collections should then flow into `classifyIssuePullRequestAttachments(...)` without changing classification logic.
- Prefer a local fixed production cap, such as `4`, with test visibility through an injectable option or local helper if needed.

## Reject These Wrong Interpretations

- `Use unbounded Promise.all over every attachment.`

## Current Truth
- `hydrateIssuePullRequestAttachments(...)` builds a deduped list of PR attachment candidates.
- It currently awaits `resolvePullRequestSnapshot(...)` serially.
- Each resolver failure adds the candidate attachment to `unknown`.
- `classifyIssuePullRequestAttachments(...)` then derives `current`, `historical`, `conflicting`, and `unknown`.

## Proposed Design
- Keep all parsing, dedupe, mapping, and classification code semantically equivalent.
- Add a bounded concurrency helper for the resolver phase only.
- Preserve deterministic result collection by associating each result with its original candidate.
- Keep all downstream classification decisions based on the same candidate/snapshot truth as today.

## Non-Goals
- No `CO-220` correctness-scope change.
- No merge-closeout, review-promotion, or readiness-gate change.
- No unbounded fanout.
- No broad GitHub API redesign.
- No Linear attachment cleanup.
- No registry, Linear, workpad, source, or test edits in this child lane.

## Parity / Alignment Matrix
- Current truth:
  - unique PR attachment snapshots are fetched serially
  - duplicate GitHub PR URLs are already suppressed
  - one failed lookup becomes one `unknown` attachment
  - classification is handled after snapshot resolution
- Reference truth:
  - independent snapshot reads can run concurrently if concurrency is bounded
  - semantics must be defined by resolver outcomes, not call order
  - adjacent correctness lanes remain authoritative for PR selection behavior
- Target truth / intended delta:
  - unique PR attachment snapshots are fetched with bounded concurrency
  - max in-flight resolver calls are capped
  - same resolver outcomes produce the same classification output
- Explicitly out-of-scope differences:
  - `CO-220` selection correctness
  - merge-closeout/review-promotion behavior
  - unbounded fanout
  - cache schema rewrite

## Not Done If
- The hydration path is still serial for multiple unique PR attachments.
- The implementation can run all PR snapshot lookups at once without a cap.
- Any unchanged resolver outcome changes `current`, `historical`, `conflicting`, or `unknown`.
- A single resolver failure fails the whole `issue-context` operation.
- The patch modifies CO-220-style stale attachment correctness or readiness gate behavior.
