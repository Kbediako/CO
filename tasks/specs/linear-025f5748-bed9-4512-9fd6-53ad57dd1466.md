## Target Contract

The parent implementation should:

- consider only live `Blocked issues`
- classify every blocker in the `blocker list`
- suppress actionable recommendations when any blocker is non-terminal or unknown
- surface all-terminal blocker issues as operator-visible advisory records
- include `issue id`, `live state`, `blocker list`, `canonical/duplicate hints`, and `recommended action`
- use `ready-to-unblock candidate` for unblock review
- use `duplicate-cleanup candidate` for duplicate or canonical cleanup review
- keep the advisory read-only
- reuse CO-259 canonical-owner behavior with the exact key `blocked-terminal-blocker-cleanup-advisory` when ownership/follow-up reuse is needed

## Parent Implementation Closeout

- Parent implementation surface: `operator-autopilot` latest result, provider workflow store, and observability read-model projection.
- Advisory records are additive and read-only: no Linear transition helper is called from terminal-blocker advisory generation.
- Terminal blocker truth uses the existing provider workflow-state classifier so completed, canceled/cancelled, duplicate, and done blocker states remain terminal while unknown or non-terminal blockers suppress the advisory.
- Duplicate-cleanup evidence comes from exact duplicate relation hints or exact canonical-owner markers in issue descriptions. Otherwise all-terminal Blocked issues become `ready_to_unblock` candidates.
- Durable evidence includes `issue_id`, `issue_identifier`, `issue_state`, `issue_state_type`, `issue_updated_at`, `blockers`, `canonical_owner_hints`, `duplicate_hints`, `recommended_action`, and `summary`.
