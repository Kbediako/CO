# Technical Specification

## Context

`CO-41` addresses a stale-live-instance incident on March 30, 2026 where the local `co-control-host` stopped admitting new `Ready` issues until restart. The stale instance left `.runs/local-mcp/cli/control-host/provider-intake-state.json` frozen at `2026-03-30 09:07:19 +1100`, continued to show stale `CO-38` truth, timed out valid control-host refresh and read requests, and failed to launch `CO-39` and `CO-40` even though the same tracked-issue query still returned them as eligible. An out-of-process replay of the current provider refresh logic finished in about 14.9 seconds and would have released the stale truth, which strongly suggests a wedged serialized refresh lifecycle rather than a pure eligibility bug.

## Requirements

1. A single wedged provider refresh or poll lifecycle must not silently freeze admission of newly eligible `Ready` issues.
2. If a refresh or poll cycle exceeds its safe budget, operator-facing health surfaces must expose an explicit degraded, stuck, or restart-required condition instead of timing out ambiguously.
3. Recovery behavior must be explicit and machine-checkable: either the host safely re-arms or releases the stuck lifecycle, or it surfaces a truthful restart-required signal.
4. The implementation must preserve serialized correctness for provider refresh work; any watchdog or recovery path must not create concurrent refresh corruption or duplicate issue claims.
5. A focused reproducer, regression, or diagnostic harness must cover the stale-live-instance / healthy-refresh-replay class.
6. Closeout must identify the actual blocking seam and how the repair addresses it.

## Current Truth

- `controlServerPublicLifecycle.ts` serializes public provider refresh work through one active operation plus queued refresh state.
- `providerIssueHandoff.ts` serializes rehydrate, refresh, and poll work through `refreshLifecycleChain`.
- If a refresh-path promise never resolves, meaningful provider progress can stop while the process still appears live.
- Once the public lifecycle sets `state.active` to that wedged promise, both queued manual refresh requests and the timer-driven `waitForProviderIssueHandoffQueueToDrain()` path can wait indefinitely, which matches the stale live incident’s ambiguous timeouts.
- `controlRuntime.ts` and `observabilityApiController.ts` currently do not expose enough durable refresh-lifecycle state to distinguish a slow refresh from a permanently wedged one.
- The stale incident evidence shows that process restart immediately rehydrated state, released stale `CO-38` truth, and launched `CO-39` / `CO-40`, while a replay of the same refresh logic succeeded without restart.
