# Technical Specification

## Context

`CO-41` originally addressed a March 30, 2026 control-host provider refresh stall where newly eligible `Ready` issues were not admitted until the host restarted. That original stuck polling / refresh watchdog repair landed in `PR #324` / commit `330581458`.

The issue was reopened on 2026-04-18 because the failure family recurred. The current recurrence evidence is specific: `provider-intake-state.json` reports `stuck=true`, `restart_required=true`, `last_error=provider_refresh_lifecycle_stuck`, `refresh_phase=refresh:claim_reconcile`, `running=2`, and `max_allowed=3`, while a `Ready` issue tracked here as `CO-252` was not admitted until restart. Free capacity makes this a lifecycle/admission stall, not a simple capacity-full condition.

The current blocking seam is in control-host supervision health evaluation. Repeated active-worker restart quarantine remains correct for full-capacity churn, but for `CO-41` the Apr 18 evidence shows open capacity, so the host must keep `restart_required` unhealthy and allow supervision to restart/recover instead of quarantining the stale stuck series. The existing `co-status --format json` operator dataset already carries `counts.max_allowed`; the supervisor must preserve that signal inside its diagnostic and quarantine decision.
