# Technical Specification

## Context

`CO-51` addresses the remaining narrow seam after `CO-40` and `CO-41`: a provider-worker continuation can start correctly in `Merging`, enter the merge shepherd loop, then stop on transient late review or check activity during the quiet window. When that late noise clears and the PR later returns to `OPEN`, `MERGEABLE`, `CLEAN`, with unresolved review threads back at `0`, the issue can remain parked in `Merging` until an operator manually reruns `pr resolve-merge`. The required baseline is the March 30, 2026 `CO-41` / PR `#324` incident captured under `/Users/kbediako/Code/CO/.runs/linear-af97d673-43a4-4a36-8738-b7f61e5b71a1/cli/2026-03-30T07-55-18-395Z-e26e1404/`.

## Requirements

1. If a `Merging`-stage merge drain is interrupted by late review or check activity that later clears, the provider continuation must not silently strand the issue in `Merging`.
2. The worker must either retry automatically when the PR returns to clean mergeable state or emit an explicit machine-checkable terminal/action-required result explaining why automatic recovery stopped.
3. Operator-facing artifacts must record merge lifecycle phases, at minimum: quiet-window start, interruption reason, retry decision, and final terminal outcome.
4. The repair must preserve fail-closed behavior when the PR never returns to `OPEN`, `MERGEABLE`, `CLEAN`, with unresolved review threads at `0`.
5. Focused regression coverage must exercise the clean start -> late review noise -> clean recovery sequence.
6. Closeout must include real or simulated proof showing automatic merge completion or explicit actionable failure for the `CO-41` / `#324` incident class.

## Current Truth

- `providerLinearWorkerRunner.ts` owns provider-worker proof output, end-reason reporting, and child-stream lineage.
- `providerIssueHandoff.ts` owns the provider continuation lifecycle, proof interpretation, and post-worker decisions about whether a tracked issue should stay active, retry, or release.
- The March 30 `CO-41` incident showed that the first merge drain exited when a late Codex review thread appeared during the quiet window, but the later clean PR state still required a manual `pr resolve-merge` rerun from the shared checkout.
- Existing artifacts do not yet guarantee a truthful operator-visible distinction between retryable merge-drain interruption, terminal failure, and successful auto-resume.
- `CO-40` already fixed stale later-`Merging` pickup truth, and `CO-41` already fixed the separate control-host refresh/poll stall class, so this lane must stay bounded to post-handoff merge-drain interruption handling.
