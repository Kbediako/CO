## Technical Requirements
- Functional requirements:
  - process inspection must define at least three lifecycle classes for parent-session `delegate-server` children: `active`, `idle`, and `stale`
  - `active` requires evidence of a live issue-worker/review/child-lane process tree or current associated manifest/session
  - `idle` covers still-parented children under a live top-level Codex parent session that no longer show active delegated-work evidence but are not yet safe cleanup candidates
  - `stale` covers long-lived leftovers with bounded age plus missing or terminal active-work evidence that make cleanup/recovery safe to recommend
  - observability must expose parent PID, cwd when recoverable, age, RSS, associated manifest, and process-tree role for each relevant `delegate-server`
  - doctor or another bounded observability surface must summarize classification counts and include per-process detail or a stable path to that detail
  - recovery guidance must never target active issue-worker/review/child-lane delegate servers; `idle` should default to advisory rather than automatic cleanup
  - manifest correlation must reuse existing `.runs/**` truth and must not depend on live Linear mutation
- Non-functional requirements:
  - uncertain associations fail closed and remain non-destructive
  - inspection stays bounded and local on the current macOS/Unix host
  - `CO-168` orphaned stale-chain handling and direct transport behavior stay intact
- Interfaces / contracts:
  - `codex-orchestrator doctor [--json]`
