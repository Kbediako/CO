## Current Truth
- `providerIntakeState.ts` currently defines `selectProviderIntakeClaim()` and `buildProviderIntakeSummary()`, and the selector sorts claims by state rank then `updated_at`.
- `controlRuntime.ts` currently threads that singular summary into both selected-run and compatibility/runtime snapshots as top-level `providerIntake`.
- `observabilityReadModel.ts`, `observabilitySurface.ts`, and `operatorDashboardPresenter.ts` currently expose that top-level `provider_intake` while other runtime paths still expose concurrent running claims.
- This makes the mismatch a truth-contract problem between the singular summary path and the concurrent-running/read-model path, not an intake-history absence problem.

## Proposed Design
- Add or clarify one bounded contract that:
  - keeps concurrent running claims authoritative for concurrent-running operator truth
  - preserves `retry/error history` for the relevant claim or claim set
  - ensures top-level `provider_intake` no longer implies contradictory single-claim authority when concurrent running claims exist
- The parent implementation may keep `selectProviderIntakeClaim()` as an internal selector, but the operator-facing contract must no longer treat that internal sort as the only current-truth surface.
