## Technical Requirements
- Functional requirements:
  1. Preserve CO-542 as detection/reporting only.
  2. Provide a separate remediation surface under the canonical owner key `quota-hygiene:operator-confirmed-stale-process-remediation:v1`.
  3. Require explicit operator confirmation before any remediation side effect.
  4. Require scoped targets from the current quota-hygiene audit.
  5. Require current-audit PID eligibility for every target immediately before action.
  6. Reject PID targets that disappeared, changed identity, are not stale, are protected infrastructure, or lack current-audit evidence.
  7. Emit durable per-PID audit output for each requested target, including skipped/refused targets.
  8. Derive top-level remediation mode from executed action status: `applied` requires a delivered signal outcome, `no_action` means operator-confirmed apply was requested but current-audit revalidation/refusal left no process action to execute, `operator_confirm_required` remains only for missing confirmation/scope, and `dry_run` remains non-mutating process preflight.
  9. Keep artifact output truth separate from process mutation: `artifact_written` records any successful artifact write, `audit_artifact_path` points only at the current final artifact, final artifact write failure clears that pointer and records `artifact_error`, and explicit `--remediation-output` is honored for dry-run/confirmation-required/no-action preflight evidence.
  10. Keep stale `gh pr view` polling detectable/remediable by argv shape while not classifying it as model-quota-burning or as an unowned critical process by default.
  11. Avoid app-server/control-host restarts, provider-intake mutation, queue mutation, model calls, and kill-by-name behavior.
- Non-functional requirements:
  - fail closed for ambiguous process identity or stale audit snapshots
  - keep output deterministic and machine-readable
  - keep the remediation command bounded and auditable
  - keep parent validation focused on the touched command surface
- Interfaces / contracts:
  - input target identity is PID-scoped and bound to current-audit evidence
  - confirmation UX is parent-owned but must be explicit and non-default
  - output includes a per-PID record with eligibility, confirmation binding, action, signal/result or refusal reason, and timestamp
  - remediation summary exposes `mode`, `artifact_written`, `audit_artifact_path`, and `artifact_error`; consumers must not treat `mode=no_action`, `mode=operator_confirm_required`, or `artifact_written=true` as proof that a process was signaled
  - mutation mode distinguishes `operator_confirmed_remediation` process signaling from `audit_artifact_write` evidence emission
