# Technical Specification

## Context
`CO STATUS` already has adjacent work that improved provider-worker activity truth, but the running issue list still has an identity leak. A provider-worker run whose manifest lacks canonical issue identity can still inherit `task_id` as `issue_identifier` and `issue_id`. When that task id looks like `linear-<uuid>`, the operator issue list renders it as if it were a real issue row. The defect is not a second real running issue; it is a projection and admission bug around fallback identity.

## Requirements
1. Fallback-only `linear-*` task ids must not count as authoritative issue identity for running issue presentation.
2. When canonical identity is available from tracked issue state or a matched intake claim, the projection should rebind to that canonical issue id and identifier before running issue presentation.
3. When no canonical identity is available, selected or discovered in-progress fallback-only sources must be excluded from the running issue table.
4. Real provider-worker rows with canonical issue identity must keep rendering correctly.
5. Explicit-identity non-provider rows that are already considered legitimate must not regress.
6. Underlying run or manifest diagnostics should still retain task id or run id visibility for debugging.
7. The change must stay bounded to identity repair, running-source admission, and focused regressions.

## Design
- Keep `buildSelectedRunManifestSnapshot(...)` permissive enough to retain fallback task and run ids for lookup or debug continuity, but classify fallback-only `linear-*` task-id identity as non-authoritative.
- In projection-context construction, prefer canonical identity from:
  - tracked issue state in the run-local advisory snapshot
  - matched provider intake claim identity
- Merge canonical identity with fallback aliases so later lookup and debug tooling can still find the run by task id or run id.
- Tighten authoritative-running checks so fallback-only synthetic identities do not become running issue rows just because the selected run is in progress or the manifest declares `issue_provider`.
- Leave compatibility and operator presenters mostly unchanged once upstream sources carry canonical identity or are suppressed.

## Protected Expectations
- Preserve the `CO-146` issue framing that this is a projection and identity bug, not proof of a second real issue.
- Preserve debug access to task id and run id even when fallback-only identity is hidden from the running issue table.
- Preserve real canonical provider-worker rows.
- Avoid reopening `CO-112`, retry-state duplication, or dashboard redesign.

## Reject These Wrong Interpretations
- "Hide the row only in the UI dataset and leave the wrong identity authoritative upstream."
- "Every `linear-*` task id is a legitimate issue identifier."
- "Touching `selectedRunProjection.ts` means we should reopen EVENT provenance while we are there."
- "Any row without explicit issue identity should always stay visible because it might be useful for debugging."

## Current Truth
- `buildSelectedRunManifestSnapshot(...)` assigns `task_id` or `run_id` to `issueIdentifier` and `issueId` when the manifest lacks explicit identity.
- `isAuthoritativeSelectedCurrentRunningSource(...)` still accepts non-`local-mcp` selected in-progress sources before canonical identity is established.
- `isAuthoritativeCurrentRunningSource(...)` still accepts provider-scoped in-progress sources when intake state is absent, even if the only visible identity is fallback.
- `buildCompatibilityIssueIndex(...)` and `buildUiDataset(...)` key their presentation off `issueIdentifier`, so fallback-only synthetic identity leaks directly into the running issue list.

## Proposed Design
- Add a bounded fallback classifier for task-id-derived synthetic `linear-*` identities.
- Resolve canonical issue identity from tracked issue or matched claim before building the compatibility source context.
- Suppress fallback-only synthetic sources from authoritative running activity when no canonical identity exists.
- Preserve fallback values as aliases for lookup and debug continuity.

## Non-Goals
- Reworking EVENT ranking or activity provenance.
- Reworking provider retry or claim lifecycle semantics.
- Changing the STATUS dashboard layout.

## Parity / Alignment Matrix
- Current truth:
  - fallback-only provider-worker task ids can still appear as running issue identifiers
  - running issue grouping can therefore treat a synthetic task id as a second issue
- Reference truth:
  - canonical issue identity should drive running issue presentation
  - task id and run id should remain debug artifacts, not user-visible issue ids
- Target truth / intended delta:
  - fallback-only synthetic `linear-*` identities are rebound or suppressed before presentation
  - real issue rows remain and synthetic duplicates disappear
- Explicitly out-of-scope differences:
  - EVENT text selection
  - retry-state dedupe
  - dashboard redesign

## Not Done If
- A fallback-only `linear-*` task id still renders as a running issue identifier.
- The real canonical issue row disappears because the fallback-only source was not rebound correctly.
- Task id or run id can no longer be inspected anywhere useful for debugging.
