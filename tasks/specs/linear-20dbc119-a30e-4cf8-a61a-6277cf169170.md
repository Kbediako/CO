# Technical Specification

## Summary
- Objective: prevent recurring baseline debt lanes from creating duplicate follow-up issues by adding a deterministic canonical-owner key contract between docs freshness maintenance output and the Linear follow-up helper.
- Scope: maintenance-report candidate cohort keys, exact marker stamping/reuse in `linear create-follow-up`, worker guidance, and focused regressions.
- Constraints: reuse only open issues in the same team and project with the exact marker; terminal issues are evidence only.

## Issue-Shaping Contract
- User-request translation carried forward: use machine-readable cohort identity to reuse or update one canonical owner before creating any new follow-up for recurring baseline debt.
- Protected terms / exact artifact and surface names: `docs:freshness:maintain`, `baseline_cohort_id`, `canonical-owner key`, `linear create-follow-up`, `provider-worker`, `CO-254`, `CO-253`, `CO-256`, `CO-257`, `CO-258`.
- Nearby wrong interpretations to reject: fuzzy title/body duplicate search, terminal issue reuse, broad same-attempt retry suppression, and docs freshness repair work.
- Explicit non-goals carried forward: no automatic reopen, no broad duplicate detector, no CO-186 retry-suppression changes.

## Parity / Alignment Matrix
- Current truth: maintenance output has candidate cohort and owner guidance, but the mutation path creates a new issue for every valid helper call.
- Reference truth: queue-safe recurring baseline ownership should converge on one open owner keyed by exact machine-readable cohort identity.
- Target truth / intended delta: candidate cohorts include `canonical_owner_key`; `create-follow-up` optionally accepts that key and searches only open same-team same-project issues stamped with the exact marker before creating.
- Explicitly out-of-scope differences: closed owners stay closed; unrelated issue creation remains unchanged when no canonical key is supplied.

## Readiness Gate
- Not done if: repeated helper calls with the same key create multiple open owners; closed `CO-254`-style issues are reused; or the key is derived from prose/title similarity.
- Pre-implementation issue-quality review evidence: parent source inspection found the bounded seam in `createProviderLinearFollowUpIssue` and `buildDocsFreshnessMaintenanceDecision`; the child-lane matrix isolated tests as a non-overlapping stream.
- Safeguard ownership split: parent owns implementation and docs; child lane owns listed test files until patch acceptance.

## Technical Requirements
- Functional requirements:
  - `docs:freshness:maintain` candidate cohorts expose `canonical_owner_key` and `canonical_owner_marker`.
  - Keys prefer `baseline_cohort_id` when present; otherwise they use a stable tuple of machine fields such as doc class, path family, last review, and cadence.
  - `linear create-follow-up` accepts a canonical owner key, builds the exact marker, and reuses an open same-team same-project issue already stamped with the marker.
  - When no open stamped owner exists, the helper creates one Backlog issue and stamps the marker into a deterministic `Canonical Owner` section.
  - Reused and created owner paths both preserve related/blocker relation behavior and audit output.
- Non-functional requirements (performance, reliability, security): keep request budget preflight conservative; fail closed on invalid key input; avoid fuzzy matching or cross-project reuse.
- Interfaces / contracts: add CLI flags for canonical owner key input and extend success `action` with `reused` for owner reuse.

## Architecture & Data
- Architecture / design adjustments: add a small marker builder/parser and an open-owner query near the existing provider Linear facade, plus maintenance-output key helpers near candidate cohort summarization.
- Data model changes / migrations: no persisted database changes; Linear issues receive a deterministic Markdown section.
- External dependencies / integrations: Linear GraphQL `issues` query, `issueCreate`, `issueUpdate`, and `issueRelationCreate`.

## Validation Plan
- Tests / checks: focused facade/CLI regressions for canonical owner reuse, terminal owner non-reuse, marker stamping, and maintenance output keys; then repo validation floor.
- Rollout verification: workpad and PR summarize the canonical marker contract and exact regression commands.
- Monitoring / alerts: existing provider Linear audit entries identify `create-follow-up` action as `created` or `reused`.
