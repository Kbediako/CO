# Technical Specification

## Context
Post-`CO-109`, the remaining `CO STATUS` defect is source-truth loss. Rich current-turn activity can still disappear before STATUS selection because the proof sidecar persists only narrowed `last_event` / `last_message`, session-log hydration restores ids and token telemetry without canonical activity provenance, and later layers can only rank from already-collapsed inputs.

## Requirements
1. Add a canonical current-turn activity object that survives provider-worker proof persistence and later hydration.
2. Minimum contract: `{ event, message_or_payload, recorded_at, source, turn_id, session_id }`, with any additional provenance or freshness timestamps needed for ranked selection/debugging.
3. Populate canonical activity from stdout JSONL first and session-log hydration second.
4. Child-stream and child-lane summaries must remain explicit derived candidates rather than silent replacements for canonical activity.
5. `selectedRunProjection.ts`, `providerIssueObservability.ts`, and `compatibilityIssuePresenter.ts` must rank candidates with explicit provenance and freshness instead of relying on generic progress collapse.
6. Generic text such as `turn running` or `Provider worker turn is active.` must remain terminal fallback only.
7. STATUS/debug surfaces must expose chosen source, candidate set, rejection reasons, and freshness timestamps such as `source_updated_at`, `message_recorded_at`, and `summary_recorded_at`.
8. `controlStatusDashboard.ts` must stay formatter-oriented rather than becoming the main policy engine.
9. Focused regressions must cover the known null-`last_message` plus rich-child-summary case, repeated hydration/refresh, and canonical-activity winner selection.

## Design
- Add an additive canonical activity field to `ProviderLinearWorkerProof`.
- Update live proof writes and hydration so canonical activity is preserved even when `last_message` is null or generic.
- Refactor observability to build a ranked candidate set:
  - canonical current-turn activity
  - newer hydrated current-turn activity
  - child-stream / child-lane summaries
  - derived progress summaries
  - generic fallback text
- Preserve candidate provenance and timestamps into compatibility payloads so STATUS and debug surfaces stay aligned.

## Reject These Wrong Interpretations
- “solve the issue only with dashboard string suppression”
- “treat progress summaries as authoritative current-turn truth by default”
- “let hydration overwrite fresher stdout truth without explicit freshness comparison”
- “reopen adjacent merge, cooldown, or rate-limit lanes because proof shape changed”

## Current Truth
- `selectedRunProjection.ts` can still turn `providerDebugSnapshot.progress` into `latestEvent`.
- `providerIssueObservability.ts` prefers `proof.last_message`, then child summaries, then generic phase text, but without canonical activity provenance.
- `providerLinearWorkerRunner.ts` persists narrowed proof telemetry and does not currently restore canonical message/event truth during hydration.

## Proposed Design
- Introduce `current_turn_activity` on the proof shape and hydrate it from both stdout JSONL and session-log sources.
- Build and expose a ranked candidate set with winning source and rejection reasons.
- Keep presenter/dashboard consumption message-first and provenance-aware.

## Non-Goals
- Reopening `CO-103`, `CO-107`, or `CO-109` renderer-only work.
- Redesigning throughput or rate-limit presentation.
- Changing merge closeout or shared-root reconciliation behavior.
