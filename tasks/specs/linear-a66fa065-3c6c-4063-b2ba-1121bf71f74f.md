## Summary
- Objective: make skipped-review summaries and feedback truthful when review is skipped by a failed prerequisite guard/stage.
- Scope: `TaskManager` skipped-review wording, `CommandBuilder` error artifact propagation, additive `BuildResult` diagnostics, and focused tests.
- Constraints: preserve existing review result schema, existing true build/test/review skip wording, and guard execution behavior.

## Issue-Shaping Contract
- User-request translation carried forward: CO-346 is a bug fix for misleading skipped-review wording after guard-stage failures.
- Protected terms / exact artifact and surface names: `delegation-guard`, `spec-guard`, `docs:check`, `diff-budget`, `errors/01-delegation-guard.json`, `cloud-env-missing`, `allowFailure`, `createSkippedReviewResult`.
- Nearby wrong interpretations to reject: broad manifest truthfulness work, guard behavior changes, generic README/model posture edits, or treating advisory command artifacts as failed-stage evidence.
- Explicit non-goals carried forward: no guard weakening, no telemetry taxonomy redesign, no stale artifact cleanup.

## Parity / Alignment Matrix
- Current truth: failed prerequisite stages can arrive as failed build results and cause `Review skipped: build stage failed.`
- Reference truth: manifest `status_detail` and failed command records can identify the failed stage, such as `stage:delegation-guard:failed` plus a failed `delegation-guard` command.
- Target truth / intended delta: skipped-review summaries name the failed prerequisite stage when known, and include an error artifact path when the builder provides one for that failed stage/command.
- Explicitly out-of-scope differences: historical run-summary rewrites, unrelated provider closeout metadata, and non-stage failure taxonomy changes.

## Readiness Gate
- Not done if: guard-stage failures still report as generic build failures; true build/test/review failures no longer report generic wording; non-stage details like `cloud-env-missing` infer false prerequisite stages; tests omit the Rework false-positive paths.
- Pre-implementation issue-quality review evidence: parent provider worker verified Linear CO-346, PR `#634` review feedback, old PR closure, old workpad deletion, and fresh `origin/main` branch reset before edits.
- Safeguard ownership split: parent owns docs and implementation; same-issue child lane owns focused test files.

## Technical Requirements
- Functional requirements:
  - derive a failed prerequisite stage from explicit stage/subpipeline status detail when available
  - derive a failed cloud target stage only when `cloud:<stage>:failed` is backed by a matching failed command record
  - derive from failed command records only when no non-empty status detail exists
  - keep fallback wording for normal build/test/review failures
  - include command error artifacts in builder artifacts for operator evidence
  - set `failureArtifactPath` only for the failed stage/command
  - avoid attaching unrelated error artifacts when no failed stage/command artifact is known
- Non-functional requirements: no extra filesystem reads in the manager path; no network or auth changes.
- Interfaces / contracts: `ReviewResult` shape remains unchanged; `BuildResult` gains additive optional failure diagnostics (`failureStage`, `failureArtifactPath`).
