# Technical Specification

## Context

`CO-83` narrows the remaining truthfulness gap from `CO-78`. The terminal frame already exists and the worker proof schema already has most of the necessary fields, but active provider-worker/appserver usage still fails to keep those fields authoritative all the way through to `CO STATUS`.

The current problem spans three hops:

1. live Codex/appserver JSONL does not yet normalize every authoritative token/session/rate-limit shape the issue depends on,
2. the control read-model still projects active stage/event semantics through raw-state compatibility logic, and
3. the terminal renderer still only understands the older generic Codex rate-limit bucket shape instead of the richer `5-hour` / `weekly` usage-window semantics the issue calls out.

This lane therefore needs one coherent pass across runtime parse, proof persistence, selected-run and compatibility projection, and terminal presentation.

## Requirements

1. During active provider-worker/appserver-backed runs, authoritative runtime token/session/turn/event/rate-limit telemetry must reach `provider-linear-worker-proof.json` when the runtime emits it.
2. Header `Tokens: in/out/total` and header `Throughput` must show truthful live values when authoritative totals exist; explicit unavailable semantics are acceptable only when the runtime truly does not expose the signal.
3. Running rows must show truthful `TOKENS`, `SESSION`, and `AGE / TURN` values when turn/session/token lineage exists.
4. `STAGE` must reflect operator-visible workflow phase/state semantics rather than collapsing to raw `in_progress`.
5. `EVENT` must reflect meaningful last Codex/runtime activity rather than low-signal generic fallback text like `item completed`.
6. `Rate Limits` must surface authoritative Codex usage/rate-limit information for the relevant operator windows, explicitly including `5-hour` and `weekly` values when present, while preserving existing Linear budget handling as a secondary signal.
7. Automated coverage must exercise the runtime parse -> proof -> projection -> terminal presentation path, including token/session/turn extraction, stage/event projection, and rate-limit window rendering.
8. Manual validation must use real screenshots captured on this device and embedded directly in the Linear workpad.

## Protected Expectations

- This lane owns the proof/read-model truth path for the terminal `CO STATUS` surfaces called out by the issue, not just the renderer.
- `Tokens`, `Throughput`, `TOKENS`, `SESSION`, `AGE / TURN`, `STAGE`, `EVENT`, and Codex `5-hour` / `weekly` windows are all in scope together.
- Symphony remains the semantics reference for operator-useful event and rate-limit summaries.
- Validation proof must be real-device screenshots embedded directly in Linear.

## Reject These Wrong Interpretations

- Patch only `controlStatusDashboard.ts` and leave upstream proof/projection telemetry unchanged.
- Render placeholders or stale historical values when the live runtime is silent.
- Treat raw `in_progress` as acceptable stage output for later workflow phases like `In Review` or `Merging`.
- Stop after generic Codex rate-limit buckets render without proving the richer usage-window surfaces.
- Call the lane complete without end-to-end tests and embedded screenshots.

## Current Truth

- `providerLinearWorkerRunner.ts` already parses thread/turn lineage, direct token totals, `turn.completed` usage, and generic `rate_limits` records, and writes live proof updates during active turns.
- `selectedRunProjection.ts` currently derives `displayStatus` only from raw run status plus pause/input overlays, so running `STAGE` still ignores operator-visible workflow phase.
- `compatibilityIssuePresenter.ts` builds running `EVENT` by mixing selected latest event/action/raw status with proof event/message, but it can still prefer low-signal generic activity.
- `controlRuntime.ts` aggregates token totals and combines Codex `rate_limits` with authoritative Linear budget, but only whatever proof normalization already captured.
- `controlStatusDashboard.ts` renders tokens, throughput, and rate limits, yet its rate-limit formatting only understands the older `limit_id` + `primary`/`secondary`/`credits` shape.
- Symphonyâ€™s status/orchestrator tests show the target behavior: authoritative session/turn/token tracking, event/message humanization, app-server PID tracking, and rate-limit summary formatting that understands usage windows.

## Proposed Design

### 1. Extend authoritative runtime proof normalization

- Expand provider-worker JSONL parsing to recognize richer Codex/appserver rate-limit payload shapes in addition to the existing generic bucket shape.
- Keep live proof updates authoritative during a turn so session id, turn id, tokens, event/message, and rate limits refresh without waiting for terminal exit.
- Preserve backwards compatibility with existing token/rate-limit records already covered by current tests.

### 2. Promote operator-visible stage semantics

- Add a bounded active-lane stage projection that prefers the tracked workflow phase when it provides more operator-useful truth than raw `in_progress`.
- Preserve pause/input overlays so `awaiting_input` and `paused` remain visible where they are the higher-value operator truth.
- Keep retry/degraded semantics distinct from running-stage semantics.

### 3. Tighten running-event selection

- Prefer fresher proof-backed runtime activity when it is more specific than generic compatibility fallbacks.
- Humanize low-signal runtime method/event names where necessary so `EVENT` surfaces meaningful last activity rather than raw protocol noise.
- Keep `EVENT` distinct from `STAGE`; do not duplicate the same generic state string across both fields.

### 4. Add richer Codex usage-window rendering

- Normalize richer Codex rate-limit window payloads so terminal formatting can explicitly render `5-hour` and `weekly` usage/limit windows when present.
- Preserve the existing generic `primary`/`secondary`/`credits` rendering for older payloads.
- Continue combining Codex rate limits with authoritative Linear budget without letting Linear budget overwrite newer Codex data.

### 5. Re-test the full telemetry truth path

- Expand provider-worker parser tests for richer rate-limit payloads and proof updates.
- Add control-runtime/projection tests for stage/event/rate-limit preference.
- Add terminal dashboard tests for `STAGE`, `EVENT`, `AGE / TURN`, `SESSION`, `TOKENS`, throughput, and richer Codex rate-limit line rendering.

## Non-Goals

- No terminal layout redesign.
- No browser `/ui` redesign.
- No unrelated provider-worker lifecycle or merge-closeout work.
- No broad proof schema rewrite if additive normalization is sufficient.
