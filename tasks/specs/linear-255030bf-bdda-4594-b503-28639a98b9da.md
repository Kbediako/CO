# Technical Specification
## Context
- CO already has webhook intake, startup recovery, and shared-budget fail-fast behavior.
- The avoidable steady-state cost is ordinary full tracked-issue polling during refresh ticks.
## Requirements
1. Keep webhook-by-id intake as the normal fast path.
2. Preserve startup recovery, slow full sweeps, retry/release/relaunch semantics, and operator budget truth.
3. Run ordinary refresh as targeted reconcile plus bounded fresh discovery only when capacity exists.
4. Stop discovery once enough eligible candidates are found for current global and per-state slots.
5. Record enough request-burn evidence for a later quota decision.
## Design
- `controlServerPublicLifecycle.ts` separates slow recovery sweeps from ordinary steady-state ticks.
- `providerIssueHandoff.ts` requests deferred fresh discovery only when dispatch budget still has slots.
- `linearDispatchSource.ts` supports `full`, `recovery_sweep`, and `fresh_discovery` query modes plus eligible-target stopping.
- The live-compatible discovery path preserves dispatch ordering by scanning priority buckets `[1, 2, 3, 4, 0]` and reverse-paging each bucket by `createdAt`.
## Validation Plan
- audited `linear child-stream --pipeline docs-review`
- focused lifecycle, handoff, and dispatch-source tests
- repo validation floor plus standalone review/elegance before handoff
