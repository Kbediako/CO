# Technical Specification

## Context

`CO-144` is not a "remove polling" lane. The current repo already has:

- direct webhook intake through `linearWebhookController.ts`
- startup or refresh recovery and claim rehydrate through `providerIssueHandoff.ts`
- shared-budget fail-fast, reservations, and cooldown-aware polling through the Linear budget helpers and `controlServerPublicLifecycle.ts`

The steady-state pressure problem is narrower and more concrete:

- ordinary scheduled refresh ticks still call `resolveLiveLinearTrackedIssues(...)` across the active Linear set
- `providerIssueHandoff.refresh()` still preloads that full tracked-issue set before reconciling existing claims
- the tracked-issues query still fetches heavy fields for every scanned issue
- the scheduler does not use the existing early-stop hook in a free-slot-aware way

This lane must preserve recovery and budget truth while removing the avoidable full-scan behavior from ordinary refresh ticks.

## Requirements

1. Preserve webhook create or update intake as a direct by-id fast path for new or changed issues.
2. Preserve startup rehydrate and full recovery behavior for existing claims and older unchanged ready issues.
3. Split ordinary refresh behavior into:
   - targeted reconcile for owned or recovery-tracked issues
   - bounded fresh discovery only when current dispatch capacity has free slots
4. Stop bounded fresh discovery once enough eligible candidates exist for the currently free global or per-state slots.
5. Keep a slower periodic full recovery sweep for missed webhooks, stale webhook health, restart rehydrate, and older unchanged ready issues.
6. Trim fresh-discovery payloads so the scheduler fetches only the fields needed to decide eligibility and ordering, deferring richer by-id reads until a candidate is selected or a claim is reconciled.
7. Preserve claim, retry, release, reopen, reattach, and relaunch-after-success behavior.
8. Preserve shared-budget cooldown, exhaustion truth, and operator-visible next-refresh state or reason.
9. Record enough request-burn evidence from shared-budget snapshots, response headers, or both to support the later quota-increase decision.
10. Focused tests must cover startup recovery, webhook-first intake, bounded discovery stop conditions, slow full-sweep recovery, and operator-visible budget truth.

## Current Truth

- `handleLinearWebhookRequest(...)` already validates the webhook, resolves the issue by id, and hands accepted issue deliveries directly into `providerIssueHandoff.handleAcceptedTrackedIssue(...)`.
- `createProviderRefreshCoordinator(...)` still calls `resolveProviderPollTrackedIssues()` on ordinary scheduled ticks and hands the resulting full tracked-issue list into `providerIssueHandoff.poll(...)`.
- `resolveRefreshPollInput()` in `providerIssueHandoff.ts` preloads the full tracked-issue set before `runRefreshCycle(...)`, even though `resolveRefreshTrackedIssueResolution(...)` can already fall back to direct per-claim by-id refresh reads through `resolveTrackedIssue(...)`.
- `resolveLiveLinearTrackedIssues(...)` already supports `limit` and `stopWhenEligibleForExecution`, but current scheduler call sites do not pass those options, and the early-stop path only stops after the first eligible issue rather than after enough free-slot candidates.
- `buildLinearTrackedIssuesQuery(...)` still requests `description`, blocker relations, and recent history for every scanned active issue.
- `providerPollingHealth.ts` already exposes interval, next refresh, reason, and Linear budget state; that truth should remain authoritative.

## Proposed Design

### 1. Split ordinary refresh modes

- Keep `rehydrate()` plus startup-triggered refresh as the full recovery entrypoint.
- Change ordinary periodic refresh ticks so they do not automatically preload the full tracked-issue set.
- For existing claims:
  - reconcile via direct by-id reads through `resolveTrackedIssue(...)`
  - preserve the existing fallback and release or reopen logic in `providerIssueHandoff.ts`
- For fresh work:
  - run a separate bounded discovery pass only when `createProviderPollDispatchBudget(...)` still reports free global slots

### 2. Bounded discovery contract

- Extend `resolveLiveLinearTrackedIssues(...)` with a free-slot-aware stop condition such as an eligible-candidate target count instead of the current boolean first-hit stop.
- Keep dispatch ordering semantics identical: priority, oldest `created_at`, then identifier.
- Let the discovery caller request only the number of eligible candidates needed to fill current free slots.
- Break once the eligible-candidate target is satisfied or no more pages remain.

### 3. Lean discovery payload plus by-id hydrate

- Split tracked-issue reads into:
  - lean discovery fields for eligibility and ordering
  - richer by-id or reconcile reads when a claim is selected or refreshed
- Keep the fields required for eligibility, ownership, and blocked-state decisions in the lean query.
- Move description, history, and other richer fields behind the by-id reconcile path when they are not needed for discovery.

### 4. Slow full recovery sweep

- Add a slower full-scan sweep on top of startup recovery so missed webhook events and older unchanged ready issues are still discovered eventually.
- The first landed policy may be fixed-cadence as long as it is explicit, test-backed, and slower than ordinary targeted reconcile ticks.
- Keep the slower sweep authoritative for older ready work that webhook-first intake will not surface if nothing changes on the issue.

### 5. Observability and evidence

- Keep `providerPollingHealth.ts` truthful about next refresh timing, reason, and Linear budget state.
- Expose enough mode or reason detail to distinguish bounded discovery versus slow recovery sweep without widening the public contract unnecessarily.
- Record before or after evidence from shared-budget state and request headers so the post-optimization request-burn comparison is grounded in persisted artifacts.
