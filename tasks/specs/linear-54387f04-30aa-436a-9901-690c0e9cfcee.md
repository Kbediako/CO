# Technical Specification

## Context
Recent successful GitHub `Core Lane` jobs still spend roughly `21.5m` to `22.1m`, with the `Test` step averaging about `20.9m`. The issue already narrows the long tail to `tests/run-review.spec.ts` and `tests/cli-command-surface.spec.ts`, and current source audit confirms both suites repeatedly boot TypeScript entrypoints through `node --loader ts-node/esm`. The lane is therefore a test-harness performance redesign, not a general validation reduction or hang-truth investigation.

## Requirements
1. Keep `tests/run-review.spec.ts` and `tests/cli-command-surface.spec.ts` as the primary optimization targets.
2. Move pure contract assertions that do not require real entrypoint boot into faster in-process coverage.
3. Preserve a bounded subprocess smoke matrix for real CLI and review-wrapper end-to-end confidence.
4. Reduce remaining subprocess startup overhead where a lighter shared or built harness preserves the same contract.
5. Keep build/lint/spec/docs/pack-smoke guardrails intact.
6. Record before/after timing evidence for the two hot suites and the full test-step proxy.
7. Keep the diff bounded to the hot suites, their direct harness helpers, and the required docs packet updates unless fresh evidence proves a broader seam.

## Design
- Extract importable execution seams from:
  - `bin/codex-orchestrator.ts` for root CLI contract coverage
  - `scripts/run-review.ts` for review-wrapper contract coverage
- Reallocate tests:
  - pure help/arg/contract assertions move to in-process execution
  - a smaller subprocess smoke layer still boots the real entrypoints
- Optimize subprocess smoke:
  - prefer built JS or another lighter shared harness over repeated `ts-node/esm` cold boot when the same contract is preserved
- Keep coverage truth explicit:
  - document which assertions remain subprocess-only and why
  - avoid changing product behavior merely to make tests faster unless a small export seam is the minimal path

## Implementation Surface
- Expected codepaths:
  - `bin/codex-orchestrator.ts`
  - `scripts/run-review.ts`
  - `tests/cli-command-surface.spec.ts`
  - `tests/run-review.spec.ts`
  - any extracted test helpers under `tests/helpers/**`
- Expected evidence:
  - focused before/after timings
  - retained subprocess smoke proof
  - full validation floor logs

## Protected Expectations
- Preserve the current Core Lane guardrail categories.
- Preserve real entrypoint smoke coverage for CLI and review-wrapper behavior.
- Keep the performance claim backed by timing artifacts, not anecdotal impressions.
- Keep the implementation focused on the hot suites and direct seams.

## Reject These Wrong Interpretations
- "delete or skip slow checks to get the number down"
- "reopen historical hang-truth work without new evidence"
- "move every assertion out of subprocesses and stop testing the real entrypoints"
- "accept remaining `ts-node/esm` cost everywhere because the suites are already passing"

## Current Truth
- `tests/cli-command-surface.spec.ts` defines `runCli()` around `execFileAsync(process.execPath, ['--loader', 'ts-node/esm', CLI_ENTRY, ...args])`.
- `tests/run-review.spec.ts` defines `runReviewCommand()` around `execFileAsync(process.execPath, ['--loader', 'ts-node/esm', runReviewScript, ...extraArgs])`.
- Both target entrypoints currently expose behavior only through local `main()` functions, which makes many pure contract assertions expensive to exercise.

## Proposed Design
- Add importable entrypoint runners or equivalent shell-facing helpers for the root CLI and review-wrapper.
- Convert the bulk of pure contract assertions to in-process coverage using those runners.
- Keep a reduced subprocess smoke matrix to cover true end-to-end launch semantics.
- Make the subprocess smoke layer use a lighter/shared harness when that keeps the same confidence.

## Non-Goals
- Reopening older hang-truth or teardown-stall lanes without fresh evidence.
- Removing build/lint/spec/docs/pack-smoke categories.
- Broad repo-wide CI cleanup.
- Product-surface redesign unrelated to making the hot suites faster and still truthful.

## Parity / Alignment Matrix
- Current truth:
  - the two cited suites dominate the hot tail
  - both repeatedly cold-boot TypeScript entrypoints
- Reference truth:
  - the same contracts should be covered with less repeated startup cost
- Target truth / intended delta:
  - most pure contracts run in-process
  - a bounded subprocess matrix still proves the real entrypoints
  - remaining subprocess cases use a lighter harness
- Explicitly out-of-scope differences:
  - unrelated CI categories
  - older hang investigations

## Not Done If
- The same two suites still dominate the test wall time without meaningful improvement.
- Speedup mainly comes from dropping coverage rather than reallocating it.
- The final packet lacks before/after timing evidence.
- The subprocess smoke matrix no longer proves the real entrypoints.
