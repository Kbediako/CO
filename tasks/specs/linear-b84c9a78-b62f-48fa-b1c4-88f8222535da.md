# Technical Specification

## Context

`CO-163` is the restart/orphan follow-up after `CO-152` duplicate-host protection. Current `main` already has:

- same-task startup ownership acquisition and duplicate/stale owner artifacts from `CO-152`
- supervision state tracking with `child_pid` and restart reason under `controlHostSupervision`
- stuck polling detection and `restart_required=true` projection after the 45s watchdog budget
- startup and deferred poll no-burn work from `CO-160` and `CO-162`

The remaining gap is the incident path where supervision restart and stuck refresh overlap:

- `control-host supervise restart` currently calls `launchctl kickstart -k` and returns success immediately
- if launchd leaves the old supervised child tree alive, the replacement host can race into the existing ownership lock and the old host can keep polling
- the active `providerIssueHandoff.poll(...)` loop can continue into later direct `resolveTrackedIssue(...)` calls or fresh discovery even after the watchdog has already classified the lifecycle as `provider_refresh_lifecycle_stuck`

This lane must make restart a real replacement boundary and stop additional read burn from the already-stuck refresh cycle.

## Requirements
1. Supervised restart must capture the previously tracked supervised child pid before restart and must not report success until that old child tree is gone.
2. If launchd leaves the old child tree alive beyond the configured timeout, restart must force-clean only that old control-host tree and surface the cleanup in machine-readable output.
3. Before any forced cleanup, restart must verify that a timed-out persisted `child_pid` still matches the expected supervised control-host process identity; reused or unrelated PIDs must not be killed.
4. Restart/orphan evidence must preserve owner-token/process diagnostics through the existing ownership artifacts rather than hiding the duplicate race.
5. Active provider-linear-worker issue processes must not be killed as collateral by the restart cleanup path.
6. Once polling health is already stuck, the active refresh cycle must stop before later direct issue-by-id reads or fresh discovery.
7. Focused regression coverage must exercise both the stuck refresh boundary and the restart/orphan cleanup seam, together proving no orphaned supervised host remains.
8. Operator-facing status/runbook guidance must clearly distinguish the supervised control-host child pid from detached provider worker issue processes.

## Design
- Add a focused restart helper under `controlHostSupervisionCliShell.ts` that:
  - reads the stored supervision state to capture the old `child_pid`
  - triggers `launchctl kickstart -k`
  - waits for the old child process group to exit
  - escalates to the existing tracked process-group cleanup path if the old tree remains alive
  - optionally reads current owner diagnostics for restart/orphan evidence
- Reuse the current process-group wait/kill helpers and descendant lookup instead of inventing a new process cleanup surface, but keep out-of-group descendants diagnostic-only so detached provider workers are preserved.
- Validate the timed-out tracked PID against the expected supervised control-host command identity before any forced process-group kill so stale persisted PIDs cannot target unrelated processes.
- Add a sticky refresh-abort callback or equivalent seam from `controlServerPublicLifecycle` / polling health into `providerIssueHandoff` so the active poll loop can fail closed before later `resolveTrackedIssue(...)` or fresh discovery work after stuck is set.
- Keep status/runbook changes additive:
  - expose the supervised child pid more clearly in status text
  - document that detached provider workers are separate issue processes and are not the restart cleanup target
