# Technical Specification

## Context
`CO-148` is a follow-up to `CO-120`, not a reopening of it. The repo already owns future leak prevention for linked-worktree helper flows, but historical shared-checkout config pollution in `/Users/kbediako/Code/CO/.git/config` remains silent local state unless the operator knows exact manual git commands. The missing contract is explicit shared-checkout detection and cleanup that is intentional, opt-in, and clearly distinct from the linked-worktree preventive helper.

## Requirements
1. Provide a deterministic shared-checkout command path that reports whether `/Users/kbediako/Code/CO/.git/config` contains repo-local `user.name` and/or `user.email`.
2. Provide an explicit, opt-in shared-checkout command path that clears only those shared repo-local `user.*` overrides.
3. Refuse remediation when invoked from a linked worktree or non-root path.
4. Keep `scripts/worktree-git-identity.mjs` scoped to future leak prevention only; it must not auto-clear shared repo-local `user.*`.
5. Make the helper and packet wording explicit that `future leak prevention` and `historical cleanup` are separate contracts.
6. Add focused regression coverage for status, clear, and linked-worktree refusal.
7. Record exact verification commands for `/Users/kbediako/Code/CO` in the workpad and validation notes.

## Design
- Add one shared-checkout helper that:
  - requires `--repo-root <path>`
  - resolves the shared config path from the named repo root
  - reports present repo-local `user.name` / `user.email` values in a deterministic status path
  - clears those keys only when an explicit opt-in flag is supplied
  - fails closed when the supplied path is not the shared checkout root
- Keep the existing worktree helper behavior unchanged apart from usage/help text that points historical cleanup to the new shared-checkout helper.
- Use `scripts/shared-repo-git-identity.mjs --repo-root /Users/kbediako/Code/CO` for exact detection and the same command with `--clear` for explicit remediation.
- Add focused tests that create a temporary main repo plus linked worktree and assert:
  - status reports no override when `/Users/kbediako/Code/CO/.git/config` lacks repo-local `user.*`
  - status reports the override and config path when present
  - opt-in clear removes only `user.name` / `user.email`
  - linked-worktree invocation is rejected

## Protected Expectations
- Preserve the CO-120 future leak prevention behavior.
- Preserve explicit separation between shared-checkout cleanup and linked-worktree helper behavior.
- Preserve the issue's non-goals around historical rewriting and global config changes.
- Preserve the bounded single-issue scope; no unrelated workflow or provider changes.

## Reject These Wrong Interpretations
- "CO-148 should make the worktree helper auto-clear shared config every time."
- "The right fix is to rewrite historical commits."
- "The cleanup helper should change global git config."
- "Because CO-120 and CO-148 touch git identity, they are the same contract."

## Current Truth
- `scripts/worktree-git-identity.mjs` already protects future linked-worktree writes by using `git config --worktree`.
- `.github/workflows/archive-automation-base.yml` already uses the preventive helper.
- The repo still exposes no dedicated shared-checkout detection/cleanup path for historical overrides in `/Users/kbediako/Code/CO/.git/config`.
- Existing helper messaging does not explicitly redirect historical cleanup to a separate shared-checkout path.

## Proposed Design
- Introduce a tiny shared-checkout helper dedicated to historical cleanup.
- Keep the preventive worktree helper narrow and explanatory.
- Let focused tests and exact command paths carry the verification burden.

## Non-Goals
- Rewriting historical commits or contributor attribution.
- Modifying user-global git config.
- Auto-clearing shared config from linked-worktree helpers.
- Reopening workflow semantics already fixed in CO-120.

## Parity / Alignment Matrix
- Current truth:
  - future leak prevention exists
  - historical shared-checkout config cleanup is implicit/manual
  - contract wording does not clearly separate preventive and remediation paths
- Reference truth:
  - preventive helper handles only future leak prevention
  - shared checkout exposes an explicit inspect/clear path for historical pollution
- Target truth / intended delta:
  - repo-owned shared-checkout helper provides deterministic status and opt-in clear
  - worktree helper usage text and docs explicitly point to the separate remediation path
- Explicitly out-of-scope differences:
  - global git policy
  - history rewriting
  - unrelated helper/workflow refactors

## Not Done If
- Operators still need ad hoc raw git commands to discover historical shared repo-local overrides.
- The remediation helper can mutate shared config from a linked-worktree path.
- The worktree helper or docs imply automatic historical cleanup.
- The closeout lacks focused proof or exact rerunnable commands.
