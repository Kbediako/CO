# Technical Specification

## Context
`CO-102` exposed a live late-turn recovery seam after the broader docs-baseline and adjacent merge-closeout work had already moved forward. The preserved incident is no longer a still-live blocked issue, so this follow-up cannot be framed as "the issue is still stuck right now." The accurate current statement is narrower: the archived run self-recovered through review and merge states even while repeated control-host refresh requests kept timing out. That means the remaining defect is the refresh acknowledgement contract under pressure, not the absence of any eventual recovery.

## Requirements
1. Repeated late-turn refresh requests from `provider-linear-worker` must not time out merely because the control-host already has equivalent refresh work queued or running.
2. The control-host refresh/public lifecycle must return prompt accepted truth for queued/coalesced work while preserving auditability about why the request was not executed immediately.
3. Genuine lifecycle stalls must still surface `provider_refresh_lifecycle_stuck` / `restart_required` truth and must not be hidden behind the queued/coalesced fast path.
4. Provider-worker late-turn recovery must be able to advance an issue out of `In Progress` once the active PR and issue state are actually ready.
5. Historical attached PRs must remain disambiguated on the already-landed bounded path from `CO-104`.
6. The implementation must stay bounded to refresh acknowledgement, proof/health truth, and focused regressions.

## Design
- Keep the refresh queue/coalescing substrate in `controlServerPublicLifecycle.ts`, but change the caller-facing contract so queued/coalesced work can return an immediate accepted response instead of waiting for the active lifecycle promise to settle.
- Preserve actual lifecycle execution, stuck detection, and restart-required decisions in their existing control-host ownership paths.
- Keep `providerLinearWorkerRunner.ts` strict about actual unresponsive refresh requests while teaching it to treat explicit accepted queued/coalesced responses as success rather than timeout-worthy failure.
- Leave historical attached-PR filtering and review/merge handoff policy on existing bounded lanes.

## Implementation Surface
- Expected codepaths:
  - `orchestrator/src/cli/providerLinearWorkerRunner.ts`

  - `orchestrator/src/cli/control/providerIssueHandoff.ts`

  - `orchestrator/tests/ProviderIssueHandoff.test.ts` if late-turn claim assertions change
  - `orchestrator/tests/ProviderPollingHealth.test.ts` if stuck truth changes

## Protected Expectations
- Preserve the worker-side refresh timeout as a failure signal for truly unresponsive refresh requests.
- Preserve `provider_refresh_lifecycle_stuck` / `restart_required` truth for actual unhealthy lifecycle states.
- Preserve the existing same-repo attached-PR selector and reopened/historical closeout handling from `CO-104`.
- Keep review-to-`Merging` promotion ownership on the existing `CO-116` path rather than slipping new policy into this lane.

## Reject These Wrong Interpretations
- "This means CO-102 is still currently stuck, so the fix should replay that exact terminal state."
- "A timeout increase is good enough even if the control-host already knows the refresh is queued."
- "If queued work returns early, we no longer need stuck/restart-required truth."
- "Historical attached-PR cleanup should be redesigned again as part of this refresh fix."

## Current Truth
- Archived `CO-102` evidence proves repeated refresh timeouts and eventual workflow recovery can coexist in the same late-turn run.
- `providerLinearWorkerRunner.ts` currently waits for `/api/v1/refresh` to return within 15s even when the control-host already has pending refresh work.
- `controlServerPublicLifecycle.ts` already understands queued/coalesced refresh work, but the current caller-facing promise shape is slow enough to look like failure upstream.
- `providerIssueHandoff.ts` and `providerPollingHealth.ts` already own the later claim-routing and stuck-truth surfaces that this lane must preserve.

## Proposed Design
- Return immediate accepted results for queued/coalesced refresh requests from the control-host/public lifecycle, with explicit metadata showing the request was accepted into an already-busy lifecycle rather than fully executed inline.
- Keep actual refresh execution and stuck classification unchanged behind that acknowledgement seam.
- Use focused provider-worker handling updates only as needed to interpret the new accepted response correctly.
- Add tests that model the archived late-turn shape: repeated refresh requests while the active issue can still later progress through review/merge closeout.

## Non-Goals
- Reopening `CO-102` docs-baseline scope.
- Replacing the control-host lifecycle queue/coalescing model.
- Generic polling-budget or dashboard redesign.
- Reworking `CO-104` or `CO-116`.

## Parity / Alignment Matrix
- Current truth:
  - busy control-host refresh work can still look like a timeout failure to the provider worker
  - actual stuck truth and historical attached-PR handling already exist on adjacent bounded paths
  - source incident evidence is now archived rather than actively live
- Reference truth:
  - a queued/coalesced refresh request is still a truthful acceptance outcome
  - actual stuck truth should remain explicit
  - one attached-PR selector should continue to own historical ambiguity
- Target truth / intended delta:
  - provider-worker refresh requests stop timing out on busy-but-healthy control-host work
  - actual stuck states still fail closed
  - historical attached-PR handling remains unchanged and truthful
- Explicitly out-of-scope differences:
  - review-state promotion policy
  - merge-closeout redesign
  - stale-event/UI rendering cleanup

## Not Done If
- Busy-but-healthy queued refresh work can still spend 15s and be reported as a refresh failure.
- Active claims can still remain pinned in `In Progress` under the archived late-turn shape.
- Actual stuck lifecycle states lose `provider_refresh_lifecycle_stuck` / `restart_required` truth.
- Historical attached PRs need a second new selector to keep the flow working.
