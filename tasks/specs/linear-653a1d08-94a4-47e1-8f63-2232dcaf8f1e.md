# Technical Specification

## Context
`CO-134` is a validation-stability follow-up, not a feature continuation lane. `CO-57` already narrowed one historical `implementation-gate` problem to stale manifest heartbeats during long-running tests, and the current tree already carries a temp-package-root fix path inside `tests/cli-frontend-test.spec.ts`. The remaining blocker is therefore not assumed to be the same old late-suite symptom. Instead, the issue now names a fresh cluster: provider-worker `implementation-gate` terminated while `npm run test` was still in progress, `ProviderLinearWorkerRunner.test.ts` reproduces a broad failure or timeout cluster in isolation, and a bounded standalone review previously failed closed on a `failed-boundary` outcome after crossing into direct validation commands.

## Requirements
1. Reproduce the current blocking validation or review behavior with saved manifest or log evidence on current `HEAD`.
2. Classify the implementation-gate blocker truthfully as stage timeout, actual suite regression, review-boundary drift, or an evidence-backed combination.
3. Restore a machine-checkable path where `implementation-gate` and forced standalone review expose truthful terminal outcomes.
4. Keep the lane scoped to validation and review-owner seams; do not broaden it into CO-117 feature work.
5. Preserve bounded-review enforcement for genuinely heavy commands while removing avoidable false handoff blockers.
6. Keep the final diff bounded to the classified owner surfaces plus the required docs and evidence paths.

## Design
- Classification ladder:
  - rerun the named focused suites and a manifest-backed `implementation-gate` path first
  - separate stage-timeout signals from suite-red signals and from review-telemetry boundary signals
  - only after classification, patch the minimal owner surface
- Validation-path repair:
  - if `implementation-gate` stage timeout is the owner, make the timeout or closeout evidence truthful instead of collapsing into generic failure text
  - if a real suite regression is the owner, keep the fix local to the failing tests or harness
- Review-path repair:
  - if the owner is avoidable bounded-review drift, narrow the contract so provider-worker handoff evidence remains allowed without weakening true heavy-command enforcement
  - keep `clean-success`, `bounded-success`, `failed-boundary`, and `failed-other` interpretations aligned between telemetry generation and command-runner closeout
