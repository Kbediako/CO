# Technical Specification

## Context
`CO-100` made shared-root reconciliation explicit and retryable for real `Merging` closeout. `CO-138` handled stale merge-closeout residue once an issue had clearly left `Merging` and returned to active work. `CO-214` is the merged-terminal sibling of those lanes: after `CO-211` / `PR #506` are already merged and live Linear truth is `Done`, local `provider-intake-state.json` can still retain older `Merging` closeout residue with `state=handoff_failed`, reason `provider_issue_merge_closeout_action_required`, top-level or nested `merge_closeout.status=action_required`, and shared-root detail such as `pending_shared_root_reconciliation`, `shared_root_not_on_main`, and `linear_transition=null`.

That stale residue is not just historical noise. It can keep `fresh discovery suppression` active, make `CO STATUS active/backoff projection` look like live work is still pending, and keep stale closeout state participating in `provider_refresh_lifecycle_stuck` / `restart_required` control-flow even though merged PR truth and live Linear Done truth should now dominate. The fix needs one bounded stale-residue decision seam that preserves true current `Merging` pending action while clearing or superseding stale merged/Done residue.

## Requirements
1. Reproduce the `CO-211` / `PR #506` stale residue shape: merged PR truth plus live Linear `Done` truth coexist with retained local `issue_state=Merging`, `state=handoff_failed`, `provider_issue_merge_closeout_action_required`, `merge_closeout.status=action_required`, `pending_shared_root_reconciliation`, `shared_root_not_on_main`, and `linear_transition=null`.
2. Fresh discovery and later handoff/recovery logic must not keep the issue suppressed once merged PR truth and newer Linear Done truth prove the local `Merging` action-required residue is stale.
3. `CO STATUS active/backoff projection` must not report the stale merged/Done issue as active, retrying, or otherwise current work.
4. Real current `pending_shared_root_reconciliation` behavior for actually live `Merging` issues must remain unchanged.
5. Genuine `provider_refresh_lifecycle_stuck` / `restart_required` outcomes must remain visible for actual refresh-health failures.
6. The repair must remain separate from `CO-212 Ready reclaim`, shared-root policy redesign, and docs/spec-guard-only reinterpretation.

## Design
- Introduce one authoritative stale merged-closeout residue predicate that compares retained local closeout state against merged PR truth and the newest tracked Linear issue truth.
- Use that predicate before fresh-discovery suppression, handoff-failed recovery, and active/backoff projection consume the retained closeout state.
- When merged PR truth is authoritative and newer Linear truth is terminal `Done`, treat older local `Merging` action-required shared-root residue as stale and clear or safely supersede it.
- Keep the current shared-root pending-action contract intact when the issue is still truly `Merging`, when merged PR truth is absent, or when the local closeout residue still matches the live state lineage.
- Reuse the same stale-residue decision across provider handoff and `CO STATUS` projection so runtime classification and read-side projection do not drift apart.

## Current Truth
- Local retained closeout state can outlive the real workflow state that produced it.
- The retained residue can still look like current `Merging` action-required work even after merged PR truth and live Linear `Done` truth are authoritative.
- Existing suppression/projection seams can still treat that residue as active or blocking.
- Adjacent lanes prove the required boundaries already exist separately: `CO-100` for true shared-root pending action, `CO-138` for stale merge-closeout after leaving `Merging`, `CO-182` / `CO-194` for stale terminal projection and refresh-health stabilization, and `CO-212` for genuine Ready reclaim.

## Proposed Design
- Add a small shared predicate that answers whether retained closeout residue is stale because newer merged PR truth and live Linear Done truth supersede it.
- Run that predicate before:
  - any handoff-failed recovery or rehydrate path classifies the issue as still blocked
  - any fresh-discovery suppression logic treats the issue as still occupying a live blocking state
  - any active/backoff presentation path reports the issue as current work
- Persist or project the cleaned/superseded result so later refresh and projection paths do not revive the stale state.
- Leave all current behavior untouched for truly current `Merging` shared-root pending action and for genuinely unhealthy refresh lifecycles.

## Non-Goals
- Reworking `CO-212` Ready reclaim behavior.
- Redesigning shared-root reconciliation policy or forcing shared-root mutation.
- Reopening merge-promotion, attached-PR, or generic merge-closeout policy lanes.
- Replacing runtime fixes with docs-only or spec-guard-only changes.
