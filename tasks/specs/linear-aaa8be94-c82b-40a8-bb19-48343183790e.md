# Technical Specification

## Context

CO-183 made the immediate `explorer_fast` / `gpt-5.3-codex-spark` exception policy safe enough to ship by adding targeted docs-hygiene regression coverage. The remaining technical debt is the maintainability of the spark-policy guard: allowed and forbidden wording lives in line-level detection and inline test phrases, making future wording changes harder to audit.

CO-191 is a behavior-preserving refactor. The implementation must keep `npm run docs:check` on the existing production rule while moving the spark-policy decision into a focused parser-backed classifier and structured fixture harness.

## Issue-Shaping Contract
- User-request translation carried forward: preserve CO-183 final spark file/codebase-search-only policy behavior while replacing regex-heavy clause/window heuristics with a parser-backed classifier and fixture-driven evaluator.
- Protected terms / exact artifact and surface names: `scripts/docs-hygiene.ts`, `scripts/lib/docs-catalog.js`, `scripts/lib/docs-catalog.d.ts`, `tests/docs-hygiene.spec.ts`, `tests/fixtures/docs-hygiene/spark-policy-cases.json`, `docs:check`, `explorer_fast`, `gpt-5.3-codex-spark`, `file/codebase search only`.
- Nearby wrong interpretations to reject: reopening Codex CLI 0.120 adoption posture, weakening the spark policy, blocking CO-183 after its immediate regressions are fixed, building a general docs natural-language parser, or storing private Linear/review text as fixtures.
- Explicit non-goals carried forward: no spark policy change, no file/codebase-search-only weakening, no docs:freshness work, no CO-184/CO-188 maintenance, no broad natural-language parser.

## Technical Requirements
- Functional requirements:
  - preserve current accepted docs-hygiene behavior before refactoring internals;
  - extract spark-policy classification into a focused parser/classifier module;
  - represent concepts explicitly: marker, clause, assertion window, scope qualifier, forbidden usage, restrictive non-use, neutral reference;
  - normalize Markdown emphasis/backticks and wrapped continuation lines enough to classify the accepted CO-183 cases;
  - move phrase coverage into structured fixtures with stable ids, category labels, input text, expected verdict, and expected reason;
  - add table-driven tests that fail with fixture id, expected verdict, actual verdict, and reason;
  - keep `npm run docs:check` wired through the same classifier path.
- Non-functional requirements:
  - deterministic local behavior with no network calls;
  - readable fixture JSON that avoids raw private workpads or unsanitized operator text;
  - small module surface, no broad natural-language parser;
  - TypeScript declaration coverage for any exported JS helper used by tests.
- Interfaces / contracts:
  - production entrypoint: `runDocsCheck()` in `scripts/docs-hygiene.ts`;
  - posture helpers: `scripts/lib/docs-catalog.js` and `scripts/lib/docs-catalog.d.ts`;
  - classifier module: `scripts/lib/spark-policy-classifier.js` and matching declaration if added;
  - test harness: `tests/docs-hygiene.spec.ts`;
  - fixtures: `tests/fixtures/docs-hygiene/spark-policy-cases.json`.

## Architecture & Data
- Architecture / design adjustments:
  - add a small classifier module that parses candidate lines into marker/scope/usage classifications;
  - keep model mention extraction in docs-catalog but use the classifier to decide whether a spark mention is allowed, forbidden, restrictive non-use, or neutral;
  - expose a test-friendly evaluation function that returns `{ verdict, reason }` for fixture assertions;
  - keep unsupported-review warning logic separate from spark-policy classification.
- Data model changes / migrations:
  - add fixture JSON with stable `id`, `category`, `text`, `expected`, and `reason` fields;
  - no runtime data migration.
- External dependencies / integrations:
  - no new runtime dependency expected.
