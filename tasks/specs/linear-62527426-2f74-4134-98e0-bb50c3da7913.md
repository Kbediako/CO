# Technical Specification

## Context

`CO-14` is no longer a greenfield architecture bet. It now sits above two concrete inputs:

- `CO-13` established bounded provider-worker child-stream support inside the current single-worker-per-issue seam.
- The 2026-03-27 Codex `0.117.0` / upstream-main canary lowered substrate risk for future multi-agent work, but it still kept provider-worker supervision on the existing `codex exec` / `codex exec resume` seam.

That combination makes issue-parent orchestration worth documenting, but not worth implementing by assumption.

## Requirements

1. Compare the future issue-parent architecture against:
   - current Symphony single-owner behavior
   - `CO-13` bounded child-stream support
   - the `0.117/main` adoption-canary findings
   - current bounded parity slices for intake, continuation, runtime snapshot, retry truth, and current Linear operational parity
2. Preserve one authoritative issue claim and running owner, one externally authoritative workspace, workpad, thread, and handoff state machine, and serialized Linear lifecycle mutations through the issue parent.
3. Define lineage, workspace, observability, and conflict rules for subordinate same-issue lanes without weakening `delegation-guard`.
4. Keep implementation explicitly deferred until evidence shows the bounded child-stream seam or current single-worker path is insufficient.
5. Validate with delegated scout evidence, docs-review, and the required repo validation floor for the final docs diff.

## Documented Contract

- Parent authority:
  - one issue-parent owner remains the only actor allowed to mutate Linear state, the single active workpad comment, PR attachment state, and review or merge handoffs
  - subordinate lanes are internal helpers, not peer issue owners
- Lineage:
  - parent identity remains the authoritative issue task id and run id
  - child lanes use `<parent-task-id>-<stream>` identity plus `parent_run_id`
  - child-lane lineage stays nested under the parent proof surface rather than appearing as peer scheduler-owned provider runs
- Workspace and branch truth:
  - the authoritative issue workspace remains parent-owned
  - child lanes may use lane-local worktrees, branches, or patch artifacts only inside the per-issue workspace boundary
  - the parent accepts, rejects, or reruns child output as issue state changes
- Conflict rules:
  - same-issue parallel lanes require explicit file-scope or phase-scope ownership
  - overlapping scopes must be serialized or re-scoped by the parent
  - stale child outputs are invalid once review feedback, issue state, or the base branch changes in ways that affect their declared scope
- Activation gate:
  - no implementation work starts from this issue unless `CO-13` or the current single-worker seam produces evidence-backed throughput or quality failures around review, merge, workpad, or supervision that bounded child streams cannot solve
