# Technical Specification

## Context
`CO-116` removed the manual `In Review` -> `Merging` state hop for clean PRs, and `CO-104` / `CO-111` tightened deterministic merge-closeout truth, but the remaining green-but-dirty seam is still real. GitHub can report a PR as approved with green checks while `mergeStateStatus` is `BEHIND` or `DIRTY`. Today that leaves the PR outside the autonomous path: the shared watcher exits, review-handoff promotion refuses the PR, and merge closeout refuses the PR. `CO-140` closes that remaining operator bottleneck without redesigning merge-closeout ownership.

## Requirements
1. `BEHIND` PRs must trigger a bounded branch-refresh attempt before review handoff or merge closeout gives up.
2. `DIRTY` PRs must trigger the same bounded recovery primitive before the system concludes that rework is required.
3. Exact recovery-attempt metadata must be persisted in machine-checkable review-promotion or merge-closeout records.
4. Successful recovery requests must return the PR to ordinary snapshot monitoring instead of immediately surfacing terminal manual action.
5. Conflict-shaped recovery failures must transition the Linear issue to `Rework` with exact blocker and attempt truth.
6. Unclassified recovery-command failures must fail closed with exact recovery metadata and without silent state changes.
7. The implementation must stay bounded to watcher recovery, provider reuse, and proof/observability truth.

## Design
- Export one branch-recovery helper from `pr-watch-merge.js` that:
  - detects recoverable `merge_state=BEHIND` and `merge_state=DIRTY` blockers
  - runs `gh pr update-branch <pr> --repo <owner>/<repo>` non-interactively
  - records exact command, args, exit status, stdout, stderr, and the triggering merge-state reason
- Teach `runPrWatchMerge(...)` to use that helper before exiting on `BEHIND` or `DIRTY`, and to keep watching after a successful refresh request.
- Reuse the same helper in `runProviderReviewHandoffPromotion(...)` and `runProviderDeterministicMergeCloseout(...)`.
- Extend review-promotion and merge-closeout records with additive recovery-attempt metadata.
- When the recovery helper fails with conflict-shaped GitHub output, transition the issue to `Rework` and persist the transition plus exact recovery-attempt details.

## Protected Expectations
- Preserve the current merge-ready blocker classifier after recovery; do not silently bypass review or check blockers.
- Preserve the current attached-PR disambiguation rules from `CO-104`.
- Preserve ordinary merged closeout and shared-root reconciliation semantics from `CO-111` / `CO-100`.
- Keep `Rework` as an explicit workflow transition, not an implied operator suggestion.

## Reject These Wrong Interpretations
- "Any green PR should move to `Merging` even if GitHub still reports conflicts."
- "This needs a general local git merge-and-push subsystem."
- "A successful branch-update request means the PR is immediately merge-ready without another snapshot."
- "The quickest fix is to keep waiting longer in `In Review` and call that recovery."

## Current Truth
- `resolveActionRequiredReasons(...)` already marks `merge_state=BEHIND` and `merge_state=DIRTY` as action-required blockers.
- `runPrWatchMerge(...)` logs those blockers and exits before any branch refresh attempt.
- `runProviderReviewHandoffPromotion(...)` and `runProviderDeterministicMergeCloseout(...)` both rely on that same blocker classifier but do not add recovery around it.
- Provider claim records do not currently carry a structured recovery-attempt payload.

## Proposed Design
- Introduce an exported recovery helper and recovery-attempt record type in the watcher seam.
- Persist the recovery-attempt payload on both `ProviderReviewHandoffPromotionRecord` and `ProviderMergeCloseoutRecord`.
- On successful refresh request, return a non-terminal watching result with the refreshed snapshot so later polling can continue truthfully.
- On conflict-shaped refresh failure, transition the issue to `Rework` and persist the transition plus exact command output.
- Keep generic non-conflict command failures as explicit action-required or failed states without hiding them behind review or check summaries.

## Non-Goals
- Solving arbitrary merge conflicts automatically by editing code.
- Reworking provider launch/dispatch lifecycle.
- Broad GitHub CLI surface redesign.
- Reopening merge-closeout truth lanes that do not involve green-but-dirty recovery.

## Parity / Alignment Matrix
- Current truth:
  - `BEHIND` and `DIRTY` are terminal blockers with no autonomous recovery attempt
  - conflicting PRs can remain in `In Review` or `Merging` with only generic action-required truth
  - recovery-attempt provenance is absent from provider proof
- Reference truth:
  - green-but-behind PRs should request branch refresh automatically
  - conflicting PRs should leave review/merge limbo and enter explicit rework when bounded recovery fails
  - review promotion and merge closeout should share one recovery contract
- Target truth / intended delta:
  - the watcher and provider code share one bounded recovery helper
  - successful refresh requests return the PR to ordinary monitoring
  - failed conflict recovery records exact attempt truth and transitions the issue to `Rework`
- Explicitly out-of-scope differences:
  - local conflict resolution and push
  - merge-queue or status-dashboard redesign
  - unrelated cooldown policy changes

## Not Done If
- A green `BEHIND` PR still stalls until a human clicks "Update branch".
- A `DIRTY` PR stays in `In Review` or `Merging` after bounded recovery fails.
- Provider proof still lacks exact recovery-attempt command/output metadata.
- Recovery success or failure can weaken the existing merge-safety classifier.
