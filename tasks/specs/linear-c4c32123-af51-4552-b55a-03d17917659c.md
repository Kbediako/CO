# Technical Specification

## Context

`CO-55` and `CO-60` improved the launched `CO STATUS` path, but they deliberately kept `co-status` as a foreground launch alias over `control-host`. The current operator gap is now different: when a JSON `control-host` is already running, there is still no dedicated read-only terminal viewer that attaches to that host. Operators who only want terminal status must still launch the host path itself, and the launched live monitor uses alternate-screen semantics that hide live scrollback until pause.

This lane must preserve the current launched flow while adding a separate attach flow that uses the already-running host's authenticated read-only dashboard surface.

## Requirements

1. A dedicated attach/viewer command must exist under the shipped CLI help surface.
2. The attach viewer must resolve the local host from persisted run-dir endpoint/auth artifacts and/or explicit task/run/run-dir flags.
3. The attach viewer must remain read-only and must not launch another `control-host`, Linear poller, or Telegram poller.
4. Live attach viewing must keep primary terminal scrollback available without requiring `p` first.
5. Useful local inspect affordances (`p`, `c`, `s`) should remain available when they stay truthful and read-only.
6. Help/docs must make launch vs attach behavior explicit.
7. Focused tests and real local proof must cover the attach flow, not just static render output.

## Current Truth

- `bin/codex-orchestrator.ts` routes `co-status` to `runControlHostCliShell(...)`.
- `printCoStatusHelp()` describes `co-status` as a launch alias over `control-host`.
- `startControlStatusDashboard(...)` defaults live interactive output to the terminal alternate screen.
- The running host already exposes authenticated read-only dashboard data at `/ui/data.json`.
- The host run dir already persists `manifest.json`, `control_endpoint.json`, and `control_auth.json`.

## Proposed Design

### 1. `co-status attach` command surface

Add an explicit attach subcommand under `co-status` while preserving the current default `co-status` launch behavior.

Proposed CLI shape:

- `codex-orchestrator co-status` -> current launch path
- `codex-orchestrator co-status attach` -> new read-only attach viewer

The attach help should document:

- default `task=local-mcp`
- default `run=control-host`
- optional explicit `--run-dir`
- launch-vs-attach difference

### 2. Artifact-backed host resolution

The attach viewer should resolve the target host from the persisted control-host run dir:

- default run dir from `task` + `run`
- or explicit `--run-dir`
- load and validate `control_endpoint.json`
- load and validate `control_auth.json` (or the token sidecar referenced by `control_endpoint.json`)

This lane should not guess internal routes or require the operator to manually assemble URLs.

### 3. Read-only dataset fetch

The attach viewer should fetch the authenticated `/ui/data.json` payload from the host and must not invoke mutating host actions.

This means:

- no `/control/action`
- no new provider polling loop in the viewer
- no second `control-host`

The viewer process is only a read-side terminal surface.

### 4. Primary-scrollback attach rendering

The attach viewer should render live in primary scrollback rather than the alternate screen. This keeps scrollback available while live.

The existing renderer can likely stay the visual source of truth if it gains a narrow way to distinguish:

- launched live mode: alternate screen
- attached live mode: primary scrollback
- paused inspect mode: primary snapshot

If direct reuse would widen the design too much, create a narrow attach-viewer path that still reuses `renderControlStatusFrame(...)` rather than forking a second frame presenter.

### 5. Local-only inspect affordances

`p`, `c`, and `s` should remain local viewer behaviors:

- `p` freezes local redraw only
- `c` toggles compact/full local rendering
- `s` exports a local snapshot under the resolved run dir

None of these should mutate the running host.
