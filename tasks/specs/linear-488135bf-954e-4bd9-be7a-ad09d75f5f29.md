# Technical Specification

## Context

Current provider-worker lanes already satisfy several Symphony-informed constraints: one live app-server thread, same-workspace continuation, and a single main worker that owns issue execution. The missing parity is within-issue bounded child streams. Today a provider worker cannot launch a sanctioned subordinate review/planning lane from inside that worker lifetime, so review-style steps still resort to a blanket delegation override even though `delegation-guard` already recognizes a legitimate provider-child lineage model.

## Requirements

1. Add a provider-worker child-stream launcher that can start at least one bounded review/planning pipeline as a sanctioned child run of the active provider worker.
2. Require the launcher to run only inside an active `provider-linear-worker` run and to fail closed if `CODEX_ORCHESTRATOR_MANIFEST_PATH`, `CODEX_ORCHESTRATOR_RUN_ID`, `CODEX_ORCHESTRATOR_TASK_ID`, current issue metadata, or provider-control-host provenance are missing or mismatched.
3. Reuse the current sanctioned provider-child task identity contract: child task ids must be `<provider-task-id>-<stream>` and child manifests must carry `parent_run_id` pointing at the active provider-worker run.
4. Keep child streams bounded to read-only, planning, and review surfaces; they must not directly mutate scheduling state or widen authority outside the parent issue workspace/root.
5. Preserve the current single-stream provider-worker success path when no child streams are used.
6. Record truthful child-stream lineage in provider-worker proof/read-model artifacts so operators can audit launches and outcomes after the run.
7. Ensure scheduler-owned provider-run discovery ignores nested provider-worker child manifests so it continues to track only the authoritative provider worker for an issue.
8. Prove docs-review and review-style validation can use actual child-stream lineage instead of the blanket provider-worker delegation override when a real child stream is launched.
9. Add focused tests covering launcher fail-closed behavior, lineage surfacing, and nested-manifest exclusion without weakening existing delegation or provider-run contracts.
