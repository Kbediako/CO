# Technical Specification

## Context

The standalone review wrapper already knows that current Codex CLI behavior rejects combining a prompt argument with `--commit`, `--base`, or `--uncommitted`. The bug is that the launch builder still appends the prompt payload for explicit scoped runs anyway. Because the wrapper correctly refuses to silently remove explicit scope flags, current commit/base review requests fail instead of executing truthfully on the requested review surface.

## Requirements

1. `npm run review -- --commit <sha>`, `npm run review -- --base <ref>`, and `npm run review -- --uncommitted` must launch `codex review` without the unsupported prompt argument and must keep the requested explicit scope intact.
2. The wrapper must continue saving prompt/context artifacts beside the manifest for audit continuity even when explicit scoped launches cannot pass that prompt inline to Codex.
3. Wrapper help/docs must state the scoped-launch contract truthfully: explicit scoped runs use scoped review without a prompt argument, while prompt-only focus runs stay unscoped.
4. Persisted review evidence must remain truthful about scoped launch behavior so operators can tell whether prompt context was delivered inline or retained only as wrapper-side evidence.
5. Focused regressions must cover the shared scoped launch builder and end-to-end wrapper execution for commit/base/uncommitted review, plus any retry guard that keeps failure telemetry truthful when no scope-dropping retry would change the launched command.
6. The change must stay bounded to standalone-review wrapper launch behavior, docs, and focused tests.

## Current Truth

- `buildReviewArgs(...)` in `scripts/lib/review-launch-attempt.ts` currently appends `prompt` unconditionally, even when `--commit`, `--base`, or `--uncommitted` are present.
- `runReviewLaunchAttemptShell(...)` intentionally refuses to retry without explicit scope flags when the CLI rejects prompt-plus-scope, because dropping scope would make the review surface unauditable.
- `scripts/run-review.ts`, `docs/standalone-review-guide.md`, `skills/standalone-review/SKILL.md`, `AGENTS.md`, and `docs/AGENTS.md` already state the upstream compatibility guard, so the runtime behavior is currently drifting away from the documented contract.
- Existing tests already cover prompt artifacts, scope-path notes, and prompt-plus-scope failure handling, but they do not yet pin the truthful success path for explicit commit/base/uncommitted launch without inline prompt delivery or the no-op retry short-circuit for unchanged resolved commands.
