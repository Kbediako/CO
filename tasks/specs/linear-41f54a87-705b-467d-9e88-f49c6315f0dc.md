# Technical Specification

## Context

Symphony's Elixir runtime maintains a workflow store that fails startup on missing or invalid workflow configuration but, after startup, logs reload errors and keeps serving the last known good workflow until the file is fixed. CO does not yet provide the same safety contract for the equivalent control-host provider workflow path. Today the provider launch flow resolves repo-local `codex.orchestrator.json` directly, so a later invalid config edit can break provider-child launch resolution instead of preserving a working effective snapshot.

## Requirements

1. Fail closed at control-host startup when the repo-local provider workflow/config is missing or invalid for the required provider launch path.
2. After successful startup, preserve the last known good effective provider workflow/config when a later reload attempt fails because the repo config is missing or invalid.
3. Allow a later valid config edit to replace the cached snapshot and clear the reload error state.
4. Log reload failures with enough detail for operators to identify the failing path and reason.
5. Surface the current provider workflow health, including degraded reload state, through the existing operator-visible control-host observability path.
6. Keep the change scoped to the control-host provider workflow/config seam and avoid altering unrelated runtime or provider behavior.
7. Add focused tests covering startup failure, good-to-bad reload retention, and bad-to-good recovery.
8. Retry the same source revision after transient reload failures instead of staying permanently degraded until source metadata changes.
9. Ensure provider launches never return a dead cached snapshot path; when the snapshot disappears but the source config still validates, recreate or revalidate the snapshot before launch.

## Current Truth

- `orchestrator/src/cli/config/userConfig.ts` currently loads repo-local `codex.orchestrator.json` directly from `env.repoRoot`.
- `orchestrator/src/cli/controlHostCliShell.ts` launches provider child runs for the `provider-linear-worker` pipeline but does not maintain a validated cached repo-config snapshot.
- `orchestrator/src/cli/control/providerIssueHandoff.ts` depends on the control-host launcher path, so provider child starts and resumes inherit any repo-config read failures at launch time.
- Existing observability controllers already publish control-host state, but there is no dedicated provider workflow reload health field today.
- Symphony's `workflow_store.ex` is the parity baseline: initial load failure stops startup, reload errors log and keep the last known good configuration, and recovery clears the error on the next successful reload.
- The prior implementation attempt proved the provider seam and review surface, but `origin/main` does not contain the feature yet because the unmerged branch was reset during `Rework`.
