# Technical Specification

## Context
The `1093` through `1109` packet cohort is already completed, but those packet specs still live under `tasks/specs/` with March 10 review dates. `spec-guard` currently treats every Markdown file in `tasks/specs/` and `docs/design/specs/` as an active freshness target, so unrelated merge-ready PRs fail on this baseline before they reach their own issue-specific validation. The repo already has an implementation-doc archive convention, but archived spec stubs still reside under `tasks/specs/`, which means archival and guard semantics are currently out of sync.

## Requirements
1. Completed packet docs for `1093` through `1109` must be archived or quarantined truthfully.
2. Archived spec stubs must not participate in active-spec freshness gating.
3. Active specs must continue to require a valid `last_review` within the existing 30-day policy.
4. `CO-102` must remain scoped to the separate archive-selection / `docs:freshness` seam.
5. Representative blocked PR evidence must show that this stale cohort is no longer the baseline blocker.

## Design
- Quarantine the full `1093` through `1109` packet cohort from active freshness by treating inactive task-spec statuses as non-active during freshness checks.
- Preserve the existing archive-stub skip so already archived specs that match the repo-standard stub shape stay out of active freshness too.
- Keep `tasks/index.json` and packet docs otherwise stable so this lane does not silently reopen archive-selection work owned elsewhere.

## Implementation Surface
- Expected codepaths:
  - `scripts/spec-guard.mjs`
  - `tests/spec-guard.spec.ts`
- Expected proof surfaces:
  - `node scripts/spec-guard.mjs`
  - representative PR metadata or rerun evidence for `#395`, `#397`, `#399`, `#400`, and `#401`

## Protected Expectations
- Preserve the current active-spec freshness rule for non-archived specs.
- Preserve the repo-standard archive stub shape (a top-level Markdown heading, optional `last_review`, `<!-- docs-archive:stub -->`, `Archived on`, `Archive branch`, and a same-file `Archive path`) and existing inactive task-spec statuses.
- Keep the quarantine set bounded to the completed `1093` through `1109` cohort semantics rather than broad repo cleanup.
- Keep `CO-102` ownership boundaries explicit.

## Reject These Wrong Interpretations
- Archival alone is sufficient even if completed specs are still treated as active freshness targets.
- Refreshing dates on completed packets is equivalent to a truthful archive or quarantine decision.
- The issue should be folded into generic docs-freshness cleanup.
- A PR waiver is an acceptable substitute for fixing the baseline.

## Current Truth
- the completed packet specs from `tasks/specs/1093-coordinator-symphony-aligned-standalone-review-diff-audit-surface-split.md` through `tasks/specs/1109-coordinator-symphony-aligned-standalone-review-audit-exported-env-startup-anchor-propagation.md` are stale.
- implementation-doc archival already exists and writes archive stubs into tracked paths.
- `spec-guard` does not yet distinguish archived spec stubs from active specs.

## Proposed Design
- Treat the full repo-standard archive stub shape as the authoritative signal that a spec is archived, including a top-level heading and a same-file `Archive path`.
- Treat inactive task-spec statuses as authoritative signals that a spec is no longer part of active freshness.
- Skip inactive specs and archived stubs in `spec-guard` freshness checks.

## Non-Goals
- Reworking the broader implementation-doc archive policy.
- Cleaning unrelated stale docs across the repo.
- Reopening merge-conflict recovery or representative PR branch refresh work.
