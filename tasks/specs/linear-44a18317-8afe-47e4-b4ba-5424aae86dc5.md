# Technical Specification

## Context

`CO-55`, `CO-67`, `CO-73`, and `CO-74` already landed narrow STATUS improvements: launch ergonomics, paused handoff truth, null-provider current-running authority, and attach viewing. The remaining work in `CO-76` is different. The shipped STATUS surface now needs one hardening pass that proves the whole operator product is coherent: every rendered field and state needs a documented coverage story, the machine-readable STATUS contract needs to stop being ambiguous, real screenshots on this device need to show the representative states inline in Linear, and any stale legacy STATUS naming that obscures current authority should be reduced when the cleanup is obviously bounded.

This lane must keep the existing truthful read-side sources while making `co-status --format json` an explicit STATUS snapshot contract and documenting how the terminal surface, `/ui/data.json`, and `/api/v1/state` agree.

## Requirements

1. A documented coverage matrix must enumerate every currently rendered `CO STATUS` field/state and map it to automated coverage or explicit manual proof.
2. `co-status --format json` must emit a machine-readable STATUS snapshot rather than launch-readiness metadata.
3. The STATUS JSON snapshot must align on shared STATUS fields with the interactive terminal dashboard and the authoritative `/ui/data.json` or `/api/v1/state` sources that back those values.
4. Time-relative values, especially aggregate runtime and rate-limit reset countdowns, must have an explicit comparison contract and deterministic tests.
5. Representative inline screenshot proof on this device must cover normal live, paused/inspect, compact/constrained-height, empty/idle, and retry/degraded or unavailable telemetry states.
6. Any stale selected-run or legacy STATUS naming/helper surfaces touched by this lane must either be clarified, renamed, or explicitly retained for a reviewed reason.

## Current Truth

- The interactive terminal dashboard already renders from `operatorDashboardPresenter.ts`, which reads the compatibility projection built by `controlRuntime.ts`.
- `/ui/data.json` already returns the same operator-dashboard dataset (`mode: operator_dashboard`, `read_only: true`).
- `/api/v1/state` exposes overlapping running/totals/rate-limit/polling truth from the same compatibility runtime, but with a different payload shape.
- `co-status` still routes interactive launch through `runControlHostCliShell(...)`.
- `control-host --format json` and `co-status` help still describe machine-readable readiness output, not a STATUS snapshot.
- `co-status attach --format json` currently returns attach-target metadata (`mode: attach`) rather than the STATUS dataset itself.
- The live local host under `/Users/kbediako/Code/CO/.runs/local-mcp/cli/control-host` currently shows selected issue `CO-76` with `running = 1`, null token totals, live rate-limit reset timestamps, and active polling-checking state on both `/ui/data.json` and `/api/v1/state`; runtime seconds differ across requests because they are sampled at different times.
- Older selected-run presenter names and tests remain in the repo even though the shipped STATUS surface is no longer built from that old selected-run UI dataset.

## Proposed Design

### 1. Explicit STATUS JSON contract

- Keep `control-host --format json` as launch/readiness output.
- Change `co-status --format json` so it resolves the current local control-host status source and emits the machine-readable STATUS snapshot instead of readiness metadata.
- Reuse the current operator-dashboard dataset shape for the shared STATUS fields instead of inventing another payload.
- Fail closed with a clear error if the STATUS snapshot cannot be resolved from the current local host state.

### 2. Shared STATUS snapshot helper

- Introduce or extract a small shared helper for:
  - resolving the local control-host endpoint/auth target
  - reading the current operator-dashboard dataset snapshot
- Reuse that helper in the new STATUS JSON path so the terminal and JSON flows share the same read-side truth inputs.
- Do not widen this into host lifecycle or mutating-control changes.

### 3. Coverage matrix + agreement tests

- Add one explicit coverage matrix for the current STATUS surface.
- Tighten tests so each of these areas has a clear automated owner:
  - summary/header fields
  - running and retry rows
  - polling and next-refresh behavior
  - token unavailable semantics
  - rate-limit source and countdown rendering
  - empty/idle, paused/inspect, compact, attach, and snapshot/export states
  - STATUS JSON snapshot agreement with the operator-dashboard dataset
- Use the matrix to capture which rows require manual screenshot proof rather than pretending every visual state is fully automated.

### 4. Bounded legacy cleanup

- If stale selected-run or legacy STATUS naming remains directly on the current STATUS path, rename or narrow it when the change is obviously local and reduces truth ambiguity.
- Do not delete broad historical presenter/controller surfaces unless they are provably unused and the cleanup stays small.
- File a follow-up instead of broadening the lane if cleanup grows beyond status-contract clarity.
