## Technical Requirements
- Functional requirements:
  1. `docs:freshness:maintain` exposes a deterministic `canonical_owner_key` and `canonical_owner_marker` for eligible recurrence candidates.
  2. Before mutation or follow-up guidance, the maintenance path verifies live owner state by exact marker in the same team/project.
  3. Open exact-marker owners select an update/noop path rather than a duplicate create path.
  4. Missing owners select a create path with complete copyable issue body evidence.
  5. Terminal stamped owners select a terminal replacement path; they are not reopened or reused.
  6. Route classification distinguishes:
     - `CO-428`: stale active-spec
     - `CO-429`: completed-lane registry residue
     - `CO-430`: terminal-owner replacement
  7. Report output includes action evidence for create, update, noop, and terminal replacement decisions.
  8. Dry-run and no-token runs emit the same route/action evidence and copyable bodies without mutation.
  9. Scheduled maintenance and provider-worker preflight paths consume the same decision schema.
- Non-functional requirements:
  - fail closed on ambiguous canonical keys, missing required route fields, or mismatched owner scope
  - avoid extra Linear calls when local report-only dry-run/no-token mode is active
  - keep output deterministic enough for tests and parent workpad evidence
  - no new runtime dependency unless the parent validates the need
- Interfaces / contracts:
  - maintenance report fields should include `canonical_owner_key`, `canonical_owner_marker`, route id, route reason, owner state, owner issue identifiers when known, intended action, dry-run/no-token mutation status, and copyable body text
  - provider-worker preflight must consume report JSON rather than parse human console text
  - scheduled workflow output must store the same JSON contract under the task/out artifact path
