## Objective

Repair the provider recovery/control-host admission contract so explicit recovery of an accepted pending-revalidation no-run claim produces durable pending, launch, retry, release, or failure evidence instead of terminal `provider_worker_recover_no_launch_evidence` or refreshed accepted/no-run state.

## Protected Evidence

- CO-575 and CO-574 both produced accepted claims with no `launch_token`, no `run_id`, and no `run_manifest_path`.
- CO-574 nudge after CO-575 release returned `provider_worker_recover_no_launch_evidence` while recovery was still represented as accepted pending-revalidation.
- GPT Pro advisory recommends treating no-launch accepted state as intermediate, preserving a stable manifestless stale anchor, and splitting active-worker probe timeout from true dead-host restart.

## Non-Goals

- No manual provider-intake edits.
- No weakening WIP/admission caps.
- No docs freshness owner repair in this lane.
- No broad worker lifecycle rewrite without additional evidence.

## Acceptance Criteria

- [ ] Observability API recovery acknowledgements no longer classify accepted/no-run pending-revalidation as terminal no-launch while recovery is in flight.
- [ ] Provider rehydrate/recovery preserves a stable stale anchor for manifestless accepted claims.
- [ ] Capacity-blocked recovery remains truthful and test-covered.
- [ ] Active-worker machine-status/probe timeout behavior is either fixed or explicitly left unchanged with evidence that it is not part of the patch.
- [ ] Focused tests and required repo gates are recorded before handoff.
