# Technical Specification

## Context

CO already has strong docs truth and docs freshness lanes, but they are structurally scoped to markdown/doc surfaces. The missing contract is not another stale-doc sweep. The missing contract is a recurring tracked-file stewardship system that inspects the whole repo, decides what each tracked surface needs, and publishes bounded upkeep output that reviewers and future agents can act on.

## Requirements

1. Inventory every tracked file via `git ls-files`, not just markdown/doc roots.
2. Assign every tracked surface one machine-checkable decision: `validate`, `update`, `delete`, or `retain_with_rationale`.
3. Cover all major repo areas explicitly, including root/front-door docs, code, scripts, configs, tests, fixtures, skills, templates, references, archives, and assets.
4. Keep README/public onboarding truthfulness tied to the existing CLI/runtime/skills docs checks rather than file age alone.
5. Publish reviewable JSON and markdown stewardship artifacts.
6. Add recurring automation for the stewardship lane.
7. Detect at least one real non-doc-root action candidate from the current repo and, if needed, split broad cleanup into a same-project follow-up.

## Current Truth

- `scripts/lib/docs-helpers.js` only returns markdown files from `.agent`, `.ai-dev-tasks`, `docs`, `skills`, `tasks`, `templates`, plus root `README.md` / `AGENTS.md`.
- `scripts/docs-freshness.mjs` only evaluates that doc inventory against `docs/docs-freshness-registry.json`.
- `.github/workflows/docs-truthfulness-weekly.yml` currently runs `docs:freshness` only, so weekly automation is still markdown-centric.
- `docs/docs-catalog.json` carries truth checks for docs surfaces, not repo-wide tracked-file stewardship decisions.
- Tracked historical payloads already exist under `reference/**` and `archives/**`, but they are not under a repo-wide retain/update/delete rationale contract.
