## Summary
- Objective: implement a cheap, authenticated status-machine contract so `co-status --format json` and control-host supervision do not depend on the heavy `/ui/data.json` dashboard projection to prove fresh quiescent health.
- Scope: status read model/controller, co-status current-endpoint degraded behavior, supervision probe decoupling, progress-field propagation, bounded dashboard degradation, focused tests, docs mirrors, and validation.
- Constraints: no timeout-only fix, no public-health-only proof, no manual provider-intake edits, no broad provider-intake/admission rewrite, no hiding failed runs or true `restart_required` state.

## Issue-Shaping Contract
- User-request translation carried forward: CO-572 exists because repeated status/probe failures are still happening after CO-571 fixed provider progress-stall classification. The fix must address the status architecture root cause and can be a larger refactor if that is the correct solution.
- Protected terms / exact artifact and surface names: `co-status --format json`, `/ui/data.json`, `same-endpoint current-endpoint timeout`, `probe_failed`, `probe_timeout`, `provider-intake-state.json`, `control_endpoint.json`, `control_auth.json`, `polling.restart_required=false`, `polling.stuck=false`, `active_claims=[]`, `progress_updated_at`, `machine status endpoint`, `readCompatibilityProjection()`.
- Nearby wrong interpretations to reject: timeout bump, launchd restart, public `/health` only proof, treating zero active claims as no proof, treating all timeouts as healthy, hiding failed runs, hand-editing provider-intake artifacts, or widening into provider-worker admission policy.
- Explicit non-goals carried forward: no CO-571 rework without new progress-stall evidence, no CO-407/CO-376/CO-336 history rewrite, no docs/spec gate weakening, and no fabricated complete dashboard snapshot.

## Parity / Alignment Matrix
- Current truth: provider polling can be fresh and idle while `co-status` still fails because dashboard projection is the machine-health read and degraded fallback expects active claims.
- Reference truth: machine health should be cheap, bounded, authenticated, current-source aware, and independent from dashboard compatibility projection.
- Target truth / intended delta: cheap machine status proves fresh quiescent health; `co-status` and supervisor consume it; dashboard timeouts are explicit degraded dashboard events; negative cases remain fail-closed.
- Explicitly out-of-scope differences: optimizing every dashboard subprojection is secondary if the first implementation already removes dashboard projection from machine health and preserves fail-closed semantics.

## Technical Requirements
- Functional requirements:
  - Add cheap machine status read model/controller that is authenticated, GET-only, no-store, and bounded.
  - Machine status must not call `readCompatibilityProjection()` or full dashboard presenter paths.
  - Machine status includes source owner/freshness, endpoint identity, provider polling status, `progress_updated_at`, `progress_elapsed_ms`, provider-intake update times, active/running/retrying counts, max allowed, and degraded metadata.
  - `co-status --format json` uses machine status as the authoritative current endpoint health path and treats dashboard timeout/reset as explicit degraded read when machine status proves freshness.
  - Zero active claims with fresh polling/progress and no restart/stuck reason must produce fresh quiescent status, not a thrown timeout.
  - Supervisor probes machine status or an equivalent bounded local diagnostic rather than full `/ui/data.json`.
  - `/ui/data.json` dashboard read is bounded and returns explicit dashboard-degraded/error metadata when it cannot complete within budget.
- Non-functional requirements:
  - Fail closed on auth, endpoint corruption, stale source, dead endpoint without fresh local proof, true provider stuck, active no-progress refresh, and failed run evidence.
  - Preserve source labels and reason metadata in every degraded response.
  - Keep status path fast enough for supervision cadence without scanning full run history or docs freshness detail.
- Interfaces / contracts:
  - Candidate HTTP endpoint: `GET /api/v1/machine-status`, `GET /machine/status.json`, or `GET /status.json`.
  - CLI output should remain JSON-compatible with existing `co-status --format json` consumers and add explicit `degraded_read` / `dashboard_degraded` metadata rather than pretending partial data is complete.
  - Supervision evaluator accepts machine-status payloads and preserves current fail-closed decision classes.
