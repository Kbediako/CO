# Technical Specification

## Context
CO already has the control-host and provider-worker structure needed for a bounded worker-host parity slice, but all of the worker-host specifics are missing. Workflow config has no host inventory, capacity tracking is not host-aware, launches are local-only, and observability mostly assumes the local host. The issue is therefore to add one optional distributed path to the current provider control plane, not to replace the default model or redesign runtime ownership.

## Requirements
1. `provider-linear-worker` must expose additive worker-host inventory and per-host capacity configuration.
2. The control host must be able to select an eligible configured worker host for bounded remote execution.
3. Host selection must remain explicit and truthful, including retry preference for the prior host when safe.
4. Remote dispatch must use SSH or an equivalently bounded transport and keep the local single-host path as the default.
5. `worker_host` must persist into provider proof, read-model, and operator surfaces where relevant.
6. Failure or recovery semantics must remain truthful and must not silently reroute past side effects.
7. Focused tests must cover host selection, capacity behavior, launch routing, and observability.
8. The change must stay bounded to this parity lane and not absorb `CO-82`, `CO-83`, or `CO-88`.

## Design
- Config:
  - extend effective provider workflow metadata with optional `worker_hosts`
  - each host record should include a stable `name`, `ssh_destination`, optional `ssh_options`, optional `node_path`, and optional `max_concurrent_agents`
  - this first lane assumes the remote host can access the same absolute repo, workspace, `.runs`, and `out` paths as the control host; it does not add host-local path remapping or repo sync
- Selection:
  - retain current global or per-state gating
  - add host-aware load accounting from active provider state
  - choose the least-loaded eligible host under explicit capacity
  - prefer the prior `worker_host` on retries when that preserves truthful ownership
- Launch:
  - build remote launch specs at the control-host seam
  - use SSH-backed command execution from the control host into the shared checkout paths already persisted by the control host
  - keep environment propagation bounded and reuse the existing provider-worker command shape as much as possible
- Observability:
  - persist `worker_host` in provider proof and state
  - surface worker-host truth in provider issue observability, compatibility, and operator read-model output

## Protected Expectations
- Keep the feature optional and bounded.
- Keep the single-host default intact.
- Keep SSH host routing, capacity selection, and `worker_host` observability in the shipped behavior, not only in docs.
- Keep retry or recovery semantics truthful.

## Reject These Wrong Interpretations
- Replace the existing local single-host model.
- Add only placeholder config or documentation.
- Treat remote-capacity exhaustion as a reason to silently fall back to local execution.
- Fold unrelated provider observability or runtime rewrite work into this lane.

## Current Truth
- Provider workflow config has no worker-host inventory.
- Capacity is global or per-state only.
- Provider launch specs are local-only.
- `worker_host` is absent or inconsistent across proof and operator surfaces.

## Proposed Design
- Add additive worker-host metadata to `provider-linear-worker`.
- Teach the control host to select hosts using explicit per-host capacity and retry truth.
- Launch provider workers remotely over SSH through the existing provider-launcher seam.
- Thread `worker_host` through proof, state, and read-model surfaces so operators can see which host owns a run.

## Non-Goals
- Mandatory distributed runtime.
- General runtime continuity or app-server rewrite work.
- Provider-debug-snapshot expansion already owned elsewhere.
- Reopening `CO-82`, `CO-83`, or `CO-88`.

## Parity / Alignment Matrix
- Current truth:
  - local-only provider launch path
  - no worker-host inventory or host-aware capacity
  - no consistent `worker_host` projection
- Reference truth:
  - Symphony already supports optional SSH worker hosts, per-host concurrency, retry stickiness, and `worker_host` observability
- Target truth / intended delta:
  - CO keeps the same control-host ownership model while adding one optional distributed worker-host path with truthful host selection and observability
- Explicitly out-of-scope differences:
  - hot-reload or runtime rewrite work
  - broader telemetry or debug-snapshot parity

## Not Done If
- The feature is only documented and not executable.
- Capacity selection is not enforced in runtime truth.
- `worker_host` is not visible in the relevant operator surfaces.
- The single-host default regresses.
