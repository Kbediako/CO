# Technical Specification

## Context
`operator-autopilot` already promotes the first safe `Backlog` head to `Ready` and already uses the shared Linear transition guard with `expectedStateName`, `expectedStateType`, and `expectedUpdatedAt`. That single-transition safety is not enough for the cross-cycle churn case in `CO-216`: after autopilot promotes an issue to `Ready`, an operator can manually demote the same issue back to `Backlog`, and the next autopilot cycle can treat that same issue as the first safe backlog head again because backlog promotion currently reads only live tracked issues, existing claims, and blocked-by truth. The result and observability path also flatten transition metadata down to a smaller payload, which obscures the exact snapshot and force-path truth the parent needs for explanation.

## Requirements
1. `CO-216` must stay scoped to the operator-autopilot backlog re-promotion churn seam.
2. Parent implementation must detect a bounded manual-demotion signal for the same issue after a prior autopilot backlog promotion.
3. Parent implementation must hold the exact manual-demotion snapshot instead of re-promoting it immediately on the next cycle.
4. Normal safe backlog-head promotion must remain unchanged when no manual-demotion signal exists.
5. Backlog promotion must continue to pass expected-state metadata through the shared transition helper.
6. Stored autopilot result and observability payloads must expose the relevant `issue updated_at`, hold reason, and whether a force path was used.
7. The hold must be keyed narrowly enough that later distinct backlog snapshots can still be reevaluated.
8. The lane must not widen into scheduler redesign, reclaim, refresh-stall recovery, or generic `CO-215` transition work.
9. Parent validation must include the focused regression for promote -> manual demote -> hold plus a normal-promotion preserve case.

## Design
- Reuse the existing backlog-promotion ordering, ownership, claim-state, and blocked-by checks.
- Reuse the existing `transitionProviderLinearIssueState(...)` contract rather than redesigning it.
- Derive the manual-demotion hold from existing autopilot result plus live tracked-issue truth, preferably using the prior promoted snapshot's `issue_updated_at` and the newer live `Backlog` snapshot.
- Extend autopilot action/hold projection so operators can see the snapshot and force-path truth without inspecting raw lower-level transition payloads.
- Keep the hold specific to the exact manual-demotion snapshot rather than building a broad backlog lifecycle framework.

## Protected Expectations

- Preserve `expectedStateName`, `expectedStateType`, and `expectedUpdatedAt` on backlog promotion.

- Preserve hold reason visibility in observability and extend it rather than replacing it with summary-only text.
- Preserve narrow scope; do not redesign queue ranking or refresh behavior.

## Reject These Wrong Interpretations
- "This is really a generic scheduler rewrite."
- "This should be fixed by broadening `CO-215`."
- "Any later `Backlog` appearance should block forever."
- "We can solve this by hiding the promotion in summary text only."
- "A refresh-stall or reclaim fix should absorb this issue."

## Current Truth
- `runProviderOperatorAutopilot(...)` receives `previous_result`, but `maybeRunBacklogPromotion(...)` does not currently use it.
- `maybeRunBacklogPromotion(...)` already passes `expectedStateName`, `expectedStateType`, and `expectedUpdatedAt`.
- `transitionProviderLinearIssueState(...)` already returns `transition_guard` with expected-state and force metadata.
- `ProviderOperatorAutopilotLinearTransitionRecord` currently stores only `previous_state`, `target_state`, `issue_state`, `issue_state_type`, `issue_updated_at`, and `error`.
- `ProviderOperatorAutopilotHoldRecord` currently stores `reason` and `summary`, but no hold-side snapshot or force-path flag.
- `providerWorkflowConfigStore.ts` persists `operator_autopilot.last_result`, and `observabilityReadModel.ts` projects it.

## Proposed Design
- Detect a manual-demotion hold candidate when:
  - the same issue was recently autopilot-promoted to `Ready`, and
  - the live tracked issue is again `Backlog`, and
  - the live `updated_at` is newer than the promoted snapshot in a way consistent with a later manual demotion.
- Emit a bounded backlog hold instead of a new transition for that same snapshot.
- Preserve normal promotion for:
  - a different backlog head
  - the same issue without a matching manual-demotion signal
  - a later distinct backlog snapshot that no longer matches the held demotion signal
- Extend stored autopilot result / observability payload so the parent can expose:
  - expected-state inputs at the shared transition-helper seam
  - relevant `issue updated_at`
  - hold reason
  - whether a force path was used

## Non-Goals
- Queue-order redesign.
- Generic transition redesign.
- Reclaim or refresh-stall remediation.
- New broad backlog lifecycle store unless the parent proves existing result data is insufficient.
- Child-lane implementation or test edits.

## Parity / Alignment Matrix
- Current truth:

  - transition metadata is partially lost in autopilot projection
- Reference truth:
  - manual demotion should stop immediate churn
  - operators should still see why the hold happened
  - normal safe promotion should remain intact
- Target truth / intended delta:
  - bounded manual-demotion hold
  - preserved expected-state inputs at the shared transition-helper seam
  - projected `issue updated_at`, hold reason, and force-path usage
- Explicitly out-of-scope differences:
  - scheduler redesign
  - reclaim/stall repair
  - generic transition feature work

## Not Done If
- a manual demotion is immediately undone by autopilot on the next cycle
- normal safe backlog promotion is blocked when no manual-demotion signal exists
- expected-state inputs are no longer passed through the shared backlog-promotion transition helper
- observability cannot tell the operator which snapshot and hold reason were involved
- the implementation requires broad queue redesign
