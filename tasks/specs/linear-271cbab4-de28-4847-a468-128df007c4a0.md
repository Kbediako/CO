# Technical Specification

## Context

`CO-14` deliberately stopped at the architecture contract. Current CO still has:

- one external provider-owner per issue (`control-host` -> `provider-linear-worker`)
- one authoritative workpad and PR lifecycle
- one existing subordinate helper path (`linear child-stream`) that is limited to docs or review pipelines

That means the contract is documented, but not yet real for coding lanes. `CO-35` is the first runtime slice that makes parent-owned same-issue multi-worker behavior real without changing who owns the issue externally.

## Requirements

1. Keep `control-host` plus `provider-linear-worker` as the single external issue owner and treat the provider worker as the issue parent.
2. Add a real subordinate same-issue child-lane runtime that launches with:
   - `<parent-task-id>-<stream>` task identity
   - `parent_run_id` lineage
   - declared purpose
   - explicit file-scope or phase-scope ownership
3. Keep all Linear mutation paths parent-owned only, including workpad, issue state, PR attachment, and review or merge handoffs.
4. Keep subordinate execution inside the authoritative per-issue workspace boundary and force the parent to accept, reject, rerun, or invalidate child output explicitly.
5. Extend parent proof and read-side observability so nested `child_lanes` are visible without surfacing subordinate runs as peer issue owners.
6. Capture real runtime proof where one parent issue run uses more than one subordinate lane and still keeps the same issue, workpad, and PR truth.

## Runtime Contract

- Parent authority:
  - one `provider-linear-worker` parent remains the only actor allowed to mutate Linear state, the persistent workpad, PR attachments, and review or merge handoffs
  - subordinate lanes are internal helpers only
- Subordinate lane identity:
  - child task id is `<parent-task-id>-<stream>`
  - child manifest records `parent_run_id` pointing at the parent provider-worker run
  - child pipeline id is subordinate-specific and must be excluded from peer-owner discovery
- Declared purpose and scope:
  - each child lane must declare `purpose`
  - each child lane must declare either file-scope ownership, phase-scope ownership, or both
  - overlapping unresolved scopes are rejected rather than silently queued
- Workspace and output truth:
  - the authoritative issue workspace stays parent-owned
  - child execution runs in a lane-local worktree rooted under the parent workspace boundary
  - each child lane emits a patch artifact and parent snapshot metadata (`base_sha`, issue freshness)
  - the parent explicitly accepts, rejects, or invalidates the lane output
- Stale invalidation:
  - a child lane becomes stale when the parent base SHA or parent issue freshness has moved beyond the lane snapshot in a way that makes the patch unsafe
  - stale lanes cannot be accepted until invalidated and relaunched
- Observability:
  - parent proof surfaces nested `child_lanes`
  - read-side surfaces either expose `child_lanes` directly or surface them via the parent proof payload
  - subordinate manifests never become peer issue-owner rows in compatibility or selected-run views

## Planned Implementation

1. Add a new same-issue child-lane helper surface under `linear`:
   - `launch`
   - `accept`
   - `reject`
   - `invalidate`
2. Add a new subordinate pipeline plus runner that:
   - executes a bounded lane prompt on the existing `codex exec` seam
   - works in a lane-local worktree under the parent workspace
   - writes a patch artifact and child-lane proof sidecar
3. Widen the parent child-stream ledger into a richer child-lane ledger that still preserves compatibility for existing bounded review lanes.
4. Add a parent-only Linear mutation guard in the packaged Linear helper layer so subordinate manifests cannot use mutation commands directly.
5. Extend parent proof hydration and read-side presenters to surface nested `child_lanes`.
6. Keep provider run discovery parent-centric by excluding subordinate child-lane pipeline manifests from peer-owner discovery.
