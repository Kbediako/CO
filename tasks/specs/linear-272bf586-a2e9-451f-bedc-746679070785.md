# Technical Specification

## Context

`CO-26` achieved a live terminal surface, but it stopped at "dataset plus redraw loop." The local Symphony reference is narrower and stricter than that: the operator-facing shell is a first-class ANSI dashboard with compact summary lines, a table-shaped running panel, and a queue-shaped backoff panel. CO already has the authoritative read model for the same underlying operational truth, so this follow-up lane should stay focused on terminal parity rather than reopening `/ui` or another backend data path.

## Requirements

1. When `control-host` runs in normal text mode on a TTY, render a styled terminal dashboard headed exactly `CO STATUS`.
2. Keep the frame structure close to Symphony's operator layout:
   - title + top border
   - compact summary/header lines
   - running section with table header + separator
   - backoff queue section
   - closing border
3. Add a rolling throughput line derived from live token totals so CO can present a Symphony-style `Throughput` summary.
4. Replace the current running prose rows with a width-aware table that preserves scanability across explicit width bands.
5. Sort retry rows by next due time ascending and render relative due times plus single-line sanitized error text.
6. Preserve read-only behavior and the current runtime refresh posture.
7. Preserve `control-host --format json` as machine-readable readiness output and preserve readable non-TTY text behavior.
8. Add focused regression coverage that verifies the full rendered frame (or equivalent full-frame sections) for representative wide/narrow datasets and startup gating.
9. Record any remaining justified deviations from Symphony explicitly in the spec/workpad before review handoff.

## Intended Parity Model

- Mirror from Symphony:
  - ANSI clear/home redraw behavior
  - bordered header with compact summary lines
  - running table with width budgeting
  - sorted one-line backoff queue rows
  - dashboard link / refresh context in the summary block
  - snapshot-style regression expectations
- Shipped CO-specific deviations:
  - title remains `CO STATUS` instead of `SYMPHONY STATUS`
  - no PID column because the current CO dataset does not expose a truthful equivalent field
  - the header uses `running/tracked` counts instead of a true `current/max` agent summary because no lightweight max-agent field is currently available in the dashboard dataset

## Current Truth

- `orchestrator/src/cli/control/operatorDashboardPresenter.ts` already exposes the read-only `OperatorDashboardDataset` needed for running rows, retry rows, totals, rate limits, polling health, and tracked Linear project metadata.
- `orchestrator/src/cli/control/controlStatusDashboard.ts` currently emits a plain-text report with summary prose, prose running rows, prose retry rows, and a verbose `ISSUES` block that is not part of Symphony's terminal frame.
- `orchestrator/src/cli/controlHostCliShell.ts` already owns the startup gate, which should remain the only place that decides whether the live dashboard attaches.
- Symphony's local snapshot fixtures make the parity bar concrete enough to test directly against section structure and frame shape rather than qualitative descriptions alone.
