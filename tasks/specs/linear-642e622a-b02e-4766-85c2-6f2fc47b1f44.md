## Parity / Alignment Matrix
- Current truth:
  - `ProviderOperatorAutopilotPendingActionRecord` has `kind: 'local_rollout'`, issue identity, summary, merge-closeout reason, shared-root status, and Linear transition status.
  - Pending actions are generated from merged provider-intake claims when post-merge rollout is enabled.
  - Existing cleanup drops stale pending actions when rollout is disabled or no live merged closeout remains.
  - There is no operator lifecycle state for acknowledged, cleared, or dismissed rollout actions.
- Reference truth:
  - Operator-owned follow-up queues need a durable distinction between pending, seen, completed, and intentionally skipped.
  - Audit records should explain operator decisions without mutating source-of-truth merge-closeout records.
  - A resolved action should not reappear unless a new rollout identity exists.
- Target truth / intended delta:
  - Add a stable rollout action identity derived from protected pending-action fields and any required merge-closeout identity.
  - Represent lifecycle states for pending, acknowledged, cleared, and dismissed actions.
  - Persist lifecycle decisions across control-host refreshes and restarts.
  - Filter cleared/dismissed actions out of unresolved `pending_actions` while keeping them in audit/read-model history.
  - Keep acknowledge visible as unresolved but acknowledged follow-up.
- Explicitly out-of-scope differences:
  - automatically running rollout commands
  - changing Linear state or labels
  - clearing provider-intake claims
  - changing merge-closeout completion conditions
  - adding unrelated dashboard redesign

## Technical Requirements
- Functional requirements:
  - Define a stable action identity for each `local_rollout` pending action. The identity must include issue identity and enough merge-closeout/action fields to distinguish a later rollout from an already cleared or dismissed one.
  - Add or pair lifecycle state with `ProviderOperatorAutopilotPendingActionRecord` while preserving the protected pending-action fields.
  - Support idempotent acknowledge for a matching unresolved `local_rollout` action.
  - Support clear for a matching action only when the operator indicates local rollout work is complete.
  - Support dismiss for a matching action only when the operator intentionally decides local rollout work is not applicable, with durable reason/provenance.
  - Persist lifecycle decisions so refresh cycles and control-host restarts do not lose them.
  - Project lifecycle state through operator-autopilot result/read-model surfaces without requiring live JSONL inspection.
  - Keep generated stale-action cleanup separate from explicit clear/dismiss lifecycle transitions.
- Non-functional requirements:
  - fail closed on ambiguous action identity.
  - avoid new Linear/GitHub request burn for lifecycle-only transitions.
  - keep audit append-only.
  - keep implementation narrow to operator-autopilot lifecycle and control/observability seams.
- Interfaces / contracts:
  - `ProviderOperatorAutopilotPendingActionRecord`
  - `ProviderOperatorAutopilotResult.pending_actions`
  - `provider-operator-autopilot.jsonl`
  - control/observability read-model payloads for operator-autopilot
  - any parent-selected CLI/API control path for acknowledge, clear, and dismiss
