# Technical Specification

## Context
Post-`CO-107`, the remaining `CO STATUS` live gap is no longer about runtime ticking, throughput, or current Linear complexity semantics. The remaining defect is that authoritative running-event truth is still too generic in some live turns, and the top-level STATUS surface does not explicitly say how fresh the underlying rate-limit or progress data is. Because the dashboard rerenders every second, operators can otherwise misread local redraw cadence as source freshness.

## Requirements
1. Running `display_event` must prefer richer authoritative worker, debug, or nearby child-stream or child-lane truth over generic summaries such as `Provider worker turn is active.` when that richer truth exists.
2. Generic worker-progress text may only remain when no richer authoritative message is available.
3. Operator-facing STATUS output must expose explicit freshness or staleness age for rate-limit or progress telemetry, or safely improve the underlying cadence.
4. Freshness surfacing must come from authoritative timestamps, not local rerender cadence.
5. Header `Runtime`, row `AGE / TURN`, and relative fallback event-age ticking must remain unchanged and regression-tested.
6. Throughput must remain the current 5-second rolling window unless a separate approved lane changes that contract.
7. Remaining-based Linear complexity truth must remain unchanged and regression-tested.
8. Final proof must explicitly record which original complaints were still valid on the `2026-04-08` baseline versus already-correct behavior.

## Design
- Use `compatibilityIssuePresenter.ts` as the authoritative `display_event` seam and tighten the ranking of available non-generic source text instead of adding more dashboard-local inference.
- Expose freshness through explicit projection data sourced from authoritative timestamps:
  - shared Linear budget freshness should come from `providerPollingHealth.ts` or the authoritative budget observation timestamp it carries
  - running progress freshness should come from the authoritative event or progress timestamp associated with the chosen running-event truth
- Keep `controlStatusDashboard.ts` focused on formatting those authoritative fields so operators can see both the content and the age of that content.

## Protected Expectations
- Preserve current 1-second live ticking for header `Runtime`, row `AGE / TURN`, and fallback event age.
- Preserve the current 5-second rolling throughput window.
- Preserve remaining-based Linear complexity display semantics.
- Keep `display_event` authoritative and specific when richer source truth exists.

## Reject These Wrong Interpretations
- “change throughput or complexity semantics because freshness is being discussed”
- “claim the live runtime ticking fix regressed without a new repro”
- “solve the lane only by changing string formatting in `controlStatusDashboard.ts`”
- “assume a generic worker-progress summary is authoritative enough even when richer proof/debug truth exists nearby”

## Current Truth
- `compatibilityIssuePresenter.ts` prefers budget-exhaustion text and some high-signal progress summaries, but still lets live running rows go generic when the authoritative source path is thin.
- `providerPollingHealth.ts` already projects cooldown/checking/scheduled truth and next-refresh countdowns, but not explicit operator-facing freshness age.
- `controlStatusDashboard.ts` shows live cadence truth every second without explicitly distinguishing local redraw cadence from source freshness.

## Proposed Design
- Add the smallest explicit freshness data needed for operator truth:
  - shared polling or Linear-budget freshness timestamp from `providerPollingHealth.ts`
  - authoritative running-event freshness timestamp alongside the chosen running-event text
- Tighten `display_event` selection so current-turn non-generic worker or debug truth outranks generic filler.
- Render the freshness contract in `controlStatusDashboard.ts` in a bounded way that does not change the already-correct runtime, throughput, or complexity surfaces.

## Non-Goals
- Reopening `CO-107` runtime ticking work.
- Reworking the throughput window or adding smoothing.
- Reopening `CO-108` next-refresh wording beyond what is needed for explicit freshness.
- Reopening Linear complexity semantics without new evidence.
- Redesigning the broader dashboard layout.

## Parity / Alignment Matrix
- Current truth:
  - live worker rows can still show generic authoritative progress summaries
  - top-level operator freshness is still implied by rerender cadence instead of explicit source age
  - current runtime ticking, throughput, and Linear complexity semantics are already correct
- Reference truth:
  - Symphony uses authoritative event or message text and explicit timing semantics rather than renderer-local inference
- Target truth / intended delta:
  - operator-facing running rows stay specific when source truth exists
  - freshness age is explicit for rate-limit or progress telemetry
  - correct runtime, throughput, and complexity semantics remain unchanged
- Explicitly out-of-scope differences:
  - throughput redesign
  - runtime-ticking redesign
  - broad UI redesign

## Not Done If
- Generic worker-progress text still wins over richer authoritative truth.
- Operators still cannot tell whether telemetry is actually fresh.
- Runtime, throughput, or complexity semantics drift.
- The proof packet cannot show which complaints were already correct on the audited baseline.
