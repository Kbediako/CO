# Technical Specification

## Summary
- Objective: stop passive retained released provider-intake claims from repeatedly churning provider refresh while preserving retained audit evidence and current actionable revalidation paths.
- Scope: docs-first packet in this child lane; parent-owned source diagnosis and implementation in provider-intake claim refresh/projection seams; focused regressions for passive released idempotence and actionable exceptions.
- Constraints: no source or test edits in this child lane, no Linear/GitHub/workpad mutation, no manual `provider-intake-state.json` cleanup, no broad provider scheduler redesign, and no unbounded direct issue-by-id reads.

## Issue-Shaping Contract
- User-request translation carried forward: CO-549 is about retained released provider-intake claim refresh churn. Passive retained released/non-active rows should stay visible as audit history, but repeated refreshes should not keep rechecking, rewriting, or projecting them as live work unless newer same-issue truth or current actionable evidence exists.
- Protected terms / exact artifact and surface names: `provider-intake-state.json`, `state=released`, `released/not_active`, `provider_issue_released:not_active`, `refresh:claim_reconcile`, `refresh:claim_issue_by_id_reconcile`, `claim_reconcile:released`, `claim_issue_by_id:released`, `passive_release`, `issue_state`, `issue_state_type`, `issue_updated_at`, `run_id`, `run_manifest_path`, `retry`, `review_promotion`, `merge_closeout`, `co-status`, `/ui/data.json`, `selected-run`, `freshness-gauge`, `providerIssueHandoff.ts`, `ProviderIntakeState`.
- Nearby wrong interpretations to reject: retained-row deletion, blanket released-claim suppression, active/retry/Ready reclassification by default, unbounded issue-by-id polling, PR/Linear lifecycle policy changes, provider-worker launch changes, docs-freshness owner work, and manual control-host state edits.
- Explicit non-goals carried forward: no implementation/test changes by this docs child lane, no Linear mutation, no GitHub mutation, no workpad/PR lifecycle action, no provider scheduler redesign, and no broad polling increase.

## Technical Requirements
- Functional requirements:
  - identify passive retained rows with `state=released` / released non-active reason shapes such as `provider_issue_released:not_active`
  - make passive released refresh idempotent when observed truth is unchanged and no current actionable evidence exists
  - preserve claim state, reason, `run_id`, `run_manifest_path`, retry fields, `review_promotion`, `merge_closeout`, claim history, and source-labeled terminal evidence
  - re-enter refresh only for newer same-issue truth or explicit current actionable evidence, including active/reopened issue truth, active retry truth, current review promotion, current merge closeout, cancelable retained-run evidence, or bounded stale-proof revalidation
  - keep passive released rows out of active WIP, retry occupancy, and current closeout alarm projection
  - avoid unbounded direct issue-by-id sweeps over retained released residue
- Non-functional requirements:
  - localize parent implementation to `providerIssueHandoff.ts` or existing provider-intake refresh/projection helpers
  - fail closed when the evidence is missing, older, stale, or not same-issue
  - keep request-budget behavior bounded and explainable
  - avoid adding a parallel provider-intake authority surface
- Interfaces / contracts:
  - `provider-intake-state.json`
  - `providerIssueHandoff.ts`
