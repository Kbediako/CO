# Technical Specification

## Context
The packaged supervision feature already exists. The remaining defect is that real runtime ownership on this host still lives outside that feature boundary. Today, the live LaunchAgent still points to the old host-local shim and the active root-host process lineage still descends from that shim. That means the shipped `CO-117` surface is present in source, but bypassed in live operation. `CO-141` is therefore an adoption and rollout-truth lane: operators need one bounded way to move an existing shim baseline onto the managed supervision path, and `status` must report whether that rollout actually happened.

## Requirements
1. The supported operator-facing CLI artifact for this lane must expose `control-host supervise install|status|restart|uninstall|run`.
2. `control-host supervise status` must classify whether the current machine is on `legacy_shim`, `managed_supervision`, `not_installed`, or another explicit fail-closed drift state when rollout truth is mixed.
3. The canonical rollout or install path must migrate an existing shim-backed LaunchAgent to the managed supervision artifacts without requiring a user to hand-edit `~/Library/LaunchAgents`.
4. After rollout or restart, the active root `control-host` must launch via the managed supervision path rather than the shim.
5. The rollout must preserve the existing health-poll and unhealthy-restart behavior from `CO-117`.
6. Operator docs must explain the upgrade path from the real legacy shim baseline.
7. The lane must remain separate from queue-management, review-state, merge-policy, and unrelated host-package cleanup work.

## Design
- Extend rollout-state inspection to compare the live LaunchAgent payload and/or launchctl truth against both:
  - the managed `CO-117` supervision contract
  - the known legacy shim baseline (`co-control-host-supervisor.sh`)
- Teach `status` to report rollout mode and the active launch path, not only whether the managed config file exists.
- Treat the legacy shim-backed LaunchAgent as a first-class migration baseline:
  - detect it explicitly
  - replace it with the managed LaunchAgent and support-dir artifacts in one bounded command flow
  - keep restart and health semantics aligned with the existing supervision runner
- Ensure the blessed operator entrypoint for supported environments is truthful:
  - source checkouts should use the current repo-owned surface
  - packaged installs should use the shipped JS surface
  - rollout should not silently depend on a stale ambient global CLI that lacks the shipped command family

## Implementation Surface
- Expected codepaths:
  - `bin/codex-orchestrator.ts`
  - `orchestrator/src/cli/controlHostSupervisionCliShell.ts`
  - `orchestrator/src/cli/control/controlHostSupervision.ts`
  - any shared bootstrap or entrypoint helpers required to make the supported operator path truthful
- Expected docs:
  - `docs/public/provider-onboarding.md`
  - the CO-141 docs packet and workpad
- Expected tests:
  - rollout classification and status formatting
  - legacy-shim migration or install behavior
  - LaunchAgent rendering or migration helpers
  - top-level CLI help or parsing if the supported entrypoint contract changes

## Protected Expectations
- Preserve the current plain `control-host` behavior when supervision is not installed.
- Preserve the current health semantic: repeated unhealthy or `restart_required` observations still lead to a bounded launchd restart.
- Keep queue-management and merge-policy behavior out of the rollout lane.
- Fail closed when rollout truth is ambiguous instead of reporting a false managed state.

## Reject These Wrong Interpretations
- "rewrite the CO-117 supervision feature from scratch"
- "just tell operators to edit the plist by hand"
- "leave the live host on the shim and only update docs"
- "global npm on PATH is the only rollout target worth supporting"
- "expand this into queue-management, merge automation, or broad package-manager cleanup"

## Current Truth
- `CO-117` merged the shipped supervision surface into source and public onboarding docs.
- The live LaunchAgent on this host still points at `/Users/kbediako/.local/bin/co-control-host-supervisor.sh`.
- The active process lineage is still shim -> `node ... codex-orchestrator.js control-host ...`.
- The current operator-facing `codex-orchestrator` on `PATH` on this host does not recognize `control-host supervise status --format json`.

## Proposed Design
- Status:
  - inspect the managed config/state paths already defined by `CO-117`
  - inspect the live LaunchAgent payload for the current label
  - classify rollout mode and include the active launch path in output
- Install or migration:
  - keep `install` idempotent for already-managed hosts
  - detect a legacy shim baseline and migrate it without manual shell or plist edits
  - render or refresh any managed launcher/artifact path needed for truthful runtime ownership
- Docs:
  - teach operators how to inspect legacy status, run the rollout path, and verify after migration

## Non-Goals
- Queue-management automation
- Merge or review orchestration
- Non-macOS service managers
- Broad host-wide cleanup outside the named rollout seam

## Parity / Alignment Matrix
- Current truth:
  - shipped supervision exists in source, but live host ownership still sits with the shim
- Reference truth:
  - the shipped supervision surface should also be the real operator/runtime surface
- Target truth / intended delta:
  - the host can migrate from legacy shim to managed supervision through a bounded CLI workflow and status reports that truth accurately
- Explicitly out-of-scope differences:
  - redesigning the runner semantics from `CO-117`
  - queue policy
  - merge policy

## Not Done If
- The final path still leaves launchd pointing at `co-control-host-supervisor.sh`.
- Operators still need manual plist editing to migrate.
- `status` still cannot distinguish legacy from managed rollout.
- The active root host still launches beneath the shim after the migration path runs.
