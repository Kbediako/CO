# Technical Specification

## Context

`CO-63` exposed a narrower follow-up than pickup or refresh-stall recovery. The provider worker and live Linear already moved the issue into `In Progress`, but the local persisted claim snapshot stayed on `Ready`. That leaves operator truth stale underneath CO STATUS and any other read model backed by `provider-intake-state.json`.

## Requirements

1. Refresh persisted tracked-issue metadata for active eligible claims while the run remains in progress, including `issue_state`, `issue_state_type`, and `issue_updated_at`.
2. Keep the claim lifecycle state truthful and unchanged where appropriate:
   - active run still in progress -> claim remains `running`
   - terminal manifest or proof authority paths keep their current classification behavior
   - release or non-eligible behavior stays unchanged
3. Ensure a `Ready` to `In Progress` transition does not remain cached locally as `Ready` after control-host refresh or rehydration.
4. Inspect all active-run persistence paths that can write `provider_issue_rehydrated_active_run` so the fix does not leave a second stale path behind.
5. Add focused regressions proving active-run metadata refresh works while lifecycle ownership stays intact.
6. Preserve operator-facing truth by fixing the persisted claim source rather than adding read-side-only compensation.

## Current Truth

- `providerIntakeState.ts` already stores `issue_state`, `issue_state_type`, and `issue_updated_at`; the schema is not the limiting seam.
- `providerIssueHandoff.ts` already merges fresher tracked-issue metadata in several terminal or retry-related paths, so this lane is about aligning the active-run path with that existing truth bar.
- The active-run branch in `rehydrateNow()` currently re-persists `state: running` and reason `provider_issue_rehydrated_active_run` with the discovered run identity, but it does not explicitly merge newer tracked-issue metadata from live Linear before persisting.
- Existing tests in `orchestrator/tests/ProviderIssueHandoff.test.ts` already encode the stale behavior for active claims: when `resolveTrackedIssue` reports newer `In Progress` metadata, the claim can still stay on `Ready` while remaining `running`.

  - active running claim + newer live tracked issue metadata -> refreshed `issue_state`, `issue_state_type`, and `issue_updated_at` while keeping `state: running`
  - nearby active-run cases that preserve manifest-authority or proof-authority behavior when live data is unavailable or older

- Run the required validation floor, then record standalone review and elegance review before any review handoff.
