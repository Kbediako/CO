# TECH_SPEC - CO-380 authoritative review gate

## Summary
- Objective: Replace routine review-wrapper fallbacks with one authoritative gate for implementation handoff review.
- Scope: Review prompt input validation, non-interactive wrapper behavior, pipeline gate env posture, docs, and focused tests.
- Constraints: Preserve prompt-only advisory behavior for `docs-relevance-advisory`; do not alter PR review feedback handling or merge shepherding.

## Issue-Shaping Contract
- User-request translation carried forward: The worker must inventory review wrapper entry points, define one repo-owned gate, demote fallback branches to break-glass only, and prove fallback activation is exceptional.
- Protected terms / exact artifact and surface names: `codex-orchestrator review`, `npm run review`, `review/telemetry.json`, `docs-review`, `implementation-gate`, `provider-linear-worker`, `CODEX_REVIEW_NON_INTERACTIVE`, `FORCE_CODEX_REVIEW`, `NOTES`.
- Nearby wrong interpretations to reject: Generated `NOTES` as routine success; handoff-only prompt output as review success; automatic manual fallback after a failed authoritative review; changing advisory docs relevance semantics.
- Explicit non-goals carried forward: No broad review prompt redesign, no review model change, no GitHub PR review-thread workflow change.

## Parity / Alignment Matrix
- Current truth: Review helper code can generate fallback `NOTES`, non-interactive mode can return handoff-only output when not forced, and authoritative gate semantics are spread across wrapper flags and pipeline env.
- Reference truth: Implementation handoff review should be a forced wrapper-led review with a classified terminal telemetry outcome.
- Target truth / intended delta: Introduce an explicit authoritative gate contract that requires authored `NOTES` or break-glass waiver metadata, disallows handoff-only success, and classifies telemetry outcomes deterministically.
- Explicitly out-of-scope differences: `docs-relevance-advisory` stays advisory and may remain prompt-only; scoped review title fallback remains only a bounded transport helper and not a success verdict.

## Readiness Gate
- Not done if: Primary review still succeeds through generated fallback notes, handoff-only output, or silent manual review after a failed authoritative review.
- Pre-implementation issue-quality review evidence: Parent reviewed CO-380 acceptance criteria and source-0 anchor `ctx:sha256:a2d50baa89917b57a325011e1552d5a4879ede05cf4c8466094c5758087bf39e#chunk:c000001`; same-issue child lane `notes-contract-tests` added failing contract tests before implementation.
- Safeguard ownership split: Parent owns implementation/docs; accepted child lane owns focused tests in `tests/review-prompt-context.spec.ts`.

## Technical Requirements
- Functional requirements:
  - Add an authoritative review gate signal for review wrapper invocations used by provider-worker, `docs-review`, and `implementation-gate`.
  - Require explicit authored `NOTES` for authoritative review unless break-glass waiver fields are present.
  - Break-glass waiver fields must include owner/approver, reason, expiry, and evidence, and must be visible in review context.
  - Authoritative non-interactive review must execute when forced; prompt-only handoff output is not a successful gate result.
  - Keep telemetry outcome handling explicit: `clean-success` and `bounded-success` are success; `failed-boundary` and `failed-other` are failed gate verdicts.
- Non-functional requirements (performance, reliability, security): No network or dependency changes; wrapper failures should fail closed with clear messages.
- Interfaces / contracts:
  - Environment: `CODEX_REVIEW_AUTHORITATIVE_GATE=1`.
  - Break-glass environment: `CODEX_REVIEW_BREAK_GLASS_NOTES_FALLBACK=1`, `CODEX_REVIEW_BREAK_GLASS_OWNER`, `CODEX_REVIEW_BREAK_GLASS_EXPIRES_AT`, `CODEX_REVIEW_BREAK_GLASS_REASON`, `CODEX_REVIEW_BREAK_GLASS_EVIDENCE`.
  - Review prompt context: resolved notes must either be authored notes or waiver-tagged break-glass notes.

## Architecture & Data
- Architecture / design adjustments: Centralize authoritative input validation in review prompt context and enforce gate-level non-interactive behavior in the run-review wrapper before handoff output can be considered success.
- Data model changes / migrations: None.
- External dependencies / integrations: None.

## Validation Plan
- Tests / checks: Focused review wrapper tests, build, lint, full tests, docs checks, docs freshness, repo stewardship, diff budget, forced standalone review, and pack smoke if downstream-facing wrapper/package paths move.
- Rollout verification: Provider-worker workpad records child-lane manifest, review telemetry verdict, and validation commands.
- Monitoring / alerts: Existing `review/telemetry.json` remains the review outcome artifact.
