## Architecture & Data
- Current behavior:
  - `create-follow-up` can derive source labels from live source issue labels and apply them via `labelIds` or `addedLabelIds`.
  - The helper can still be tempted to rely on mutation response labels or local projection for output and success decisions.
- Target behavior:
  1. Resolve expected source-derived label ids using the existing CO-482 policy.
  2. Create or reuse the follow-up issue through existing guarded paths.
  3. Read the follow-up issue live after the mutation using an authority equivalent to `live linear issue-context`.
  4. Compare expected source-derived labels to observed live labels by id.
  5. If labels are missing, perform one bounded `addedLabelIds` repair with the missing expected ids.
  6. Reread the follow-up issue live.
  7. Return clean success only when the terminal live read includes all expected source-derived labels.
  8. On persistent missing labels, emit partial failure with issue id or identifier, expected labels, observed labels, missing labels, and repair attempt metadata.
  9. JSON and human terminal output report terminal live labels, not mutation-return labels as authority.
- Data model changes:
  - No persistent schema migration is expected.
  - Facade result types may need explicit `live_labels`, `label_verification`, or equivalent fields.
  - Failure details should separate issue creation/reuse success from label verification failure.
- External dependencies / integrations:
  - Linear live issue read through the existing provider Linear client/facade path.

## Technical Requirements
- Functional requirements:
  1. Preserve CO-482 source-derived label resolution from live source issue labels.
  2. Perform post-create live verification after a created follow-up issue.
  3. Perform post-reuse live verification after canonical owner reuse.
  4. Use a bounded `addedLabelIds` repair for missing expected labels.
  5. Reread live target issue labels after repair before reporting success.
  6. Fail closed when terminal live labels still miss expected source-derived labels.
  7. Include follow-up issue id or identifier, expected labels, observed labels, and missing labels in partial failure output.
  8. Show terminal live labels in JSON output and human terminal output.
  9. Preserve existing follow-up issue-shaping guards, canonical owner reuse, related issue behavior, and CO-482 label policy.
- Non-functional requirements:
  - Bounded retries; no unbounded polling or manual sweep.
  - Deterministic tests with mocked propagation and persistent-missing responses.
  - No new label catalog or projection cache.
  - Fail-closed behavior for missing, malformed, paginated, stale, or incomplete live label evidence.
