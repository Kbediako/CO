# Technical Specification

## Context
The repo already persists reusable experiences and injects them through prompt packs, but the selection step is under-specified and under-instrumented. Prompt packs declare slot count only, the store ranks by a raw reward sum, and the manifest only preserves final snippet strings. CO-95 is therefore a bounded retrieval-policy issue: define the policy on the controller-owned prompt-pack surface, run a competitive scorer at retrieval time, and persist enough diagnostics to explain why repeated sources were penalized or excluded.

## Requirements
1. Retrieval-stage scoring must be explicit and implemented as a competitive scorer rather than raw top-K reward ordering.
2. Anti-dominance normalization must use existing experience provenance surfaces and be inspectable.
3. The retrieval policy must be controller-consumable through prompt-pack configuration instead of store-local hard-coded heuristics.
4. Run manifests must expose the policy and bounded selection metadata while preserving existing snippet compatibility.
5. Focused tests must cover repeated-source suppression and ranking changes.
6. The slice must not broaden into TF-GRPO reward persistence, telemetry, runtime continuity, or a new memory backend.

## Design
- Controller policy:
  - extend prompt-pack manifests with an additive retrieval-policy block
  - include score weights, min-score threshold, provenance grouping strategy, and anti-dominance strength
- Retrieval engine:
  - add a structured selection method to `ExperienceStore` that returns both selected records and diagnostics
  - compute raw score from weighted reward terms, then recompute competitive score per slot with a dominance penalty derived from previously selected records sharing the same provenance source
- Inspectability:
  - derive a source key from existing provenance in priority order and record the chosen source kind/key in diagnostics
  - persist a per-pack retrieval diagnostics artifact plus manifest summary fields such as selected IDs, suppressed source keys, and diagnostics path
- Compatibility:
  - keep `prompt_packs[].experiences` as bounded strings
  - optionally enrich string snippets with the final score and source key while keeping the deeper explanation in diagnostics

- Keep literal `competitive scoring` and `anti-dominance normalization` wording tied to retrieval-stage memory selection.
- Keep the policy inspectable and controller-consumable.
- Reuse existing experience provenance rather than inventing a separate source system.
- Preserve existing string snippet consumers.

- reuse TF-GRPO rewarders unchanged and call that retrieval policy.
- raw top-K reward or recency already satisfies the issue.
- inspectability can stay in comments or only in tests.
- this lane should absorb `CO-82`, `CO-83`, `CO-89`, or `CO-90`.

## Current Truth
- Retrieval is still raw reward top-K with a floor.
- Controller config only exposes slot count.
- Manifests expose final snippets only.
- Existing provenance is already present on stored records.

## Proposed Design
- Add an additive prompt-pack retrieval-policy contract.
- Add Experience Store competitive selection and diagnostics.
- Persist manifest-visible retrieval policy and selection summary per pack.
- Keep string snippet injection stable for prompt consumers.

## Non-Goals
- Model changes.
- TF-GRPO reward persistence or rewarder redesign.
- Telemetry/status projection work.
- Runtime continuity or worker-host parity work.
- New memory backends.
- Reopening `0303`.

  - prompt packs declare `experienceSlots` only
  - store ranks by raw reward sum
  - manifests only show string snippets
- Reference truth:
  - controller-owned prompt-pack manifests already define experience injection shape
  - stored records already expose provenance surfaces that can drive grouping
- Target truth / intended delta:
  - controller-declared retrieval policy
  - competitive scoring with anti-dominance normalization
  - manifest-visible diagnostics for ranking and suppression
- Explicitly out-of-scope differences:
  - rewarder semantics
  - runtime/telemetry work
  - new memory substrates

## Not Done If
- selection is still plain top-K reward or recency.

- repeated-source suppression and ranking changes are untested.
- the issue broadens into unrelated telemetry or runtime continuity work.
