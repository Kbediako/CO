# Technical Specification

## Context
`CO-91` added `memory.source_0`, but that contract stops at the descriptor. The shared manifest can tell downstream readers where a bounded memory object lives, yet it cannot tell them whether an inherited memory candidate was selected or rejected, whether the run had to rediscover memory after a miss, whether any source-artifact contradiction was detected, or whether later manual resume/repair activity changed the observable memory outcome. `run:summary` and `metrics.json` currently omit those signals entirely. This issue therefore needs one additive memory observability contract that projects the existing raw seams into shared run artifacts without widening into provider-worker status or transcript storage.

## Requirements
1. The shared manifest memory contract must expose an additive bounded observability block in addition to `memory.source_0`.
2. The memory observability block must make selected-memory provenance inspectable from source-artifact fields such as pointer, object id, artifact paths, and origin/inheritance lineage.
3. Rejected and rediscovered memory decisions must be inspectable without transcript dumps.
4. Manual repair and resume latency must be derived from existing persisted artifacts, not inferred from chat text.
5. Counters must exist for contradiction count, rediscovery count, resume latency, manual repair, repeated-failure streak, retrieval hits/misses, and selected-memory provenance.
6. At least one emitted event surface must carry the memory observability payload.
7. `metrics.json` must mirror the bounded counters and selected-memory provenance summary needed for evaluation/debugging.
8. Older manifests without the new memory observability block must remain valid and readable.
9. Tests must cover both emission and absence behavior.

## Design
- Contract:
  - extend `schemas/manifest.json` and generated shared manifest types with `memory.observability`
  - keep `memory.source_0` unchanged as the selected source-artifact descriptor
- Selected-memory provenance:
  - expose the currently selected source-0 pointer, object id, artifact paths, origin, and inheritance lineage
  - explicitly label whether the selection came from inherited reuse or a fresh child-local rebuild
- Rejected-memory decisions:
  - record bounded rejected-candidate entries only when an inherited source-0 candidate exists but is rejected
  - keep each rejection tied to source artifacts and a bounded reason code, not transcript text
- Rediscovery:
  - record rediscovery when a child run rejects an inherited candidate and rebuilds a fresh local source-0 object
  - increment rediscovery count and the repeated-failure streak on that bounded path
- Contradictions:
  - count explicit source-artifact provenance contradictions only, such as invalid source-0 payload lineage or descriptor/payload mismatch
  - keep plain missing-artifact misses separate from contradictions
- Manual repair and resume latency:
  - derive manual repair only from explicitly marked memory-repair resume records already persisted in `resume_events`
  - derive resume latency from the latest accepted manual resume timestamp to the current memory observability snapshot timestamp
- Event and metrics projection:
  - mirror the manifest memory observability payload into `run:summary`
  - mirror counters plus selected-memory provenance summary into `metrics.json`

## Protected Expectations
- Keep the lane memory-specific and artifact-backed.
- Preserve literal `memory provenance`, `outcome schema`, and `selected-memory provenance` wording in the docs and contract comments.
- Keep the contract additive and backward-compatible.
- Avoid transcript dumps or general provider-worker telemetry expansion.

## Reject These Wrong Interpretations
- `CO-82` already owns this slice because it added provider-worker progress telemetry.
- The right fix is to emit more raw transcript output.
- `memory.source_0` already makes rejected or rediscovered decisions inspectable.
- Manual repair must wait for a new transcript parser instead of using persisted `resume_events`.
- The lane should absorb `CO STATUS`, resident continuity, or distributed parity work.

## Current Truth
- `memory.source_0` exists and is pointer-based, traceable, and inheritable across child manifests.
- Source-0 inheritance already has the bounded raw seam needed to distinguish inherited reuse from fresh rebuilds after a miss.
- No shared manifest/event/metrics payload currently summarizes those decisions.
- `resume_events` already persist accepted/blocked resume metadata, including `manual-resume`, but memory artifacts do not currently project explicitly marked memory-repair truth.

## Proposed Design
- Add `memory.observability` to the manifest.
- Store bounded decision objects for:
  - selected memory
  - rejected inherited candidates
  - rediscovered memory
  - manual repairs derived from explicitly marked `resume_events`
- Store counters for:
  - `contradiction_count`
  - `rediscovery_count`
  - `resume_latency_ms`
  - `manual_repair_count`
  - `repeated_failure_streak`
  - `retrieval_hits`
  - `retrieval_misses`
- Mirror the same observability block into `run:summary`.
- Mirror the counters plus selected-memory provenance summary into `metrics.json`.

## Non-Goals
- Full eval-lane implementation.
- Broad provider-worker progress or stall telemetry.
- `CO STATUS` semantic changes.
- Resident app-server continuity.
- Distributed worker-host parity.
- Transcript dump surfaces.
- Reopening `0303`.

## Parity / Alignment Matrix
- Current truth:
  - manifest exposes only `memory.source_0`
  - no shared event or metrics payload for memory outcomes
  - source-artifact hit/miss/rediscovery truth exists only implicitly in the source-0 inheritance path
- Reference truth:
  - `source_0` context objects already provide stable object ids, artifact paths, and lineage fields
  - `resume_events` already provide bounded persisted manual-resume truth when the event is explicitly marked as a memory repair
- Target truth / intended delta:
  - one additive manifest memory observability block
  - the same bounded memory observability payload in `run:summary`
  - counters and selected-memory provenance summary in `metrics.json`
- Explicitly out-of-scope differences:
  - general provider observability
  - review-wrapper telemetry
  - full task-memory controller behavior

## Not Done If
- manifests/events/metrics still cannot expose the requested counters and selected-memory provenance.
- rejected or rediscovered decisions still require transcript inspection.
- contradictions are counted from guesswork instead of explicit artifact mismatches.
- the issue widens into adjacent observability or continuity work.
