## Technical Requirements
- Functional requirements:
  - track file-level Vitest lifecycle updates for collected test files
  - detect file tasks that remain in `run` state beyond a short bounded delay
  - emit a human-readable progress line naming active file path(s) and elapsed time before the file reaches a terminal state
  - stop reporting files once they pass or fail
  - wire the reporter into the `vitest.config.core.ts` path used by `CI=1 npm run test` and standard truthy CI full-suite runs
- Non-functional requirements (performance, reliability, security):
  - keep output bounded to long-running file activity instead of per-test chatter
  - avoid leaking timers or state between reruns
  - preserve compatibility with existing Vitest runs and review wrappers
- Interfaces / contracts:
  - config surface: `vitest.config.core.ts`
  - reporter helper: `scripts/lib/vitest-progress-reporter.ts`
  - focused regression: `tests/vitest-progress-reporter.spec.ts`

## Architecture & Data
- Architecture / design adjustments:
  - add a reporter helper that implements the Vitest `Reporter` hooks already available on the current tree
  - build a file-id -> filepath map during `onCollected`
  - track active file start times from file-level `run` updates in `onTaskUpdate`
  - emit progress lines on a bounded cadence only after the configured announce delay has elapsed
  - clear timer state when files reach terminal states or the run ends
- Data model changes / migrations:
  - no production data-model changes
  - only in-memory reporter bookkeeping for active file ids, filepaths, start times, and next log deadlines
- External dependencies / integrations:
  - Vitest reporter API from the current repo dependency set
  - existing repo test command surface through `npm run test`
