# Technical Specification

## Context

`CO-17` is not an implementation-ready runtime change. Current CO already has a clear global reasoning baseline, but it does not yet have a truthful issue-level or run-level contract for reasoning selection:

- `codexDefaultsSetup.ts` writes the CO-wide baseline `model_reasoning_effort = "xhigh"`.
- `AGENTS.md` keeps docs-first and top-level CO work at `xhigh` unless a role-specific exception is explicitly documented.
- `providerLinearWorkerRunner.ts` resolves runtime mode and launches `codex exec` / `codex exec resume`, but does not choose reasoning per issue.
- `providerLinearWorkerRunner.ts` and its tests show token telemetry can be parsed into proof artifacts, but only opportunistically from runtime/session output.
- `linearDispatchSource.ts` does not model issue estimate, issue risk, or reasoning metadata.
- Live Linear team settings still report `issueEstimationType = "notUsed"`.

## Requirements

1. State clearly whether dynamic provider-worker reasoning selection is worth doing at all for CO.
2. Preserve the current docs-first authoring baseline at highest reasoning (`xhigh`).
3. Reject estimate-driven automatic routing today because the live team still has estimation disabled and the tracked-issue contract does not model estimate/risk.
4. If the idea remains worth revisiting later, define a staged experiment:
   - close truthful token-telemetry gaps first
   - add explicit issue/run reasoning metadata second
   - pilot a manifest-backed human-auditable override third
5. Keep dispatch ordering unchanged in this lane and avoid runtime implementation changes.

## Documented Contract

- Current recommendation:
  - do not implement dynamic provider-worker reasoning selection now
  - keep docs-first authoring on the current highest reasoning baseline
  - treat any future implementation-worker reasoning override as an explicit, auditable experiment rather than a default routing rule
- Future experiment gate:
  - only start after provider-worker token telemetry is truthful enough to compare cost and value by run or phase
  - only start after issue or run metadata explicitly carries reasoning intent
  - only use Linear estimate as an optional secondary signal if the live team actually enables estimation and CO adds truthful contract support for it
- Override carrier:
  - use per-run manifest-backed env/config overrides such as `CODEX_CONFIG_OVERRIDES`
  - do not mutate global config defaults to serve one issue
- Explicit no-go conditions for now:
  - no scheduler-driven reasoning selection from current token telemetry
  - no estimate-driven routing while `issueEstimationType = "notUsed"`
  - no dispatch-order changes in `linearDispatchSource.ts`
