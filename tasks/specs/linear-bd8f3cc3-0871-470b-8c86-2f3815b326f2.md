# Technical Specification

## Context

`CO-74` intentionally put `co-status attach` on `primary scrollback`, while `CO-67` fixed a different paused-snapshot duplication defect for the launched monitor. The current defect is neither of those older decisions alone: attach already runs live on primary, but it still rewrites the entire frame each refresh, which accumulates full-screen history in scrollback. Operators therefore get both the worst of live redraw and the worst of scrollback duplication.

The same live attach surface still carries additional readability and truth issues that are explicitly part of this ticket: large countdowns stay in raw seconds or minute-only form, Codex and Linear rate-limit text is hard to scan, retry rows are vague, and event summaries still collapse to generic worker text. This lane must fix the live attach surface as one coherent operator-facing experience without silently widening back into the broader telemetry restoration work.

## Requirements

1. `co-status attach` must remain a read-only live viewer for an already-running control-host.
2. Live attach viewing must preserve `primary scrollback` without accumulated duplicate full-frame `CO STATUS` history.
3. The fix must not simply revert attach live mode to `alternate screen`.
4. Live attach rendering must use a bounded in-place or pinned-region update strategy in primary scrollback rather than appending a full new frame on every refresh.
5. Launch-mode `co-status` alternate-screen semantics must remain truthful and unregressed.
6. `Next refresh`, runtime/age values, retry countdowns, and rate-limit reset windows must render in higher-order readable units when values are large.
7. Codex rate-limit rendering must use the requested percentage semantics and spacing, and Linear rate-limit rendering must use clearer request/complexity spacing.
8. Event and retry text must be more meaningful than the current generic fallbacks, and stale `Merging`/`Done` truth must not survive if it still reproduces in the attach view.
9. Focused tests plus real-device screenshot proof must cover the bounded live attach and operator-readability contract.

## Current Truth

- `startAttachedControlStatusDashboard(...)` already sets `liveSurfaceMode: 'primary'`.
- `writeFrame(...)` still handles all primary live redraws through `ANSI_CLEAR_HOME${frame}`, which means attach refreshes append full-frame history into primary scrollback.
- `renderNextRefreshLine(...)` and compact status still render large poll windows as raw seconds.
- `formatRuntimeSeconds(...)`, `formatRuntimeAndTurns(...)`, and `formatRelativeDue(...)` still use short minute/second or second.millisecond-only formats that become hard to scan for long windows.
- `formatRateLimitBucket(...)` and `formatCompactRateLimitBucket(...)` still treat Codex percent values as used-percent output, while compact Linear budget segments still omit the requested readable spacing.
- `summarizeRunningEvent(...)` and `renderRetryRow(...)` still prefer generic last-message or terse `attempt=... in ...` output.
- `selectedRunProjection.ts` and `compatibilityIssuePresenter.ts` already try to suppress some stale terminal-state projections, so not every truth concern necessarily requires more code there.

## Proposed Design

### 1. Pinned live attach region on primary scrollback

- Keep live launched mode on alternate screen.
- Keep live attached mode on primary scrollback.
- For live primary attach redraws, stop writing `ANSI_CLEAR_HOME` full-frame replacements.
- Instead, track the previous rendered live-frame height and rewrite the current live block in place using cursor-relative repositioning plus clear-from-cursor-down semantics.
- Preserve paused primary snapshot behavior and shell-prompt cleanliness when attach leaves live mode or exits.

### 2. Higher-order live operator duration formatting

- Introduce one dashboard-local duration formatter that can render:
  - short windows as seconds where appropriate
  - medium windows as `Xm Ys`
  - longer windows as `Xh Ym`
  - multi-day windows as `Xd Xh`
- Reuse that formatter for:
  - `Runtime`
  - `AGE / TURN`
  - `Next refresh`
  - retry/backoff countdowns
  - rate-limit reset windows

### 3. Clearer rate-limit formatting

- Keep existing rate-limit source selection intact unless validation proves a source-truth defect.
- Change only the dashboard presentation:
  - Codex percent windows should render as percent remaining with the requested spacing, including `5-hour` and `weekly`
  - Linear compact and full views should add the requested spacing around request/complexity buckets
  - long reset windows should use the shared higher-order duration formatter

### 4. More meaningful event and retry summaries

- Tighten running-event summarization so clearly generic fallback text does not outrank more useful current activity when better event/action context already exists.
- Improve retry-row wording so the surface explains what is being retried or why it is waiting, rather than only showing `attempt=1 in 0.000s`.
- Only widen into presenter/read-model logic if validation proves the stale `Merging` / retry projection still survives the existing freshness guards.

### 5. Proof and truth checks

- Capture device-local screenshot proof directly in Linear after visually inspecting the images against the terminal.
- During proof, cross-check token totals and Codex rate-limit values against the authoritative root-host payload; record mismatches explicitly instead of assuming rendered numbers are correct because they are non-empty.
- Ensure proof capture cleans up any temporary sessions or windows it opens.
