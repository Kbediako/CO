# Technical Specification

## Context

`CO-67` already fixed the paused snapshot race and broader authoritative telemetry semantics, but the current tree still overstates live activity because historical non-provider manifests with raw `in_progress` status can survive the current-running filter. The local control-host intake snapshot on this device shows no active current claims, yet the live host still reports a large running set and inflated runtime. The defect is narrower than `CO-67`: it is a current-source selection problem in the compatibility/runtime path, not another paused UI or rate-limit lane.

This lane must keep the existing `CO STATUS` and `/api/v1/state` surfaces while making current-running selection truthful when historical generic manifests have no active claim backing them.

## Requirements

1. When provider intake has no active current claims and there is no other explicitly authoritative current activity, live `CO STATUS` and `/api/v1/state` must show `running = 0`.
2. Historical non-provider `.runs/**/cli/*` manifests must not appear as current running activity unless they are explicitly current by contract.
3. Aggregate runtime must be computed from the same authoritative current source set as the running table.
4. Focused regression coverage must prove null-provider historical manifests without an active current claim do not survive the current filter.
5. Final validation must include live host/API verification plus terminal and attach-viewer proof on this device, embedded directly in Linear.

## Current Truth

- `.runs/local-mcp/cli/control-host/provider-intake-state.json` currently reports `latest_reason: provider_issue_released:not_active` and no active claim state.
- `readCompatibilityRuntimeSnapshot()` still builds the running set from discovered raw `in_progress` sources plus a permissive authority check.
- `isAuthoritativeCurrentRunningSource(...)` currently returns `!hasProviderBoundIssueIdentity(source)` for null-provider sources with no matching claim, which lets generic historical manifests survive.
- Initial audit also noted that `buildSelectedRunManifestSnapshot(...)` still falls back `issueIdentifier` and `issueId` to generic `taskId` / `runId`, but the bounded fix should avoid rewriting that selected-manifest compatibility contract unless new evidence proves it necessary.

## Proposed Design

### 1. Keep selected-run identity behavior unchanged for this lane

- Keep selected-run identity available for the active selected manifest.
- Treat the selected-manifest compatibility fallback as an observed input seam, not as the implementation target for this issue.
- File a follow-up instead of rewriting `selectedRunProjection.ts` if later evidence shows that contract itself is still too permissive after the runtime filter fix.

### 2. Exclude null-provider historical manifests from current-running authority

- Keep provider-intake claims authoritative for current provider-backed activity.
- Keep the selected run authoritative for the currently selected issue when it is actually active, but require an active claim for the null-provider `local-mcp` fallback manifest.
- For discovered historical null-provider sources with no active current claim, fail closed instead of treating "not provider-bound" as current authority.
- Preserve explicit fallback behavior only where a manifest is clearly current by contract, not merely because its raw status remains `in_progress`.

### 3. Align aggregate runtime to the filtered running set

- Continue deriving runtime totals from the same authoritative running source collection used for the table.
- Do not let historical non-provider entries inflate seconds-running once they have been excluded from `running`.

### 4. Tests and proof

- Add focused regressions covering:
  - current-running exclusion for null-provider historical manifests without an active claim
  - runtime totals aligned to the filtered running set
  - live `/api/v1/state` compatibility output reflecting the same filtered source set
- Capture terminal and attach-viewer screenshots directly in Linear after the fix on this device.
