# Technical Specification

## Context

`CO-179` ties together two existing contracts that currently meet at an operator failure point. `co-status attach` is a read-only viewer for an already-running local control-host, resolved through `control_endpoint.json` and `control_auth.json`. Provider refresh stuck detection can mark polling as `provider_refresh_lifecycle_stuck` / `restart_required`, and supervision can restart the control-host after repeated unhealthy probes. When the restart publishes a new endpoint, an attach viewer that cached the old `base_url` can remain stranded on a dead port just when the operator needs status. The fix should be a bounded attach endpoint-rotation and recovery lane, not a dashboard or provider scheduler redesign.

## Requirements

1. `co-status attach` must be able to reload `control_endpoint.json` and `control_auth.json` from the same run dir after recoverable fetch/probe failures.
2. A stale endpoint rotation must recover when the old endpoint is dead and the current sidecar points to a healthy replacement `/ui/data.json`.
3. Dead-port fetches must be classified distinctly from malformed payload, HTTP non-OK, and generic dashboard failures.
4. Raw network errors must be classified with safe detail such as error name/code/message where available, without leaking tokens.
5. Auth failures (`401` / `403`) must be classified distinctly and should recover when the auth sidecar rotates to a valid token where practical.
6. Timeout failures must be classified distinctly from user cancellation.
7. Attach retry/reload behavior must stay bounded by the configured refresh interval and a small per-cycle reload budget.
8. `restart_required` and `probe_timeout` supervision behavior must remain thresholded, observable, and finite.
9. Recovery must preserve active detached `provider-linear-worker` issue processes.
10. Implementation must not redesign `CO STATUS`, provider scheduling, provider admission, or Linear workflow policy.

## Current Truth

- `orchestrator/src/cli/coStatusAttachCliShell.ts` resolves a target once through `loadControlEndpoint(...)` and passes one `baseUrl` and token to `startAttachedControlStatusDashboard(...)`.
- `fetchUiDataset(...)` fetches `/ui/data.json`, maps aborts to timeout/cancel messages, and throws generic errors for non-OK status or invalid datasets.
- `controlHostSupervision` health evaluation treats polling `restart_required=true` as unhealthy, while `controlHostSupervisionCliShell` classifies command probe timeouts as `probe_timeout`.
- Existing provider refresh stuck work already treats `provider_refresh_lifecycle_stuck` and `restart_required` as request-burn stop signals.
- Existing restart cleanup lanes require active provider worker preservation.

## Target Design

- Add a reloadable attach target/session boundary around the existing endpoint loader.
- Keep attach endpoint/auth authority in `control_endpoint.json` and `control_auth.json`; do not infer ports or launch another host.
- Add an attach fetch result taxonomy:
  - `stale_endpoint_dead_port`
  - `attach_network_error`
  - `attach_auth_failed`
  - `attach_request_timeout`
  - `attach_request_cancelled`
  - `attach_http_error`
  - `attach_invalid_dataset`
  - `attach_current_endpoint_recovered`
- Use the taxonomy to decide whether to reload artifacts and retry once within the bounded cycle.
- Keep dashboard rendering mostly unchanged by delivering either a valid dataset or a classified safe error.
- Preserve supervision semantics for `restart_required` and `probe_timeout`; add narrow tests or fixes only for bounded loops or worker-preservation gaps.

## Validation Plan

- Focused attach tests in `orchestrator/tests/CoStatusAttachCliShell.test.ts`:
  - stale endpoint rotation from a stopped server/old port to a healthy current endpoint
  - dead-port fetch classification
  - current endpoint recovery and rendered dataset continuation
  - raw network error classification through injected failing fetch or controlled server failure
  - auth failure classification and token sidecar recovery where practical
  - timeout classification for a hanging `/ui/data.json` request
- Focused supervision tests in `orchestrator/tests/ControlHostSupervision.test.ts`:
  - `restart_required` remains unhealthy
  - `probe_timeout` thresholds into bounded restart state/exit behavior
  - cleanup/preservation fixtures prove active provider workers are not killed
- Add `ControlServerPublicLifecycle` or `ProviderIssueHandoff` regressions only if implementation changes provider refresh recovery.
- Parent lane runs docs-review and full repo gates before review handoff.
