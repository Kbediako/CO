# Technical Specification

## Context
`CO-120` is about a local linked-worktree git-config leak, not a cloud-authorship event. The issue evidence already shows the exact failure mode: plain `git config user.*` inside a linked worktree writes to the shared repo git config file in the main checkout, so later commits in the main checkout inherit the wrong author identity. The repo still contains the same unsafe pattern in the archive automation workflow, and it currently has no repo-owned helper or regression test that locks the correct `git config --worktree` behavior.

## Requirements
1. Repo-owned linked-worktree helper/archive flows must inherit global git identity by default.
2. Any explicit alternate identity must be written with `git config --worktree`, never plain `git config user.*`.
3. If `extensions.worktreeConfig` is not already enabled, the helper must enable it before the first `--worktree` identity write and treat that as the only allowed shared-config precondition.
4. The checked-in archive automation workflow must use the same safe worktree-local identity path.
5. A reproducible regression test must prove the helper leaves the shared repo git config file free of a repo-local `[user]` override.
6. The closeout must include exact verification commands that operators or reviewers can rerun locally, including any one-time `extensions.worktreeConfig=true` precondition.
7. The implementation must preserve the issue's root-cause framing: linked-worktree git-config leakage, not cloud/Codex upgrade behavior.
8. The lane must stay bounded to future behavior and verification, not historical commit rewriting or global git-config policy.

## Design
- Add one small repo-owned helper that:
  - leaves identity untouched when no alternate identity is requested
  - requires both name and email when an alternate identity is requested
  - enables `extensions.worktreeConfig` when needed
  - applies alternate identity with `git config --worktree`
- Update `.github/workflows/archive-automation-base.yml` to call the helper instead of plain `git config user.*`.
- Add focused Vitest coverage that creates a temporary repo, adds a linked worktree, runs the helper, and asserts:
  - the worktree-local identity exists when explicitly requested
  - the shared repo git config file does not gain a repo-local `[user]` override
- Record the exact verification path in the workpad and validation notes so the closeout is machine-checkable instead of narrative-only.

## Implementation Surface
- Expected codepaths:
  - `.github/workflows/archive-automation-base.yml`
  - new `scripts/` helper for linked-worktree git identity handling
- Expected tests:
  - new `tests/` coverage for linked-worktree git identity isolation

## Protected Expectations
- Preserve global git identity inheritance by default.
- Preserve the current archive automation behavior apart from the identity-isolation seam.
- Preserve the issue's explicit non-goals around historical commit rewriting and global config changes.
- Preserve the issue's root-cause framing unless new evidence appears.

## Reject These Wrong Interpretations
- "This proves Codex cloud upgraded the repo and rewrote authorship."
- "The fix is to force `Codex <codex@openai.com>` across repo-owned helpers."
- "We should clean this up by rewriting old commits."
- "Turning on `extensions.worktreeConfig` alone is enough even if we keep using plain `git config user.*`."

## Current Truth
- The issue evidence reproduces that plain `git -C <linked-worktree> config user.* ...` writes to the shared repo git config file.
- The repo currently contains no `git config --worktree` identity writes.
- The repo also contains no seam that enables `extensions.worktreeConfig` before worktree-local config is attempted.
- `.github/workflows/archive-automation-base.yml` still configures an alternate linked-worktree identity with plain `git config user.*`.
- `orchestrator/src/cli/run/workspacePath.ts` only bootstraps detached worktrees and does not set git identity, so the leak is not in workspace creation itself.

## Proposed Design
- Introduce a tiny helper script dedicated to linked-worktree git identity handling.
- Let that helper enable `extensions.worktreeConfig` once when Git requires it, then write user identity via `--worktree`.
- Route the existing archive automation workflow through that helper.
- Keep the verification burden local and explicit with a temp-repo regression test plus exact manual verification commands.

## Non-Goals
- Rewriting historical commits or contributor attribution.
- Modifying user-global git config.
- Broad workflow redesign outside linked-worktree identity handling.
- Blaming cloud/runtime changes without separate evidence.

## Parity / Alignment Matrix
- Current truth:
  - linked-worktree alternate identity writes can leak into the shared repo git config file
  - no repo-owned helper or test locks the safe pattern
  - no repo-owned path handles the `extensions.worktreeConfig` precondition
- Reference truth:
  - default behavior inherits global identity
  - explicit alternate identity is isolated to worktree-local config only
- Target truth / intended delta:
  - repo-owned helper/workflow enables `extensions.worktreeConfig` when needed and uses `git config --worktree` only when alternate identity is explicit
  - regression coverage proves the shared repo git config file gains no repo-local `[user]` override
- Explicitly out-of-scope differences:
  - history rewriting
  - global git policy
  - unrelated provider/runtime changes

## Not Done If
- Shared repo git config can still gain a `[user]` override from a repo-owned linked-worktree helper flow.
- The implementation still assumes `git config --worktree` works without `extensions.worktreeConfig` already enabled.
- The workflow still calls plain `git config user.name` / `user.email` inside a linked worktree.
- The issue closeout cannot point to a reproducible proof path for non-mutation.
