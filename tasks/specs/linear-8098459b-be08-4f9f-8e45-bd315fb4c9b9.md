## Functional Requirements
1. Child-lane records or companion audit entries must carry durable parent turn lineage and parallelization decision lineage.
2. The lineage must identify the exact `parallelize_now` decision satisfied by the child lane, not just the issue and stream.
3. Retry recovery must prefer lineage over timestamp inference.
4. Launch-before-decision ordering is accepted only when durable lineage proves the child lane satisfies the relevant decision.
5. Timestamp inference is retained only for legacy records where lineage is absent and the existing bounded skew check is explicitly safe.
6. Recovery fails closed for stale older child lanes, unrelated issue ids, stream-only matches, lineage mismatches, ambiguous lineage, failed child lanes, missing lineage without safe legacy fallback, and unsafe timestamp ordering.
7. Pending parent acceptance checks remain required before recovered child-lane success can satisfy enforcement.
8. `provider-linear-worker-proof.json` and `co-status` expose recovered lineage, safe legacy fallback, or fail-closed rejection truthfully.

## Target Data Shape
- The implementation can use direct child-lane fields, companion audit rows, or both. Whatever shape is selected must preserve:
  - parent issue id
  - parent run id or equivalent attempt id
  - stable parent turn id
  - `parallelize_now` decision id or equivalent decision audit key
  - decision stream
  - decision recorded timestamp
  - child stream
  - child run id
  - child launch timestamp
  - child completion state
  - pending parent acceptance state
  - recovery markers such as `recover_child_lane:<stream>` and `recover_run:<run_id>` as diagnostics only

## Recovery Algorithm Contract
1. Build the candidate set from same-issue child-lane records and companion audit entries.
2. Prefer candidates with durable parent turn and decision lineage matching the active or latest prior `parallelize_now` decision.
3. Reject candidates where lineage points to a different decision, different parent turn, unrelated issue, different stream, failed child lane, completed but not pending parent acceptance, or stale older lineage.
4. If lineage is absent, use the legacy timestamp inference path only when records predate lineage support and the existing narrow timestamp-skew window still proves the launch belongs to the decision.
5. Emit proof/status fields that identify whether recovery accepted durable lineage, accepted safe legacy timestamp inference, or rejected the candidate fail-closed.

## Risks
- Risk: lineage is added but retry recovery still falls back to timestamp inference first.
  - Mitigation: require lineage-first recovery and tests that fail if timestamp ordering masks a lineage mismatch.
- Risk: implementation treats `recover_child_lane:<stream>` or `recover_run:<run_id>` as acceptance proof.
  - Mitigation: mark both as diagnostics only unless paired with latest prior decision lineage.
- Risk: legacy fallback becomes a broad timestamp bypass.
  - Mitigation: keep the existing narrow skew policy and require proof/status to classify legacy fallback explicitly.
- Risk: packet scope gets widened into active provider current-state authority work.
  - Mitigation: keep CO-408 limited to child-lane decision lineage and retry recovery.

## Not Done If
- Durable parent turn and parallelization decision lineage are missing.
- Retry recovery counts any successful same-issue child lane by stream alone.
- A stale older child lane can satisfy a newer `parallelize_now` decision.
- Timestamp inference is broadened without durable lineage.
- Pending parent acceptance checks are weakened.
- Proof/status surfaces omit whether recovery used lineage, safe legacy fallback, or fail-closed rejection.
- Implementation leaves provider-worker source, tests, proof/status projection, or packet mirrors inconsistent with the lineage contract.
