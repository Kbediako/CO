# Technical Specification

## Context

`CO-160` is the remaining released-claim follow-up after `CO-159`. Current `main` already has:

- dead active claim cached fail-closed handling
- completed review-wait and merge-wait cached fail-closed handling
- released-claim reopen helpers such as `provider_issue_released_pending_reopen:*`
- deferred-poll routing that can suppress fresh discovery when a poll resolves to cached fail-closed state

The remaining gap is ordinary deferred polls for retained released claims. The issue evidence and current code reading show:

- local intake can retain 128 released claims, mostly `provider_issue_released:not_active` plus a small `provider_issue_released:not_mutable` set
- ordinary deferred polls pass `trackedIssues: []` and `deferFreshDiscovery: true`
- `resolveTrackedIssuePollResolutionWithFallback(...)` uses `resolveProviderIssuePollFailClosedReason(...)` only for certain accepted, completed, or handoff-failed claim classes
- ordinary retained `released` claims therefore fall through to direct `resolveTrackedIssue(...)`, which maps to `dispatch_source_issue_by_id`

This lane must make those ordinary released claims fail closed from cached local truth during deferred polls, while preserving the existing bounded recovery seams that can reopen a released claim on newer evidence.

## Requirements

1. Preserve the already-landed `CO-159` dead-active claim handling and existing review-wait or merge-wait cached fail-closed behavior.
2. Ensure a deferred poll with empty `trackedIssues` performs zero direct issue-by-id reads for retained released inactive or non-mutable claims.
3. Ensure those released claims remain locally visible and do not churn `updated_at` on every deferred poll.
4. Ensure all-released retained claims do not trigger fresh discovery because of those claims during deferred polls.
5. Preserve the existing bounded reopen path for released claims when newer tracked-issue, webhook, or snapshot evidence shows the issue is runnable again.
6. Preserve queued or in-progress release cleanup behavior where cancellation or provider cleanup is still pending.
7. Cover the deferred-poll no-burn behavior with focused regressions over at least two consecutive polls with 128 retained released claims.

## Current Truth

- `resolveProviderIssuePollFailClosedReason(...)` currently returns cached reasons for:
  - `accepted` claims pending revalidation
  - `accepted` claims whose workflow state is not active
  - review-wait claims in `completed` or `handoff_failed`
  - merge-wait claims in `completed` or `handoff_failed`
- It currently does not return a cached fail-closed reason for ordinary `released` claims such as `provider_issue_released:not_active` or `provider_issue_released:not_mutable`.
- `resolveTrackedIssuePollResolutionWithFallback(...)` only avoids direct `resolveTrackedIssue(...)` when the tracked-issue map already resolves the claim, the claim is not a `not_found` release, or `resolveProviderIssuePollFailClosedReason(...)` returns a cached reason.
- `runRefreshCycle(...)` already treats `skip` results with `provider_issue_poll_cached_*` reasons as grounds to suppress fresh discovery in deferred polls.
- Released-claim reopen helpers already exist through `markProviderIssueReleasedPendingReopen(...)`, `shouldReopenReleasedClaimAtCurrentTimestamp(...)`, and `shouldReopenReleasedClaimOnRefresh(...)`, which means the reopen seam should be preserved rather than rebuilt.

## Proposed Design

### 1. Extend deferred-poll cached fail-closed classification for released claims

- Add a narrow released-claim branch to the deferred-poll fail-closed path so locally terminal released claims such as `provider_issue_released:not_active` and `provider_issue_released:not_mutable` return a cached poll reason instead of falling through to direct `resolveTrackedIssue(...)`.
- Keep the cached reason under the existing `provider_issue_poll_cached_*` family so the current fresh-discovery suppression path continues to work without a second routing mechanism.

### 2. Preserve bounded reopen semantics

- Do not fail closed for released claims that are already marked `provider_issue_released_pending_reopen:*`.
- Keep the current live reopen checks for newer tracked-issue or snapshot evidence, including assignee or mutability-driven reopen paths where the existing helpers already allow them.
- Leave non-deferred refresh behavior available for explicit live revalidation paths outside the bounded deferred-poll case.

### 3. Keep release cleanup behavior intact

- Do not interfere with `retryReleaseCancel(...)`, `shouldAttemptReleaseCancel(...)`, or similar cleanup behavior for released claims still tied to queued or in-progress runs.
- The cached fail-closed behavior should only stop unnecessary direct issue reads, not prevent genuine cleanup or cancellation follow-up.

### 4. Add focused regression coverage

- Add a regression that seeds 128 retained released claims, runs at least two deferred polls with `trackedIssues: []` and `deferFreshDiscovery: true`, and asserts:
  - zero direct `resolveTrackedIssue(...)` calls
  - zero fresh discovery refetches
  - stable released claim timestamps and local visibility
- Add coverage for both `provider_issue_released:not_active` and `provider_issue_released:not_mutable` in the same lane or adjacent focused cases.
