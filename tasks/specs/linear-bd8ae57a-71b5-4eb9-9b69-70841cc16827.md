# Technical Specification

## Context

`CO-110` is not a retry of `CO-106`; it is a narrower throughput follow-up. The current tree already has:

- user-scoped shared Linear budget persistence in `linearBudgetState.ts`
- reservation-aware and complexity-aware preflight in `linearBudgetState.ts`
- explicit cooldown precedence and safe observation merge semantics in `linearBudgetState.ts`
- provider helper fail-fast before `fetch` in `providerLinearWorkflowFacade.ts`
- control-host polling that already widens under low, exhausted, and cooldown states through `resolveLinearPollingInterval(...)`

The remaining gap is practical request burn:

- `getProviderLinearIssueContext(...)` still always performs a live GraphQL read after budget preflight, even when the same worker run already holds a fresh scoped cache record
- mutation helpers already reuse cache while it is fresh, but the top-level read path does not
- request-pressure thresholds for polling are still keyed to very late static request counts, so low-headroom operation can continue polling at or near the default `15000ms` cadence longer than it should
- local evidence still shows request spend as the dominant limiter while complexity remains effectively untouched

This lane must reduce request burn without weakening truthful fail-fast semantics.

## Requirements

1. Keep the current `CO-106` contract intact: user-scoped budgeting, reservation-aware preflight, complexity-aware checks, safe observation merge behavior, and truthful fail-fast semantics.
2. Reduce request-bucket exhaustion frequency under a reproduced low-concurrency single-issue workload.
3. Start degrading tracked-issue polling before `requests.remaining` reaches `0`, using request headroom that is aware of reset timing rather than only current static low/exhausted cutoffs.
4. Avoid redundant live rereads on the read-only `provider-linear:issue-context` path when a recent authoritative scoped cache record is already available and request headroom is degraded.
5. Do not return cached success during active shared-budget cooldown instead of truthful `linear_rate_limited` failure.
6. Preserve the distinction between request-bucket exhaustion and complexity-bucket exhaustion in both behavior and regression coverage.
7. Keep proof artifacts showing current request-burn shape, low-headroom polling, and helper-operation audit counts.

## Current Truth

- `providerLinearWorkflowFacade.ts` writes a scoped issue-context cache after successful `issue-context`, `transition`, `upsert-workpad`, and related mutations, but `getProviderLinearIssueContext(...)` does not attempt to reuse that cache before reading live data.
- The direct mutation cache freshness bound is `10_000ms`, and mutation helpers already revalidate or reuse cache within that window depending on risk.
- `resolveBucketPressure(...)` treats request pressure as `constrained` at `<= 10%` of limit capped at `50`, and `low` at `<= 2%` capped at `10`; for a `5000` request bucket this means request-based slowdown does not begin until `remaining <= 50`.
- `resolveLinearPollingInterval(...)` already applies deterministic positive jitter and preserves cooldown precedence, but its pre-cooldown slowdown still depends on those static thresholds rather than reset-aware remaining headroom.
- Local current intake truth at `/Users/kbediako/Code/CO/.runs/local-mcp/cli/control-host/provider-intake-state.json` still shows request bucket consumption as the practical limiter:
  - `polling.interval_ms` remains `15000`
  - `linear_budget.requests.remaining` falls independently of the complexity bucket
  - operation-level endpoint aliases include `provider-linear:issue-context:read-issue-context`, `provider-linear:transition:read-issue-summary`, and `provider-linear:upsert-workpad:read-issue-context`
- The cited `CO-109` worker proof and audit artifacts preserve a concrete fail-fast example where helper `issue-context` failed closed during shared cooldown while other operations had already spent request budget.

## Proposed Design

### 1. Low-headroom read-only cache reuse for `issue-context`

- Add a bounded cache-reuse gate inside `getProviderLinearIssueContext(...)`:
  - read the existing scoped cache record before the live GraphQL read
  - keep the current budget preflight first so active cooldown or exhausted shortfall still fail closed
  - if the cache record is within a bounded reuse window and request headroom is already degraded, return the cached issue context instead of burning another live read
- Keep the reuse scope narrow:
  - read-only `issue-context` only
  - same issue id and same resolved source setup only
  - no cache success during cooldown
  - no widening of mutation trust windows in this lane

### 2. Reset-aware request headroom guard for polling

- Extend `resolveLinearPollingInterval(...)` to compute a request-headroom-derived interval floor from the selected request bucket:
  - global `requests` bucket when no endpoint request bucket applies
  - endpoint request bucket when the current operation resolves one
- Use `remaining` plus `reset_at` to derive a minimum safe interval so the poll scheduler starts stretching before static low/exhausted thresholds are hit.
- Preserve dominance order:
  - cooldown still wins outright
  - explicit exhausted buckets still remain stronger than the new headroom guard
  - complexity pressure remains unchanged and separate
- Surface the new reason truthfully through `reason` / `suppression_reason` so operator and proof surfaces can explain why the poll interval stretched.

### 3. Request-vs-complexity coverage and proof

- Add focused tests that lock:
  - cache reuse on `issue-context` under degraded request headroom
  - live reads when headroom is healthy
  - reset-aware polling stretch before request exhaustion
  - unchanged cooldown dominance
  - explicit distinction between request exhaustion and complexity exhaustion
- Use existing `.runs` artifacts and the local provider intake snapshot to document the before/after request-burn shape and helper audit counts.
