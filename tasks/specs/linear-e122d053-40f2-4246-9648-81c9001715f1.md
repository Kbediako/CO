## Parity / Alignment Matrix
- Current truth:
  - poll dispatch computes global/state occupancy through `createProviderPollDispatchBudget(...)` inside `runRefreshCycle(...)`
  - webhook/direct accepted issues reach `processTrackedIssueCandidate(...)`, which can call `launchStartForTrackedIssue(...)` directly
  - queued retry dispatch can call both `launchResumeForRun(...)` and `launchStartForTrackedIssue(...)` without going through the poll gate
  - `discoverProviderIssueRuns(...)` plus persisted claims already represent the authoritative active-work seam closest to the real bug
- Reference truth:
  - every provider admission source should use one shared capacity contract
  - occupancy should be based on the same authoritative active claims and discovered active runs regardless of admission source
  - already-running work should survive restart/rehydration even when the host is over cap
- Target truth / intended delta:
  - start and resume helpers refuse new admissions when authoritative occupancy has reached cap
  - poll dispatch no longer has a unique global gate that non-poll paths can bypass
  - focused tests prove webhook/direct and queued retry paths cannot over-admit

## Technical Requirements
- Functional requirements:
  - one shared admission helper must gate both starts and resumes before the underlying launcher call
  - occupancy must count authoritative active work from both discovered `in_progress` runs and persisted claims already in active launch/running states
  - the gate must allow currently-owned active work to continue but refuse new admissions when occupancy is at or above cap
  - the poll loop may keep state-specific dispatch limits, but it cannot be the only global-capacity check
- Non-functional requirements (performance, reliability, security):
  - keep the helper deterministic and local to the repo/workspace state
  - avoid extra network calls in the admission decision
  - preserve persisted claim/run provenance so operator proofs remain truthful
- Interfaces / contracts:
  - `orchestrator/src/cli/control/providerIssueHandoff.ts`
  - `orchestrator/tests/ProviderIssueHandoff.test.ts`
  - provider-worker manifest discovery under `.runs/**`

## Architecture & Data
- Architecture / design adjustments:
  - factor the global-capacity decision out of the poll loop and into a shared start/resume boundary
  - compute occupancy from attachable discovered runs plus persisted claims, with de-duplication by provider issue key / active run identity so the count reflects real active work without double counting the same issue
  - preserve the existing over-cap recovery policy by refusing only new admissions
- Data model changes / migrations:

## Implementation Notes
- Added a shared provider admission helper in `providerIssueHandoff.ts` that seeds occupancy from authoritative active claims plus discovered attachable `in_progress` manifests, then reuses the current capacity-budget contract to answer launch eligibility.
- Applied that gate at all shared launch boundaries: `launchResumeForRun(...)`, `launchStartForTrackedIssue(...)`, and the direct `processTrackedIssueCandidate(...)` launch path, so webhook/direct, queued retry start, queued retry resume, and poll-driven launches all consult the same max-concurrency decision.
- Follow-up review work narrows repeated run discovery to one per async operation with `AsyncLocalStorage`, serializes direct webhook admission through the refresh lifecycle lock, and makes host-wide run discovery skip unreadable manifests instead of blocking every new admission.
- Kept poll dispatch state limits intact while tightening poll-budget accounting so refresh only marks a slot as consumed when a launch actually happened or a queued retry still owns capacity.
- Added focused regressions for direct webhook blocking, concurrent webhook serialization, unreadable-manifest tolerance, queued retry resume blocking, queued retry start blocking, and already-over-cap hosts with real `in_progress` manifests.
