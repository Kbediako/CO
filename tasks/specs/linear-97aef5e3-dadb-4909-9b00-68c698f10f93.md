# Technical Specification

## Context

Provider-worker lanes are supposed to use `linear child-stream` as the audited path for bounded docs/review/planning help. The current helper launches the child run correctly, but the parse step in `parseProviderChildRunResult(...)` rejects any stdout that does not start with `{`. Real `CO-26` evidence shows successful child runs with leading wrapper logs before the final JSON payload, which makes the helper fail even though the child manifest reports `status: succeeded`.

## Requirements

1. `linear child-stream --pipeline docs-review --format json` must succeed when stdout contains leading informational log lines before a valid final JSON object.
2. Malformed or missing final JSON must still return `provider_worker_child_stream_output_invalid` with a clear error instead of a false success.
3. Existing child-run normalization and path confinement checks must remain unchanged after JSON parsing succeeds.
4. Successful parse output must keep the existing payload structure and field names unchanged.
5. Focused regressions must cover both a valid prelude-log success case and a malformed-output failure case.
6. The change must stay bounded to the child-stream parse seam plus the docs/tests needed to keep it truthful.

## Current Truth

- `orchestrator/src/cli/providerLinearChildStreamShell.ts` currently gates parsing on `trimmed.startsWith('{')`, which rejects valid child output with any leading wrapper log lines.
- Once JSON is parsed, the current implementation already enforces safe `run_id`, artifact-root, manifest-path, and log-path confinement under the expected child run root. That behavior should not change.
- `orchestrator/tests/ProviderLinearChildStreamShell.test.ts` currently covers clean JSON success, unsafe path rejection, and record-write failures, but it does not yet cover leading prelude logs or malformed trailing JSON.
- `orchestrator/src/cli/delegationServer.ts` already has a bounded log-tolerant object extraction pattern that can serve as the narrow reference posture.

## Validation Plan

- Run `docs-review` via the audited `linear child-stream` path before implementation.
- Add focused regressions in `orchestrator/tests/ProviderLinearChildStreamShell.test.ts`.
- Run the required validation floor for the touched parser/test/docs surface.
- Run forced manifest-backed standalone review plus an explicit elegance/minimality pass before review handoff.
