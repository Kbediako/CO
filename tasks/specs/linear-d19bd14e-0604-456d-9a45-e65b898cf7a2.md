# Technical Specification

## Context
`CO-80`, `CO-100`, and `CO-111` made merge-closeout explicit and authoritative for live `Merging` lanes, but `CO-138` exposes the inverse stale-data failure mode. When a worker run completes and later rehydrates, the claim can still carry an older `merge_closeout.status=action_required` snapshot from when the issue was `Merging`, even though newer tracked-issue truth now shows the same issue back in `In Progress`. `buildProviderCompletedRunRehydrateState(...)` still treats that stale record as authoritative, so the claim remains `handoff_failed` with reason `provider_issue_merge_closeout_action_required` and the issue cannot be picked up again until someone manually clears local intake state. The repair is to make newer non-`Merging` issue truth dominate stale merge-closeout residue.

## Requirements
1. Reproduce the `CO-120` shape: a newer live issue update in `In Progress` plus an older `merge_closeout` record whose `issue_state` is `Merging` and whose closeout status resolves to `handoff_failed`.
2. Completed-run rehydrate must not preserve `handoff_failed` / `provider_issue_merge_closeout_action_required` when newer tracked-issue truth proves the issue is no longer in `Merging`.
3. The stale `merge_closeout` record must be cleared or safely superseded so subsequent refresh or poll logic can treat the issue as runnable again without manual file edits.
4. Existing behavior for truly current `Merging` issues and fresh merge-closeout outcomes must remain unchanged.
5. Focused regressions must cover completed-run rehydrate and poll / rehydrate recovery for the stale-record shape.

## Design
- Add a stale-merge-closeout freshness check at the rehydrate boundary that compares persisted merge-closeout issue-state truth against newer tracked-issue truth.
- When the persisted merge-closeout came from `Merging` but the current tracked issue state has newer non-`Merging` truth, treat the merge-closeout record as stale and drop or supersede it before the claim-state resolver classifies the claim.
- Keep current merge-closeout semantics untouched when the live tracked issue still resolves to `Merging` or when the persisted merge-closeout record remains current for the same state lineage.
- Make the stale-record cleanup monotonic in persisted intake state so later refreshes and observability no longer keep reviving the stale `handoff_failed` reason.

## Implementation Surface
- Expected codepaths:
  - `orchestrator/src/cli/control/providerIssueHandoff.ts`
  - `orchestrator/src/cli/control/providerIssueObservability.ts` only if projected summaries need the cleared or superseded state reflected explicitly
- Expected tests:
  - `orchestrator/tests/ProviderIssueHandoff.test.ts`
  - `orchestrator/tests/ControlServerStartupInputPreparation.test.ts` or equivalent poll / rehydrate recovery coverage if that path owns the reproduced shape

## Protected Expectations
- Preserve the exact surfaces `merge_closeout`, `provider_issue_merge_closeout_action_required`, `handoff_failed`, `Merging`, `In Progress`, and `provider-intake-state.json`.
- Preserve current deterministic merge-closeout behavior while the issue is actually still in `Merging`.
- Prefer a narrow freshness / invalidation repair over broader provider lifecycle redesign.
- Keep the issue eligible for normal pickup through existing revalidation rather than inventing a new manual-recovery flow.

## Reject These Wrong Interpretations
- `Any action_required merge-closeout should be cleared automatically regardless of current issue truth.`
- `This is only a STATUS or observability presentation bug.`
- `Fix it by reopening merged-PR retirement, historical attached-PR ambiguity, or review-to-merge automation.`
- `A fresh non-Merging issue update should still be blocked until someone manually edits intake state.`

## Current Truth
- `providerIssueHandoff.ts` still resolves preserved merge-closeout truth during completed-run rehydrate before new active-pickup logic can recover.
- `providerIntakeState.ts` preserves `merge_closeout` from the existing claim whenever no replacement record is written.
- The `CO-120` incident artifact shows newer live issue truth (`In Progress`) coexisting with stale `merge_closeout.issue_state: "Merging"` and `merge_state_status: "BEHIND"` in shared intake state.
- Manually clearing that one stale `merge_closeout` record was sufficient to make the issue runnable again.

## Proposed Design
- Introduce one authoritative stale-merge-closeout filter that can answer whether newer tracked-issue truth proves the persisted merge-closeout no longer matches the live workflow state lineage.
- Invoke that filter before `buildProviderCompletedRunRehydrateState(...)` resolves the persisted claim state.
- Persist the filtered result from `providerIssueHandoff.ts` so stale merge-closeout records do not survive later rehydrate or poll recovery cycles even though `providerIntakeState.ts` still preserves omitted fields by design.
- Leave all other merge-closeout reasons and current `Merging` closeout behavior unchanged.

## Non-Goals
- Reworking merged-PR retirement from `CO-111`.
- Reworking historical attached-PR ambiguity from `CO-104`.
- Reworking `In Review` / `Merging` promotion from `CO-116`.
- Reworking refresh-timeout recovery from `CO-119`.
- Reworking concurrency admission from `CO-125`.

## Parity / Alignment Matrix
- Current truth:
  - stale `merge_closeout` failure records can outlive the workflow state that produced them
  - completed-run rehydrate still lets that stale record dominate claim classification
  - normal pickup remains blocked until someone edits local intake state
- Reference truth:
  - newer tracked issue state should dominate stale merge-closeout residue once the issue has clearly left `Merging`
  - live `Merging` closeout outcomes should remain authoritative while the issue is still in that state
- Target truth / intended delta:
  - stale merge-closeout records are cleared or superseded when newer non-`Merging` issue truth exists
  - completed-run rehydrate and poll recovery produce a runnable claim for the reproduced shape
  - current `Merging` closeout behavior stays unchanged
- Explicitly out-of-scope differences:
  - review-promotion automation
  - merged-lane retirement redesign
  - PR attachment disambiguation
  - refresh-timeout or admission-policy changes

## Not Done If
- A reproduced `CO-120` stale merge-closeout still leaves the issue in `handoff_failed`.
- The only way to relaunch the issue is still manual `provider-intake-state.json` editing.
- Truly current `Merging` issues stop honoring their active merge-closeout result.
- Poll / rehydrate recovery can reintroduce the stale merge-closeout after the fix.
