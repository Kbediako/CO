# TECH_SPEC - CO-404 Control-Host Provider-Worker Acknowledgement Timeouts

## Issue-Shaping Contract
- User-request translation: `control-host recover|relaunch|nudge --issue-id <id>` must acknowledge accepted work inside the request budget. If launch/resume continues, return `queued` or `starting` metadata instead of a hard timeout after side effects.
- Protected terms: `control-host provider-worker nudge request timed out after 25000ms`, `provider_launch_source=control-host`, `provider_issue_rehydration_pending_revalidation`, `provider_issue_start_blocked:max_concurrency`, `run_id=null`, `run_manifest_path=null`, `launch_started_at=null`, `launch_token=null`, `queued`, `starting`.
- Protected surfaces: `controlHostProviderWorkerRecoverCliShell.ts`, `observabilityApiController.ts`, `provider-intake-state.json`, `provider-linear-worker-proof.json`, `provider-linear-worker-linear-audit.jsonl`.
- Reject: timeout-only tuning, delayed `202` until full completion, duplicate launches on retry, counting accepted-without-run rows as active capacity, reopening CO-393, or folding in CO-403 without shared-root proof.

## Requirements
- Return truthful acknowledgement once accepted recover/relaunch/nudge work is durable.
- Represent async continuation with machine-checkable metadata and no raw launch token exposure.
- Preserve retry idempotency across ambiguous acknowledgement retries, including UUID/human aliases.
- Preserve accepted pending-revalidation rows but exclude rows without launch/run identity from running/starting capacity.
- Add focused regressions for acknowledgement timing, failure/no-queued truth, retry coalescing, and accepted-without-run capacity.

## Parity Matrix

| Variant | Current Truth | Target Truth | Out Of Scope |
| --- | --- | --- | --- |
| Timeout after successful launch | Request times out while manifest/proof/audit show launch and `Ready -> In Progress`. | Accepted work returns `queued` or `starting` truth inside the request budget. | CO-393 command guardrails and CO-403 stale proof-lock recovery. |
| Accepted without run | `state=accepted` with null run/manifest/launch/token can look active. | Pending truth is retained but not counted as active capacity. | Broad provider-worker concurrency redesign. |

## Fallback / Refactor Decision

| Surface | Seam | Decision | Validation |
| --- | --- | --- | --- |
| `controlHostProviderWorkerRecoverCliShell.ts` | 25s hard timeout after accepted side effects. | `remove fallback` | Timeout-after-launch regression. |
| `observabilityApiController.ts` | `202` waits for full recovery action. | `remove fallback` | Accepted acknowledgement regression. |
| `provider-intake-state.json` | Accepted-without-run consumes capacity. | `remove fallback` | Null launch/run capacity regression. |
| Provider retry/idempotency | Ambiguous retry safety. | `justify retaining fallback` | Duplicate launch/intake prevention regression. |

## Not Done If
- Hard timeout still surfaces after accepted launch/resume side effects.
- API acknowledgement still waits for full action completion.
- Retry duplicates launches or intake truth.
- Accepted pending-revalidation rows with null launch/run identity consume active capacity.
