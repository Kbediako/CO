# Technical Specification

## Context
`CO-104` tightened historical attached-PR disambiguation for deterministic merge closeout. `CO-138` handled stale merge-closeout `action_required` residue once newer live issue truth had already left `Merging` and returned to active work. `CO-214` handled the merged-terminal sibling where live Linear truth was already `Done` but local closeout state still retained `action_required` residue. `CO-210` separately fixed current-progress hydration for active child-lane truth.

`CO-228` is the stale merged-terminal `transition_failed` sibling of those lanes: after `CO-210` / `PR #512` are already merged and live Linear truth is `Done`, local `provider-intake-state.json` can still retain older merge-closeout residue with top-level claim reason `provider_issue_merge_closeout_transition_failed`, nested `merge_closeout.status=transition_failed`, `reason=linear_done_transition_failed`, `linear_transition.status=failed`, and `error=linear_transition_conflict`.

That residue is not just historical noise. It can remain authoritative for provider handoff or read-side current-state classification even though merged PR truth and live Linear `Done` truth have already advanced. The repair needs one bounded stale-residue decision seam that preserves real current transition failures while clearing or superseding only stale merged-and-Done `transition_failed` residue.

## Requirements
1. Reproduce the `CO-210` / `PR #512` stale merged-terminal shape: merged PR truth plus live Linear `Done` truth coexist with retained local `provider_issue_merge_closeout_transition_failed`, `merge_closeout.status=transition_failed`, `reason=linear_done_transition_failed`, `linear_transition.status=failed`, and `error=linear_transition_conflict`.
2. Provider handoff and later recovery logic must not keep the issue governed by stale merged-and-Done residue once merged PR truth and newer live Linear `Done` truth prove the residue is stale.
3. `CO STATUS` active/backoff or equivalent read-side projection must not report the stale merged-and-Done `transition_failed` shape as current work.
4. Real current `transition_failed` truth must remain visible for issues whose live Linear truth has not advanced to terminal `Done`.
5. The repair must preserve existing attached-PR selector truth from `CO-104`.
6. The repair must remain separate from `CO-138`, `CO-214`, and broader `CO-210` semantics.
7. The repair must remain separate from shared-root reconciliation redesign and new attachment-selector regressions.

## Design
- Introduce one authoritative stale merged-terminal residue predicate that compares retained local closeout state against merged PR truth plus the newest tracked Linear issue truth.
- Use that predicate before provider handoff or current-state projection consumes retained `transition_failed` closeout state as if it were still current.
- When merged PR truth is authoritative and newer live Linear truth is terminal `Done`, treat older local merged-terminal `transition_failed` residue as stale and clear or safely supersede it for current-state classification.
- Keep current behavior intact when the issue is still truly non-terminal, when merged PR truth is absent, or when the retained failure residue still matches live state lineage.
- Reuse the same stale-residue decision across provider handoff and read-side / active-backoff projection so write-side and read-side truth do not drift apart.

## Current Truth
- `providerMergeCloseout.ts` records `status='transition_failed'` and `reason='linear_done_transition_failed'` when the PR merges but the Linear `Done` transition fails.
- `providerIssueHandoff.ts` maps that retained closeout record to claim reason `provider_issue_merge_closeout_transition_failed`.
- The retained merged-terminal residue can outlive the real workflow state that produced it.
- Current handoff or read-side seams can still treat that residue as current blocked work even after live Linear truth is already `Done`.
- Adjacent lanes already define the required boundaries: `CO-104` for attached-PR selection, `CO-138` for stale active-work residue, `CO-214` for stale merged-terminal `action_required` residue, and `CO-210` for current-progress hydration.

## Proposed Design
- Add a small shared predicate that answers whether retained merged-terminal `transition_failed` residue is stale because merged PR truth plus newer live Linear `Done` truth supersede it.
- Run that predicate before:
  - any provider handoff or recovery path classifies the issue as still blocked by stale closeout failure
  - any read-side / active-backoff presentation path reports the issue as current work
- Persist or project the cleaned or superseded result so later handoff and projection paths do not revive stale state.
- Leave all current behavior untouched for genuinely current transition failures and for adjacent seams owned by `CO-104`, `CO-138`, `CO-214`, and `CO-210`.

## Non-Goals
- Reworking shared-root reconciliation policy.
- Reopening attached-PR selector work from `CO-104`.
- Rewriting `CO-138` or `CO-214`.
- Changing broader `CO-210` hydration semantics.
- Replacing runtime fixes with docs-only reinterpretation.
