## Added by Docs Packet 2026-04-27

## Summary
- Objective: Make provider-worker retries recover a completed same-issue child lane for the active `parallelize_now` decision and recover stale orphaned proof locks without corrupting `provider-linear-worker-proof.json`.
- Scope: Docs packet for CO-403; implementation scope is expected to cover `providerLinearWorkerRunner.ts`, `withProviderLinearWorkerProofLock`, `PROVIDER_LINEAR_WORKER_PROOF_LOCK_RETRY`, `orchestrator/src/persistence/lockFile.ts`, proof sidecar projection, `co-status`, and focused tests.
- Constraints: Preserve fail-closed `parallelize_now` enforcement, reject unrelated/older/failed child lanes, avoid duplicate no-op launches, and keep CO-400 current-state authority out of scope.

## Issue-Shaping Contract
- User-request translation carried forward: A resumed provider-worker attempt should not call the active same-issue child lane missing when that lane launched in the prior attempt, succeeded, and is pending parent acceptance; stale proof locks should be reclaimed only through an explicit stale-lock policy with live-lock keepalive protection.
- Protected terms / exact artifact and surface names: `parallelize_now`, `parallelization_launch_missing`, `provider-linear-worker-proof.json.lock`, `provider-linear-worker-proof.json`, `co-status`, `providerLinearWorkerRunner.ts`, `withProviderLinearWorkerProofLock`, `PROVIDER_LINEAR_WORKER_PROOF_LOCK_RETRY`, `orchestrator/src/persistence/lockFile.ts`, same-issue child lane, pending parent acceptance, stale orphaned proof lock.
- Nearby wrong interpretations to reject: Counting any successful child lane regardless of lineage, weakening child-lane ownership, bypassing enforcement by launching duplicate no-op lanes, deleting stale locks manually, hand-editing proof JSON, or hiding recovery in stale snapshots.
- Explicit non-goals carried forward: Do not disable `parallelize_now` enforcement; do not launch duplicate no-op child lanes; do not fold unrelated CO-400 authority work into this bug; do not weaken patch acceptance or missing-child fail-closed handling.

## Parity / Alignment Matrix
- Current truth: Provider-worker retry proof can be attempt-local and stale, leading to `parallelization_launch_missing` after prior successful child-lane completion; leaked proof locks can indefinitely block proof writes.
- Reference truth: Same-issue child-lane enforcement is required, but a completed child lane launched for the active decision should be recoverable as pending parent acceptance.
- Target truth / intended delta: Retry recovery uses issue id, stream, turn lineage, and parent decision lineage to accept only the correct completed child lane; stale lock recovery validates the stale/dead owner before reclaiming; proof and status surfaces show recovered truth.
- Explicitly out-of-scope differences: Linear/GitHub lifecycle, workpad updates, source/test edits in this child lane, duplicate child-lane launches, CO-400 authority consolidation, and manual proof repairs.

## Technical Requirements
- Functional requirements:
  - Rehydrate or discover the active `parallelize_now` decision's same-issue child-lane proof on retry.
  - Continue to parent patch acceptance when the child lane launched in the prior attempt, completed successfully, matches issue/stream/turn/decision lineage, and is pending parent acceptance.
  - Keep `parallelization_launch_missing` for no launch, failed child, unrelated child, older child, or lineage mismatch.
  - Recover stale orphaned `provider-linear-worker-proof.json.lock` files via explicit stale-lock policy or equivalent owner/dead-process validation.
  - Preserve exclusive proof writes so concurrent writers cannot corrupt `provider-linear-worker-proof.json`.
  - Project recovered child-lane and lock state truthfully through `co-status` and proof surfaces.
- Non-functional requirements: Recovery must be deterministic, auditable from proof/status fields, conservative while locks are fresh, and must not increase Linear/GitHub read pressure unless parent scope expands.
- Interfaces / contracts:
  - `providerLinearWorkerRunner.ts`: active-decision child-lane recovery and `parallelization_launch_missing` classification.
  - `withProviderLinearWorkerProofLock` and `PROVIDER_LINEAR_WORKER_PROOF_LOCK_RETRY`: proof lock acquisition/retry behavior.
  - `orchestrator/src/persistence/lockFile.ts`: shared or provider-scoped stale-lock policy surface.
  - `provider-linear-worker-proof.json` / `provider-linear-worker-proof.json.lock`: sidecar truth and lock owner metadata.
  - `co-status`: operator-visible recovered state.
