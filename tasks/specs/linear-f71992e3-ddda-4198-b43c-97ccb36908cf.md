## Requirements
1. The steady-state top-line `Rate Limits` row renders as `Codex ... || Linear ...`.

3. Exhausted buckets show `resets Xm`, and only the exhausted buckets do.
4. The displayed Linear buckets on that row are `requests` and `complexity`.
5. Active running `STAGE` preserves the current Linear state instead of generic runtime/provider phase.
6. Active running `EVENT` reaches Symphony-level specificity for shared concepts plus truthful CO-specific budget/backoff/review-drain semantics.
7. The live default STATUS surface does not render `Dashboard:`.
8. The live frame renders a separate `Status controls` section below `Backoff queue`.
9. The implementation explicitly preserves the already-correct upstream budget substrate and stays distinct from `CO-98/99/100/81/101/17`.
10. Focused tests and payload/output proof cover both projection truth and terminal rendering.

## Design
- Preserve the authoritative combined budget source in `controlRuntime.ts`; do not re-implement shared Linear budget logic.
- Update `selectedRunProjection.ts` so active started workflow state returns the actual Linear state unless the tracked state is queued/unstarted/terminal or the run is in a stronger local paused/input override.
- Promote authoritative operator-facing running event text into projection/presenter output, using current proof/debug/polling evidence where available, so `controlStatusDashboard.ts` stops deciding most event meaning from shallow fallbacks.
- Keep `controlStatusDashboard.ts` responsible mainly for:
  - bounded rate-limit bucket formatting
  - rendering the `Codex ... || Linear ...` divider row
  - omitting `Dashboard:`
