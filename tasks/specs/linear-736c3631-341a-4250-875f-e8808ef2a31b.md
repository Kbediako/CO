# Technical Specification

## Context

`CO-79`, `CO-80`, and `CO-81` each narrow one truth problem in the provider-worker lifecycle, but none of them creates the single operator-facing or self-repair-facing contract that `CO-82` asks for. Current active-lane truth is still fractured:

- provider-worker proof says more than the status surfaces but is still mostly ownership and turn data
- merge-closeout truth lives in `merge_closeout` claim records inside intake state
- `co-status` and the control-host API project raw proof and claim data without an explicit semantic progress model
- appserver-backed workers can keep fresh proof heartbeats while the normal runner/event stream says almost nothing meaningful

The lane therefore needs one shared provider progress and debug snapshot contract that reuses the existing proof, claim, merge-closeout, and tracked-issue seams instead of inventing another artifact family.

## Requirements

1. Active provider-worker lanes must emit structured progress and stall telemetry, including a meaningful phase model for merge-closeout and other active states.
2. Appserver-backed worker progress must be mirrored into the normal runner/event stream with semantic events, not only proof-sidecar updates.
3. Control-host claim and read-model surfaces must expose latest semantic progress time, progress kind, stall reason, and whether the claim is fresh or rehydrated.
4. `co-status --format json` or an equivalent control-host API route must expose one authoritative per-issue debug snapshot containing at least live Linear state, attached PR state, worker progress phase, last audit operation, last semantic progress time, stall classification, and recovery recommendation.
5. Regression coverage must prove the `CO-77` / `CO-78` case can be classified from one surface without manual forensic correlation.

## Protected Expectations

- Observability is the product contract for this lane, not just additional logging.
- A fresh heartbeat is not good enough when semantic progress is unknown.
- Manual correlation across provider intake, worker proof, audits, GitHub, Linear, and `ps` is not an acceptable steady-state debugging requirement.
- Backend or read-model changes are required; terminal-only presentation changes are not sufficient.

## Reject These Wrong Interpretations

- “The existing proof sidecar is already the authoritative surface.”
- “Current control-host ownership truth is enough because it says `running`.”
- “Merge-closeout progress can stay hidden until the issue reaches `Done`.”
- “The fix is only to add more raw log lines.”

## Current Truth

- `orchestrator/src/cli/providerLinearWorkerRunner.ts` writes `provider-linear-worker-proof.json` with `owner_phase`, `owner_status`, `last_event`, `last_message`, tokens, audit summary, child lineage, and budget metadata, but no first-class semantic progress or stall contract.
- `orchestrator/src/cli/control/providerIntakeState.ts` and `providerIssueHandoff.ts` persist claim ownership, retry state, and `merge_closeout` records, but those semantics are not promoted into one authoritative debug snapshot.
- `orchestrator/src/cli/control/selectedRunProjection.ts`, `compatibilityIssuePresenter.ts`, `observabilityReadModel.ts`, and `operatorDashboardPresenter.ts` project raw proof and claim data into STATUS/API payloads without an explicit progress kind, stall classification, or recovery recommendation surface.
- `orchestrator/src/cli/control/providerMergeCloseout.ts` already knows the attached-PR readiness or merge outcome contract, but that truth is still isolated inside intake `merge_closeout` records rather than a unified per-issue debug payload.

## Proposed Design

### 1. Shared provider progress and debug snapshot contract

- Add one shared helper under `orchestrator/src/cli/control/` to define and derive:
  - structured provider progress records for worker proof
  - authoritative per-issue debug snapshots for control-host read surfaces
  - stall classification and recovery recommendation mapping
- The helper must consume existing truth sources, especially worker proof, intake claim data, merge-closeout records, tracked Linear state, and audit summaries.

### 2. Structured progress in provider-worker proof

- Extend `ProviderLinearWorkerProof` with semantic progress metadata such as:
  - current phase and progress kind
  - last semantic progress summary and timestamp
  - stall classification and reason
  - attached-PR or merge-closeout state when available
- Update `providerLinearWorkerRunner.ts` so progress changes are persisted alongside existing proof snapshots instead of creating another sidecar file.

### 3. Semantic progress mirrored into the runner stream

- When semantic progress changes, emit a normal process log line or event from the provider worker so the command-stage `runner.ndjson` captures it through the existing command-runner exec chunk path.
- Keep the mirror bounded: emit on semantic changes, not on every heartbeat or proof refresh.
- The normal runner/event stream must carry enough information for an operator to distinguish active progress from waiting or stalled states without opening the proof file directly.

### 4. Control-host claim and STATUS projection

- Extend selected-run and compatibility issue payloads so they expose the authoritative debug snapshot directly instead of leaving callers to reconstruct it from raw `provider_linear_worker_proof` plus intake state.
- Project claim freshness and rehydrate truth explicitly from intake data, including whether a claim was rehydrated and whether the projected state is current or derived from historical recovery.
- Reuse `merge_closeout` claim records so attached-PR state and merge wait reasons appear in the same snapshot when the issue is in `Merging` or when a recent merge-closeout attempt exists.

### 5. Regression-first validation

- Extend existing provider-worker and control-host tests rather than building a parallel harness.
- Cover:
  - proof progress derivation
  - claim freshness or rehydrate projection
  - merge-closeout PR state inclusion
  - STATUS/API issue payloads exposing one authoritative snapshot
  - runner-stream semantic progress mirroring
- Add at least one classification-style regression that encodes the `CO-77` / `CO-78` manual-correlation family and proves one-surface classification.

## Non-Goals

- Replacing the provider-worker lifecycle model.
- Rewriting historical proof or claim artifacts for old runs.
- Redesigning the STATUS dashboard UI.
- Changing merge or recovery behavior already scoped in `CO-79`, `CO-80`, or `CO-81`.

## Parity / Alignment Matrix

| Surface | Current CO truth | Required target truth |
| --- | --- | --- |
| Provider-worker proof | raw ownership/turn telemetry with audit and lineage | semantic progress/stall contract that stays authoritative for active lanes |
| Runner/event stream | generic exec output can stay semantically quiet while proof keeps updating | semantic progress changes are visible in ordinary runner events |
| Intake claims | ownership, retry, and merge-closeout records are present but indirect | fresh/rehydrated truth plus progress semantics are directly projected |
| STATUS/API issue view | callers must correlate proof, claims, merge-closeout, and tracked issue state manually | one authoritative debug snapshot answers active/waiting/stalled/completed directly |

## Not Done If

- An operator still needs to manually correlate provider intake, proof, audit, GitHub, Linear, and `ps` to understand active-lane state.
- Appserver-backed workers can still look heartbeat-alive while `runner.ndjson` stays semantically silent.
- Active claim payloads still omit fresh-vs-rehydrated truth, latest semantic progress time, or stall classification.
- `co-status --format json` or `/api/v1/<issue>` still lacks attached PR state or recovery guidance for merge-closeout cases.
