# Technical Specification

## Context

`CO-35` intentionally shipped the first truthful same-issue child-lane runtime with explicit parent acceptance, but it left phase scope incomplete. The current contract is:

- explicit file scope can be machine-checked before `git apply`
- raw phase names can be recorded, but they do not map to persisted allowed repo paths
- pure phase-only lanes therefore still fail closed instead of being verifiable bounded helpers

`CO-52` is the missing trust-boundary slice that makes declared phase scope real without weakening the parent-owned acceptance model.

## Requirements

1. Define a deterministic repo-local child-lane phase contract that maps supported declared phases to bounded repo path selectors.
2. Preserve raw declared `files` and `phases`, but extend scope data so proof and ledger records also carry the resolved contract version and allowed selectors.
3. Allow phase-only launch when the declared phases resolve through the supported contract.
4. During parent acceptance, recompute the expected selector set from the declared phase contract and reject stale or tampered proof or ledger scope data before inspecting or applying the patch.
5. Accept a phase-scoped lane only when every patch path matches either the explicit file scope or the resolved phase selectors.
6. Preserve existing file-scoped child-lane behavior.

## Runtime Contract

- Phase contract:
  - supported phase names are explicit and versioned in repo-local code
  - each supported phase expands to one or more repo-relative exact or prefix selectors
  - unknown phase names fail closed
- Persisted scope truth:
  - child-lane proof and parent ledger records carry:
    - raw declared `files`
    - raw declared `phases`
    - `phase_contract_version`
    - resolved `allowed_path_selectors`
- Parent verification:
  - acceptance recomputes the expected contract from the declared raw scope
  - acceptance rejects when persisted proof or ledger selector data is missing for phase-scoped lanes, mismatched, or stale
  - acceptance then verifies every patch target path against the resolved selector union before `git apply`
- Compatibility:
  - file-only lanes continue to work with the same patch-inspection flow
  - older phase-only records without the new contract metadata remain fail-closed rather than silently trusted

## Planned Implementation

1. Add a dedicated phase-contract helper that resolves supported phase names into deterministic repo path selectors and can compare or match selectors against changed patch paths.
2. Extend child-lane scope typing, normalization, proof generation, and ledger serialization to carry `phase_contract_version` plus `allowed_path_selectors`.
3. Update child-lane launch validation so supported phase-only lanes can start without requiring explicit files.
4. Update parent acceptance to:
   - read the proof bundle again from the child artifact root when phase scope is involved
   - recompute and compare expected selector data against both ledger and proof scope payloads
   - reject out-of-phase, stale, or tampered scope data before parent patch apply
5. Add focused regressions for valid phase acceptance, out-of-phase rejection, and missing or mismatched contract data.
