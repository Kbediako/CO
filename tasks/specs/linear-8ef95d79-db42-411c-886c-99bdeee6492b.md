## Summary
- Objective: make deterministic merge closeout for reopened `Merging` issues tolerate clearly historical merged same-repo PR attachments while preserving explicit action-required truth for unresolved ambiguity.
- Scope:
  - docs-first packet and workpad for `CO-104`
  - deterministic attached-PR disambiguation in `providerMergeCloseout.ts`
  - merge-closeout persistence coverage in `providerIssueHandoff.ts`
  - focused regressions for the `CO-81` historical-plus-replacement shape, true ambiguity, and unchanged repo-mismatch / no-attached-PR behavior
  - standard validation, standalone review, and elegance review gates
- Constraints:
  - preserve `multiple_attached_prs` action-required semantics when multiple current candidates remain
  - ignore only deterministically historical merged PR attachments
  - keep the implementation additive and machine-checkable without deleting attachments or redesigning the broader attachment lifecycle

## Technical Requirements
- Functional requirements:
  - deterministic merge closeout must inspect same-repo attached PRs and safely classify historical merged PR attachments separately from current candidates
  - when exactly one current same-repo candidate remains after historical filtering, merge closeout must select it automatically
  - when more than one current same-repo candidate remains, merge closeout must stay action-required and persist the conflicting PR URLs explicitly
  - `merge_closeout` records must preserve the selected PR, ignored historical PR URLs, and conflicting PR URLs in machine-checkable form
  - `providerIssueHandoff.ts` must continue to persist the enriched merge-closeout record into intake claims without dropping the new fields
- Non-functional requirements (performance, reliability, security):
  - reuse existing GitHub snapshot infrastructure and keep disambiguation deterministic
  - preserve existing no-PR and repo-mismatch fail-closed behavior
  - avoid destructive attachment mutations or heuristic selection rules that cannot be justified from persisted evidence
- Interfaces / contracts:
  - `orchestrator/src/cli/control/providerMergeCloseout.ts`
  - `orchestrator/src/cli/control/providerIssueHandoff.ts`
  - `orchestrator/tests/ProviderMergeCloseout.test.ts`
  - `orchestrator/tests/ProviderIssueHandoff.test.ts`

## Architecture & Data
- Architecture / design adjustments:
  - add a candidate-resolution step in `providerMergeCloseout.ts` that can read per-PR GitHub state and partition same-repo attachments into selected, ignored historical, and conflicting groups without treating every merged PR as automatically historical
  - preserve the top-level merge-closeout flow for the selected PR once disambiguation is complete
  - keep handoff persistence generic by extending the merge-closeout record shape rather than adding bespoke handoff-only fields
- Data model changes / migrations:
  - additive `ProviderMergeCloseoutRecord` fields for ignored historical and conflicting attached PR URLs, plus any structured disambiguation metadata needed to explain why a PR was selected or ignored
  - no destructive migrations to intake state or historical claim data
- External dependencies / integrations:
  - GitHub PR snapshots already used for merge readiness
  - live Linear issue attachments
  - provider intake claim persistence / replay
