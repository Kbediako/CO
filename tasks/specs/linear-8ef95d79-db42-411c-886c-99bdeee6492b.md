## Summary
- Objective: make reopened deterministic merge closeout tolerate clearly historical merged same-repo PR attachments while keeping unresolved ambiguity explicit and machine-checkable.
- Scope: `providerMergeCloseout.ts`, persistence continuity through `providerIssueHandoff.ts`, debug-snapshot parity for the new record fields, focused regressions, and the standard validation / review gates.
- Constraints: preserve `multiple_attached_prs`, ignore only deterministically historical merged attachments, and avoid attachment deletion or broader lifecycle redesign.

## Issue-Shaping Contract
- User-request translation: reopened merge closeout must no longer depend on manual stale-attachment cleanup when one obvious replacement PR remains; ambiguous multiple-current-PR cases must remain action-required truth; `merge_closeout` artifacts must show selected, ignored historical, and conflicting PR URLs.
- Protected terms: `providerMergeCloseout.ts`, `providerIssueHandoff.ts`, `issue.attachments`, `multiple_attached_prs`, `Merging`, `provider-intake-state.json`, `merge_closeout`, reopened issue, historical merged PR attachment, replacement PR attachment, `CO-81`, `https://github.com/Kbediako/CO/pull/360`, `https://github.com/Kbediako/CO/pull/372`.
- Reject these interpretations: this was already solved by `CO-81` or `CO-100`; manual stale-attachment removal is the intended path; auto-deleting attachments is the fix; status/workpad wording alone is sufficient.
- Non-goals: stale-proof rehydrate / reclaim from `CO-81`; shared-root reconciliation from `CO-100`; full PR-attachment lifecycle redesign or garbage collection.
- Not done if: manual cleanup is still required for the historical-plus-current shape; ambiguity still collapses into generic `multiple_attached_prs`; provider/control-host artifacts still cannot show which PR was selected, ignored, or left conflicting.

## Parity / Alignment Matrix
- Current: `providerMergeCloseout.ts` fails immediately on multiple same-repo attachments, and `providerIssueHandoff.ts` persists the record without explicit ignored-historical or conflicting-URL truth.
- Reference: reopened issues with historical merged baggage should still close out when one replacement PR is unambiguous, while true ambiguity remains explicit.
- Target: historical merged baggage is persisted in `ignored_historical_pr_urls`, unresolved ambiguity is persisted in `conflicting_attached_pr_urls`, and the selected PR remains deterministic.

## Technical Requirements
- Use GitHub snapshots only when multiple same-repo attached PRs exist.
- Ignore only merged attachments that are strictly older than the selected replacement candidate.
- Recover an already-merged replacement PR over older stale open same-repo attachments.
- Keep repo-mismatch and no-attached-PR branches unchanged.
- Preserve the enriched `ProviderMergeCloseoutRecord` through intake claim persistence and the provider debug snapshot.
