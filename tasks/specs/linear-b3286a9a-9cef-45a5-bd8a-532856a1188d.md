# Technical Specification

## Context
The issue is a narrow hardening follow-up to `CO-114`, not another performance lane. The raw source parity observation was valid: `scripts/run-review.ts` was still stricter than `bin/codex-orchestrator.ts` under `NODE_OPTIONS=--preserve-symlinks-main`. The product surface, however, is narrower than that raw comparison suggests. Repo-local review still runs `scripts/run-review.ts` directly, packaged review still spawns `dist/scripts/run-review.js` through `codex-orchestrator review`, and the package installs no standalone `run-review` bin. The correct closeout is therefore to prove that narrower surface and codify it without destabilizing the subprocess harness.

## Requirements
1. Prove whether a supported symlinked `run-review` entrypoint exists today for repo-local or packaged workflows.
2. Keep `npm run review` behavior unchanged.
3. Keep packaged `codex-orchestrator review` launch behavior unchanged.
4. Do not edit `bin/codex-orchestrator.ts`.
5. Add focused regression coverage for the supported direct-exec path and the intentionally unsupported symlink-preserved direct-exec case.
6. Preserve `tests/run-review.spec.ts`, especially `does not crash when stdout pipe closes early`, without weakening it.
7. Keep the diff bounded to the direct-exec seam, its focused tests, and the required docs/evidence updates.

## Design
- Supported-path proof:
  - treat `package.json` bin/script/file declarations as the authoritative shipped-surface contract
  - confirm the repo-local supported path remains `npm run review`
  - confirm packaged review still reaches `dist/scripts/run-review.js` only through `codex-orchestrator review`
  - confirm user-facing docs do not advertise a standalone installed `run-review` bin
- Accepted implementation:
  - export `isDirectExecution(...)` from `scripts/run-review.ts`
  - keep the current realpath-first direct-exec contract and explain why it remains intentionally stricter than `bin/codex-orchestrator.ts`
  - add a focused direct regression in `tests/run-review.spec.ts` that proves the supported direct path and explicitly leaves symlink-preserved direct exec unsupported
- Regression strategy:
  - retain the existing pipe-close subprocess smoke unchanged
  - use the focused direct regression to cover the contract seam without adding more heavy subprocess coverage

## Current Truth
- `package.json` declares `review: "node --loader ts-node/esm scripts/run-review.ts"`.
- `package.json` publishes `dist/scripts/run-review.js` in `files`, but only `codex-orchestrator` / `codex-orch` in `bin`.
- `orchestrator/src/cli/reviewCliLaunchShell.ts` resolves review mode to the direct source or dist helper path.
- `orchestrator/tests/ReviewCliLaunchShell.test.ts` already locks those direct source/dist launch paths.
- `scripts/run-review.ts` now exports `isDirectExecution(...)` and documents why the stricter contract is intentional.
- `tests/run-review.spec.ts` now covers direct-path success, canonical realpath matching, and symlink-preserved rejection, while keeping the pipe-close subprocess smoke.

## Proposed Design
- No broader runtime parity widening ships in this lane.
- The lane closes by making the narrower supported contract explicit in tests and docs.
- If a future lane introduces a supported standalone `run-review` symlink-preserved entrypoint, that lane can widen `isDirectExecution(...)` with evidence and add the corresponding subprocess-safe regression coverage.
