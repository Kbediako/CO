# Technical Specification

## Context
`CO-160` and `CO-161` established released-claim no-burn behavior and `replay-block` for deferred polls so unrelated released claims do not keep burning direct issue reads or re-entering `fresh discovery` incorrectly. `CO-189`, `CO-202`, and `CO-203` separately cover live same-issue worker protection and safe reclaim of retained plain `provider_issue_released:not_active` claims. `CO-212` covers the narrower `Ready reclaim` case where completed blockers should stop suppressing a plain released claim. `CO-138` covers stale `merge_closeout residue` after an issue leaves `Merging` and returns to active work. `CO-214` and `CO-228` cover stale merged-terminal closeout residue after live `Done` truth dominates.

`CO-238` is the reopened-terminal sibling across those seams: a previously terminal same issue can later reopen to runnable truth, yet retained local `provider_issue_released:not_active`, stale `merge_closeout residue`, or release-family `replay-block` can still keep `fresh discovery` and same-issue re-admission closed. The repair needs one bounded rule that allows reopened same-issue admission when live truth is fresher and runnable and no live same-issue worker exists, without weakening all-released deferred-poll safeguards, duplicate-worker protection, or adjacent-lane ownership.

## Requirements
1. Reproduce the reopened terminal claim shape: retained local `provider_issue_released:not_active` remains after prior terminal truth, while live issue truth later reopens to runnable state for the same issue.
2. Same-issue re-admission must be possible when reopened live truth is fresher and runnable and no live same-issue worker remains.
3. Stale `merge_closeout residue` must not remain authoritative once fresher reopened truth is explicit for the same issue.
4. `replay-block` must continue preserving all-released deferred-poll no-burn behavior for unrelated released claims; only the reopened same-issue path may bypass that suppression.
5. The repair must stay explicitly separate from `CO-138` stale merge-closeout cleanup after leaving `Merging`.
6. The repair must stay explicitly separate from `CO-212` completed-blocker `Ready reclaim`.
7. The repair must not imply any `CO-175` docs-freshness ownership or policy change simply because registry mirrors are touched in this packet.

## Design
- Introduce one authoritative reopened-terminal re-admission predicate that compares retained local release and closeout truth against fresher live issue truth for the same issue.
- Use that predicate before released-claim `replay-block`, stale closeout residue, or equivalent current-state classification can keep the same reopened issue excluded from bounded re-admission.
- Preserve the current released-claim zero-burn behavior when live reopened truth is absent, stale, or contradicted by live same-issue worker evidence.
- Keep any stale closeout supersession bounded to reopened same-issue admission; do not reopen the broader stale `merge_closeout residue` ownership already held by `CO-138`, `CO-214`, or `CO-228`.
- Keep the same no-live-worker proof and duplicate-worker protections already established by the same issue family.

## Current Truth
- Released-claim fail-closed and replay-blocking behavior already protect all-released deferred-poll paths from direct-read churn.
- Stale `merge_closeout residue` and retained released truth can remain locally visible after an issue becomes terminal.
- A later reopen of the same issue can still be stranded if those retained terminal signals continue to outrank fresher runnable truth.
- Adjacent issues already prove the key boundaries exist separately: `CO-138` for stale merge-closeout cleanup after leaving `Merging`, `CO-212` for completed-blocker `Ready reclaim`, `CO-160` / `CO-161` for replay-block and zero-burn release handling, and `CO-175` for docs-freshness ownership.

## Proposed Design
- Add a small shared predicate that answers whether a retained terminal released claim is now eligible for same-issue re-admission because fresher live truth reopened the issue to runnable state and no live same-issue worker remains.
- Evaluate that predicate before:
  - release-family replay-block keeps the same issue out of bounded re-admission
  - stale `merge_closeout residue` continues to classify the same issue as blocked
  - runtime/read-side surfaces, if any, present retained terminal truth as still authoritative for the reopened issue
- Leave existing all-released deferred-poll replay-block behavior untouched when reopened live truth is not present or not trustworthy.
- Keep the repair local to reopened same-issue admission truth instead of broadening merge-closeout, poll policy, or docs-freshness ownership.

## Non-Goals
- Reworking broader merge-closeout cleanup policy.
- Rewriting completed-blocker `Ready reclaim`.
- Redesigning deferred poll or request-budget behavior.
- Weakening same-issue live-worker protection.
- Reopening `CO-175` docs-freshness policy, capacity, or ownership.

## Parity / Alignment Matrix
- Current truth:
  - a reopened same issue can remain suppressed by retained terminal release or closeout truth
  - release-family replay-block can keep `fresh discovery` closed for that same issue even after reopen
  - adjacent lanes separately own stale merge-closeout cleanup, completed-blocker reclaim, no-burn replay-block, and docs-freshness ownership
- Reference truth:
  - reopened live truth should outrank stale retained terminal signals for the same issue
  - replay-block should remain zero-burn for unrelated released claims
  - adjacent issue ownership boundaries should remain explicit
- Target truth / intended delta:
  - reopened same issues become eligible for bounded re-admission when live truth is fresher and no live same-issue worker remains
  - stale `merge_closeout residue` and release-family replay-block no longer strand that same issue
  - `CO-138`, `CO-212`, and `CO-175` remain separate
- Explicitly out-of-scope differences:
  - broad replay-block redesign
  - broad merge-closeout redesign
  - completed-blocker policy change
  - docs-freshness ownership or policy changes

## Not Done If
- A reopened terminal same issue is still stranded behind retained `provider_issue_released:not_active`, stale `merge_closeout residue`, or release-family `replay-block`.
- The repair broadens into `CO-138`, `CO-212`, or `CO-175`.
- The repair weakens same-issue live-worker protection.
- The repair removes release-family zero-burn behavior for unrelated claims.
