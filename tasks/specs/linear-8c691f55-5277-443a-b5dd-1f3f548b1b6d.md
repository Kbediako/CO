# Technical Specification

## Context

`CO-162` is the startup-recovery follow-up after `CO-160`. Current `main` already has:

- the deferred-poll released-claim cached fail-closed handling from `CO-160`
- the dead-active and shared request-headroom work from `CO-159` and `CO-156`
- `mode: recovery_sweep` tracked-issue fetches during startup in `controlServerPublicLifecycle.ts`
- a separate released-claim cached poll helper in `providerIssueHandoff.ts`

The remaining gap is specifically the first startup full recovery sweep. The issue evidence and current code reading show:

- startup schedules an immediate refresh at timeout `0`, and the first pass uses `mode: recovery_sweep`
- startup recovery sweep fetches tracked issues once, then calls `providerIssueHandoff.poll(...)` without enabling the released-claim cached fail-closed path
- `resolveTrackedIssuePollResolutionWithFallback(...)` only applies released-claim cached fail-closed reasons when the poll caller explicitly allows that behavior
- retained released claims that are absent from the recovery-sweep tracked-issue map therefore still fall through to direct `resolveTrackedIssue(...)`, which maps to `dispatch_source_issue_by_id`

This lane must make startup recovery-sweep polls fail closed for locally terminal retained released claims, while preserving the recovery-sweep handling of runnable tracked issues already returned by live Linear.

## Requirements

1. Preserve the already-landed deferred-poll no-burn behavior from `CO-160`.
2. Ensure startup full recovery sweep performs zero direct issue-by-id reads for retained released inactive or non-mutable claims that are absent from the recovery-sweep tracked-issue map.
3. Ensure recovery sweep still processes runnable tracked issues returned by `mode: recovery_sweep` and does not hide genuinely runnable work.
4. Preserve non-deferred explicit refresh or reopen paths and active-run recovery for claims that still need live revalidation.
5. Keep release cleanup and pending-reopen behavior intact for claim classes that are not locally terminal.
6. Distinguish deferred poll, startup recovery sweep, non-deferred explicit refresh or reopen, and active-run recovery in focused regression coverage.
7. Capture a guarded restart proof showing no new `dispatch_source_issue_by_id` entries after the first startup recovery sweep.

## Current Truth

- `ProviderIssueHandoffPollInput` now exposes `trackedIssues`, `refetchTrackedIssues`, `deferFreshDiscovery`, and `allowPollFailClosed`.
- `resolveTrackedIssuePollResolutionWithFallback(...)` already uses:
  - `resolveProviderIssuePollFailClosedReason(...)` for accepted, review-wait, or merge-wait cached cases
  - `resolveReleasedProviderIssuePollFailClosedReason(...)` for locally terminal released claims
- startup callers may set `allowPollFailClosed` to enable released-claim cached fail-closed handling, while broader cached fail-closed reasons still remain tied to `deferFreshDiscovery === true`
- startup full recovery sweep in `controlServerPublicLifecycle.ts` currently calls `providerIssueHandoff.poll(...)` with returned `trackedIssues` plus `refetchTrackedIssues`, but without that released-claim fail-closed opt-in
- ordinary deferred polls already pass `deferFreshDiscovery: true`, so `CO-160` behavior stays bounded to that lane today

## Proposed Design

### 1. Decouple startup fail-closed admission from deferred fresh-discovery behavior

- Add a bounded poll input signal that allows startup recovery-sweep callers to enable the released-claim cached fail-closed helper without implying a broader fresh-discovery policy change.
- Preserve the current deferred-poll behavior so `CO-160` remains intact.

### 2. Keep recovery-sweep tracked issues authoritative

- When startup recovery sweep already returns runnable tracked issues, those tracked issues must continue through the existing owned/start or resume paths.
- Only claims absent from that tracked-issue map should take the cached released-claim fail-closed path during the startup sweep.

### 3. Preserve explicit live revalidation paths

- Non-deferred poll or refresh paths should still be able to resolve a released claim live when newer evidence says the issue is runnable again.
- Existing active-run rehydrate and explicit reopen helpers remain the authoritative recovery seams for those cases.

### 4. Add focused regressions

- Cover the existing deferred-poll all-released case from `CO-160`.
- Add startup recovery-sweep coverage that proves zero direct issue-by-id reads for retained released claims while still handling at least one runnable tracked issue returned by the sweep.
- Keep non-deferred reopen and active-run recovery coverage explicit.
