# Technical Specification

## Context
`CO-103` already improved stage/event/rate-limit truth on current `main`, but the issue’s three remaining STATUS gaps are still live and operator-visible. The terminal header still renders `Agents` using tracked issue count rather than allowed concurrency, the viewer’s 1-second refresh loop does not translate into genuinely live runtime/age/event-recency text because cached rerenders reuse snapshot-derived reference times, and the running-row `EVENT` surface still permits generic dashboard fallback text to outrank richer upstream message/progress text. Symphony’s current terminal/dashboard handling model is explicit on all three seams.

## Requirements
1. Header `Agents` must render `running/max_allowed`.
2. `max_allowed` must come from the same live concurrency/runtime contract CO already uses for provider admission (`max_concurrent_agents`), not from tracked issue count.
3. Header `Runtime` must advance locally every second during live monitoring even when a fresh dataset has not arrived.
4. Running-row `AGE / TURN` must advance locally every second during live monitoring even when a fresh dataset has not arrived.
5. Relative event-age text must advance locally every second during live monitoring even when a fresh dataset has not arrived.
6. Running-row `EVENT` must be message-first and projection-authored upstream.
7. Generic fallback text such as `turn running` or `provider worker turn is active` may only appear when no richer authoritative message/progress text exists.
8. Tests must lock the exact Symphony mapping for agent denominator, live clock behavior, and event authorship.
9. The lane must stay bounded to the post-`CO-103` remainder and not reopen unrelated STATUS work.
10. Final validation must include real screenshot proof embedded directly in Linear.

## Design
- Add the smallest truthful read-model field needed to expose `max_allowed` from the live provider concurrency contract into the compatibility/operator dataset.
- Source that field from the same contract currently read by provider poll dispatch admission (`max_concurrent_agents`), preserving existing defaults/fallback behavior.
- Carry a live wall-clock anchor through the dashboard render path:
  - when a fresh dataset is read, capture the current wall-clock time
  - retain the dataset snapshot as authoritative data
  - on cached rerenders, derive a fresh current reference time from elapsed wall-clock delta since that capture
  - use that re-derived reference time when formatting runtime, running age, and relative event age
- Tighten `compatibilityIssuePresenter.ts` so `display_event` is the authoritative message-first running event field, with raw event and timestamp preserved as secondary semantics/debug context.
- Keep `controlStatusDashboard.ts` mostly responsible for formatting authoritative fields, not for deciding event meaning from shallow fallback heuristics.
