# Technical Specification

## Context

`CO-79` follows the `CO-78` lane after contradictory provider-worker closeout surfaced on a real issue attempt. The main lane could fail terminally while related summaries and workpad text still implied a clean or review-ready outcome, deterministic invalid follow-up creation (`linear_follow_up_parity_matrix_missing`) was retried in the same attempt, and auxiliary proof helper manifests remained `in_progress` long enough to pollute operator-facing run status.

The repair must keep the lane bounded: authoritative terminal truth plus authoritative review telemetry should shape closeout everywhere that matters, deterministic invalid provider mutations must stop retrying within the attempt, and proof sidecars must be finalized or excluded from active provider truth surfaces.

## Requirements

1. A failed or degraded provider-worker lane must read consistently across manifest-backed closeout surfaces, including manifest-derived summary state, run-summary text, out/state rollups, and workpad closeout content.
2. Closeout wording must reflect authoritative review telemetry. `bounded-success` is successful bounded completion; `failed-boundary` is an explicit wrapper-boundary failure; `failed-other` is a review-command failure without mislabeling unrelated blockers.
3. Deterministic invalid provider mutations must be classified as non-retryable within the same attempt.
4. Auxiliary proof harness runs must not appear as active provider-worker work when they are helper artifacts rather than the main lane.
5. Focused regression coverage must reproduce the `CO-78` class: partial progress before terminal failure, deterministic invalid mutation, proof sidecar artifacts, and closeout truth agreement.
6. The fix must stay bounded to the current provider-worker closeout/reporting path and not weaken existing validation contracts.

## Protected Expectations

- A failed provider-worker lane must read as failed everywhere that matters.
- Deterministic invalid mutations are not transient noise and should not be retried blindly.
- Auxiliary proof runs must not pollute operator-facing run truth.
- Review telemetry classifications must remain explicit and authoritative in closeout.

## Reject These Wrong Interpretations

- It is acceptable if the manifest fails but the workpad still reads as clean-success.
- Retrying a parity-matrix-missing follow-up in the same attempt is harmless.
- Manual proof helper runs can stay `in_progress` forever because they are not the main lane.
- Any non-null review telemetry means the lane should sound successful.

## Current Truth

- Provider-worker proof carries attempt state, child-stream or child-lane lineage, and review-adjacent evidence that nearby surfaces consume.
- `CO-78` showed command-runner closeout and manifest-derived summary could disagree: the stage summary recorded failure while command-level closeout still said the provider worker completed cleanly for handoff.
- Follow-up creation already validates parity lanes fail-closed with `linear_follow_up_parity_matrix_missing`, but the provider-worker lane could still retry deterministic invalid writes within the same attempt.
- Runtime/status truth reads provider-worker proof and sibling run manifests, so lingering helper-sidecar `in_progress` manifests can pollute active-running surfaces if not finalized or excluded.

## Proposed Design

### 1. Make one terminal result authoritative

- Introduce a shared provider-worker truth helper that converts proof state plus review telemetry into one terminal status, reason, and summary string.
- Ensure command-runner closeout and selected-run/status projection consume that helper instead of inferring success from completed milestones or stale manifest text.
- Keep review telemetry classification as a secondary explicit input, not a substitute for terminal lane status.

### 2. Preserve review telemetry truth in closeout

- Map `clean-success`, `bounded-success`, `failed-boundary`, and `failed-other` to truthful closeout wording without collapsing them into generic success or wrapper-failure claims.
- Ensure failed terminal lanes remain failed even if earlier review or milestone text was positive.

### 3. Suppress deterministic invalid mutations

- Detect deterministic invalid provider write outcomes such as missing parity matrix on follow-up creation.
- Carry that classification through the same attempt and render it into the next provider-worker prompt so already-proven invalid mutations are not retried blindly.
- Leave retryable or transient failures untouched.

### 4. Remove proof-sidecar noise from runtime truth

- Detect auxiliary proof harness runs such as `manual-live-proof` via shared truth helpers and exclude them from active provider-worker/runtime truth projection.
- Preserve real parent worker truth while hiding helper noise.

### 5. Add regression coverage for mixed partial-success / terminal-failure truth

- Cover the specific `CO-78` class in focused tests instead of relying only on one surface’s behavior.
- Verify closeout text, mutation suppression, and runtime truth together where possible.

## Non-Goals

- Historical artifact rewrite/backfill for already completed tasks.
- Redesigning the workpad model or review wrapper architecture.
- Generic status/report copy polishing unrelated to truthfulness.
