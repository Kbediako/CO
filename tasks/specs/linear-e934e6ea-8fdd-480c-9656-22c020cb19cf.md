# Technical Specification

## Context

`CO-156` is a bounded follow-up to the shared-budget work already landed in `CO-62`, `CO-110`, and `CO-144`, plus the live evidence captured in `CO-147`. The current tree already has:

- shared Linear budget persistence keyed by user/token scope in `linearBudgetState.ts`
- reservation-aware and complexity-aware helper preflight
- request-aware polling slowdown and cooldown precedence in `resolveLinearPollingInterval(...)`
- helper-side validation and no-op workpad avoidance in `providerLinearWorkflowFacade.ts`
- live request-burn evidence proving requests, not complexity, are the scarce resource

The remaining gap is the reserve boundary itself. The current shared preflight blocks only when budget is already exhausted, cooldown is active, or an operation-specific minimum is smaller than remaining budget. That still lets nonessential callers spend protected shared request headroom during low-headroom operation, especially on:

- issue-by-id refresh during restart/rehydrate/reconcile
- recovery sweep and fresh discovery tracked-issue reads
- helper entrypoints whose planned request count is smaller than the configured reserve
- shared-budget observations that overwrite the latest snapshot without preserving burn deltas over time

This lane must make the shared request reserve authoritative for those consumers while preserving safe local fallbacks and existing landed contracts.

## Requirements

1. Keep the landed shared-budget substrate intact: user/token scope resolution, reservation-aware materialization, explicit cooldown precedence, helper validation, webhook-first targeted reconcile, and truthful fail-fast semantics.
2. Add a central reserve-aware request-headroom gate in `linearBudgetState.ts` that can fail nonessential live Linear requests before `fetch` when remaining request headroom would violate the configured reserve.
3. Route helper operations (`issue-context`, `upsert-workpad`, `transition`, `attach-pr`, `create-follow-up`, and similar) through that gate without regressing existing safe cache or validation behavior.
4. Route dispatch-source operations (`dispatch_source_tracked_issues`, `dispatch_source_tracked_issues:recovery_sweep`, `dispatch_source_tracked_issues:fresh_discovery`, `dispatch_source_issue_by_id`) through that same gate so restart/rehydrate/reconcile cannot spend reserve-blind live requests.
5. Preserve local validation before live read/mutation for `create-follow-up`, and preserve workpad noop/coalescing behavior for unchanged bodies.
6. Persist shared request-burn telemetry with enough fields to attribute burn over time by source, run_id, process, operation, request id, remaining delta, reset time, and cooldown reason.
7. Keep the telemetry available to local diagnostics or existing status consumers without requiring a separate manual evidence lane.

## Current Truth

- `resolveLinearBudgetPreflight(...)` currently rejects cooldown, explicit request shortfall, explicit complexity shortfall, and exhausted buckets, but it does not yet protect the configured shared request reserve.
- `reserveLinearBudgetReservation(...)` already accounts for concurrent reservations, so the shared governor is the right place to apply a stricter reserve gate.
- `providerLinearWorkflowFacade.ts` already validates `create-follow-up` payload pieces before session resolution and live reads, and `upsert-workpad` already noops unchanged workpads instead of re-mutating Linear.
- `linearDispatchSource.ts` already uses shared preflight before every tracked-issue query and issue-by-id refresh, so strengthening shared preflight should cover recovery, fresh discovery, and rehydrate callers without inventing a second policy.
- `providerIssueHandoff.ts` already treats skipped refresh resolutions as local no-op or release paths, which means reserve-aware query failures can safely defer live metadata refresh instead of corrupting local state.
- `CO-147` proved that the shared budget snapshot records source, request id, reset times, cooldown state, and endpoint aliases, but it does not yet preserve a shared time-series request-burn history.

## Proposed Design

### 1. Shared reserve-aware preflight

- Extend `resolveLinearBudgetPreflight(...)` so request buckets fail fast not only on exhaustion and explicit minimum shortfall, but also when a nonessential operation would spend below the configured shared request reserve.
- Reuse the existing request reserve math already used by polling slowdown so helper and dispatch preflight share one source of truth.
- Preserve explicit cooldown and explicit shortfall semantics as higher-signal failures when they apply.

### 2. Shared request-burn telemetry

- Extend shared budget observation recording to persist bounded request-burn telemetry entries alongside the latest snapshot.
- Each entry should include:
  - observed/recorded time
  - source and normalized operation
  - run id from environment when available
  - process metadata
  - request id
  - selected request bucket and remaining delta
  - reset time
  - cooldown or suppression reason
- Keep the history bounded so the shared artifact stays practical for local diagnostics.

### 3. Restart/rehydrate/reconcile behavior

- Preserve the current local-state-first rehydrate/refresh behavior in `providerIssueHandoff.ts`.
- Let reserve-aware query preflight prevent live tracked-issue reads when the shared reserve is not healthy, so restart and rehydrate continue locally but defer nonessential metadata refresh.
- Keep recovery sweep and fresh discovery fail-closed before `fetch` under the same shared reserve gate.
