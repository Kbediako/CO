# Technical Specification

## Context

`CO-54` exists because validation in a provider-worker-adjacent lane surfaced two different ways a test can stop being hermetic. The first is host/provider repo-config state leaking into what should be a temp-fixture load path. The second is a temp-workspace CLI test silently depending on package-root assets outside the fixture. On the current tree, the first seam appears to be mostly fixed already for the simple local case, while the second seam is the active reproducer: `tests/cli-frontend-test.spec.ts` launches `frontend-test` from a temp root with no local `codex.orchestrator.json`, so the CLI falls back to the package config and then shells into `node "$CODEX_ORCHESTRATOR_PACKAGE_ROOT/dist/orchestrator/src/cli/frontendTestingRunner.js"`, which fails in this checkout because `dist/` is not present.

## Requirements

1. `orchestrator/tests/UserConfigStageSets.test.ts` must prove that temp-fixture config remains authoritative even when provider-style host snapshot env is present.
2. `tests/cli-frontend-test.spec.ts` must pass from a temp workspace without requiring host `.runs/local-mcp` state, repo-root `dist/` artifacts, or package-fallback-owned guardrail assets.
3. The `frontend-test` CLI test must remain on the real command entrypoint; do not replace it with a lower-level unit test that no longer covers CLI request shaping/runtime output.
4. Keep the patch narrowly scoped to the two cited tests and any immediate fixture helpers they need.
5. Record the narrowed current truth honestly in docs/workpad/checklists so later workers do not reopen the broader older hypothesis.

## Current Truth

- `orchestrator/tests/UserConfigStageSets.test.ts` now writes both a temp-fixture config and a provider-style host snapshot path, then proves `loadUserConfig` still resolves the temp-fixture `diagnostics` pipeline when hostile provider env vars are present.
- `tests/cli-frontend-test.spec.ts` now writes a temp-local `codex.orchestrator.json` for the default/devtools/runtime cases so the real `frontend-test` CLI executes a self-contained fixture stage and records `CODEX_REVIEW_DEVTOOLS` into a temp log instead of falling back to repo-root package assets.
