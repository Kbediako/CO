# Technical Specification

## Context
`CO-116` added truthful `In Review` -> `Merging` review-handoff promotion. `CO-104` added bounded historical attached-PR disambiguation for deterministic `Merging` closeout. Those slices are real and should remain intact.

`CO-237` is the remaining review-handoff sibling. A provider-owned issue can still remain in `In Review` with top-level claim reason `provider_issue_review_promotion_action_required` and nested review-promotion reason `multiple_attached_prs` when same-repo attachments include non-primary PRs. The current checksum is not hypothetical:

- `CO-196` carries closed superseded PRs `#508` and `#515`
- `CO-219` carries current PR `#523` plus auxiliary `#522`, whose title uses `blocking CO-219 handoff` wording rather than the narrower exact `blocker` wording

Current review-handoff promotion still routes these shapes through the shared attached-PR selector in `providerMergeCloseout.ts`, then persists the refusal through `providerIssueHandoff.ts`. That means the issue can stay parked in `In Review` even when there is one truthful current candidate and the remaining attachments are bounded non-primary baggage rather than real ambiguity.

## Requirements
1. Reproduce the `CO-196` checksum where closed `#508` and `#515` remain attached and current review-handoff truth still resolves to `provider_issue_review_promotion_action_required` / `multiple_attached_prs`.
2. Reproduce the `CO-219` checksum where current `#523` coexists with auxiliary `#522` and current review-handoff truth still resolves to generic `multiple_attached_prs`.
3. Review-handoff promotion must not keep the issue parked in `In Review` when one truthful current candidate exists and the remaining same-repo attachments are bounded non-primary baggage.
4. Real ambiguity must remain explicit as `multiple_attached_prs` with machine-checkable conflicting truth.
5. Existing `CO-104` historical merged-attachment behavior and ordinary `Merging` closeout semantics must remain intact.
6. Manual attachment cleanup must not remain the intended steady-state product path for these bounded shapes.
7. The repair must remain explicitly separate from `CO-154` attachment-mutability parity and `CO-226` full-suite timeout stabilization.

## Design
- Keep one authoritative attached-PR current-candidate truth instead of adding a review-handoff-only heuristic that drifts from merge-closeout logic.
- Expand or refine the current candidate-selection seam so it can distinguish real ambiguity from bounded non-primary same-repo attachments:
  - closed superseded attempts such as `closed #508` and `#515`
  - auxiliary attached PRs such as `#522` when the issue's truthful current candidate is `#523`
- Preserve explicit `multiple_attached_prs` fail-closed behavior when more than one current candidate truly remains after bounded filtering.
- Preserve or extend additive proof surfaces so parent-visible artifacts still explain what was ignored, what stayed conflicting, and why review-handoff promotion proceeded or refused.

## Current Truth
- `providerMergeCloseout.ts` already owns the shared same-repo attached-PR selection logic.
- That logic can ignore older merged baggage from `CO-104`, but still returns `multiple_attached_prs` when more than one non-primary same-repo attachment remains.
- `providerIssueHandoff.ts` reuses that logic during review-handoff promotion and persists the refusal as `provider_issue_review_promotion_action_required`.
- That means provider-owned issues can stay in `In Review` on generic ambiguity even when the remaining extra attachments are bounded, concrete baggage rather than real current-candidate conflict.

## Proposed Design
- Keep the shared selection seam but add the smallest truthful distinction needed for bounded non-primary attachments.
- Use the same distinction before review-handoff promotion persists `provider_issue_review_promotion_action_required`.
- As implemented for the reopened `CO-219` regression, review-promotion cross-issue filtering treats `block`, `blocked`, `blocker`, and `blocking` wording as blocker evidence when the attachment's leading issue key is the issue that blocks the current lane; follow-up wording remains ignored cross-issue baggage.
- Preserve or extend additive proof so parent-visible records can still show which attached PRs were treated as current, ignored, or still conflicting.
- Leave ordinary `Merging` closeout unchanged unless the same bounded selection rule is reused there without reopening broader scope.
