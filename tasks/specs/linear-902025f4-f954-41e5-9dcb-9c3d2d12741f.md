## Technical Requirements
- Functional requirements:
  1. Define one explicit authority mode with at least `repo-authoritative` and `downstream-compatibility` values.
  2. Make repo-authoritative mode require repo-local config and canonical CO pipelines by default.
  3. Deny packaged fallback config/pipelines in repo-authoritative mode.
  4. Permit packaged fallback config/pipelines only through explicit downstream compatibility mode or adapter.
  5. Ensure operator output names the active mode and why it was selected.
  6. Ensure manifest/status output carries machine-readable mode, reason, config source, pipeline source, and fallback allowance.
  7. Add separate CI coverage for repo-authoritative fail-closed behavior and downstream compatibility success behavior.
- Non-functional requirements:
  - keep behavior auditable for parent review and release decisions
  - avoid breaking downstream compatibility without migration
  - keep source changes narrowly scoped to config/pipeline resolution, output, and tests
- Interfaces / contracts:
  - `codex.orchestrator.json` remains the repo-local config authority in repo-authoritative mode
  - packaged fallback config/pipelines remain compatibility-only
  - manifest/status/operator output must not collapse the two modes into a generic fallback warning
  - `config_resolution` is the machine-readable manifest and plan-output field carrying `mode`, `reason`, and `config_source`
