# Technical Specification

## Context
`CO-152` follows an incident where a second same-task `control-host --task local-mcp --run control-host --pipeline provider-linear-worker` process appeared alongside the launchd-supervised host while sharing the same CO root and `.runs/local-mcp/cli/control-host/` provider-intake state. The local intake snapshot already exposes polling stuck and Linear budget data, but startup has no same-task ownership guard, so a duplicate process can plausibly enter provider refresh scheduling before the operator gets a distinct duplicate-owner diagnosis.

## Requirements
1. Control-host startup must acquire a same-task provider-intake ownership lock before provider refresh polling is scheduled.
2. The lock identity must include repo root, run directory, task id, run id, pipeline id where available, pid, hostname, process start timestamp, and metadata file path.
3. Startup must fail closed when an active same-task owner already owns the lock.
4. Startup may clean up a stale lock only when process liveness proves the recorded owner is not active.
5. Lock acquisition, duplicate rejection, and stale cleanup must not kill provider workers and must not remove, rewrite, or release provider-intake claims.
6. Operator diagnostics must include owner metadata and a classified reason distinct from provider cooldown, GitHub throttling, and Linear API exhaustion.
7. Provider polling/stuck diagnostics and runtime/status reads should expose duplicate-owner metadata additively when a duplicate or stale owner condition is relevant.
8. Shutdown must release only the current process owner lock and must leave another owner untouched if ownership changed unexpectedly.

## Design
- Add a small control-host ownership module under `orchestrator/src/cli/control/`.
- Use an atomic owner lock directory or exclusive file create under the control-host run directory, with a JSON metadata artifact such as `control-host-owner.json`.
- Treat the lock scope as the resolved repo root plus `paths.runDir` plus `runId`; include pipeline id from config or environment when available without making that field mandatory for existing tests.
- Acquire the owner in `startControlServerPublicLifecycle(...)` before `startControlServerReadyInstanceLifecycle(...)` and before `scheduleStartupProviderRefresh(...)`.
- On active duplicate, throw a structured `duplicate_control_host_owner` error carrying the existing owner metadata and current attempted owner metadata.
- On stale owner, persist a stale-owner diagnostic, remove only the stale lock path, and then acquire ownership for the current process.
- Store latest ownership metadata in the run directory so an operator can inspect it even if `co-status` cannot connect to the duplicate process.
- Thread owner metadata into provider polling health or compatibility runtime snapshots additively when a duplicate or stale owner was detected.
- Release the lock during `closeControlServerPublicLifecycle(...)` only when the current token still owns it.

## Protected Expectations
- Do not kill duplicate or owner processes.
- Do not kill provider workers.
- Do not wipe provider-intake claims.
- Do not relabel Linear API budget or GitHub API throttling as duplicate-host ownership.
- Do not remove launchd supervision or require manual restart as the durable behavior.

## Reject These Wrong Interpretations
- "The fix is to pkill stale control-host processes."
- "Provider-intake claims should be cleared when duplicate ownership is detected."
- "A duplicate host can continue polling if it logs a warning."
- "CO-151 GitHub budget backoff should be implemented here."

## Parity / Alignment Matrix
- Not applicable as a formal parity lane.
- Current truth: same-task `control-host` processes can start against the same run directory without an ownership guard before provider refresh scheduling.
- Reference truth: a single owner should acquire durable metadata before polling and duplicates should fail closed against active owner identity.
- Target truth / intended delta: atomic same-task ownership, stale-owner cleanup, owner diagnostics, and focused tests while preserving active workers and provider claims.
- Explicitly out-of-scope differences: API budget policy, broad process cleanup, launchd replacement, and provider claim reconciliation changes.

## Not Done If
- Two same-task control-host processes can poll the same provider-intake state concurrently.
- Duplicate startup proceeds past provider refresh scheduling.
- The only diagnostic is a generic `provider_refresh_lifecycle_stuck` or timeout.
- Stale cleanup can delete provider-intake claims or terminate provider workers.
