## Metadata-Safe Matching Requirements
- A child-lane manifest may hydrate a reservation only when all available lineage fields agree:
  - `parent_run_id` matches the parent provider-worker run id.
  - `issue_id` matches the child-lane ledger record and parent issue.
  - `issue_identifier` matches the child-lane ledger record and parent issue.
  - `pipeline_id` is exactly `provider-linear-child-lane`.
  - child `task_id` matches the reservation record, while the manifest `run_id` is a valid real child run id inside the expected child task artifact tree and may replace a synthetic `launching-*` reservation id in the hydrated proof snapshot.
- A missing, unreadable, malformed, or lineage-mismatched manifest must not be used as active progress.
- Hydration must not scan or attach unrelated historical child runs solely by stream name, filesystem recency, or Linear `updated_at`.
