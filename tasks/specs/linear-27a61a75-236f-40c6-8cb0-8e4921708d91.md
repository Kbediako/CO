# Technical Specification

## Context
`CO-220` is the reset-attempt sibling of the stale merged-closeout cleanup lanes. The live issue can already have reset into a fresh `Rework` attempt with no new PR attached, while local rehydrated claim/proof truth still carries the retired attempt forward as current `running` / `Merging` state and continues surfacing a closed stale attached PR such as `#509`. The result is a cross-surface truth split:

- `co-status` and intake-derived projection can still look like the retired attempt is active or merging.
- `issue-context` can still surface the stale closed attachment as if it were the current attached PR.
- historical evidence exists, but it is not clearly separated from current truth.

The repair must make newer reset truth authoritative without weakening real live `Merging` lanes or real attached PR truth.

## Requirements
1. Reproduce the reset shape where a prior attempt is retired, its stale PR attachment is closed, the live issue is `Rework`, and no new PR is attached yet.
2. Current-truth selection for operator-facing projection must prefer newer tracked/live `Rework` truth over stale rehydrated `running` / `Merging` state from the retired attempt.
3. `issue-context` must stop presenting the stale closed attachment `#509` as the current attached PR when the reset attempt has no new PR yet.
4. Historical evidence for the retired attempt must remain inspectable for audit/debug purposes.
5. Real live `Merging` issues and real current attached PRs must continue to project correctly.
6. The lane must stay bounded and not regress `CO-210`, `CO-214`, or `CO-219`.

## Design
- Introduce one bounded current-truth rule for reset attempts: when tracked/live issue truth has advanced from the retired attempt into newer `Rework` with no new attached PR, stale rehydrated active/merge-closeout state becomes historical-only.
- Apply that rule consistently across:
  - selected-run / `CO STATUS` projection
  - provider issue observability or related projection ranking
  - `issue-context` attachment/current-PR presentation
- Preserve the existing historical evidence path by separating current attached PR truth from historical attachments rather than deleting raw history.
- Keep current `Merging` / attached PR truth untouched whenever the live issue is actually still `Merging` or a live current PR remains attached.

## Protected Expectations
- Preserve the exact surfaces `CO STATUS`, intake-derived projection, `issue-context`, `Rework`, stale rehydrated `Merging`, stale rehydrated `running`, closed PR attachment, `#509`, `no attached PR yet`, and historical evidence.
- Preserve real live `Merging` behavior and real current attached PR truth.
- Preserve historical inspectability for audit/debug.
- Prefer a narrow current-truth predicate over broad workflow redesign or artifact deletion.

## Reject These Wrong Interpretations
- `This is just a CO STATUS display bug.`
- `This is only issue-context attachment filtering.`
- `Delete or hide all historical stale attachments.`
- `Any closed attached PR means the issue has no attached PR truth.`
- `Suppress Merging truth whenever Rework exists anywhere in history.`

## Current Truth
- Newer tracked/live `Rework` truth can exist while rehydrated claim/proof truth still projects the retired attempt as current `running` / `Merging`.
- Stale merge-closeout/current-PR ranking does not yet clearly treat non-terminal reset truth as authoritative.
- `issue-context` exposes raw attachments but does not yet clearly separate current attached PR truth from historical stale attachments for reset attempts.

## Proposed Design
- Broaden the stale-current-truth suppression seam so newer live `Rework` truth can demote stale rehydrated merge-closeout/active state, not only newer terminal truth.
- Add or reuse a current-vs-historical attached PR classifier so `issue-context` can report `no attached PR yet` as current truth while retaining stale closed attachments as historical evidence.
- Reuse the same current-truth rule across projection and `issue-context` so the surfaces do not drift.

## Parity / Alignment Matrix
- Current truth:
  - live issue can be `Rework`
  - stale rehydrated `running` / `Merging` can still rank as current
  - stale closed attachment `#509` can still read like the current attached PR
- Reference truth:
  - a reset attempt with no new PR should read as `Rework` plus `no attached PR yet`
  - historical stale attachments remain inspectable but not current
  - live `Merging` and live attached PRs should still read as current where real
- Target truth / intended delta:
  - current `Rework` reset truth outranks stale rehydrated `running` / `Merging`
  - `issue-context` exposes `no attached PR yet` as current truth for the reset attempt
  - historical stale attachment `#509` remains inspectable without being current
- Explicitly out-of-scope differences:
  - `CO-210` child-lane hydration/progress changes
  - `CO-214` stale merged-closeout cleanup
  - `CO-219` dead-proof running-count fixes
  - historical artifact deletion

## Not Done If
- Reset `Rework` truth still loses to stale rehydrated `Merging` / `running` on any relevant surface.
- `issue-context` still implies closed stale attachment `#509` is current for the reset attempt.
- The fix hides real live `Merging` or real attached PR truth.
- The fix only changes one surface and leaves the others split.
