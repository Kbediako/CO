# Technical Specification

## Context

`CO-50` extracted `parseTrailingJsonObject(...)` into `orchestrator/src/cli/utils/trailingJsonObject.ts` so the provider-worker child-stream seam and delegation-server spawn seam would stop drifting. `providerLinearChildLaneShell.ts` still carries a third local parser copy, and that copy is looser because it will attempt line-suffix parsing without first proving the trimmed output ends in `}`. `CO-68` is the bounded follow-up that finishes the deduplication by moving the child-lane seam onto the same shared helper while preserving the existing child-lane output and confinement contracts.

## Requirements

1. Replace the local trailing JSON-tail parser in `providerLinearChildLaneShell.ts` with the shared helper from `orchestrator/src/cli/utils/trailingJsonObject.ts`.
2. Preserve fail-closed behavior:
   - object-only parse
   - direct full-string parse first
   - strict trailing `}` contract before suffix scanning
3. Keep the child-lane `parseProviderChildLaneRunResult(...)` contract unchanged: success still normalizes into `ProviderLinearChildLaneRunResult`, and parse failure still returns `null`.
4. Preserve existing child-lane run-root and path confinement validation for `artifact_root`, `manifest`, and optional `log_path`.
5. Add focused regression coverage so the child-lane seam proves:
   - success with prelude logs before a valid final JSON object
   - fail-closed handling for malformed or truncated final payload
6. Stay bounded to the parser replacement, focused tests, and truthful docs/task updates.

## Planned Interaction Contract

- Shared helper contract:
  - input: raw stdout string
  - output: `Record<string, unknown> | null`
  - behavior:
    - trim the input
    - try parsing the full trimmed string first
    - in strict default mode, reject when the trimmed output does not end with `}`
    - if full-string parse fails, scan suffixes that begin on lines whose trimmed content starts with `{`
    - return the first successfully parsed object suffix, else `null`
- Child-lane adaptation:
  - use the helper in strict default mode
  - keep local field normalization and safe `run_id` plus path-confinement validation unchanged

## Pre-change Baseline

- `providerLinearChildLaneShell.ts` currently uses a local `parseTrailingJsonObject(...)` that scans from any line beginning with `{` and parses the remainder, even when the overall trimmed output does not end with `}`.
- `orchestrator/src/cli/utils/trailingJsonObject.ts` already exists from `CO-50` and holds the stricter shared-helper contract that the issue wants child lanes to adopt.
- `ProviderLinearChildLaneShell.test.ts` already proves successful parsing when prelude logs appear before the final JSON object, but it does not yet cover malformed-final-payload failure for the child-lane seam.
