# Technical Specification

## Context
The useful local supervision behavior already exists, but only as one operator-owned shell script and one LaunchAgent outside the repo. That setup keeps the root `control-host` alive under launchd, sources local env/bootstrap files, launches with an explicit Node executable, polls `co-status`, and exits after repeated unhealthy samples so launchd restarts it. The missing piece is packaging and supportability: there is no installable CO command surface, no configurable rendered artifact path, and no machine-checkable status surface for restart reasons.

## Requirements
1. CO must ship a supported macOS launchd supervision surface for the root `control-host`.
2. The shipped path must not hard-code a specific user home, repo root, or Homebrew Node path in repo-tracked source.
3. Install-time defaults may discover current machine paths, but those values must be configurable and rendered into local artifacts rather than baked into shared source.
4. The supervision runner must preserve env/bootstrap sourcing, explicit runtime selection, `co-status` polling, and unhealthy restart threshold behavior.
5. Install or uninstall or status or restart flows must be machine-checkable CLI operations.
6. Failure and restart reasons must be visible through logs or status output.
7. The lane must stay separate from queue-management, review-state, and merge-policy behavior.

## Design
- Add a nested CLI family for supervision management, likely `control-host supervise <subcommand>`.
- Persist a local supervision config plus local state file under a user-local support directory.
- Generate and manage a LaunchAgent plist that calls the packaged CLI plus a supervision `run` subcommand.
- Use current-runtime defaults such as `process.execPath` and current CLI entrypoint or package-root discovery instead of fixed Homebrew assumptions.
- Launch the child `control-host` through a shell-owned bootstrap step so existing env files remain source-compatible.
- Poll `co-status --format json`, track consecutive `restart_required` samples, and write explicit restart-reason state before exiting for launchd restart.
- Keep status reporting bounded to supervision truth: config, launchd presence, local state, logs, and optional current health.

## Implementation Surface
- Expected codepaths:
  - `bin/codex-orchestrator.ts`
  - `orchestrator/src/cli/controlHostCliShell.ts`
  - new supervision CLI and helper modules under `orchestrator/src/cli/`
  - supporting tests under `orchestrator/tests/`
- Expected docs:
  - `README.md` or public downstream docs if command usage needs front-door documentation
- Expected tests:
  - top-level help coverage for the new command family
  - focused supervision install/config/plist/status/runtime tests

## Protected Expectations
- Preserve the current plain `control-host` start path for existing operators who do not use launchd supervision.
- Preserve the current health semantic: repeated `co-status.polling.restart_required === true` should cause a bounded restart.
- Keep queue-management and merge-policy behavior out of the supervision runner.
- Keep local-artifact ownership explicit so uninstall only removes generated files the CLI owns.

## Reject These Wrong Interpretations
- "just commit the existing hard-coded shell script into the repo"
- "ship only docs and leave install steps manual"
- "use the supervision runner to decide which issues to pick up or merge"
- "this issue should also include systemd or Windows service management"

## Current Truth
- The host-local supervisor script sources `~/.local/bin/env` and `~/.co_provider_env`, exports a Homebrew-flavored PATH, launches `control-host`, polls `co-status`, and exits `75` after repeated unhealthy samples.
- The current LaunchAgent uses `KeepAlive`, `RunAtLoad`, and log paths under `~/Library/Logs/co-control-host/`.
- None of that behavior is a repo-tracked, installable CO CLI surface today.

## Proposed Design
- Install command writes:
  - supervision config JSON
  - supervision state/status JSON path
  - LaunchAgent plist
  - log directory when needed
- Runtime `run` command:
  - loads config
  - launches `control-host` under the configured shell bootstrap
  - polls health on the configured cadence
  - records reasons and exits for launchd restart on threshold breach
- Status command:
  - reports config presence, launchctl state, local state file contents, and log paths in JSON or text
- Restart or uninstall commands:
  - use launchctl for bounded lifecycle operations and return explicit structured results

## Non-Goals
- Queue-management automation
- Merge or review orchestration
- Non-macOS service managers
- Reworking the underlying `control-host` provider logic

## Parity / Alignment Matrix
- Current truth:
  - local supervision exists, but only as copied host-local files
- Reference truth:
  - supervision should be a supported CLI-managed package surface
- Target truth / intended delta:
  - operators install and manage supervision from CO itself
- Explicitly out-of-scope differences:
  - queue policy
  - merge policy
  - non-macOS services

## Not Done If
- The final path still requires copying a raw script from one machine to another.
- Repo-tracked source still embeds one machine's repo root or Node path.
- Restart reasons remain opaque to the packaged status surface.
- The command surface mutates anything beyond root-host supervision and health.
