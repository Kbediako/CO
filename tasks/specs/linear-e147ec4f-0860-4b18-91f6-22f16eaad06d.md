# Technical Specification

## Context

CO's worker-visible Linear helper surface already covers reads, workpad management, state transitions, and PR attachment, but it still lacks the last Symphony workflow tool needed for autonomous scope control: filing a separate follow-up issue in the same project when out-of-scope improvements are discovered. The missing piece is not a generic GraphQL tunnel; it is a bounded provider-worker helper that keeps creation scoped to the current issue's team/project and returns enough structured data for workpad/audit usage.

## Requirements

1. Preserve the worker-owned tracker mutation boundary:
   - keep follow-up creation inside the existing provider-worker Linear helper surface
   - do not move issue creation into provider reconciliation or generic completion logic
2. Add a new worker-visible `create-follow-up` helper contract that:
   - requires the source issue id
   - requires title, description, and acceptance criteria
   - defaults the new issue into the live team's `Backlog` state
   - preserves the current team/project assignment
   - always creates a `related` relation to the source issue
   - optionally creates a source-to-follow-up `blocks` relation when the follow-up depends on the source issue
3. Fail closed when:
   - the source issue is missing
   - the source issue does not belong to the expected scoped team/project
   - the source issue has no project
   - the live team does not expose `Backlog`
4. Return created issue metadata sufficient for worker flow usage:
   - issue id
   - identifier
   - URL
   - title
   - state/team/project summary
   - relation metadata
5. Return truthful failure details when create/relation calls only partially succeed.
6. Extend prompt/skill docs so workers are told to create a follow-up issue instead of widening scope.
7. Preserve existing audit logging and add a new operation entry for the new helper.

## Current Truth

- Current Symphony:
  - `SPEC.md` keeps ticket writes as worker-owned optional tooling rather than orchestrator-owned tracker mutations.
  - `elixir/README.md` explicitly points adopters at the workflow plus repo-local `linear` skill.
  - `elixir/WORKFLOW.md` requires a separate same-project `Backlog` follow-up issue with `related` and optional blocker linkage when meaningful out-of-scope work is discovered.
- Current CO:
  - `providerLinearWorkflowFacade.ts` exposes `issue-context`, `upsert-workpad`, `delete-workpad`, `transition`, and `attach-pr`, but no issue creation path.
  - `linearCliShell.ts` and `providerLinearWorkflowAudit.ts` only understand those existing operations.
  - `providerLinearWorkerRunner.ts` prompt and `skills/linear/SKILL.md` do not describe a follow-up issue helper.
- Live schema truth:
  - Linear exposes `issueCreate`, `issueRelationCreate`, `IssueCreateInput`, and `IssueRelationType={blocks, duplicate, related, similar}`.
  - Existing CO/Symphony blocker reads prove that `blocks` must be created from the blocker issue to the blocked issue.
