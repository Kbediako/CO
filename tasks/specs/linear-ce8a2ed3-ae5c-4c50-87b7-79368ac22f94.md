## Objective
Make docs freshness preventive and lifecycle-driven by joining terminal task state, docs freshness metadata, archive automation, scheduled maintenance, repo-gate status, and provider intake/handoff around one strict lifecycle model.

## Requirements
1. Packet creation and registration paths write or validate source issue, doc class, lifecycle state, owner, created_at, last_review, cadence_days, and next_review metadata.
2. Terminal task/Linear state classifies every packet surface (`.agent/task`, PRD, TECH_SPEC, ACTION_PLAN, `tasks/specs`, `tasks/tasks-*`, and `docs/findings`) before ordinary active stale classification.
3. Mechanical completed-packet archive/reclassification produces deterministic dry-run actions and a self-heal PR path where workflow credentials permit it.
4. Public/current/shipped docs never enter rolling deferral and receive direct pre-expiry action routing.
5. `docs:freshness:maintain -- --format json` exposes actionable forecast and repo-gate context: severity, owner state, spec_guard, capacity, next expiry, action-required counts, and action path.
6. `co-status --format json` and status UI expose `repo_gates.docs_freshness_maintain` without counting repo-gate owner state as provider WIP.
7. Provider intake/handoff records repo-gate context early and distinguishes unrelated-lane blocks from handoff/doc-touch blocks.
8. Guide/catalog declared-cohort parity is machine checked, including the `co-420-apr-28-march-28-task-packet-mirror` drift candidate.

## Non-Requirements
- No weakening of `docs:freshness`, `docs:freshness:maintain`, `spec-guard`, `docs:check`, archive policies, provider-worker review gates, or canonical `next_review` metadata.
- No blind `last_review` bumps.
- No stale historical packet deletion.
- No duplicate canonical-owner issue creation.
- No CO-431 or CO-522 scope expansion without fresh regression evidence.
