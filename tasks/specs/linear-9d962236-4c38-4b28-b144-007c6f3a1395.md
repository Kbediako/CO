# Technical Specification

## Context

CO already instructs provider workers to sweep PR feedback before moving an issue into `In Review`, but the current contract still leaves a gap between "latest PR update pushed" and "review handoff now." The existing PR monitor is strong enough to close that gap if CO adds a pre-review mode with the right blocking policy and wires the worker contract to use it.

## Requirements

1. Add a bounded pre-review monitor mode on top of the existing PR quiet-window monitor.
2. Keep required checks, unresolved review threads, actionable bot inline feedback, and current-head rereview signals as blockers before `In Review`.
3. Treat `CHANGES_REQUESTED` as blocking, but do not block the pre-review drain on normal GitHub `REVIEW_REQUIRED`.
4. Preserve a bounded timeout path and existing action-required exit-code behavior.
5. Update the provider-worker prompt and repo-local Linear skill so the worker runs the bounded pre-review drain after PR open/update and before `In Review`.
6. Add focused tests for clean no-feedback success, actionable-feedback-handled recovery, and bounded timeout.
7. Keep the issue active until a PR is attached and the new drain is satisfied.

## Current Truth

- `scripts/lib/pr-watch-merge.js` already polls required checks, review threads, bot inline feedback, rereview signals, and a quiet window, but it is merge-specific today.
- `orchestrator/src/cli/providerLinearWorkerRunner.ts` and `skills/linear/SKILL.md` currently require PR feedback sweeps before `In Review`, but they do not yet name a bounded automated-feedback drain command.
- Symphony `SPEC.md` keeps tracker writes worker-owned, and current Elixir `WORKFLOW.md` requires PR feedback sweeps before review handoff, but no separate bounded pre-review drain is currently documented there.
- The baseline audit is captured in `out/linear-9d962236-4c38-4b28-b144-007c6f3a1395/manual/20260325T070120Z-baseline-audit.md`.
