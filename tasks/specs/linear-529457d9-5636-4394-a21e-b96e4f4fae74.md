# Technical Specification

## Context

`CO-81` follows the `CO-78` recovery incident rather than the visible `CO STATUS` surface. The failure chain crossed three truth boundaries at once: the provider worker tried to request a control-host refresh and failed because it expected `.runs/local-mcp/cli/control-host/manifest.json` to exist; the local host then remained pinned to stale shared-budget cooldown state until an operator replaced it; and restart-time or rehydrate-time claim truth still needed to distinguish genuinely live runs from stale non-terminal artifacts.

The lane therefore needs one coherent recovery pass over provider-worker refresh discovery, cooldown recovery, and claim rehydrate or reclaim truth without widening into a larger lifecycle redesign.

## Requirements

1. Provider-worker recovery must no longer fail just because the local control-host manifest path is absent or different from the caller’s expectation.
2. After a shared Linear budget cooldown clears, local intake must resume truthful reconciliation without requiring manual control-host replacement.
3. Rehydrate logic must only mark a claim as active when the underlying run is demonstrably alive, and only mark it resumable when an actual resume path exists.
4. Eligible live issues must be reclaimed automatically after cooldown or restart instead of depending on another operator intervention.
5. Regression coverage must explicitly cover the `CO-78` class: missing local host manifest, shared-budget cooldown, host restart or recovery, and truthful post-cooldown reclaim.

## Protected Expectations

- Missing local control-host manifest state must not break provider-worker recovery.
- Linear shared-budget cooldown is expected pressure, not a reason for the host to become operationally dead until restart.
- Claim rehydration must be truthful about whether a run is actually alive.
- `control-host/manifest.json`, `provider-intake-state.json`, `linear_budget_shared_cooldown`, and `dispatch_source_provider_rate_limited` are part of the tracked evidence and naming contract for this lane.

## Reject These Wrong Interpretations

- “The operator can just restart the host manually every time.”
- “Any `in_progress` manifest is good enough to treat as active after restart.”
- “A cooldown window justifies stale local truth indefinitely.”
- “A log-warning downgrade alone closes the issue.”

## Current Truth

- `orchestrator/src/cli/providerLinearWorkerRunner.ts` derives the control-host manifest path from the current run’s canonical `.runs/<task>/cli/<run>/manifest.json` layout plus `provider_control_host_task_id` / `provider_control_host_run_id`, then reads endpoint/auth metadata from that run directory before posting `/api/v1/refresh`.
- `orchestrator/src/cli/control/providerIssueHandoff.ts` owns startup rehydrate and refresh lifecycle truth, classifying claims as running, resumable, completed, queued, or released based on discovered provider issue runs.
- `orchestrator/src/cli/control/linearBudgetState.ts` persists shared cooldown state, while `orchestrator/src/cli/control/linearDispatchSource.ts` fails tracked-issue reads closed while cooldown is active.
- Existing tests already cover nearby refresh-path and rehydrate scenarios, which makes this lane a bounded recovery extension rather than a greenfield design.

## Proposed Design

### 1. Durable control-host refresh discovery

- Keep the existing provider control-host locator contract (`provider_control_host_task_id`, `provider_control_host_run_id`) but relax the single manifest-path assumption.
- Prefer whichever persisted control-host artifact remains authoritative and trustable for the current locator, for example a live manifest workspace path or endpoint/auth files rooted under the same run directory when the manifest itself is absent.
- Preserve root-boundary checks, allowed-bind-host validation, and auth-token requirements even if manifest discovery becomes more flexible.

### 2. Truthful cooldown recovery

- Audit the shared Linear budget read/write path to confirm whether expired cooldown metadata self-clears correctly in the live host recovery loop.
- Ensure the refresh lifecycle resumes normal tracked-issue reconciliation once cooldown expires instead of leaving stale suppression truth pinned until a restart.
- Keep rate-limit exhaustion itself out of scope; only fix the local recovery seam that keeps truth stale past the valid cooldown window.

### 3. Honest rehydrate and reclaim

- Keep `providerIssueHandoff` as the authority for claim rehydrate.
- Tighten or extend the liveness classification only as needed so stale non-terminal artifacts do not regain active truth.
- Ensure the first healthy post-cooldown or post-restart reconciliation can reclaim eligible live issues automatically.

### 4. Regression-first validation

- Extend provider-worker/control-host tests rather than inventing a parallel recovery harness.
- Cover both worker-side refresh recovery and host-side truthful rehydrate or reclaim behavior.
- Use the workpad and review artifacts as the human-auditable proof surface for the lane.

## Non-Goals

- Eliminating Linear rate limits entirely.
- Replacing the provider-worker lifecycle model.
- Broad `CO STATUS` redesign or unrelated control-host refactors.
- Treating manual host replacement as the accepted steady-state recovery.

## Parity / Alignment Matrix

| Surface | Current CO truth | Recovery reference truth | Target CO truth |
| --- | --- | --- | --- |
| Worker refresh discovery | assumes one canonical control-host manifest path | recovery should survive path mismatch or missing manifest when durable control-host artifacts still exist | refresh request resolves through a trustable fallback and does not abort on missing manifest alone |
| Shared cooldown recovery | stale cooldown can suppress truthful intake until manual intervention | cooldown expiry should return the host to normal reconciliation automatically | expired cooldown no longer leaves intake or claim truth frozen |
| Rehydrate claim truth | discovered runs are reclassified, but stale non-terminal artifacts remain a risk to guard against | only live or resumable runs should regain claim truth | stale or dead runs stay released/completed while live eligible issues are reclaimed |

## Not Done If

- A missing `control-host/manifest.json` can still abort provider refresh requests.
- Local intake can remain frozen on an expired cooldown until a human restarts the host.
- Rehydrated claims can still point at stale or dead runs as if they were active workers.
- Eligible live issues are still not reclaimed automatically after cooldown or restart.
