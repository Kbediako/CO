# Technical Specification

## Context

`1312` solved one important inner seam: inside a single provider-worker lifetime, CO can keep a live thread/session lineage across normal successful turns. CO-89 is the next outer seam. The control-host and provider handoff layers can already rehydrate active runs, notice terminal proof, and queue continuation or retry. What they do not yet provide is an explicit guarded resident-session contract across supervised worker restarts. After a safe completed-turn boundary, the next worker start still behaves like a fresh worker launch contract instead of an intentional resident drain/restart/resume boundary with stable logical continuity rules.

## Requirements

1. Add one guarded resident continuity contract above the existing worker-lifetime continuation loop.
2. Only allow resident continuity reuse after explicit restart-safe completed-turn boundaries proven by a fresh provider proof.
3. Preserve stable logical resident session/thread continuity across guarded worker restarts where possible.
4. Keep a monotonic logical turn count across guarded restarts so continuity is not reset silently at the worker boundary.
5. Surface resident continuity metadata in provider proof and selected debug/read-model state.
6. Keep mid-turn failures, unsafe exits, or stale proof conditions fresh-start or fail-closed; do not label them resident continuity.
7. Keep `resume(runId)` truthful and separate; this lane does not redefine top-level orchestrator run resume semantics.
8. Document config/policy reload truthfully as restart-boundary behavior only, not live hot reload.

## Design

- Resident continuity envelope:
  - source run id
  - source thread id
  - source logical turn count
  - restart count
  - source end reason / boundary reason
  - stable logical session id
- Launch seam:
  - `providerIssueHandoff.ts` derives the continuity envelope from the prior fresh proof when the previous run ended at an allowed completed-turn boundary
  - `controlHostCliShell.ts` carries that envelope into the next `provider-linear-worker` launch without widening unrelated launch authority
- Worker behavior:
  - `providerLinearWorkerRunner.ts` reads the continuity envelope at startup
  - on a guarded resident resume, the first turn uses the previous `thread_id` rather than starting a brand-new thread
  - proof updates preserve stable logical session id and monotonic turn count across the new worker lifetime
- Observability:
  - provider proof and selected debug/read-model state show fresh vs guarded-resume startup, restart count, source run id, and boundary reason
- Boundary rules:
  - restart continuity is allowed only at safe completed-turn drain boundaries
  - reload semantics are explicit: new policy/config applies on the next guarded restart boundary
  - no mid-turn hot upgrade and no in-memory process-state preservation

- Preserve the existing single external issue owner and Linear mutation authority.

## Reject These Wrong Interpretations

- "just treat the next worker launch as same-session without carrying explicit continuity data"
- "mid-turn failures can safely resume the same resident state"
- "the fix requires replacing Node/TypeScript or switching to Elixir"
- "this lane should reopen unrelated telemetry or distributed worker-host work"

## Current Truth

- `providerLinearWorkerRunner.ts` already uses `exec --json` on turn 1 and `exec resume --json <thread_id>` on later turns inside one worker lifetime.
- `provider-linear-worker-proof.json` already captures runtime lineage fields, but they are scoped to one worker run and do not define cross-worker continuity semantics.
- `providerIssueHandoff.ts` rehydrates active runs and treats failed runs as resumable, while completed successful runs fall into completed or queued-continuation behavior at the run level.
- `max_turns_reached_issue_still_active` is already a real restart-safe completed-turn boundary in proof, but the next launch does not yet explicitly resume the same logical resident thread/session lineage.

## Proposed Design

- Introduce additive resident continuity metadata derived from the previous fresh proof when the previous boundary is restart-safe.
- Carry that metadata through the control-host launch seam into the next provider-worker.
- Let the restarted worker use the previous `thread_id` on its first turn, keep a stable logical session identifier, and continue the logical turn count.
- Expose the continuity envelope in proof/read-model state so restart behavior is operator-visible.
- Keep unsafe boundaries fresh-start or fail-closed.

## Non-Goals

- Rewriting top-level `resume(runId)` semantics.
- Supporting mid-turn hot reload or in-memory process continuation.
- Expanding remote app-server authority surfaces.
- Reopening CO-82, CO-83, CO-88, or distributed worker scheduling.

## Parity / Alignment Matrix

- Current truth:
  - worker-lifetime continuation exists inside one `provider-linear-worker`
  - cross-worker restart continuity is implicit and not guaranteed
- Reference truth:
  - long-lived orchestration should expose truthful restart continuity at explicit safe boundaries
- Target truth / intended delta:
  - cross-worker restart continuity becomes explicit, guarded, and visible at safe completed-turn boundaries
- Explicitly out-of-scope differences:
  - hot upgrade or preserved in-memory process state
  - distributed scheduling

## Not Done If

- A restarted worker still cannot truthfully resume the same logical resident thread/session lineage after a safe completed-turn boundary.
- Proof/read-model state still hides whether the current worker is a fresh start or a guarded resident resume.
- Unsafe boundaries can accidentally reuse the prior resident state.
- Docs imply hot reload or live upgrade semantics that the implementation does not support.
