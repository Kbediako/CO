# Technical Specification

## Context

`CO-185` is a focused follow-up from live `CO-184` provider-worker friction. The source trace showed an otherwise active/progressing provider worker burning attempts on deterministic helper failures:

1. `provider_worker_child_lane_scope_missing` after the worker requested unsupported child-lane phases `classification, analysis`; the helper reported supported phases `docs, implementation, tests`.
2. `provider_worker_child_lane_parent_dirty` after .tmp/workpad.md dirtied the parent before a phase-scoped child-lane launch.
3. `provider_worker_child_lane_scope_conflict` after another overlapping child-lane launch was attempted before resolving the pending lane.
4. A child lane later reported `succeeded`, but parent disposition rejected it because the patch artifact was zero-byte and carried no usable classification payload.
5. `create-follow-up` failed twice with `linear_follow_up_parity_matrix_missing`, showing same-attempt duplicate retry without adding the required parity matrix or reclassifying the follow-up as non-parity/alignment.

The issue is about provider helper constraints and retry behavior before worker retries. It is not permission to weaken child-lane validation, clean-parent requirements, parity matrix enforcement, or parent-owned acceptance authority.

## Requirements

1. Surface supported child-lane phases before unsupported calls:
   - expose `docs`, `implementation`, and `tests` in first-turn guidance, continuation guidance, helper preflight output, or a combination;
   - tell workers how to handle `classification` / `analysis` needs without issuing `child-lane --phases classification,analysis`;
   - do not silently add `classification` or `analysis` as supported phases unless the phase-to-path contract, docs, and tests are updated together.
2. Make parent-dirty launch feedback actionable:
   - preserve clean-parent checks;
   - report a deterministic recovery path for .tmp/workpad.md or other temp/workpad dirt;
   - avoid letting read-only classification-style work fail without worker-visible sequencing or recovery instructions.
3. Prevent same-attempt overlapping child-lane retries:
   - preserve parent-owned pending lane disposition;
   - steer workers to accept, reject, invalidate, serialize, or change scope instead of launching overlapping work blindly.
4. Suppress duplicate deterministic `create-follow-up` retries:
   - after `linear_follow_up_parity_matrix_missing`, same-attempt provider truth must not retry unchanged inputs;
   - allow retry only when the next attempt supplies a parity matrix or explicitly reclassifies the follow-up as non-parity/alignment.
5. Classify zero-byte read-only child-lane output truthfully:
   - a zero-byte patch is not useful implementation evidence by itself;
   - read-only lanes must produce a useful summary/artifact or be classified as no-output advisory with an explicit parent-owned evidence path.
6. Add focused regression/eval evidence from the `CO-184` trace:
   - prove preflight;
   - prove intentional child-lane launch or serial fallback;
   - prove no duplicate deterministic mutation failures;
   - prove useful parent evidence.
