# Technical Specification

## Context
The repo already has the components of a bounded `source 0` slice, but they are disconnected. The shared run manifest has no memory contract, review and cloud prompt builders create context without a durable pointer-based anchor, provider-worker and child-lane prompts do the same, and RLM already knows how to build and copy pointer-based context objects without a shared run-contract seam exposing them. The issue is therefore to connect those seams through one additive contract, not to reopen the full run-memory or telemetry umbrella.

## Requirements
1. The shared run contract must expose additive `memory.source_0` backing with schema and generated type support.
2. `memory.source_0` must describe a pointer-based, traceable payload rather than a bare path or flattened prose blob.
3. Reviewer, cloud, provider-worker, child-lane, and RLM consumers must be able to read the same anchor through one shared reader path.
4. The stored payload must be bounded and traceable back to originating run metadata.
5. Child or delegated runs must preserve the same `source 0` identity semantically while keeping local file paths readable in the child artifact tree.
6. Readers must remain backward-compatible when `memory.source_0` is absent from older manifests.
7. Tests must cover contract creation plus at least two consumer surfaces.
8. The slice must not broaden into run-memory-controller, telemetry, continuity, or distributed-parity work.

## Design
- Contract:
  - extend `schemas/manifest.json` and generated types with additive `memory.source_0`
  - model `source_0` as a context-object descriptor, not as inline prose
- Descriptor shape:
  - `kind: "context_object"`
  - `object_id`
  - `dir_path`
  - `index_path`
  - `source_path`
  - `byte_length`
  - `chunk_count`
  - `created_at`
  - traceability fields such as `origin_run_id`, `origin_manifest_path`, and optional inheritance metadata
- Storage:
  - reuse `buildContextObject(...)` from `orchestrator/src/cli/rlm/context.ts`
  - store structured JSON content as the context-object source payload so the anchor is pointer-based without being prose-flattened
- Reader path:
  - add one shared helper to materialize, inherit, resolve, and optionally format `source 0`
  - targeted consumers call that helper instead of reconstructing their own anchor shape

## Implementation Surface
- Expected codepaths:
  - `schemas/manifest.json`
  - `packages/shared/manifest/types.ts`
  - `orchestrator/src/cli/run/manifest.ts`
  - shared new helper for `source 0` materialization and reads
  - `scripts/lib/review-prompt-context.ts`
  - `scripts/run-review.ts`
  - `orchestrator/src/cli/services/orchestratorCloudPromptBuilder.ts`
  - `orchestrator/src/cli/providerLinearWorkerRunner.ts`
  - `orchestrator/src/cli/providerLinearChildLaneRunner.ts`
  - `orchestrator/src/cli/rlmRunner.ts`
- Expected tests:
  - manifest/source-0 contract coverage
  - review/cloud/provider-worker/child-lane/RLM consumer read coverage in the smallest bounded set that proves the shared path

## Protected Expectations
- Keep literal `source 0` wording in the contract and docs.
- Keep the payload pointer-based and traceable.
- Keep the change additive and backward-compatible for older manifests.
- Preserve a single shared read path across the targeted consumers.

## Reject These Wrong Interpretations
- `source 0` is just a file path.
- review prompt context already solves the memory anchor problem.
- the slice should add block memory, retrieval scoring, or experience reuse.
- the slice should duplicate `CO-82`, `CO-83`, `CO-89`, or `CO-90`.
- the payload can be flattened into prose because prompts are text.

## Current Truth
- There is no manifest-backed memory contract today.
- Review, cloud, provider-worker, and child-lane surfaces each have adjacent but separate context-building seams.
- RLM already has the closest correct pointer-based substrate, but it is not surfaced through the shared run contract.

## Proposed Design
- Add `memory.source_0` to the manifest and generated types.
- Materialize `source 0` as a context object once per run from bounded structured source data.
- Preserve stable object identity across inherited child runs while rewriting local paths plus traceability metadata as needed.
- Use one shared helper to resolve the descriptor and payload for reviewer, cloud, provider-worker, child-lane, and RLM reads.

## Non-Goals
- Full run-memory controller or retrieval policy.
- Block memory.
- Provider-worker telemetry or status projection work.
- Resident-session continuity work.
- Distributed worker-host parity work.
- Reopening `0303`.

## Parity / Alignment Matrix
- Current truth:
  - no manifest-backed `source 0`
  - multiple adjacent consumer seams
  - existing pointer-based context-object substrate already exists only inside RLM
- Reference truth:
  - RLM context objects already provide stable `object_id`, chunk metadata, and copyable directory layout
- Target truth / intended delta:
  - one shared `memory.source_0` contract and helper-backed reader across targeted consumers
  - same semantic anchor preserved across inherited child runs
  - pointer-based, traceable payload retained end to end
- Explicitly out-of-scope differences:
  - planner redesign
  - telemetry/status work
  - retrieval indexing or reuse policy

## Not Done If
- `source 0` exists in only one consumer surface.
- the payload is only a path or prose blob.
- targeted consumers still derive incompatible anchors.
- the issue broadens into adjacent telemetry or continuity work.
