## Technical Requirements
- Functional requirements:
  1. Add `provider_launch_source` to the manifest schema/types used by provider-worker runs.
  2. Persist `provider_launch_source`, `provider_control_host_task_id`, and `provider_control_host_run_id` during manifest bootstrap when the active launch source is `control-host`.
  3. During provider-worker context load, backfill missing tuple values from the live environment only when the manifest has no conflicting provenance; keep resume preparation as the lone explicit overwrite carve-out for reseeding stale provider-worker manifests.
  4. Require `provider_launch_source=control-host` plus matching task/run when deciding that manifest provenance is recorded and valid.
  5. Keep `linear child-lane` and `linear child-stream` fail-closed for missing or mismatched provenance.
  6. Prove the happy path with focused tests and at least one live child helper invocation in this provider-worker turn.
- Non-functional requirements (performance, reliability, security):
  - additive manifest persistence only
  - no acceptance of mismatched launch sources
  - no hidden env-only fallback as the authoritative steady-state path
