# Technical Specification

## Context

`CO-62` is the broader Linear budget-awareness follow-up to `CO-33` and `CO-34`. Those two lanes repaired narrow write-back and reread seams, but they did not solve the larger problem that CO still spends Linear requests faster than the shared hourly bucket refills during ordinary operation. The current repo still has:

- fixed `15s` control-host tracked-issue polling in `controlServerPublicLifecycle.ts`
- a `10s` trust window for direct-mutation `issue-context` reuse in `providerLinearWorkflowFacade.ts`
- provider-worker rereads after every turn in `providerLinearWorkerRunner.ts`
- multi-step helper flows such as `create-follow-up` that can consume several sequential GraphQL requests

The missing owner is shared budget awareness across these callers. This lane must reduce avoidable request burn while keeping issue truth explicit and preserving the narrow `CO-33`/`CO-34` fixes.

## Requirements

1. Linear GraphQL callers must be able to observe the relevant rate-limit headers on successful responses as well as rate-limited failures.
2. CO must persist a shared, token-scoped Linear budget state keyed without storing the raw API token.
3. The shared budget state must record explicit reset-window metadata after `linear_rate_limited` responses and expose whether a cooldown is currently active.
4. Control-host tracked-issue polling must stop relying only on a fixed `15s` interval and instead derive its next poll timing from a policy that can back off or suppress itself when the shared budget is exhausted or under pressure.
5. Multi-step helper flows (`upsert-workpad`, `transition`, `attach-pr`, `create-follow-up`, and similar direct mutation helpers) must fail fast before redundant live requests when the latest shared budget state is fresh and already exhausted or clearly insufficient for the planned sequence.
6. Provider-worker tracked-issue rereads must respect the shared cooldown state and persist the relevant Linear budget metadata they observe.
7. Worker proof and control-host/operator read surfaces must expose the tracked-issue rate-limit or reset metadata plus any new suppression state introduced by this lane.
8. Validation must prove lower avoidable request pressure and truthful behavior when the upstream bucket is genuinely exhausted.

## Current Truth

- `linearGraphqlClient.ts` reads the relevant Linear response headers, but only the failure path exposes them to higher layers today.
- `linearRateLimit.ts` already normalizes explicit `linear_rate_limited` details, including `retry-after`, request buckets, complexity buckets, reset timestamps, and `x-request-id`.
- `controlServerPublicLifecycle.ts` always re-arms the provider refresh timer at `15_000ms`, even when recent Linear headers show exhausted or nearly exhausted buckets.
- `providerLinearWorkflowFacade.ts` keeps a direct-mutation `issue-context` cache for only `10_000ms`, but it does not consult any shared cooldown or known remaining-budget state before executing multi-step helper flows.
- `providerLinearWorkerRunner.ts` will wait once through a short rate-limit window for tracked-issue rereads, but that knowledge stays local to the worker and is not shared with other callers that use the same token.
- Control-host observability already persists polling interval, next poll, reason, and worker proof payloads, so the repo has a narrow place to surface suppression decisions without inventing a new dashboard architecture.

## Proposed Design

### 1. Shared Linear budget-state owner

Introduce a new helper under `orchestrator/src/cli/control/` that:

- derives a token fingerprint via `resolveLinearApiTokenFingerprint(...)`
- resolves a shared state path from the token fingerprint
- parses success or failure headers into a normalized Linear budget sample
- persists the latest sample plus any active cooldown or insufficiency decision via atomic writes
- evaluates fresh shared state for:
  - active cooldown after an explicit rate limit
  - explicit remaining-budget insufficiency for a planned multi-request operation
  - control-host poll scheduling hints

The state must fail closed when it is missing or stale: a missing or stale sample must not block truthful live reads.

### 2. GraphQL header availability on success

Extend `executeLinearGraphql(...)` so successful results preserve the same relevant response headers that the failure path already captures. The budget-state helper will use these success headers to reduce steady-state burn before the bucket is fully exhausted.

### 3. Control-host polling policy

Replace the fixed `PROVIDER_REFRESH_INTERVAL_MS = 15_000` re-arm behavior with a policy-driven interval. The policy should:

- default to a higher steady-state interval than `15s`
- stretch further when the latest shared sample shows low remaining request headroom
- suppress polls until the recorded reset window expires when an explicit shared cooldown is active
- persist the chosen interval, next poll, and reason through the existing polling-health state so operators can tell whether the scheduler is healthy, backed off, or suppressed

### 4. Helper fail-fast policy

Add helper preflight budget checks in `providerLinearWorkflowFacade.ts` for multi-step direct-mutation flows. Each helper should declare the minimum request budget it needs for its planned sequence. When the latest shared budget sample is fresh and shows:

- an active cooldown, or
- explicit remaining request counts below the minimum needed for the operation

the helper must fail fast with a truthful rate-limit-style error instead of issuing doomed sequential GraphQL calls.

### 5. Provider-worker reread and proof surfacing

Update `providerLinearWorkerRunner.ts` so tracked-issue rereads:

- record successful budget samples into the shared state
- consult the shared state when deciding whether to wait, retry once, or fail explicitly
- persist a Linear budget or suppression summary in `provider-linear-worker-proof.json` that is distinct from Codex model `rate_limits`

### 6. Control-host observability surfacing

Update control-host read-model or observability payloads so the relevant shared Linear budget state is visible from operator surfaces. The minimum explanation bar is:

- current or last observed reset-window metadata when available
- whether polling is backed off or suppressed because of the shared budget state
- access to the worker’s persisted Linear budget summary when a provider-worker lane hit the limit
