# Technical Specification

## Context

`CO-136` is a bounded follow-up to `CO-96`: once the memory-observability work cleared, clean validation exposed a single unrelated command-surface blocker. The command-surface suite already proves most `rlm` and `exec` cases function correctly, and the current workspace already preserves quoted single-token `exec` commands through the existing bin-level normalizer. The remaining reproducible failure is that several source-entrypoint failure cases are close to the current per-test timeout because the source entrypoint loads the full CLI module graph.

## Requirements

1. Preserve quoted single-token `exec` commands as deterministic argv before execution.
2. Preserve backslashes for Windows-style path arguments in the same single-token `exec` path.
3. Keep existing multi-token `exec` behavior unchanged.
4. Use a truthful timeout budget for the affected source-entrypoint `rlm` / `exec` failure tests.
5. Keep the fix scoped to the command-surface seam and do not modify `CO-96` behavior.

## Design

- Parsing contract:
  - keep the existing bin-level `normalizeExecCommandTokens(...)` path as the only single-token `exec` normalizer in this lane
  - rely on its existing-path guard so space-containing executable paths remain unsplit
- Timeout truth:
  - move only the known boot-heavy source-entrypoint tests onto `CLI_BOOT_TIMEOUT`
  - keep the assertions on specific error messages unchanged so the larger timeout does not hide functional regressions

## Implementation Surface

- Expected codepaths:
  - `bin/codex-orchestrator.ts`
  - `tests/cli-command-surface.spec.ts`
- Expected tests:
  - the existing quoted single-token `exec` tests in `tests/cli-command-surface.spec.ts`
  - the existing `rlm` / `exec` failure-message tests whose timeout budget is being updated

## Protected Expectations

- Preserve `CO-96` isolation.
- Preserve the current `exec` argv contract for already-green multi-token invocations.
- Keep timeout truth explicit and limited to the affected source-entrypoint cases.

## Reject These Wrong Interpretations

- "fix the escaped-quote case by routing everything through a shell"
- "raise the timeout across the whole suite"
- "treat this as a general CLI performance lane"
- "change unrelated `rlm`, provider-worker, or memory-observability behavior"

## Current Truth

- `tests/cli-command-surface.spec.ts` uses `runCli(...)` with `runEntrypointLikeExec(...)`, so the failing `15_000ms` budget is enforced by Vitest rather than a child-process timeout.
- `bin/codex-orchestrator.ts` already normalizes one-token `exec` payloads before they reach `runExecCliShell(...)`.
- The current workspace preserves the simple `echo quoted-smoke`, backslash, and escaped-inner-quote `node -e` cases; the remaining reproducible failure is the source-entrypoint timeout budget.

## Proposed Design

- Keep timeout truth in the tests instead of changing CLI behavior for the already-correct fast-fail cases.
- Preserve the existing `bin/codex-orchestrator.ts` single-token `exec` normalization as-is.
- Avoid touching runtime parsing code unless validation proves the current behavior insufficient.

## Non-Goals

- General source-entrypoint cold-start optimization.
- Rewriting `execRuntime` for every command shape.
- Changing `CO-96` memory provenance or outcome-schema behavior.

## Parity / Alignment Matrix

- Current truth:
  - source-entrypoint boot-heavy failure cases are correct but close to the test timeout
  - escaped inner quotes still break in one-token `exec`
- Reference truth:
  - direct probes already show the correct `rlm` / `exec` errors and the desired escaped-quote `node -e` behavior
- Target truth / intended delta:
  - one-token `exec` commands preserve escaped quotes and backslashes
  - the affected timeout cases use a truthful source-entrypoint budget
- Explicitly out-of-scope differences:
  - general startup speedups
  - unrelated CLI/provider changes

- the timeout adjustments spread to unrelated tests.
