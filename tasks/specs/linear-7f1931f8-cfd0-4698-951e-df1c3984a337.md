# Technical Specification

## Context

`CO-98` restored truthful provider-worker runtime telemetry to the live root control-host path, but the selected-run refresh path still reconstructs session-log hydration from scratch each time it refreshes an in-progress proof. Repeated root `CO STATUS` refreshes therefore keep paying `O(file-size)` reread cost for the same active worker until all telemetry fields are populated. This follow-up needs one bounded refresh optimization on the authoritative proof seam without regressing the truthful `Tokens`, `SESSION`, `Throughput`, and Codex `5-hour` / `weekly` telemetry restored under `CO-98`.

## Requirements

1. Repeated in-progress proof refreshes for the same active provider-worker session must not reread the already-consumed session-log bytes from byte `0`.
2. The refresh path must still surface truthful runtime telemetry updates for `Tokens`, `SESSION`, `Throughput`, and Codex `5-hour` / `weekly` windows when they appear later in the session log.
3. The implementation must fail closed to a safe full reread whenever the active session-log file changes, truncates, or otherwise invalidates the persisted hydration cursor.
4. The change must remain bounded to the authoritative proof refresh seam; no refresh-cadence throttling, heuristic data loss, or unrelated schema redesign.
5. Tests must cover repeated refreshes against a growing `rollout-*.jsonl` file and prove both bounded rereads and preserved correctness.
6. Tests must also cover the fail-closed path when the active session-log file rotates or truncates, so stale cursor state cannot suppress later telemetry.

## Proposed Design

- Extend the proof refresh contract with an internal hydration cursor that records the currently discovered session-log path plus the minimum state required to resume parsing safely, including a proof-signature drift guard, so cursor reuse can detect when the proof snapshot moved ahead independently.
- Reuse that cursor only when the refresh discovers the same active file and the file still appears to have grown monotonically.
- Keep the cursor opaque and local to the proof refresh path so the broader control-host/read-model contract does not need to reason about it.

- If discovery returns a different `rollout-*.jsonl` path, or if the previously recorded cursor is inconsistent with the current file, drop the cursor and reread from the start.
- Keep telemetry derivation exactly on the existing parser path, so correctness remains identical to the current full-reread behavior.
- Treat bounded rereads as an optimization only; correctness must continue to prefer authoritative session-log data.

- `selectedRunProjection.ts` should keep deciding when an in-progress proof is incomplete enough to refresh.
- The optimization belongs in `refreshProviderLinearWorkerProofSnapshot(...)` / `hydrateProviderLinearWorkerProofFromSessionLog(...)`, not in refresh cadence or projection heuristics.
- The selected-run projection should continue receiving the same truthful proof fields after the change.

## Non-Goals

- No browser or `/ui` dashboard redesign.
- No synthetic or inferred telemetry placeholders.
- No refresh-throttling workaround that leaves byte-zero rereads unchanged.
- No heuristic tail read that can miss real telemetry updates.

## Not Done If

- `refreshProviderLinearWorkerProofSnapshot` still rereads the same session log from byte `0` on repeated in-progress refreshes for the same active worker.
- A long-running worker with a large `rollout-*.jsonl` file still causes repeated full-file reads during root control-host status refreshes.
- The optimization drops or delays truthful runtime telemetry that `CO-98` restored for `Tokens`, `SESSION`, `Throughput`, or Codex `5-hour` / `weekly` segments.
