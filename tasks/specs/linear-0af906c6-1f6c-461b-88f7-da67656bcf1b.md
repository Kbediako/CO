# Technical Specification

## Context
`CO-115`, `CO-116`, and `CO-111` proved that CO already has several truthful autonomous building blocks: long-running host supervision, review-handoff promotion into `Merging`, and deterministic merge closeout with shared-root reconciliation. The remaining gap is one layer above those seams. Queue shepherding still depends on a human operator deciding when the next backlog lane becomes `Ready`, when a review handoff actually needs `Rework`, and whether any local rollout follow-up remains after merge. `CO-118` closes that operator-policy gap without weakening the narrower landed contracts.

## Requirements
1. The repo must expose an explicit operator-autopilot mode in repo-tracked configuration rather than burying the behavior in ad hoc code.
2. The operator-autopilot must run from the long-running control-plane refresh loop, not from a single interactive worker turn.
3. Backlog promotion must reuse the existing tracked-issue sort order, must fail closed when a higher-ranked blocked lane still owns the queue head, and must only mutate a backlog head owned by the current viewer or left unassigned.
4. Review handoffs must continue using the existing clean promotion into `Merging`, while only author-action-required blocked handoffs can move into `Rework`.
5. `review=REVIEW_REQUIRED` must not trigger `Rework`; that state remains review wait, not author rework.
6. The autopilot must emit durable structured audit records and the latest summary payload for operator inspection.
7. Post-merge local rollout follow-up must be surfaced as an explicit pending operator action when it is not automated in the first slice.

## Design
- Add a dedicated operator-autopilot control module under `orchestrator/src/cli/control/` instead of expanding `controlHostCliShell.ts`.
- Parse repo-tracked operator-autopilot config from the existing provider workflow metadata in `codex.orchestrator.json`.
- Invoke the operator-autopilot from `providerIssueHandoff.ts` after claim reconciliation and before fresh dispatch candidate launch.
- Reuse:
  - `sortLiveLinearTrackedIssuesForDispatch(...)` for backlog order
  - `isLiveLinearTrackedIssueOwnedByCurrentViewerOrUnassigned(...)` so backlog promotion never mutates another operator's lane
  - `providerLinearTodoBlockedByNonTerminal(...)` and tracked blocker metadata for queue holds
  - persisted `review_promotion` truth for `Merging` versus `Rework` handoff decisions
  - existing merge-closeout truth to detect when post-merge local rollout follow-up should be surfaced
- Keep the first slice conservative:
  - at most one backlog-head promotion per cycle
  - only transition to `Rework` from persisted review-promotion author-action-required truth, excluding `review=REVIEW_REQUIRED`, `label:do-not-merge`, and `required_checks_query_failed`
  - only surface, not fully automate, local rollout actions unless the repo later adds an explicit hook lane

## Protected Expectations
- Preserve the current dispatch sort order as the first-slice queue order.
- Preserve current `CO-116` review-to-`Merging` behavior.
- Preserve current deterministic `Merging` closeout and shared-root reconciliation behavior.
- Preserve provider-worker stop-coding review handoff semantics; the autopilot acts after handoff, not by reopening worker coding in review.

## Reject These Wrong Interpretations
- "CO-118 is already done because `CO-116` merged."
- "A read-only status dashboard counts as operator autopilot."
- "Because background policy is desired, the code can silently mutate issue states without structured proof."
- "If a review handoff is not merge-ready yet, every blocked state should become `Rework`, including `REVIEW_REQUIRED`."

## Current Truth
- The control-host already polls live Linear and dispatches eligible active issues.
- `providerLinearWorkflowStates.ts` already knows that TODO-equivalent lanes should not run when blocked by non-terminal blockers.
- `providerMergeCloseout.ts` already promotes merge-ready review handoffs into `Merging`.
- There is no repo-tracked operator-autopilot layer that promotes backlog heads, transitions actionable handoffs into `Rework`, or surfaces pending local rollout actions.

## Proposed Design
- Introduce a dedicated operator-autopilot control module with:
  - repo-tracked config parsing
  - cycle evaluation for queue head, review handoffs, and merged closeout follow-up
  - durable latest-result projection plus append-only audit records
- Let the latest operator-autopilot result live beside provider workflow payload so the existing observability surfaces can project it without a separate transport.
- Keep issue/PR mutation rules explicit and tested:
  - backlog head promotion only when safe, in order, and owned by the current viewer or unassigned
  - `Rework` only from author-action-required review-promotion truth, not from `REVIEW_REQUIRED`, `label:do-not-merge`, or `required_checks_query_failed`
  - local rollout follow-up surfaced explicitly when merged closeout lands

## Non-Goals
- Replacing existing merge shepherding logic in `Merging`.
- Reworking launchd supervision or control-server lifecycle startup.
- Introducing a second queue order distinct from current dispatch sorting.
- Fully automating all local rebuild or restart hooks in the first slice.

## Parity / Alignment Matrix
- Current truth:
  - queue shepherding still depends on a human operator above the narrow landed seams
  - backlog items remain in `Backlog` until a human promotes them
  - actionable blocked review handoffs can stay parked in review until a human moves them to `Rework`
  - merged closeout does not surface explicit pending local rollout actions
- Reference truth:
  - bounded operator policy should be reusable, repo-tracked, and long-running
  - queue and PR truth should come from existing authoritative seams
- Target truth / intended delta:
  - a repo-tracked operator-autopilot can keep queue shepherding alive without a human chat turn
  - backlog promotion for current-viewer or unassigned queue heads, author-action-required `Rework` transitions, and pending local rollout actions are explicit and auditable
  - current review promotion and merge closeout contracts remain the same authorities for their narrow domains
- Explicitly out-of-scope differences:
  - generic workflow redesign
  - dashboard-only polish work
  - full local rollout command automation beyond explicit pending-action surfacing

## Not Done If
- The queue still only moves when a human operator manually babysits the issue set.
- Backlog promotion ignores blocked higher-priority lanes.
- Backlog promotion mutates a queue-head issue owned by another operator.
- `REVIEW_REQUIRED` handoffs fall into `Rework`.
- Pending local rollout follow-up after merge stays invisible.
