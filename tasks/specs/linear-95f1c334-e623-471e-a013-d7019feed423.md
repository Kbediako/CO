# Technical Specification

## Context

`CO-106` is a hardening follow-up to `CO-62`, not a fresh rate-limit lane. The current tree already has:

- shared Linear budget persistence in `linearBudgetState.ts`
- helper-path fail-fast before `fetch` in `providerLinearWorkflowFacade.ts` and `linearDispatchSource.ts`
- control-host polling shaped by shared budget pressure in `controlServerPublicLifecycle.ts`

That substrate is directionally correct, but the issue’s confirmed findings still hold in the live repo:

- budget state is keyed only by token fingerprint rather than Linear user scope
- endpoint request and complexity buckets are stored as one unkeyed pair rather than per endpoint identity
- `x-ratelimit-endpoint-name` and `x-complexity` are dropped
- preflight only enforces explicit request floors, not complexity floors
- newer partial observations replace the whole persisted snapshot
- cooldown precedence is implicit in code but not locked by a dedicated test
- control-host polling widens under pressure but still uses deterministic fixed tiers with no jitter
- concurrent callers still coordinate reactively rather than through shared reservation or smoothing

This lane must harden that existing substrate without reopening the full provider architecture.

## Requirements

1. Preserve `x-ratelimit-endpoint-name` and `x-complexity` on successful and rate-limited GraphQL responses.
2. Change canonical shared-budget persistence from token-only scoping to user-scoped coordination aligned with Linear’s real quota owner; any token fingerprint usage must be explicit bootstrap or reconciliation glue, not the canonical state key.
3. Store endpoint request and complexity buckets by endpoint identity, using the server-provided endpoint name when available.
4. Teach preflight to enforce explicit complexity headroom and to subtract active shared reservations before deciding whether a request can proceed.
5. Merge newer partial observations bucket-wise so they preserve richer prior bucket, alias, and reset data unless the newer observation explicitly supersedes it.
6. Make cooldown precedence explicit and test-covered for `retry-after` versus exhausted-bucket reset timestamps.
7. Jitter and degrade control-host polling/backoff behavior under low, exhausted, and endpoint-specific pressure while keeping operator-visible reasons intact.
8. Add a bounded reservation or smoothing layer for concurrent processes, or file a same-project follow-up with explicit rationale and residual risk before handoff.
9. Keep helper-path fail-fast before `fetch` for dispatch and provider helper surfaces.
10. Focused tests must cover user scope, endpoint-aware buckets, complexity shortfall, observation merge behavior, cooldown precedence, reservation behavior, fail-fast semantics, and polling degradation or jitter.

## Current Truth

- `linearGraphqlClient.ts` preserves request, endpoint-request, complexity, endpoint-complexity, `retry-after`, and `x-request-id`, but not endpoint name or per-request complexity cost.
- `linearRateLimit.ts` normalizes the captured headers into flat bucket fields, so there is nowhere to keep endpoint-name identity today.
- `providerLinearWorkflowFacade.ts` issue-context and summary queries request `viewer.organization.id` but not `viewer.id`, so the provider helper path cannot currently learn canonical user identity for shared-budget persistence.
- `linearBudgetState.ts` persists one flat snapshot keyed by `resolveLinearApiTokenFingerprint(...)`, and `recordLinearBudgetObservation(...)` replaces the whole persisted snapshot whenever the new observation is newer.
- `resolveLinearBudgetPreflight(...)` applies `minimum_requests_remaining` only to `requests` and `endpoint_requests`; complexity buckets matter only once they hit `<= 0`.
- `resolveLinearPollingInterval(...)` widens deterministic intervals for `constrained`, `low`, and `exhausted` pressure, but there is no jitter and no endpoint-aware distinction for tracked-issues polling.
- Dispatch and provider helper wrappers already consult shared preflight before `fetch`, so the right implementation seam is to strengthen the shared helper contract rather than layering duplicate local logic.

## Proposed Design

### 1. Header and query hardening

- Extend `LINEAR_GRAPHQL_RELEVANT_RESPONSE_HEADERS` with:
  - `x-ratelimit-endpoint-name`
  - `x-complexity`
- Extend `extractLinearRateLimitDetailsFromHeaders(...)` to preserve:
  - `endpoint_name`
  - `request_complexity`
- Add `viewer.id` to the provider issue-context and summary queries so provider helper flows can learn the same canonical user identity already available in dispatch queries.

### 2. User-scoped persisted budget identity

- Bump the persisted budget schema and store canonical scope metadata:
  - `workspace_id`
  - `viewer_id`
  - a derived canonical budget key
- Keep token fingerprint only as bootstrap or alias metadata so preflight can find the canonical user-scoped state after at least one identity-bearing observation.
- Preserve a fail-open cold-start path when canonical identity is not yet known; missing identity must not block truthful live reads.

### 3. Endpoint-aware buckets and aliases

- Replace the single `endpoint_requests` and `endpoint_complexity` payloads with endpoint-keyed maps that hold:
  - request bucket
  - complexity bucket
  - last observed request complexity
  - endpoint aliases such as the operation source string
- Use `x-ratelimit-endpoint-name` as the canonical endpoint key when present.
- Let preflight resolve the best endpoint bucket for an operation via:
  - explicit canonical endpoint name when already learned
  - source-to-endpoint alias mapping
  - truthful fallback to global buckets when endpoint identity is still unknown

### 4. Reservation-aware preflight and execution

- Extend `resolveLinearBudgetPreflight(...)` with request and complexity floor inputs plus effective remaining calculations that subtract active reservations.
- Introduce a bounded shared reservation lifecycle under the existing budget-state lock:
  - reserve before `executeLinearGraphql(...)` for requests and, when possible, complexity
  - expire stale reservations automatically
  - release or reconcile the reservation after the request completes
- Keep the first implementation bounded: enough to smooth concurrent callers and avoid obvious stampedes without attempting a full distributed rate limiter.

### 5. Safe observation merge and cooldown precedence

- Merge observations bucket-wise instead of replacing the entire snapshot.
- Prefer newer explicit bucket values, but preserve richer prior endpoint or reset metadata when the new observation omits it.
- Recompute cooldown from the merged state using an explicit precedence rule:
  - the effective cooldown ends at the later of `retry-after` and any exhausted-bucket reset timestamp
  - dedicated tests must lock this down

### 6. Jittered degraded polling

- Extend polling interval selection to:
  - apply bounded jitter
  - degrade more aggressively when the tracked-issues endpoint bucket is low or exhausted
  - preserve operator-visible reason and next-poll metadata
- Keep the jitter deterministic under test via injectable time or RNG seams.
