# Technical Specification

## Context
`frontend-test` was still paying a large pre-manifest startup tax when invoked through the live source entrypoint under `node --loader ts-node/esm`. The downstream `frontend-testing` stage itself was relatively short; the bulk of the delay came from eager CLI bootstrap before `prepareRun` could even create `.runs/.../manifest.json`.

## Requirements
1. Keep the lane focused on `frontend-test`, `tests/cli-frontend-test.spec.ts`, and the direct `bin/codex-orchestrator.ts frontend-test --format json` entrypoint.
2. Identify whether the delay is above manifest creation or inside the downstream runner stage.
3. Restore deterministic completion without weakening the existing test contract or just inflating timeouts.
4. Preserve the `frontend-testing` pipeline contract and runtime-sanitization behavior.
5. Record artifact-backed evidence that distinguishes this fix from provider refresh-path work.

## Design
- Parse the top-level CLI command first, then lazy-load `CodexOrchestrator` and command-shell modules only for the selected command.
- Keep lightweight shared helpers in the bootstrap file, but avoid importing the whole CLI graph on the `frontend-test` path.
- Preserve command behavior and output format; the change is only about when modules are loaded, not what `frontend-test` does once it starts.

## Implementation Surface
- `bin/codex-orchestrator.ts`
- `tests/cli-frontend-test.spec.ts`
- evidence under `out/linear-1cd22f2a-5a12-4d2f-bc32-8289f4d24be6/manual/`
