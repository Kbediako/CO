# Technical Specification

`CO-393` makes provider-worker relaunch/recover/nudge operator-supported through root control-host state and blocks unsafe direct starts. Requirements: root `control-host recover|relaunch|nudge --issue-id <id>`, existing provider admission/duplicate-worker/launch-token/launch-resume reuse, deterministic issue worktree cwd with root `.runs` / `out`, manifest `provider_launch_source=control-host`, `provider_control_host_task_id`, `provider_control_host_run_id`, provider-intake `launch_source=control-host`, non-null launch-token/intake reservation truth, and child-lane compatibility.

Protected contract: preserve `root control-host relaunch/recover/nudge`, `specific Linear issue`, `codex-orchestrator start provider-linear-worker`, `matching control-host provenance`, `actionable command hint`, `deterministic issue worktree cwd`, `provider-intake-state.json`, `CO-392`, `CO-244`, and `CO-189`. Reject broadening `CO-392`, redoing `CO-244`, weakening duplicate-worker/admission caps, hand-editing intake state, or promoting direct issue-worktree starts.

Implementation surfaces: `startCliRequestShell` guard; `controlHostProviderWorkerRecoverCliShell`; `/api/v1/provider-worker/recover`; `ProviderIssueHandoffService.recoverIssue`; recovery claim summaries with `launch_token_present`. Fallback decision: retain root control-host authority as supported non-expiring contract, remove unsafe direct-start fallback. Validation covers focused recovery/guard/handoff/child-lane tests plus full repo gates, standalone review, elegance pass, and PR ready-review.
