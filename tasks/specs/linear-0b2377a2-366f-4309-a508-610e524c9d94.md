# Technical Specification

## Context
`CO-393` makes provider-worker relaunch/recover/nudge safe and operator-supported. Direct `codex-orchestrator start provider-linear-worker` previously could bypass root control-host semantics, leaving wrong cwd/artifact truth, null manifest provenance, or missing provider-intake launch-token truth. Source anchor: `ctx:sha256:69a80e9d4a8a919410857a6ff81972d38b23e6ec4a7988ddfa79bb23dd6505e1#chunk:c000001`.

## Requirements
1. Add root `control-host recover|relaunch|nudge --issue-id <id>` for a specific Linear issue.
2. Reuse existing provider admission, duplicate-worker protection, launch-token generation, and launch/resume logic.
3. Preserve deterministic issue worktree cwd with root `.runs` / `out` artifacts.
4. Persist manifest `provider_launch_source=control-host`, `provider_control_host_task_id`, and `provider_control_host_run_id`.
5. Persist provider-intake `launch_source=control-host` and non-null launch-token/intake reservation truth.
6. Fail closed for direct provider-worker starts without matching provenance and an issue-specific recovery hint.
7. Keep same-issue child-lane compatibility and adjacent `CO-244`, `CO-392`, and `CO-189` tests green.

## Protected Contract
Preserve exact surfaces: `root control-host relaunch/recover/nudge`, `specific Linear issue`, `codex-orchestrator start provider-linear-worker`, `matching control-host provenance`, `actionable command hint`, `deterministic issue worktree cwd`, root `.runs` / `out`, `provider_launch_source=control-host`, `provider_control_host_task_id`, `provider_control_host_run_id`, provider-intake `launch_source=control-host`, launch-token/intake reservation truth, `provider-intake-state.json`, `CO-392`, `CO-244`, and `CO-189`.

Reject: broadening `CO-392`, redoing `CO-244`, weakening duplicate-worker/admission caps, hand-editing intake state, promoting direct issue-worktree starts, or requiring operators to memorize internal provenance env variables.

## Implementation Summary
- Guard `provider-linear-worker` starts in `startCliRequestShell`.
- Add `controlHostProviderWorkerRecoverCliShell` for root control-host recovery requests.
- Add `/api/v1/provider-worker/recover` to the authenticated control-host API.
- Add `ProviderIssueHandoffService.recoverIssue`, using existing candidate processing for launch/resume and release handling for gone/out-of-scope issue lookups.
- Summarize recovery claims with `launch_token_present` instead of raw tokens.

## Fallback / Refactor Decision
| Surface | Decision | Reason |
| --- | --- | --- |
| Root control-host launch authority | justify retaining fallback | Non-expiring operator authority for admission, tokens, provenance, and run truth. |
| Direct start without provenance | remove fallback | Unsafe seam; must fail before run truth is created. |
