## Summary
- Objective: add the minimal Codex marketplace packaging path for CO so marketplace-capable Codex CLIs can discover/register the existing CO MCP entrypoint while npm remains supported.
- Scope: repo-root marketplace metadata, one minimal packaged plugin descriptor directory, package/tarball inclusion, downstream docs, temp-home smoke, and the required validation/review floor.
- Constraints: no new MCP runtime seam, no hosted marketplace infra, no secrets/hooks/private artifacts, and no delegation/authority expansion.

## Issue-Shaping Contract
- User-request translation carried forward: CO-196 is about additive marketplace distribution for the current CO install surface, not replacing npm or redesigning runtime authority.
- Protected terms / exact artifact and surface names: `codex marketplace add`, `.agents/plugins/marketplace.json`, `plugins/codex-orchestrator/.codex-plugin/plugin.json`, `plugins/codex-orchestrator/.mcp.json`, `plugins/codex-orchestrator/launcher.mjs`, `codex-orchestrator mcp serve`, `package.json`, `docs/public/downstream-setup.md`, `README.md`.
- Nearby wrong interpretations to reject: treating marketplace descriptors as permission to change MCP guardrails, wiring a new binary or wrapper instead of `codex-orchestrator mcp serve`, or claiming hosted/raw-JSON marketplace support without a CLI/documentation basis.
- Explicit non-goals carried forward: replacing npm packaging, plugin UI/apps metadata, new auth flows, hosted marketplace infra, and Codex version-policy changes outside support-posture wording.

## Parity / Alignment Matrix
- Current truth: the repo exposes `codex-orchestrator mcp serve` and packages docs/templates/skills through npm, but has no shipped marketplace root or plugin descriptor.
- Reference truth: official Codex plugin docs require a plugin root descriptor; in CO's layout that means `plugins/codex-orchestrator/.codex-plugin/plugin.json` plus optional `plugins/codex-orchestrator/.mcp.json`, and repo/local `marketplace.json`; local `codex marketplace add --help` accepts local directories and Git-backed sources.
- Target truth / intended delta: CO ships a repo-root marketplace source and packaged plugin manifests under `plugins/codex-orchestrator`; the cached plugin launcher resolves the recorded marketplace source root from Codex config, reuses the existing MCP contract, and the tarball plus docs make that path downstream-safe and reversible.
- Explicitly out-of-scope differences: hosted raw JSON marketplace hosting, alternate package managers, new runtime commands, or changes to provider-worker/runtime authority.

## Technical Requirements
- Functional requirements:
  1. Add `.agents/plugins/marketplace.json` with a `codex-orchestrator` entry whose `source.path` resolves to `./plugins/codex-orchestrator`, with explicit `policy.installation`, `policy.authentication`, and `category`.
  2. Add `plugins/codex-orchestrator/.codex-plugin/plugin.json` with minimal plugin metadata and `mcpServers: "./.mcp.json"`.
  3. Add `plugins/codex-orchestrator/.mcp.json` that launches `node ./launcher.mjs mcp serve` from the cached plugin root.
  4. Add `plugins/codex-orchestrator/launcher.mjs` that resolves `[marketplaces.codex-orchestrator].source` from `~/.codex/config.toml`, validates the local source root, and execs `<source>/bin/codex-orchestrator.js mcp serve` without host PATH reliance.
  5. Include marketplace/plugin files in the packed npm tarball and keep pack audit rules truthful.
  6. Add a temp-home smoke that skips non-marketplace Codex baselines, proves the marketplace root can be added from a local package/repo source, and verifies the installed cached launcher resolves and starts from the recorded marketplace source root.
  7. Update downstream docs to explain npm install, marketplace install, rollback/removal, and current support posture.
- Non-functional requirements (performance, reliability, security):
  - no secrets, hooks, or private run-artifact references
  - no absolute local paths in shipped descriptors
  - no changes to default MCP/delegation authority
  - docs and smoke artifacts stay machine-checkable
- Interfaces / contracts:
  - `package.json` `files`
  - `.agents/plugins/marketplace.json`
  - `plugins/codex-orchestrator/.codex-plugin/plugin.json`
  - `plugins/codex-orchestrator/.mcp.json`
  - `plugins/codex-orchestrator/launcher.mjs`
  - `templates/codex/mcp-client.json`
  - `codex marketplace add --help`

## Architecture & Data
- Architecture / design adjustments: introduce a repo-root marketplace root and a single packaged plugin directory under `plugins/codex-orchestrator` that delegates to the already-owned `codex-orchestrator mcp serve` CLI surface. Because Codex caches only the plugin source directory, the launcher must jump back to the recorded marketplace source root instead of assuming the full package payload exists in the plugin cache. The marketplace path remains additive to npm distribution.
- Data model changes / migrations: none for runtime state; package contents expand to include marketplace/plugin metadata.
- External dependencies / integrations: local Codex CLI marketplace surface, official Codex plugin docs, npm pack contents, and downstream documentation.
