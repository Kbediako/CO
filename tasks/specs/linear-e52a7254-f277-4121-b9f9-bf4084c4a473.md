## Summary
- Objective: deliver a richer dedicated operator dashboard for the CO control host, aligned with Symphony's optional status/dashboard contract and current Elixir information density, while keeping the surface read-only and presenter-driven.
- Scope:
  - fresh docs-first registration for the current Linear issue
  - shared operator dashboard presenter over the existing compatibility projection and issue payloads
  - additive control-host poll-health surfacing for dashboard/status parity
  - `/ui` dataset and frontend updates for the richer operator surface
  - focused tests and required repo validation
- Constraints:
  - keep UI/API layers thin over shared presenter/read-model helpers
  - avoid another runtime-truth or tracker-mutation lane
  - keep refresh operational only and not correctness-bearing
  - record the explicit delegation override because this worker run cannot spawn subagents

## Technical Requirements
- Functional requirements:
  - `/ui` must show authoritative running sessions and retry queue rows
  - `/ui` must show aggregate token totals, aggregate runtime totals, latest rate limits, and poll/refresh health
  - `/ui` must support per-issue drilldown that reuses the existing compatibility issue payload shape for lifecycle state, session/thread id when present, turn count, recent agent activity, workspace path/host, retry state, and last error
  - the richer dashboard payload must be built by a shared presenter layer over `ControlCompatibilityProjectionSnapshot` / `CompatibilityProjectionIssueRecord`
  - poll-health data must come from the control-host public lifecycle rather than frontend inference
  - if the dashboard offers refresh, it must use the existing read-only refresh path and remain operational only
- Non-functional requirements (performance, reliability, security):
  - preserve the authenticated/public route split and existing static asset serving
  - keep the new surface resilient to partial runtime data and unavailable optional health sources
  - keep the change reviewable by reusing existing read-side helpers and minimizing controller churn
- Interfaces / contracts:
  - operator dashboard presenter input: compatibility projection plus optional poll-health reader
  - operator dashboard payload output: generated timestamp, counts/totals/rate-limits/polling summary, running rows, retry rows, and issue detail collection
  - issue detail reuse: `ControlIssuePayload` from `compatibilityIssuePresenter.ts`
  - additive poll-health payload: at least `checking`, `queued`, `interval_ms`, and `next_refresh_in_ms`

## Architecture & Data
- Architecture / design adjustments:
  - add a small poll-health reader/store near `controlServerPublicLifecycle.ts`
  - extend the presenter context so read-only controllers can consume both compatibility projection data and poll-health state
  - add a new control-local operator dashboard presenter instead of overloading the then-current selected-run presenter seam with two unrelated dataset shapes
  - remap `uiDataController.ts` to the new presenter
  - update `packages/orchestrator-status-ui` to render the operator dashboard payload, optionally preserving legacy monitor rendering behind payload-mode detection if that is the smaller change
- Data model changes / migrations:
  - additive read-side payload fields only; no persisted migrations expected
- External dependencies / integrations:
  - existing compatibility projection and issue presenter helpers
  - existing control-host public lifecycle refresh coordinator
  - existing status-ui static asset package

## Validation Plan

  - docs-review before code edits
  - focused unit tests for poll-health shaping and operator dashboard payload shaping
  - focused controller/control-server tests for `/ui/data.json`
  - required repo validation floor after implementation

  - verify issue drilldown stays consistent with `/api/v1/state` and `/api/v1/<issue_identifier>`
- Monitoring / alerts:
  - operator dashboard remains observational; refresh/poll health never becomes correctness-bearing
