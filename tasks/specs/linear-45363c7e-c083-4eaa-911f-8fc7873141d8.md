# Technical Specification

## Summary

- Objective: Normalize shipped MCP client configs from single command string startup to structured `command` plus `args`.
- Scope: Docs packet plus parent-owned implementation in `mcp-client.json` and `templates/codex/mcp-client.json`; `plugins/codex-orchestrator/.mcp.json` is protected reference surface.
- Constraints: This child lane is docs-only and must not edit implementation/config files or call Linear mutation helpers.

## Issue-Shaping Contract

- User-request translation carried forward: Root and template MCP client configs currently encode `codex-orchestrator mcp serve` as one `command` string. Parent implementation should encode the executable as `codex-orchestrator` and the argument vector as `["mcp", "serve"]`.
- Protected terms / exact artifact and surface names: `mcp-client.json`, `templates/codex/mcp-client.json`, `plugins/codex-orchestrator/.mcp.json`, `command`, `args`, `codex-orchestrator mcp serve`.
- Nearby wrong interpretations to reject:
  - Do not change MCP server runtime behavior.
  - Do not convert plugin `.mcp.json` to a single command string.
  - Do not introduce shell wrappers to preserve shell parsing.
  - Do not rename `codex-local` or `codex-orchestrator` server ids unless parent evidence proves that is required.
  - Do not broaden into marketplace metadata, release automation, or unrelated templates.
- Explicit non-goals carried forward:
  - No implementation/config edits in this docs child lane.
  - No Linear mutation from this child lane.
  - No MCP protocol or server command implementation changes.

## Parity / Alignment Matrix

| Surface | Current Truth | Reference Truth | Target Truth |
| --- | --- | --- | --- |
| `mcp-client.json` | `codex-local.command` is `codex-orchestrator mcp serve`; no `args`. | MCP clients generally spawn `command` directly with `args`. | `codex-local.command` is `codex-orchestrator`; `codex-local.args` is `["mcp", "serve"]`. |
| `templates/codex/mcp-client.json` | `codex-orchestrator.command` is `codex-orchestrator mcp serve`; no `args`. | Template should match the root shipped client launch contract. | `codex-orchestrator.command` is `codex-orchestrator`; `codex-orchestrator.args` is `["mcp", "serve"]`. |
| `plugins/codex-orchestrator/.mcp.json` | Already split with `command: "node"` and `args: ["./launcher.mjs", "mcp", "serve"]`. | Existing plugin-local launcher uses structured launch shape. | Preserve split `command` plus `args`; do not regress to a shell command string. |
| Launch behavior | Shell-less clients can treat the whole string as an executable name. | Structured launch avoids shell splitting assumptions. | Clients can spawn `codex-orchestrator` with `mcp` and `serve` arguments. |
| Docs/task registry | No CO-271 packet exists before this child lane. | Docs-first registry and checklist evidence precede implementation. | Packet is registered in `tasks/index.json`, `docs/TASKS.md`, and task mirrors. |

## Technical Requirements

1. Change `mcp-client.json` so `mcpServers.codex-local.command` is only `codex-orchestrator`.
2. Add `mcpServers.codex-local.args` with exactly `["mcp", "serve"]`.
3. Change `templates/codex/mcp-client.json` so `mcpServers.codex-orchestrator.command` is only `codex-orchestrator`.
4. Add `mcpServers.codex-orchestrator.args` with exactly `["mcp", "serve"]`.
5. Preserve JSON validity and existing top-level fields such as `templateVersion`.
6. Preserve `plugins/codex-orchestrator/.mcp.json` as structured launch config with `command` plus `args`.
7. Avoid shell-dependent command strings or platform-specific wrappers in root/template client configs.
